## 详细内容

### 3.1 局部业务模型的构成

#### 3.1.1 局部业务模型的定义

**概念**：局部业务模型是全局业务模型中的某一个核心业务环节或模块，单独拿出来进行梳理和分析的模型。

**本质特征**：
- 有稳定的input（输入）和output（输出）
- 处于业务整体链条的某一环
- 是完整商业系统中的一个子系统
- 上下游关系明确

```
┌─────────────────────────────────────────────────────────────┐
│                 全局业务模型 vs 局部业务模型                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   全局业务模型                                               │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  流量 → 转化 → 交付 → 复购 → 推荐                  │   │
│   └─────────────────────────────────────────────────────┘   │
│            ↓         ↓        ↓        ↓        ↓          │
│        局部模型  局部模型  局部模型  局部模型  局部模型  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.1.2 局部业务模型的input/output逻辑

| 要素 | 说明 | 示例 |
|------|------|------|
| 输入(Input) | 业务资源、生产资料 | 预算、用户名单、渠道、开班计划 |
| 输出(Output) | 稳定标准的业务价值 | 转化率、用户满意度、人效比、收入 |

#### 3.1.3 经典案例：凡登读书会城市合伙人体系

**背景**：凡登读书会需要在全国各地区建立强大的分销和城市合伙人体系，支撑会员数量的激进增长。

**4层结构设计**：

```
┌─────────────────────────────────────────────────────────────┐
│              凡登读书会城市合伙人体系架构                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第一层：省级分会                                           │
│         ├─ 职责：渠道发展和扶持                             │
│         ├─ 提供：政策、资源、配套支持                       │
│         └─ 目标：区域整体增长                               │
│                                                             │
│  第二层：城市分会                                           │
│         ├─ 职责：会员发展和服务                             │
│         ├─ 提供：利益空间、资源、政策支持                   │
│         └─ 目标：城市会员数量增长                           │
│                                                             │
│  第三层：县区/企业分会                                       │
│         ├─ 配套：樊登书店实体店                             │
│         ├─ 职责：活动承载、会员服务                         │
│         └─ 目标：本地化运营                                 │
│                                                             │
│  第四层：驿站                                               │
│         ├─ 定位：最小运营单元                               │
│         └─ 目标：深度渗透                                   │
│                                                             │
│  核心机制：三级分销利益分层体系                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**模型分析**：

- **Input**：资金资源、人力资源、政策支持
- **Output**：每个地区稳定的会员数量增长、收入增长
- **关键机制**：三级分销利益分层
- **支撑要素**：4层结构、实体店配套、活动承载

**成功要素**：
1. 清晰的层次分工
2. 明确的利益分配机制
3. 实体店支撑活动
4. 标准化的运营体系

### 3.2 局部业务模型的价值与必要性

#### 3.2.1 为什么需要局部业务模型

**核心问题**：如何应对业务体量的10倍-20倍增长？

**业务规模与管理复杂度的关系**：

| 规模 | 团队规模 | 管理方式 | 是否需要模型 |
|------|----------|----------|--------------|
| 小规模 | 2-3人 | 粗放管理、及时响应 | ❌ 不需要 |
| 中规模 | 几十人 | 分类管理、流程规范 | ✅ 开始需要 |
| 大规模 | 上百千人 | 系统化管理、多层级分工 | ✅ 必须需要 |

**案例：客服团队的演变**

**初始状态（2-3人）**：
- 工作方式：用户来电/留言 → 及时回复
- 管理方式：简单记录，无标准化流程
- 工作时间：9点上班，5-6点下班
- **结论**：不需要复杂的模型

**发展状态（业务量增长10倍）**：
- 订单量：从几千单发展到几万甚至几十万单
- 团队规模：从2-3人发展到几十甚至上百人
- 新增需求：
  - 客服问题分类
  - 处理流程标准化
  - 工作考核机制
  - 品控机制

**成熟状态（上千人call center）**：
- 案例：协成公司的客服体系
- 规模：几千人客服团队
- 处理能力：成百上千客服咨询
- 系统要求：
  - 高效的流转分发系统
  - 多层级管理体系
  - 全国5大区分布
  - 5-8种问题类型分类

**核心启示**：
- 小公司不要为了做作业而"无中生有"画模型
- 应该思考：如果业务体量增长10-20倍，我需要建立什么机制？
- 学习成熟公司：关注他们如何支撑大规模业务运转

#### 3.2.2 搭建局部业务模型的4种思路

当业务体量显著增长时，构建局部业务模型的4个思考方向：

```
┌─────────────────────────────────────────────────────────────┐
│           业务体量增长10倍后，局部业务模型的4个搭建思路        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  思路1：问题与事情分类                                      │
│         ├─ 用户咨询问题分类                                 │
│         ├─ 用户类型分类                                     │
│         └─ 每类问题的处理方式                               │
│                                                             │
│  思路2：建立流程和SOP                                      │
│         ├─ 每类问题的处理流程                               │
│         ├─ 标准化作业程序(SOP)                              │
│         ├─ 约束管理机制                                     │
│         └─ 满意度监控机制                                   │
│                                                             │
│  思路3：角色分工与协作                                     │
│         ├─ 前端：问题分类、分发                             │
│         ├─ 中端：问题处理                                  │
│         ├─ 后端：品控抽查                                   │
│         └─ 多角色对接协作                                   │
│                                                             │
│  思路4：成长路径设计                                       │
│         ├─ 面向用户的成长路径                               │
│         ├─ 面向团队成员的成长路径                           │
│         ├─ 关键节点设计                                     │
│         └─ 激励机制设计                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**详细说明**：

**思路1：问题与事情分类**
- 将每天从10-20个咨询扩展到上千上万个咨询时，必须分类
- 示例：酒店客服问题分类
  - 退房问题
  - 更换入住日期
  - 投诉建议
  - 其他咨询

**思路2：建立流程和SOP**
- 每类问题都要有标准处理流程
- 示例：酒店退房问题的处理流程
  - 验证订单信息
  - 确认退房原因
  - 计算退款金额
  - 处理退款
  - 满意度评价

**思路3：角色分工与协作**
- 几万咨询量需要多人协作
- 示例：客服团队角色分工
  - 一线客服：问题分类、初步处理
  - 二线客服：复杂问题处理
  - 客服主管：品控、抽查
  - 客服经理：团队管理、优化

**思路4：成长路径设计**
- 激励团队，保持稳定性
- 示例：客服团队成长路径
  - 初级客服
  - 中级客服
  - 高级客服
  - 客服主管
  - 客服经理
  - 客服总监/培训专家

### 3.3 局部业务模型的4个视觉化要素

当我们将局部业务模型看作一个系统并进行视觉化呈现时，通常包含4个常见要素：

```
┌─────────────────────────────────────────────────────────────┐
│              局部业务模型的4个视觉化要素                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  要素1：流程或成长路径                                      │
│         └─ 业务流转的完整链条                               │
│                                                             │
│  要素2：关键环节                                            │
│         └─ 重要节点 + 配套SOP                               │
│                                                             │
│  要素3：约束机制                                            │
│         └─ 预警机制、激励机制、品控机制等                    │
│                                                             │

└─────────────────────────────────────────────────────────────┘
```

#### 案例：教育公司供应链管理的局部业务模型

**背景**：某在线教育公司的师资供应链管理

**完整的业务流程**：

```
┌─────────────────────────────────────────────────────────────┐
│           教育公司供应链管理局部业务模型                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【Input】                                                  │
│   ├─ 营销数据                                               │
│   ├─ 开班计划                                               │
│   └─ 师资数量需求                                           │
│                                                             │
│  【完整流程】                                               │
│   1. 根据营销数据和开班计划 → 形成招聘计划                  │
│   2. 根据招聘计划 → 进行师资招募                            │
│   3. 师资招募 → 进行师资培训                                │
│   4. 师资培训 → 师资考核                                    │
│   5. 考核通过 → 匹配到班期                                  │
│   6. 班期服务 → 品控监测                                    │
│   7. 班期结束 → 业绩评价                                    │
│   8. 业绩评价 → 收入核算                                    │
│                                                             │
│  【Output】                                                 │
│   ├─ 稳定的师资供给数量                                     │
│   ├─ 教学服务数据达标                                       │
│   └─ 用户满意度达标                                         │
│                                                             │
│  【4个要素体现】                                            │
│   ✅ 流程：完整的师资管理流程                               │
│   ✅ 关键环节：招聘计划、考核、淘汰                         │
│   ✅ 约束机制：预警机制、绩效管理、品控机制、收入核算        │
│   ✅ 角色分工：营销负责人、HR、课程负责人、财务负责人        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键环节详解**：

1. **招聘计划形成**：根据开班计划推导下月需要多少老师
2. **师资考核**：定期对老师进行考核和品控
3. **师资淘汰**：不合格老师的淘汰机制
4. **班期匹配**：将师资匹配到具体班期
5. **收入核算**：每个老师每月的业绩和收入计算

**约束机制详解**：

1. **开班计划预警机制**：确保师资供给与前端需求匹配
2. **人员匹配预警机制**：避免师资过剩或短缺
3. **绩效管理激励机制**：激励老师提升教学质量
4. **品控机制**：监控教学质量
5. **财务核算机制**：与财务部门协作的收入核算

**角色分工详解**：

1. **营销团队负责人**：提供营销数据和开班计划
2. **HR团队**：负责师资招募和培训
3. **课程负责人**：负责师资的考核和班期匹配
4. **财务负责人**：负责收入核算
5. **供应链负责人**：整体协调和管理

### 3.4 局部业务模型的梳理方法

#### 3.4.1 4步梳理法

```
┌─────────────────────────────────────────────────────────────┐
│                局部业务模型梳理4步法                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第1步：明确Input和Output                                   │
│         ├─ 确定与其他部门的协同方式                         │
│         ├─ 明确需要投入的资源                               │
│         └─ 明确需要交付的成果                               │
│                                                             │
│  第2步：评估业务体量增长后的问题                            │
│         ├─ 用户/产品/业务方类型过多，需求不一致             │
│         ├─ 协作流程和机制不清楚                             │
│         ├─ 用户和内部成员成长路径不清晰                     │
│         └─ 缺乏清晰的流程                                   │
│                                                             │
│  第3步：问题分类并设定流程                                  │
│         ├─ 对问题进行分类                                   │
│         ├─ 设定流程和路径                                   │
│         ├─ 确定关键环节                                     │
│         ├─ 确定角色分工                                     │
│         └─ 确保逻辑自洽                                     │
│                                                             │
│  第4步：完善配套机制                                        │
│         ├─ 面向用户的激励机制                               │
│         ├─ 面向团队人员的激励机制                           │
│         ├─ 品控机制                                         │
│         ├─ 定期核算机制                                     │
│         └─ 补充SOP                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.4.2 常见的局部业务模型类型

| 类型 | Input | Output | 示例 |
|------|-------|--------|------|
| 增长导向型 | 流量、钱、其他资源 | 订单、新增用户、收入 | 信息流获客、私域转化 |
| 生产保障型 | 生产资料、初始状态用户 | 产品、用户满意度 | 内容生产、课程研发 |
| 供给交付型 | 用户订单、需求 | 匹配的供给、供应链效率 | 师资供给、商品配送 |
| 用户运营型 | 初始用户 | 用户活跃、留存、复购 | 用户分层运营 |

#### 3.4.3 简单业务的梳理方法

对于一些相对简单的业务，梳理流程时可能只需要画出业务流程，模型就基本清晰了。

**案例：三节课魔鬼训练营模式**

**教研局部业务模型**：

```
流程：讲师BD → 课程研发 → 班期教学
        ↓         ↓           ↓
     立项评审   课程大纲     开营
               课程PPT     每周看课
               课程录制     每周作业
                          毕业设计
```

**补充要素**：
- 立项评审：参与人员、评审标准
- 课程录制：录制原则、SOP
- 班期教学：数据监测、运营策略调整

**教学局部业务模型**：

```
流程：开营 → 每周看课 → 每周作业 → 毕业设计 → 反馈
```

**补充要素**：
- 每周数据监测
- 根据数据调整运营策略
- 配套工具和SOP

#### 3.4.4 复杂业务的梳理方法

对于复杂的业务，复杂性往往在于：
- 涉及n多角色
- 角色之间的关系和流程交叉
- 需要并行处理
- 针对不同角色有很多约束机制和策略

**梳理重点**：
1. 优先界定input和output
2. 梳理清楚有多少个角色
3. 明确角色之间的关系和分工
4. 再梳理协作流程

**案例：教育公司创新业务的4角色模型**

**背景**：某教育平台希望为不同用户提供个性化学习建议和反馈

**4个角色**：
1. Running Partner
2. 教研
3. 助教
4. 班主任

**业务流程**：
- Input：初始进入班级学习的学员
- Output：学员的个性化辅导、学习效果、口碑提升
- 流程：学前 → 学中 → 学后
- 核心问题：4个角色各自做什么？如何协作？

**案例：美团用户分层运营体系**

**背景**：美团站内数亿量级用户的维护和管理

**分层逻辑**：
- 按历史购买订单数分层
- 按最近一次购买时间分层
- 分成十几个用户类型

**运营策略**：
- 对每一类用户做定义
- 针对每一类用户制定不同的运营策略

**模型特点**：
- 不同类型的用户之间不需要发生关系
- 面向外部：一张图就够了
- 面向内部：需要补充部门间的协作分工流程

### 3.5 局部业务模型的层次划分

#### 3.5.1 业务模型分层次的必要性

**核心原则**：
- 业务模型一定是为了解决一类复杂的业务问题
- 简单的问题不需要模型

**判断标准**：
- 你面临的是一类非常明确的复杂业务问题 → 需要模型
- 你要解决的问题超级简单 → 不需要模型

**业务模型的层次**：
- 一级局部业务模型
- 二级局部业务模型
- 三级局部业务模型

#### 3.5.2 一级局部业务模型

**定义**：对应公司整体业务模型的核心模块，支撑公司怎么挣钱。

**常见的一级局部业务模块**：

| 行业 | 一级局部业务模块 |
|------|------------------|
| 教育公司 | 营销、教研、教学 |
| 制造业 | 生产、销售、供应链、仓储配送 |
| 金融公司 | 获客、风控、资产资金管理 |

**重要提示**：
- 一级局部业务模块 ≠ 部门划分
- 产研部门通常不对应一级局部业务模块
- 产研是支撑各个业务模块的赋能部门

#### 3.5.3 4种一级局部业务模型类型

**类型1：增长类**

**聚焦解决的业务问题**：
- 用户增长或收入增长的稳定产生
- 增长的可规模化

**案例：考研教育公司增长模型**

**整体增长导向业务模型**：

**聚焦解决的业务问题**：

- 交付人效的持续降低

#### 3.5.4 二级局部业务模型

**定义**：一级局部业务模型下的细分子模块或玩法。

**常见的二级局部业务模型**：

| 一级模型 | 二级模型示例 |
|----------|--------------|
| 增长 | 微信生态获客、信息流投放、用户分层运营 |
| 生产 | 爆品产出、生产团队人效提升 |
| 供给 | 流通模型、仓储管理、服务人员培训管理 |
| 交付 | 单个课程教学交付、单个客户解决方案交付 |

#### 3.5.5 三级局部业务模型

**定义**：二级局部业务模型下的更细分模块。

**示例**：
- 营销 → 私域流量运营 → 社群销售管理（三级）
- 私域流量下的社群销售人员怎么管、怎么培训、怎么招募和淘汰

4. ✅ 看完模型基本不用问太多问题

## 本章小结

### 3.6.1 核心知识点

**1. 局部业务模型的本质**
- 是全局业务模型中的某一个核心业务环节或模块
- 有稳定的input和output
- 是完整商业系统中的一个子系统

**2. 局部业务模型的4个视觉化要素**
- 流程或成长路径
- 关键环节
- 约束机制
- 角色分工

**3. 局部业务模型的4步梳理法**
- 明确Input和Output
- 评估业务体量增长后的问题
- 问题分类并设定流程
- 完善配套机制

**4. 局部业务模型的层次**
- 一级：对应公司整体业务模型的核心模块
- 二级：一级下的细分子模块或玩法
- 三级：二级下的更细分模块

**5. 4种一级局部业务模型类型**
- 增长类：用户增长或收入增长的稳定产生和可规模化
- 生产类：生产/研发与收入的相关性、效率与成本
- 供给类：前端需求与后端供给能力的匹配、供应链效率
- 交付类：用户满意度、NPS、续费、复购的稳定产生

### 3.6.2 关键案例总结

**1. 凡登读书会城市合伙人体系**
- 4层结构：省→市→县/企业→驿站
- 核心机制：三级分销利益分层

**2. 客服团队演变**
- 从2-3人到上千人的规模化历程
- 4种搭建思路的应用

**3. 教育公司供应链管理**
- 完整的师资管理流程
- 4个要素的完美体现

**4. 考研教育公司增长模型**
- 一级增长模型和二级局部模型的关系

**5. 微信公众号内容生产**
- 从选题到发布的完整流程
- 内容与收入的挂钩机制

**6. 2B销售团队**
- 从线索到成单的完整流程
- 成交率稳定的策略和机制

### 3.6.3 学习要求

**A/B类操盘手的基本要求**：
- 能够梳理清楚所在业务线中至少到二级局部业务模型
- 明确自己在解决什么复杂重要的业务问题
- 理解背后的整体策略和模型
- 一个好的业务模型应该让人一目了然

**关键认知**：
- 小公司不要为了做作业而"无中生有"
- 应该思考：如果业务体量增长10-20倍，需要建立什么机制？
- 业务模型背后一定对应一类明确的、有点复杂但重要的业务问题
