# 第六章：局部业务模型实战与应用

> 所属章节：模块二
> 核心内容：局部业务模型成型过程、从0搭建模型的方法、业务模型复杂度管理

---

## 本章导学

### 核心目标

本章帮助学员理解局部业务模型是如何在实战中逐渐成型的,以及如何从0开始搭建一个业务模型。通过真实案例,让大家看到模型跑通的真实过程,掌握从零搭建模型的两种逻辑,并理解业务模型复杂度对成功的影响。

### 学习重点

- 理解局部业务模型的成型过程（需要时间、动态演化）
- 掌握从0搭建模型的两种逻辑
- 理解业务模型复杂度与成功风险的关系
- 学会从简单开始逐步迭代

## 6.1 从案例看局部业务模型的成型过程

### 6.1.1 模型成型的真相

**核心认知**：
- 不是每家公司一开始就有模型
- 很多模型都是边摸索边干出来的
- 一个业务整体模型的成型一定是不断动态演化而成的

**模型成型过程**：
```
┌─────────────────────────────────────────────────────────────┐
│                  模型成型的完整过程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  一边执行 + 商业逻辑思考 + 混乱摸索                            │
│           ↓                                                 │
│  找到一些线索（感觉可以干,能带来增长/商业价值）                │
│           ↓                                                 │
│  脑海中形成假定的模型                                         │
│           ↓                                                 │
│  持续验证模型有效性                                           │
│   ├─ YES → 模型成立 → 稳定下来 → 优化它                      │
│   └─ NO → 模型不成立 → 重新思考 → 调整                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**关键提醒**：
- 我们这门课教的是思考方法和逻辑
- 并不是说有个固定答案放那就不能变了
- 业务模型成型一定是不断动态演化而成的

**时间认知**：
- 跑通任何一个局部至少都需要几个月时间
- 上完课需要消化、思考、行动、检验、内化
- 这需要一个过程

### 6.1.2 实战案例：三节课B端业务的模型成型过程

**背景**：
- 三节课B端业务诞生于2017年
- 某大厂主动找上门合作（"很多员工参加你们的课,能不能合作做个内训"）
- B端对接员工感觉"能挣钱,是大腿,可以搞搞看"

**第一阶段：野战军状态（2017年）**

**状态特征**：
- 业务非常随机、非常零散
- 获客不稳定
- 交付也不稳定（完全依赖老师）
- B端客户满意度完全看CEO陆可的生理和心理状态
- 属于"叮咚咣咚"的状态

**持续时间**：小半年

**第二阶段：正规军尝试（2017年中期）**

**内部讨论**：这事好像可以拎起来,正规军好好搞一搞

**前端获客模块搭建**：
1. **销售线索来源**：
   - 购买少量企业客户线索（招聘网站合作）
   - 找到企业内部培训负责人的联系方式

2. **销售团队组建**：
   - 组建10人销售团队
   - 定向跟进售卖训练营项目

**初步模型**：
- 有了销售线索和销售跟进转化的模型
- 依靠人来完成成交

**问题**：
- 模型不稳定
- 不同销售手中的转化率不稳定
- 不同行业客户需求不一样
- 没有做到稳定的input和稳定的output

**属于**：B端业务的中期状态

**第三阶段：交付模型优化（2017年下半年-2018年）**

**痛下决心**：
- 如果要好好搞,必须做到可规模化交付、复制
- 完全依赖老师个人状态交付不靠谱

**打磨过程**：
- 一个同学牵头做梳理
- 经过3-4个月打磨尝试
- 在10家企业做落地尝试

**输出交付模型**：
```
┌─────────────────────────────────────────────────────────────┐
│               三节课B端交付模型                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  核心要素：                                                   │
│  - 流程：交付分为几个周期,每个周期目标                        │
│  - 角色：涉及哪些重要角色                                     │
│  - 机制：中间有哪些重要机制                                    │
│  - 评估：哪些重要指标评估交付合格/不合格                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**结果**：
- 2018年初步形成模型
- 围绕客户需求,交付相对稳定
- 客户满意度基本稳定

**第四阶段：前端销售模型优化（2018年后）**

**问题**：前端销售虽然有基本模型,但不稳定

**优化方案**：在中间加入辅助决策产品/体验产品
- 符合"数理油模型"逻辑
- B端培训单价较高（几万-几十万）
- 决策成本需要降低

**具体做法**：
1. **社群**：B端HR进三节课社群
2. **低客单价产品**：体验课程让业务老大感受

**最终结果**：
- 前端：一系列流程 + 辅助决策产品,转化率相对稳定
- 后端：生产交付模型稳定,客户满意度稳定
- 整个三节课B端业务模型初步稳定

**时间周期**：一年多时间

**认知**：
- 在很多中早期创业公司,业务初步成型需要至少小一年时间

## 6.2 如何在业务中从0搭建出一个模型

### 6.2.1 从0搭建模型的两种逻辑

**逻辑一：先找外部参照模型**

**适用场景**：你要解决一个问题,但不知道怎么办

**步骤**：
1. **明确要解决什么问题**,背后本质是什么
2. **看行业里其他公司有没有成熟可参照的模型**
3. **判断**：如果对方公司业务情况跟我公司相差不太远
4. **拿过来套用**,结合自己业务问题优化加工
5. **形成SOP、工作体系等**
6. **验证**：在业务里跑模型,验证有效性

**案例**：社群转化做得不好,要通过社群获取更多销售线索
- 问题：通过社群获取更多销售线索
- 看行业里有没有成熟的社群转化模型
- 如果有,判断可借鉴性
- 拿来套用,优化加工
- 跑模型验证

**逻辑二：自己先干到七八分,再抽象规律**

**适用场景**：你自己亲自上一线干,能干到有稳定的input output,但别人干不一定

**步骤**：
1. **某件事在你这儿已经趋于稳定**
2. **抽象规律和操作标准**
   - 哪几种内容
   - 怎么做
   - 符合什么标准
   - 能驱动粉丝增长
3. **形成SOP或初步模型**
4. **放在其他人身上验证有效性**

**案例**：三节课公众号运营
- 最早一个人写
- 发现持续做好某几类内容,一周流量/阅读/粉丝增长特别稳定
- 抽象规律：一周做哪几种类型内容、怎么做、符合什么标准
- 形成SOP或初步模型
- 在其他人身上验证

### 6.2.2 两种逻辑的综合使用

**关键认知**：
- 两种逻辑不是绝对对立或孤立的
- 解决复杂问题时,需要交替着综合使用
- 才能从零搭建起一个复杂的业务模型

**实战案例：三节课品类运营培养模型搭建**

**背景**：
- 2018年,三节课课程产品分三个品类（运营、产品、营销）
- 每个品类需要有人通过策略、规划、工作方法达成品类收入目标
- 应该可以设一个品类运营岗位
- 但工作方法没完全成型
- 市面上找不到完全成熟的品类运营画像

**要解决的问题**：
- 需要一套工作方法和策略
- 需要有人能驾驭这些工作方法和策略
- **人的培养成为重要命题**

**解决步骤**：

**第一步：对外看,寻找可参照模型**

**发现**：
- 宝洁、阿里等成熟大厂
- 对于特定岗位人的培养有一套基本方法
- 叫做**EDAC模型**

**EDAC模型4个层次**：

1. **Explain（讲解）**
   - 有一套内容或教程告诉这件事要怎么做
   - 例如：宝洁的市场营销,搞定渠道代理商分三步
   - 中间还有两个重要工具帮助解决

2. **Demo（示范）**
   - 给一个真实的例子
   - 亲自搞定代理商的案例
   - 通过什么方法在什么环境下搞定

3. **Apply（应用）**
   - 不能只听,要亲自干一下
   - 给训练环境
   - 资深渠道管理者谈代理商,新人跟着去练手

4. **Consolidation（巩固）**
   - 应用完了还要巩固
   - 亲自负责一个小型项目,完整结项
   - 通过这个过程巩固

**第二步：拆分工作职责**

**品类运营岗位工作内容拆分**：
1. **基础工具和方法**
2. **常见工作模块**：
   - 单个班期的推广策划
   - 单个SKU的增长策略规划
   - 单个品类的整体增长策略规划
3. **进阶方法**

**第三步：落实到EDAC培养模型**

**输出**：三节课品类运营培养成长的EDAC模型

**详细表格**：
- 基础工具和方法要训练的模块一、二、三分别是什么
- 进阶的辅助工具和方法是什么
- 每个训练模块下要干什么
- Explain是什么
- Demo是什么（成功案例）
- Apply是什么（三节课所有同学去一线真实训练）
- Consolidation是什么

**第四步：跑模型验证**

- 真实放在几个同学身上检验
- 调整,跑1-2个月
- 几个新同学走完培养模型
- 发现模型比较稳定
- 界定品类运营的人的画像（满足一、二、三三个条件）
- 满足条件的人进来,经过培训流程,3周左右能上岗胜任初步工作

**总结**：
- 真实的在面临业务问题时
- 怎么向外看
- 怎么从内部已做过的事里拆解、分类、结构化梳理
- 怎么去跑
- 最后形成比较稳定的局部可依赖模型

## 6.3 业务模型的复杂度与成功风险

### 6.3.1 简单业务模型 vs 复杂业务模型

**简单业务模型特征**：
- 单线程
- 变量少
- 中间线条、分叉很少

**案例**：工具产品
- 超级简单
- 流量引进来 → 用户用 → 喜欢下次再来用 → 养成习惯
- 没了

**跑通时间**：1个月内能出答案（work or not work）

**复杂业务模型特征**：
1. **线程多**：
   - 前端、后端、中间配套机制和系统

2. **N多人和人的交互**：
   - 人是最大的变量
   - 不同人表达、沟通、语气不一样
   - 影响客户的状态和感受

3. **N多信息交互**：
   - A信息制约B行动
   - C信息制约D行动
   - 很多线条和机制

**跑通时间**：周期很长,风险更大,难度更大

### 6.3.2 复杂业务模型案例

**案例1：沃尔玛供应链系统**

**角色**：
- 总部、配送中心、仓库、N多门店

**信息流**：
- 仓库补货信息：总部下达
- 门店补货信息：反馈给总部
- 配送中心流动仓库存信息：总部及时查到
- 配送中心物流和补货信息：总部下达

**引入角色后**：更加复杂

**认知**：
- 沃尔玛花了100多年形成这套完善机制
- 如果从零开始跑,10-8年很难想象能跑出来

**案例2：美团外卖配送模型**

**流程**：
- 用户下单 → 商家接单 → 骑手接单 → 取货 → 配送

**后台复杂度**：
- 骑手 ↔ 商家信息互动
- 骑手 ↔ 用户信息互动
- 平台 ↔ 商家/骑手信息互动
- 商家供给管理
- 配送网络经营
- 运力供给动态测算
- 单量预测
- 订单分配机制
- 骑手补贴定价

**后台**：一套非常复杂的运营机制和系统保证配送流程稳定

**案例3：教育行业（好未来智慧教育解决方案）**

**角色**：
- 教研、教学、用户三种角色

**复杂点**：
- 教学服务用户,中间服务/互动节点非常多
- 要保证交付稳定、学习效果
- 中间涉及变量非常多

**后台支撑**：
- 老师培养
- 教学过程管理
- 学员评价
- 基础系统支撑

### 6.3.3 给大家的两个忠告

**忠告一：如果你所在公司业务比较复杂**

**参照标准**：
- 要么在很多局部有成熟经验可借鉴
- 要么有非常雄厚的资源可支撑
- 要么在好几个模块已经跑得比较稳定

**忠告二：不要在假想中把模型想得很复杂**

**风险**：
- 如果变量很多
- 很多事都没被验证
- 需要同时验证
- 只有变量没有定量

**反面案例**：某教育公司校园团队转化型产品

**问题模型**：
1. **转化产品本身**：转化力稳定不稳定,没被验证
2. **公号投放**：待验证的事
3. **老带新工作**：给用户权益
4. **系列大咖直播**：要做一年40个大咖直播

**模型逻辑**：
- 公号投放引用户到转化产品
- 转化产品用户听完课,想要年度大咖直播权益
- 帮忙转发,形成老带新

**问题**：
- 转化产品本身要达到5%-10%转化
- 投放也在跑转化产品转化率
- 系列大咖直播也在做
- 这些事互相影响

**变量多**：
- 转化产品转化率不稳定
- 怎么知道是产品有问题还是投放用户质量不高?
- 转化率不高是因为年度大咖直播让用户去老带新不付费了?

**结论**：
- 如果模型里变量很多,很多事没被验证,需要同时验证
- 只有变量没有定量
- 风险特别大
- 很难被验证成功
- 很容易跑三四周全是坑,一地鸡毛

**正面建议**：
1. 要么初始搭建的业务模型是别人跑通过、成熟可借鉴的
2. 要么尽量让初始搭建的业务模型足够简单

## 本章小结

### 核心要点回顾

**1. 局部业务模型的成型过程**
- 不是一开始就有模型
- 是不断动态演化而成的
- 需要几个月甚至一年多的时间
- 包含：执行+思考+摸索→线索→假定模型→验证→成立/调整

**2. 从0搭建模型的两种逻辑**
- **逻辑一**：找外部参照模型 → 明确问题 → 找参照 → 套用优化 → 验证
- **逻辑二**：自己干到七八分 → 抽象规律和标准 → 形成SOP → 验证
- **综合使用**：解决复杂问题时两种逻辑交替使用

**3. 业务模型复杂度与成功风险**
- **简单模型**：单线程、变量少 → 1个月内出答案
- **复杂模型**：多线程、多人互动、多信息交互 → 周期长、风险大
- **忠告**：
  - 复杂业务需要成熟经验/雄厚资源/稳定模块支撑
  - 不要在假想中把模型想得很复杂
  - 要么用成熟模型,要么从简单开始

**4. 真实案例的价值**
- 三节课B端业务：野战军 → 正规军尝试 → 交付优化 → 前端优化（一年多）
- 三节课品类运营EDAC模型：找参照 → 拆职责 → 落EDAC → 验证
- 教育公司反面案例：变量太多、互相影响、风险极大
