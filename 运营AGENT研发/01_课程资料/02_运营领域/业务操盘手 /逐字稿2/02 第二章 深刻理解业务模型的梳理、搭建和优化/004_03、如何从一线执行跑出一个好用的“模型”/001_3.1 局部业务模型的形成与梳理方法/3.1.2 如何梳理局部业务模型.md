# 3.1.2 如何梳理局部业务模型

随后我们要简单的跟各位讲一下号了，我们已经知道了局部原模型原来长这样，它背后的原理是这样的对实际我们在梳理或者是搭建一个局部业务模型的时候，对我们该怎么来去一一步一步把它去搭出来，我们给各位一个简单的小方法，供各位所参考使用。小方法它也是1基本的流程，流程约有这么4步。

第一步我们要明确我们所在的局部业务它的input output到底是什么，就像我们前边说到的樊登读书会的分销体系的例子，对它的input是说我要输入一些钱或者资源或者是人力，它的output我要保证在每个地区下通过这么一套体系对我的收入增长可以是稳定的。

而我们刚才所看的教育公司的供应链的管理，它的input说我初始输入的是我的开班计划，还有我的例如对师资服务的数量的需求，而我输出的output我要稳定的给我的前端供给这么多数量的老师，以及要确保这些老师的教学服务的数据，还有用户满意度对都是稳定的。

所以我们第一步要先明确我们所在的局务业务，然后跟其他部门怎么来去协同，我们的input是什么，我们要交付给其他部门或交付给处理个公司的output到底是什么？我们明确了第一步之后，第二步我们就可以做进一步的评估，我们如果在业务体量显著增长的情况下，要保证output的稳定，我们遇到些问题。那么在一个局部业务模块下，伴随着一家公司它的业务体量增长，我们遇到些问题？

常见会有这么几类，第一类遇到的问题，你看我的业务体量增加了，所以我的用户量，我的产品的数量或者我的业务方，我要承接的业务方的类型太多，对导致不同的用户不同的产品线不同业务方，他的需求和关注点可能不一致，这是常见的遇到的第一类问题。

第二类问题说随着我业务体量增加对然后我们的处理个协作的流程和机制在多个角色之间不清楚的。对第三类我们会遇到的问题我们的用户，还有我们的内部成员对因为他的业务体量增加了，他们的成长发展路径有不清晰的。对第四类问题我们的交付过程，或者是用户使用我们产品的过程，或者是我们的多个部门的协作过程缺乏清晰的流程。

所以在业务体量增长的情况下，我们在一个局部业务里边经常会遇到问题，约会有这么几类，这些问题我们都一一列出来了之后，我们就可以往下进入到下一步了。

下一步我们就可以对这些问题进行分类，并且分类完了之后设定流程、设定路径，确定关键环节，确定我们的角色分工等等这样的关系，确保在我们脑海中逻辑可自洽，至少在我们脑海当中

可去想见得到，你看当我们的客服的问题增加了可能几十倍之后，我们通过把问题这么分类，通过让几类人这么来去协作对应该是能保证我们客服的响应时间，还有我们用户解决问题的满意度是能得到保证的，我们脑海中至少要可基本自洽对处理个逻辑。

对再往后就可以结合一些关键环节，完善补充各种的配套约束机制了。

例如面向用户的激励机制或面向团队人员的激励机制，还有一些品控的机制，还有一些例如定期做核算的，做一些数据确认的这样的一些机制，并且在这基础上补充一些sop，这样可约去梳理和搭建出来一个常见的局部业务模型。

如果你之前对于我们的全球业务模型，包括说我们一家公司的处理体增长，业务模型的梳理已经理解的较为透彻，吃的较为透了，在这一部分请你理解说局部业务模型的搭建和梳理的时候，可能理解起来可能成本是更低的，会更简单的关于小的思路我们就先讲到这儿，后边我们还会有更多的案例来跟各位做进一步的分享。

我们继续往下，我们再进一步可能再来给各位补充一些信息，我们刚才讲到了梳理自己公司的局部业务模型里边第一步就要明确input，output到底是什么，对最常见的局部业务模型无非说我们增长导向的模型，以及我们生产交付保障的这样的一种模型，

增长导向的模型我们就一定记住它的input，一般来讲流量钱或者是其他资源，而output往往一定是订单新增用户或者是收入，就像例如我们在负责一款这种体验产品的运营，例如在一个咨询公司或在一个to b的公司里边，我们主要是做会销的，每一场我们举办的这种大会对我们一个转化或者体验的一个产品，我们就不断的要去做好说会议它怎么来去运营，会议，它的input肯定说我们投进来的人和流量和钱

Output一定要转化成订单，中间涉及到有很多的角色怎么去分工，我们一个会议典型应该什么样一个流程，约就逻辑生产保障和交付型的这种模型 Input，一般来讲是我们生产资料或者初始状态的用户，就一个用户买完了我们的产品和服务，他进来是一个初始状态，output往往要么是我们产品的成品，要么说用户在我们这儿很爽很嗨，他的满意度是很高的，十分的认可我们，甚至最后还续费了，所以生产保障交付的模型，它的input output往往这样的一个认为，这是我们想给各位重点补充的其中一块信息。

接着再给各位补充点信息，我觉得是说各位要有认为，对于一部分可能较为简单的业务来讲，按照我们刚才讲的这4步流程，然后很多时候可能你把你的业务流程稍微画出来，处理个模型就已经七七八八了，很这样的。

例如举个例子，像三节课过去课程的模式叫做魔鬼训练营模式，对在模式下教研和教学它的局部也有模型，显著是有一个流程的，从最前面的讲师BD到课程研发到上线之后班期教学对课程研发这里边它也有一个流程立项的评审，课程的大纲，然后课程PPT录制等等这一拿东西，这一块流程理出来之后，我们再往下再去补充一些，说立项评审里边有些人要参与，它的标准怎样的，然后课程录制需要有些什么配套的，什么原则和sop等等这东西补进去。

教研这块的局部业务模型差不多就清楚了，教学这块也类似班级教学可能也有一个简单的流程，从开营到每周的看课，再到每周要让很多学员去写作业，再到毕业设计和一些反馈，对中间一定会涉及到说每周看一些什么样的数据，根据每周数据做一些什么样的运营的这种策略的调处理之类的，对这些东西这些配套的一些机制和配套的一些工具和sop都补进去。

教学这块的业务模型差不多可能也会较为清晰了，所以很多时候对一些简单的业务，你所负责业务它没么复杂的，很多时候可能你业务流程稍微梳理出来，梳理出来还较为清晰，你的局部应用模型可能也就已经七七八八了。

那么当然了也会有一些复杂一点的这种业务，如果你负责业务模块或者说你负责这个部门，它的业务逻辑相对有点复杂，它的复杂往往会复杂在于什么？往往会复杂在于说我所负责这块业务，这块局部业务里边，它牵扯到有n多的角色，这角色之间的关系流程之类的很多时候是交叉的，需要并行的，可能还会针对不同的角色，有很多的一些约束机制，还有对应的策略等等，它复杂在地方。

所以如果你所负责的局部业务约是这么一个业务，你会发现它当中可能一个较为重要的事，还是我们刚才的，你得把你所负责的这块业务它银葡萄的葡萄是什么界定清楚，界定清楚完了之后中间涉及到到底有多少个角色，这角色之间的关系和分工是怎样的，你可能要优先先把层次先梳理清楚。

界定出来了这么4类的这种角色，而处理个过程中把我们的处理个的这种业务流程，因为它的input output也是较为明确的。 Input我初始进入班期学习的学员对Output一定说我要让学员经由我个性化的这种辅导方案之后，他最后处理个班级的这种学习效果口碑都得到了显著的提升，所以这里边肯定还是有个流程的，学前学中学后，这几个角色在其中各自分别都做什么，他们怎么来协作，他们的关系是怎样的。

，然后可能就把通过这么一个逻辑，就把这么一个局部的模型去给他去串清楚了，这是其中一个case，所以在这一类相对有点复杂的这种局部业务当中，就各个角色之间的这种关系，它的协作的逻辑和机制分工，还有我针对每个角色是否会有一些不同的策略尤为重要。

，这是一个case。再来看另外一个case，另外一个case我们很常见的一个用户分层运营体系的这么一个case了，这是美团APP在早年某个时间阶段的它的一个用户分层运营的这么这种体系和逻辑。

你也会发现在这样一个case里边，它也是因为它的用户的类型就较为复杂，所以它优先是说处理个模型里边我们优先把不同类型的用户先界定出来，每类用户从平台我们给予到他们的一些策略，去拎得足够清楚的时候，这么一个用户分层的运营体系相对就已经较为清晰了，因为这里边虽然分了十几类用户，但不同类型的用户之间他们是不需要发生关系的，在体系里边后边你要把运营体系把它要展现的更完处理一点，它涉及到说在我们自己公司的后端
