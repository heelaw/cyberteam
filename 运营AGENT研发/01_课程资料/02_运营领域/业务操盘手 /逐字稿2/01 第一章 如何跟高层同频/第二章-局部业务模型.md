# 2.1 老板当前阶段究竟在关注什么

## 学习目标

- 理解早期公司和成熟公司的评估逻辑差异
- 能判断自己公司所处阶段
- 能识别老板当前的核心关注点

## 核心概念：两种评估逻辑

### 早期公司的评估逻辑

> 并不是每家公司一开始就会有一个清晰的经营模型。

**核心关注**：找到能被做大的可能性

**典型特征**：
- 多方向尝试（"5个地方同时打井"）
- 不追求清晰的模型，追求增长破局
- 先有流量或收入，再谈模式

**例子**：三节课第一年（2016）
- 同时尝试N个业务方向和idea
- 微信涨粉、付费课程定价摸索、社区、工具产品等
- 哪个方向有效果，就持续投入

### 成熟公司的评估逻辑

> 对一家成熟公司的经营者、管理者而言，他说的思考最终一定要回归到模型上。

**核心关注**：模型的可驱动性和增长空间

**典型特征**：
- 所有的思考回归到商业价值和业务模型
- 评估现有模型能否支撑20倍增长
- 关注：成本结构、利润模型、增长驱动

**例子**：三节课第四年（2019）
- 评估整体业务模型是怎样的
- 现有模型下如何驱动增长
- 这个模型还有没有前途，如何优化模型获得更大的增长空间

## 案例演示：三节课的业务模式演变

### 2016年：早期探索期

**核心特点**：多方向尝试，没有清晰模型

**尝试方向**：
- 微信涨粉：三四个月涨到10万粉 → 持续投入
- 付费课定价：199→499→699→1000+ → 慢慢摸索
- 工具产品：做了 → 死了
- 社区：做死了 → 放弃

**底层逻辑**：5个地方同时打井，哪个地方出水，就接着往下干

### 2019年：成熟管理期

**核心特点**：所有思考回归商业模型

**关键问题**：
1. 整体业务模型是怎样的？
2. 现有模型下如何驱动增长？
3. 这个模型还有没有前途？
4. 如果只能增长1倍，如何优化模型获得20倍以上的增长？

## 关键洞察

> 这背后是投资人和管理者视角上，评估不同阶段公司的逻辑不一样：
> 
> - 早期公司：评估做大可能性，找到机会就行
> - 成熟公司：根据模型估算估值和增长空间

**不同阶段的评估逻辑**：

| 阶段 | 核心问题 | 评估标准 | 典型行为 |
|------|---------|---------|---------|
| **早期公司** | 有机会做大吗？ | 感觉有机会就搞 | 多方向尝试，看哪里出水 |
| **成熟公司** | 模型能做多大？ | 增长速度、盈利能力、模型质量 | 评估模型天花板，优化增长策略 |

## 总结与行动

✅ 判断你的公司处于哪个阶段？
✅ 列出老板当前最关注的3个问题
✅ 你的工作是否在支持这些核心关注点？

# 2.2 局部业务模型的构成与梳理方法（上）

## 学习目标

- 理解局部业务模型的定义和本质
- 掌握局部业务模型的4大构成要素
- 能用樊登读书会分销体系案例说明局部业务模型

## 核心概念：局部业务模型的定义

### 什么是局部业务模型

局部业务模型是全局业务模型中的某一个核心业务环节或模块，把它单独拿出来梳理清楚，说明这个环节的业务模型是怎样的。

### 本质特征

> 业务模型本质上一定有个稳定的input和output。

**局部业务模型的input**：某种特定的业务资源（人、生产资料、钱等）

**局部业务模型的output**：某种稳定标准的业务价值
- 某个环节的转化率
- 用户满意度
- 人效比
- 收入等

### 核心问题

局部业务模型本质上是一个完整商业系统当中的一个子系统，它解决的核心问题是：当公司的业务量要增加10倍或20倍时，如何建立系统确保这个局部业务仍然可以顺利运转。

**认知关键**：
- 业务体量小的时候不需要模型（如客服只有2-3人时）
- 业务体量大到一定程度必须建立模型（如客服有几十上百人时）
- 模型是为了应对规模化增长而建立的系统化机制

## 案例演示：樊登读书会的分销体系

### 体系架构

樊登读书会内部存在一个非常重要的局部业务模型——各地区的分销或城市合伙人体系。

```
│           提供渠道发展和扶持                                    │

│                     ↓                                           │
│    ┌──────────────────┬──────────────────┬──────────────────┐   │
│    │    【城市分会】   │   【县级分会】   │   【企业分会】   │   │
│    │                   │   配套樊登书店   │    设计驿站      │   │
│    │  会员发展和服务    │  承载分会活动    │    实体载体      │   │
│    │  给利润空间       │  排满活动承载量  │                 │   │
```

### 运作逻辑

**省级分会**：
- 提供渠道发展和扶持
- 给政策、资源、配套支持

**城市/县级/企业分会**：
- 承载会员发展和服务
- 给利润空间、资源、政策支持
- 县级分会配套樊登书店（实体店）
- 企业分会设计小驿站等

**三级分销利益分层**：
- 通过利益分配机制激励各级分会
- 支撑各地区强大的会员数量增长

从视觉化呈现的角度，一个局部业务模型通常包含4种基本要素：

### 模型流程

```

│           教育公司师资供应链管理模型                             │
```

### 4大要素的体现

**要素1：流程**
- 从营销数据推导开班计划

- 招募、培训、考核
- 匹配到班期
- 服务用户
- 品控监测
- 收入核算

**要素2：关键环节**
- 招聘计划形成
- 老师服务完成后的考核品控
- 老师淘汰机制

**要素3：约束机制**
- 开班计划预警机制
- 人员匹配预警机制
- 绩效管理激励机制
- 品控机制
- 财务核算机制

**要素4：角色分工**
- 营销团队负责人
- HR团队
- 课程负责人
- 财务负责人
- 供应链负责人
- 各角色之间的协作分工

## 构建局部业务模型的4步法

### 第一步：明确input和output

**增长导向模型**：
- input：流量、钱、其他资源
- output：订单、新增用户、收入

**生产保障/交付型模型**：
- input：生产资料、初始状态的用户
- output：产品、用户满意度、续费

### 第二步：识别业务量增长时可能遇到的问题

常见4类问题：
1. 用户量/产品类型/业务方类型太多，需求和关注点不一致
2. 多角色之间的协作流程和机制不清楚
3. 用户和内部成员的成长发展路径不清晰
4. 交付过程或用户使用过程缺乏清晰流程

### 第三步：对问题进行分类，设定流程、路径、关键环节、角色分工

确保逻辑自洽：当业务量增加时，通过这样的分类和协作，能够保证客服响应时间和用户满意度。

### 第四步：完善配套约束机制和SOP

- 面向用户的激励机制
- 面向团队成员的激励机制
- 品控机制
- 定期核算机制
- SOP补充

## 总结与行动

✅ 明确你所负责的局部业务的input和output
✅ 识别业务量增长时可能遇到的问题
✅ 用4步法构建你的局部业务模型
✅ 用4大要素检验你的模型是否完整

# 2.3 局部业务模型的构成与梳理方法（下）

## 学习目标

- 理解局部业务模型的层次性（一级/二级/三级）
- 掌握局部业务模型的4步梳理法
- 能用考研教育公司和2B销售团队案例说明二级模型

## 核心概念：局部业务模型的层次性

### 前提认知

局部业务模型本身是分层次的，分为一级、二级、三级。

> 一个业务模型既然是模型，本质上一定是为了解决一类复杂的业务问题。

**关键认知**：
- 只有面临明确的复杂业务问题时才需要一个模型
- 如果问题很简单，不需要模型
- 例如：管理3个客服 vs 管理几千人的call center

### 一级局部业务模型

**定义**：依据商业逻辑，公司整体业务会被分成几个重要的一级业务模块。

**典型一级模块**：
- 教育公司：营销、教研、教学
- 制造业：生产、销售、供应链、仓储配送
- 金融公司：获客、风控、资产管理

**重要提示**：一级局部业务模块和部门划分未必完全一一对应

例如：教育公司里的产研部门
- 它是支撑营销、教研、教学这三个一级模块的
- 产研做工作是在每个业务模块下赋能效率提升
- 产研可能是在二级或三级局部业务层面工作

### 二级局部业务模型

**定义**：一级局部业务模型下的一些细分的子模块或玩法。

**典型二级模块**：
- 营销模块下：信息流获客（一块独立的二级业务）
- 营销模块下：私域流量运营（另一块二级业务）

**三级局部业务模型**：
- 例如：营销模块下的私域流量销售团队如何管理、培训、招募、淘汰
- 这也需要一个模型来支撑

- 有清晰的策略和配套机制

## 四类常见一级局部业务模型

### 1. 增长类局部业务模型

**核心业务问题**：用户增长或收入增长的稳定产生和可规模化。

### 2. 生产类局部业务模型

**核心业务问题**：
1. 效率和收入的相关性
2. 效率角度的成本控制

### 3. 供给类局部业务模型

**核心业务问题**：
- 前端需求与后端供给能力的匹配
- 保证供应链运作效率，减少损耗和浪费

### 4. 交付类局部业务模型

**核心业务问题**：用户满意度、NPS、续费、复购等指标的稳定产生，以及人效的持续降低。

## 优秀二级局部业务模型标准

**标准1**：背后有一个明确的、重要的、复杂的业务问题

**标准2**：模型清晰地展示了这个问题如何能够被稳定解决

**标准3**：看到模型后，基本不需要再问或沟通就能理解所有工作要点

## AB类操盘手的基本要求

✅ 必须能够梳理清楚你所在业务线当中至少到二级局部业务模型

✅ 时刻保持清楚：
- 到底在解决什么复杂重要的业务问题
- 这个问题背后所依赖的整体策略和模型是怎样的

✅ 你的职业发展路径：
- 向上对话老板

# 2.4 如何在决策类问题上跟老板同频

## 学习目标

- 理解商业决策的基本结构
- 掌握4个决策工具：线性/非线性增长模型、成本收益模型、头部效应模型、杠铃策略+黑天鹅策略

## 商业决策的基本结构

```
┌─────────────────────────────────────────────────────────────────┐
│              商业经营决策的基本结构                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│     【判断所处赛道】                                             │
│           ↓                                                     │
│     【制定战术打法】                                             │
│           ↓                                                     │
│     【风险防范 + 机会管理】                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 决策工具1：线性 vs 非线性增长模型

### 线性增长模型

**核心逻辑**：投入越多，等比例的收入或用户增长。

**适用业务**：
- 2B行业（销售驱动）
- 线下零售（高度依赖门店数量）
- 传统线下教育（依赖门店和老师）

### 非线性（指数级）增长模型

**核心逻辑**：有机会实现边际成本为零，能够实现指数级增长。

**适用业务**：
- 社交产品
- 在线工具
- 在线娱乐平台
- 通过用户关系链或NPS的拉新玩法

### 应用场景

**1. 判断赛道**：你所在的行业是什么赛道，增长逻辑是线性还是指数级

**2. 判断玩法周期**：当前玩法处于什么周期

**3. 寻找新增长引擎**：当旧玩法到末端时，必须开辟新增长引擎

## 决策工具2：成本收益模型

> 你做所有事情最终都看收益，也需要投入成本。

**养成习惯**：成本收益模型必须成为一个非常稳定的思考习惯

**应用场景**：
- 评估一笔钱值不值得花
- 评估一件事值不值得投入
- 评估部门业绩和价值
- 评估团队成员的工作价值大小

## 决策工具3：头部效应模型

> 在任何一个赛道里，成为头部玩家所获得的收益，会远远胜于非头部玩家。

**经典案例**：彭小六的逆袭战
- 在简书（二线战场）成为"简书一哥"
- 携带势能杀回微信（一线战场）

**应用场景**：
- 产品上线后的渠道选择：集中一个渠道成为头部
- 职业生涯选择：小公司核心角色 vs 大公司不重要部门的执行者

## 决策工具4：杠铃策略 + 黑天鹅策略

**杠铃策略**：分散风险，把资源分布式投到不同领域

**黑天鹅策略**：为未来做探索和准备，投给不确定性大但可能带来回报或未来增长引擎的事情

**资源分配建议**：
- 80%以上资源：给有确定性的主战场
- 20%以内资源：给风险和不确定性负责的事

# 2.8 业务模型搭建的避坑指南

## 学习目标

- 理解简单模型vs复杂模型
- 掌握两个核心忠告
- 理解流程机制的本质

## 简单模型 vs 复杂模型

### 越复杂的业务模型，挑战和风险越大
- 需要的时间也更长
- 这是需要有的基本认知

### 什么是简单的业务模型

**特点**：
- 单线程
- 变量少
- 中间线条、分叉分支很少

**例子**：工具产品
- 外围流量引进来
- 用户用工具
- 用完喜欢就再来用
- 继续用，养成习惯
- 完了

### 什么是复杂的业务模型

**特点**：
1. 线程多（前端、后端、中间还有辅助配套机制）
2. 中间涉及n多人和人的交互
3. 有n多信息的交互（A信息制约B行动，C信息制约D行动）

**典型例子**：
1. 沃尔玛供应链系统
2. 美团外卖配送后台
3. 教育行业（教研、教学、用户之间，服务和互动节点非常多）

### 风险认知

**简单业务模型**：
- 1-2个线程，分叉很少
- 大概1个月内能出答案说这个模型OK还是不OK

**复杂业务模型**：
- 线程很多，中间环节很多
- 人和人的互动是巨大变量
- 信息交互意味着很多制约
- 周期会比较长，风险更大，难度更大

### 复杂业务模型的公司必要条件

如果你所在公司的业务逻辑本身比较复杂：
- 要么在很多地方的局部有成熟经验可借鉴
- 要么有非常雄厚的资源可支撑
- 要么在好几个模块下已经跑得比较稳定

## 两个核心忠告

### 忠告一：尽量让别人跑通过，成熟可借鉴

如果你在一个新项目中要搭一个模型：
- 要么让别人跑通过、成熟可借鉴
- 要么有非常雄厚的资源可支撑

### 忠告二：不要在假想中把模型想得很复杂

如果你在一个局部要创建一个业务模型：
- 不要自己在假想中就把这个模型想得很复杂
- 不要想变量非常多
- 如果那样的话，这个模型能不能跑出来对你而言非常有风险

### 典型反面案例：教育公司转化型产品

**模型逻辑**：
1. 外部公号投放
2. 老带新工作（给用户权益：系列年度大咖直播）
3. 转化产品（目标5%-10%转化）
4. 用户听课后想要年度大咖直播权益，帮忙转发形成老带新

**模型问题**：
- 不够简单
- 线程太多
- 验证周期长
- 变量太多

**核心问题**：变量太多且互相影响
- 转化产品本身没跑通
- 投放是待验证的事
- 系列大咖直播需要做
- 这几件事互相影响

**风险**：
- 如果转化率不稳定，不知道是转化产品本身有问题，还是外部投放用户质量不高
- 会不会因为做了年度大咖直播，用户直接去做老带新，就不付费了

**结论**：这样的模型风险特别大，不太容易被验证成功，很容易跑了三四周之后全是坑，一地鸡毛。

## 流程机制的本质

### 存在的意义

> 各种流程制度在一家公司里边存在的意义和本质只有4个字：提升效率。

- 因为有了流程，不需要每次做一件事都找人去问
- 按流程来就OK了

### 成熟公司的管理必然包含

因为一家公司的经营到最后就是看模型：
- 能够被一些模型抽象出来
- 能够被一些模型驱动和管理好
- 这时一定要配套上一些流程和制度
- 让公司的运转变得更高效、更精细

### 管理与业务的关系

**在中早期项目**：
- 管理服务于业务
- 一切职能部门、支持部门、配套流程工具都要为业务服务
- 怎么对业务更灵活、怎么对业务更好就怎么来

**在成熟公司**：
- 有些管理机制已经形成基础支撑（如腾讯员工晋升机制）
- 有些管理机制不可动摇

### 业务越成熟，配套管理工具越庞大

**认知**：
- 业务越成熟、公司越大，配套管理工具和工作必然会更加庞大
- 这是从管理角度看到的规律

### 例子
- 某世界500强房地产公司的管培生招聘系统（背后有一系列复杂的管理机制）
- 华为内部围绕组织能力提升的一整套机制

**认知**：
- 越庞大的公司，管理一定会越复杂
- 到了一定阶段，要掌握的管理工具会更多
- 甚至自己要有能力建立管理体系

# 2.9 如何与中层领导相处

## 学习目标

- 理解正确的协作关系
- 掌握向上管理的正确心态
- 理解两种视角的差异

## 正确的协作关系

如果你已经较好地理解了公司经营和管理的逻辑和层次体系，你可以站在你的角度跟你的上级leader良好协作。

**协作关系核心**：
- 他要向你提要求：告诉你要负责的问题是什么，部门怎么确保工作对公司运营有价值
- 如果他给不了你这个东西，你要向他要目标、要指令
- 他给到你目标和指令后，你应该给他答案

**辅助判断**：
- 如果上层有些事想不清楚，你可以辅助判断他的规划、思考跟高层有没有偏差
- 如果你发现他的规划有些风险，你有能力的话跟他一起探讨
- 帮他搞定信息不对称

## 向上管理的正确心态

**理想的角色定位**：
- 先支持老板能够更好地思考和判断
- 老板有了一个优质决策的时候，保证这个决策能够被推动落地
- 老板很多时候决策是错的，用行动支持他，快速用行动告诉他这个东西是错的

**错误的心态**：
- 现在就要一步登天替代掉我的老板

## 两种视角的差异

**执行层面视角（静态）**：
- 追求绝对确定性
- 老板之前给了我目标，按目标做完了，老板又说不行
- 觉得老板不靠谱

**操盘手视角（动态）**：
- 不要那么强调追求确定性
- 在不确定环境中，以商业逻辑为导向，不断迭代寻求问题的更优解
- 下面人做了一个东西，做到一半或30%，一看就不合理、不靠谱、没价值
- 如果下面人看不出来，没有及时反馈，会不高兴

**认知**：
- 管理和业务发展上很多事情和结论都是动态的
- 不要追求绝对正确的答案
- 一切的起点可能是先让自己能参与到老板们的决策过程中去

# 2.10 从案例看局部业务模型如何逐渐成型

## 学习目标

- 通过三节课B端业务案例理解模型成型的周期
- 理解模型成型的规律

## 三节课B端业务发展的4个阶段

### 2017年：诞生期（非常随机、零散的状态）

**起源**：某大厂主动找三节课做内训

**状态**：
- 整个业务非常随机、零散
- 给B端交付或让B端客户满意完全依赖老师
- 早期是CEO陆可亲自去现场讲
- 完全看陆可当天的生理和心理状态、临场发挥

**认知**：
- 前端的获客不稳定
- 后端的交付也没那么稳定
- 处于"叮咚咣咚"的状态

### 2017年下半年：销售模型初步搭建

**契机**：持续找上门来的B端客户量还可以，三节课内部几个同事讨论决定以正规军方式好好搞，不要"野战军"、"东一榔头西一棒子"、"游击队"状态。

**销售转化上的搭建**：
- 通过销售团队人肉的BD
- 购买少量企业客户线索
- 招聘网站合作，找到企业内部培训负责人的联系方式
- 销售团队定向跟进，售卖训练营项目

**状态**：
- 初步建立销售线索和销售跟进转化模型（依靠人来完成成交）
- 虽然有模型，但很不稳定
- 不同销售在手中的转化率非常不稳定
- 不同行业客户的需求都不一样
- 无法做到稳定的input和output

**认知**：这是B端业务的中期状态。

### 2018年上半年：交付模型初步成型

**决策**：跑了大半年时间后，一定要做到可规模化的交付、可规模化的满足需求。完全依赖老师个人生理心理状态去交付不靠谱。

**行动**：一个同学牵起头，做了一系列梳理，经过3-4个月打磨和尝试，先后在大概10家企业里做了落地和尝试。

**输出模型**：
```
┌─────────────────────────────────────────────────────────────────┐
│           三节课B端业务到企业端的交付模型                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│     【周期1】 → 【周期2】 → 【周期3】 → 【周期4】              │
│           ↓           ↓           ↓           ↓                │
│     【目标】   【角色】   【机制】   【指标】                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**模型特点**：
- 把整个交付分为几个周期
- 每个周期的目标是什么
- 涉及到哪些重要角色
- 中间有哪些重要机制
- 完成之后看哪些重要指标评估交付合格还是不合格

**状态**：
- 有了这个模型后，三节课在B端做的交付相对比较稳定
- 基本形成感觉：围绕着客户需求，B端可以做的交付相对比较稳定

### 2018年下半年：销售模型优化

**问题**：回归前端销售，发现虽然有基本模型和机制，但并不稳定。

**优化**：
```
┌─────────────────────────────────────────────────────────────────┐
│           三节课B端业务前端销售优化模型                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│     【线索】 → 【辅助决策产品/体验产品】 → 【转化】              │
│                    ↓                                          │
│             1. 社群（B端HR进入三节课社群）                      │
│             2. 低客单价产品（体验课程）                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**逻辑**：符合"数理油模型"
- B端培训单价高（少则几万，多则几十万），决策成本高
- 中间大概率需要辅助决策或体验产品

**状态**：
- 跑完之后，三节课B端整个业务模型相对比较稳定
- 前端：一系列流程 + 辅助决策产品，转化率相对稳定
- 后端：生产交付模型，稳定后端客户满意度

## 模型成型的周期和规律

### 周期认知

**典型周期**：小一年时间

三节课B端业务：
- 2017年诞生
- 2017年下半年：销售模型初步搭建
- 2018年上半年：交付模型初步成型
- 2018年下半年：销售模型优化
- 整整一年多时间，整个B端业务才初步趋于稳定

**规律认知**：
- 在很多中早期初创公司，让你的业务初步成型，这个周期怎么着也得小一年时间

### 模型成型的真相

> 一个业务整体模型的成型一定是不断的动态演化而成的。

**三节课B端业务验证**：
- 交付模型：经历了十几家用户的验证，初步成型，稳定下来，然后优化
- 销售模型：一开始有一个版本上线后发现不ok，回过头来优化思考调整

**结论**：
- 不可能一开始就有一个固定答案就不能变了
- 这门课教给你的是思考的方法和逻辑
- 需要不断验证、检验、内化

## 章节核心总结

### 关键认知

1. **局部业务模型是全局业务模型的子系统**
   - 有稳定的input和output
   - 为了应对规模化增长而建立的系统化机制

2. **局部业务模型有4大构成要素**
   - 流程/成长路径
   - 关键环节 + SOP
   - 约束机制
   - 角色分工

3. **老板问题背后有5层需求**
   - L1：验证战略方向的假设
   - L2：评估商业模式/成本模型
   - L3：了解业务模型的分支模块怎么搭建
   - L4：了解执行落地路径
   - L5：讨论怎么执行更好

4. **高层视角有4类有价值成果**
   - 提升效率，降低成本
   - 稳定某个重要的交付
   - 放大收入
   - 创造具有商业想象力的新产品

5. **模型成型是动态演化过程**
   - 不是一开始就有固定答案
   - 需要不断验证、调整、优化
   - 从0搭建有2种逻辑：外部参照、自己先做到再复制

### 实践建议

1. **判断公司所处阶段**：早期公司 vs 成熟公司
2. **明确自己在成本中心还是利润中心**
3. **用4步法构建你的局部业务模型**
4. **理解老板问题背后的需求层次**
5. **用数据和模型证明工作价值**
