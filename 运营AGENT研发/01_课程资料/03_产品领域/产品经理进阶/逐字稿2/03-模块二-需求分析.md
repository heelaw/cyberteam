## 模块二：需求分析

## 本章导学

需求分析是产品经理将用户需求转化为产品方案的关键环节。本模块将讲解用户故事、需求判断方法论（HMW分析、Kano模型）等核心技能，帮助学员学会如何讲好用户故事、如何判断需求真伪、如何通过系统化方法分析需求并找到解决方案。

## 详细内容

### 主题一：用户故事

#### 1.1 为什么要讲故事

**产品经理的核心能力：讲故事**

- 把问题从一个枯燥的讲解变成有感情的说服
- 让听众更容易理解
- 更好地完成工作

**讲故事的应用场景：**

| 场景 | 说明 |
|------|------|
| 讲需求亮点和卖点 | 让文案和推广物料更游刃有余 |
| 对内沟通 | 向研发、运营等讲明白需求 |
| 用户访谈 | 引导用户分享故事 |
| 对外宣讲 | 让用户认同产品价值 |

#### 1.2 说教vs讲故事

**说教：** 直接告诉别人应该怎么做
- "你应该干这个1234"
- 被动接受，效果差

**讲故事：** 用例子让人自己理解
- "我给你讲个例子吧"
- 主动理解，效果更好

#### 1.3 如何讲好用户故事

**用户故事四个要素：**

1. **用户是谁**：一群什么样的用户/一个用户
2. **场景**：在什么情况下想解决什么问题
3. **问题**：解决过程中碰到了什么具体难处
4. **解决方案**：用户是怎么解决的/现在怎么解决的

**案例：租房找房**
- 用户：在北京创业公司工作的女生小白
- 场景：找一居室和男朋友同居，希望离中关村近，低于3000块
- 问题：在58同城刷了半小时，遇到假房源、中介、二房东
- 解决：通过朋友推荐到豆瓣小组找到回龙观一居室

**提炼一句话：**
- "小白在58找一居室时总是碰到假房源、假中介"

#### 1.4 延伸分析：四限分析法

**纵向：产品问题vs用户问题**

- 产品问题：产品认知、功能缺失
- 用户问题：用户认知、用户功能

**横向：认知问题vs功能问题**

- 认知问题：知不知道
- 功能问题：有没有功能

**四限定位：**

| 位置 | 问题类型 | 解决策略 |
|------|---------|---------|
| 左上 | 产品认知问题 | 产品经理需检讨 |
| 右上 | 产品功能问题 | 排期考虑 |
| 左下 | 用户认知问题 | 运营通知 |
| 右下 | 用户功能问题 | 需求优先级 |

#### 1.5 讲故事的技巧

**基本原则：**

1. **真实**：反映实际用户故事，不过分扩大
2. **有用**：选择对听的人有用的信息
3. **简练**：30秒内抓住注意力

**不同场景的讲故事方法：**

| 场景 | 方法 |
|------|------|
| 讲需求 | 站在制高点，代表用户 |
| 推广产品 | 以卖点为起始点，先勾引再剖析 |
| 给运营讲 | 先讲对比，效率提升 |

### 主题二：判断需求

#### 2.1 需求判断的重要性

**问题根源：**
- 用户的需求可能并不代表真实想法
- 说一套做一套在产品工作中非常常见
- 用户想要的不一定是真实想要的

**案例：买车**
- 小明想要省油保养便宜的车
- 推荐其他品牌后，还是选择了奔驰
- 嘴上想要和实际行动截然不同

#### 2.2 需求具有迷惑性的原因

**原因一：用户认知有限**
- 用户只在认知范围内提需求
- 案例：没吃过牛排的人只选"十分熟"

**原因二：需求是方案不是问题**
- 用户提出的是具体方案，而非背后问题
- 案例：小明想买车，真实原因可能是上班不方便

#### 2.3 判断需求的三步法

**第一步：挖掘需求本质**

使用五外分析法（连续询问为什么）：
- 询问用户产生问题的真实场景
- 寻找需求的因果关系
- 打破砂锅问到底，找到根本原因

案例：借款产品取消订单功能
- 用户：想要增加取消订单功能
- 为什么后悔？：利息太高
- 为什么借款成功后才后悔？：前面不知道具体利息
- 本质：用户想知道具体利息是多少

**第二步：带入场景验证**

假设需求已满足，带入用户场景验证：
- 能否解决用户本质需求？
- 能解决 → 需求成立
- 不能解决 → 需求有问题

**第三步：多维度判断与权衡**

| 维度 | 说明 |
|------|------|
| 用户层面 | 体验问题、操作问题 |
| 产品层面 | 日活、约活、新增 |
| 业务层面 | 收入、盈利（最重要） |
| 历史层面 | 之前为什么没做 |

### 主题三：HMW分析法

#### 3.1 什么是HMW

HMW（How Might We）分析法是一种系统化需求分析方法，通过提问"我们可以用什么方式解决..."来激发解决方案。

#### 3.2 HMW分解方法

**常用HMW问题类型：**

- 删减型：我们如何能去掉某个功能/步骤？
- 组合型：我们如何能把某两个功能组合？
- 修改型：我们如何能改变某功能的形式？
- 替代型：我们如何能用其他方式满足用户？

---

### 主题四：Kano模型

#### 4.1 Kano模型概述

Kano模型将需求分为五类：

| 类型 | 说明 | 用户反应 |
|------|------|---------|
| 兴奋型 | 超出预期的功能 | 惊喜、满意 |
| 期望型 | 应该有功能 | 有则满意，无则不满 |
| 基本型 | 必须有的功能 | 无则不满，有则一般 |
| 无差异型 | 有没有都无所谓 | 无影响 |
| 反向型 | 不想有的功能 | 有则不满 |

#### 4.2 应用原则

- 优先满足基本型需求
- 努力实现期望型需求
- 适当提供兴奋型需求
- 避免反向型需求
- 无差异型需求少做

## 核心概念

| 概念 | 定义 |
|------|------|
| 用户故事 | 描述用户在特定场景下遇到问题及解决方案的叙事 |
| HMW分析 | 通过"我们如何能..."问题系统化分析方法 |
| Kano模型 | 将需求分为五类来分析用户满意度 |
| 需求本质 | 用户需求背后真实的想法和原因 |
| 五外分析法 | 连续询问挖掘需求本质的方法 |

## 本章小结

本章系统讲解了需求分析的核心方法：

1. **用户故事**：用叙事方式讲需求，让研发和用户更有代入感
2. **需求判断三步法**：挖掘本质→场景验证→多维权衡
3. **HMW分析**：系统化思考解决方案
4. **Kano模型**：理解不同类型需求对用户满意度的影响
