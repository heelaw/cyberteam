# 考核标准

## 基础门禁（五项必须同时满足）

- [ ] **有痛点**：必须有至少一条用户原话或具体情境描述，不能只剩抽象总结（如"用户觉得不好用"）
- [ ] **有场景**：场景还原必须包含"谁、在什么任务里、卡在哪一步"三要素
- [ ] **有阻碍**：阻碍必须说清楚"现在卡在哪里"，不能只写"质量差/效率低"
- [ ] **有代价**：代价必须可感知可量化，不能写"很严重/有影响"
- [ ] **有结果翻译**：结果必须翻译成用户能感知的状态，不能只写功能描述

## 卖点成立条件（三项全部满足）

- [ ] **卖点能回到产品能力**：卖点表达背后有对应的产品功能或服务能力支撑
- [ ] **卖点用户能验证**：用户能自己判断这个卖点是不是真的实现了
- [ ] **卖点用户能转述**：用户能把卖点用自己的话跟别人说清楚

## 证据质量标准

- [ ] **有原话来源**：至少有一条用户原话或能对应到具体反馈记录
- [ ] **有交叉验证**：同一痛点有≥2条独立证据（评论/工单/录音/行为数据）
- [ ] **有频率说明**：知道这个痛点是高频、偶发还是一次性情绪

## 证据缺口标注标准

- [ ] **缺口被诚实标注**：所有没有证据支撑的部分都标注了[证据缺口]
- [ ] **缺口类型说明**：标注了是频率缺口、量化缺口还是行为验证缺口
- [ ] **缺口没有被填补**：没有用假设或推断代替真实证据

## 待验证池标准

- [ ] **无法归入卖点的假设都被放进池中**：不能归入产品能力但有合理方向的部分 → [待验证]
- [ ] **池中条目有来源说明**：每条待验证假设都有对应的用户反馈依据

## 一票否决项（任何一条触发，输出不合格）

1. **只有抱怨，没有结果翻译**
   - 表现：只写了痛点描述，没有进入第5步结果翻译
   - 例：写了"用户说太麻烦了"就结束，没有翻译成"用户想要变成XX"

2. **只有结果，没有阻碍**
   - 表现：卖点表达悬浮，没有说明"现在为什么做不到"
   - 例：写了"让你轻松提升效率"，但没有说明"现在为什么效率低"

3. **把愿望句当可交付卖点**
   - 表现：结果翻译成的是抽象愿望，不是可感知状态
   - 例："提升生活品质"、"让你更专业"（无法验证、无法感知）

4. **没有场景，只写"大家/用户都需要"**
   - 表现：场景还原跳过，写的是普遍现象，不是具体情境
   - 例："现在很多人都有这个问题"（没有具体Who/When/Where）

5. **代价是抽象判断，不是可感知损失**
   - 表现：代价写成"很严重、影响很大"，没有具体的人、时间、损失量
   - 例："严重影响了工作效率"（什么效率？损失多少？谁来感知？）

6. **跳过证据恢复，直接假设成立**
   - 表现：证据不足时用推断代替，没有诚实标注缺口
   - 例：只有1条评论就说"用户普遍认为"，没有继续收集

7. **卖点表达是产品功能描述，不是用户语言**
   - 表现：卖点写成"我们的产品有XX功能"，而不是"你能XX"
   - 例："采用AI智能算法"（产品语言）vs "不用你自己想"（用户语言）

## 分级标准

### 合格（60分）

满足所有基础门禁和卖点成立条件，证据缺口诚实标注。

### 良好（75分）

满足合格标准，且：
- 场景还原包含五要素（Who/What/When/Where/Why lost）
- 阻碍分类清晰（工具/流程/能力/资源/心理）
- 代价分层（直接损失/间接损失/机会损失）
- 至少有一条证据达到★★★★★层级

### 优秀（90分）

满足良好标准，且：
- 痛点按身份/心理/业务/功能分层，并说明优先级依据
- 卖点分类清晰（功能型/信任型/效率型/身份型）
- 有完整的证据恢复记录（按优先级逐层补证据）
- 失败案例有自我预检（主动识别了潜在的替代解释）

## 新人实操考核题

### 第1题：痛点原话提炼

**输入**：用户说"每次项目一多我就怕漏事"
**预期输出**：能提炼出"漏事恐惧"为核心痛点，保留"怕"这个情绪词，区分"漏跟进"和"怕漏"的不同层次

**评分**：
- 能区分情绪词和实质痛点的 ✓
- 能写出至少2个具体表现的 ✓
- 能说清楚这个痛点属于哪层（功能/业务/心理/身份）的 ✓

### 第2题：场景还原

**输入**：用户说"开会的时候经常记不住谁负责什么"
**预期输出**：
- Who：项目负责人/跨部门协作
- When：周例会/月初/项目启动会
- Where：会议室/线上会议
- What-task：会后要执行具体任务
- Broken：散会后人各做各的，没有追踪机制

**评分**：
- 五要素齐全的 ✓
- 粒度适中的（不能太粗"开会的时候"，也不能太细"11月3日上午10点"） ✓
- 能识别出"反复出现"模式的 ✓

### 第3题：阻碍识别

**输入**：用户说"我想要随时看到所有项目进度"
**预期输出**：
- 阻碍1（信息阻碍）：不知道项目进度在哪里看
- 阻碍2（工具阻碍）：现有工具没有汇总视图
- 阻碍3（流程阻碍）：没有统一的进度更新机制
- 阻碍4（能力阻碍）：不知道什么算"进度正常"

**评分**：
- 能区分不同类型阻碍的 ✓
- 阻碍具体的（不是"没有系统"而是"有系统但看不到汇总"） ✓

### 第4题：代价估算

**输入**：用户说"我觉得进度管理很麻烦"
**追问后**：每月手动整理数据表花6小时；每次汇报前要临时找数字，被领导追问"这个数字对不对"；因为数据对不上会议迟到3次

**预期输出**：
- 直接损失：每月6小时手动整理 = 约等于一次完整复盘的时间
- 信任损失：被领导问数字对不对 = 专业形象受损
- 机会损失：这6小时本来可以用来看真正的洞察

**评分**：
- 能把抽象麻烦翻译成具体时间/金钱/信任损失 ✓
- 能区分直接损失和间接损失的 ✓
- 损失有量级的（不是"有影响"而是"每月6小时"） ✓

### 第5题：结果翻译

**输入**：
- 痛点："进度不知道问谁"
- 阻碍：没有统一的地方能看到谁负责什么
- 代价：反复在群里@人、开会时答不上来

**预期输出**：
- 结果1：打开一个页面就知道所有人的进度
- 结果2：不用再@人问，直接有人负责按时更新
- 结果3：开会时点开就能看到，不用临时找

**评分**：
- 结果是可感知状态的 ✓
- 用户能验证的（不是"提升效率"这种抽象词） ✓
- 能区分功能结果和心理结果的 ✓

### 第6题：卖点表达

**输入**：结果"打开一个页面就知道所有人的进度"

**预期输出**：
- ✗ "实时同步项目管理系统"（功能描述）
- ✓ "不用再满世界问人，一个页面看清楚"（用户感知）
- ✓ "开会不用临时抱佛脚，打开就有答案"（可验证+有画面）

**评分**：
- 卖点是用户语言而不是产品语言的 ✓
- 用户能想象自己用起来是什么感觉的 ✓
- 用户能转述给别人的 ✓

### 综合题：找出缺失

**任务**：审查以下输出，指出缺失项和原因

```
痛点：用户说"太麻烦了"
场景：上班的时候
阻碍：效率低
代价：影响工作
结果：提升效率
卖点：让工作更高效
```

**参考答案**：
1. 痛点原话缺失"麻烦"具体指什么 → 追问：什么操作让你觉得麻烦？
2. 场景只有"上班的时候"，缺少Who/What/Where/具体时刻
3. 阻碍太抽象，"效率低"是结果不是阻碍 → 补：卡在哪里？是找不到功能、数据对不上、还是操作步骤太多？
4. 代价太抽象，"影响工作"无法感知 → 补：影响什么工作？每月损失多少时间？
5. 结果"提升效率"不可验证 → 改：每天节省多少时间？
6. 卖点"让工作更高效"是功能描述 → 改：具体能让用户感受到什么变化
