# 痛点反推卖点器（反向卖点提炼器）

## 作用

从用户痛点出发，反推出可表达的卖点。

**核心逻辑**：用户不说"我要买这个功能"，用户说"我受不了这个"。从"受不了"到"正好能解决"，中间隔着一整条推理链。这条链断了，卖点就浮不上来。

这一步先回到用户语言，再翻译成产品能接住的表达。

## 启动前检查

- [ ] 只有痛点、评论、抱怨或需求描述时才用这一步。
- [ ] 如果已经有完整产品资料，优先用正向卖点（research-sellpoint-forward）。
- [ ] 如果连痛点都没有，只有模糊需求，先去补用户研究，不要硬反推。
- [ ] 如果有多条痛点，按频率×损失×产品匹配度排序，先处理最强的。

## SOP 六步链

```
第1步：痛点  → 第2步：场景  → 第3步：阻碍  → 第4步：代价
  → 第5步：结果  → 第6步：卖点
```

每一步都要有实质内容才能进入下一步。任何一步缺失，都回退补齐。

## 第1步：痛点提炼

### 操作定义

收集用户原话，从中提取真实的痛点。不是总结，是还原。

### 追问 Prompt（用于补原话）

```
用户这句话里，最让他难受的是哪一点？
有没有具体的动作、重复出现的场景、或者情绪词？
如果只能改一件事，他最想改什么？
```

### 判定标准

- 能找到原话中具体抱怨词的 ✓
- 只有一个抽象情绪词的（如"不好用"），需继续追问
- 多个痛点混杂的，拆成独立痛点

### 痛点分层

| 层级 | 特征 | 例子 |
|------|------|------|
| **功能痛点** | 做不到、做不完、做不顺 | 太慢、容易漏、总是错 |
| **业务痛点** | 导致转化差、流失高、协作断 | 漏单、返工、项目延期 |
| **心理痛点** | 担心出错、怕被问责、怕丢体面 | 怕忘、怕被说、怕不专业 |
| **身份痛点** | 跟身份认同相关的失落感 | 显得业余、没面子、不是"我" |

**优先级**：身份痛点 > 心理痛点 > 业务痛点 > 功能痛点。
身份和心理痛点最容易驱动行为，但必须有场景支撑，不能凭空拔高。

## 第2步：场景还原

### 操作定义

还原痛点发生的具体情境。时间、地点、人物、任务、触发动作、反复出现模式。

### 场景还原五要素

```
1. 谁——用户是什么角色（运营？老板？新手？）
2. 在什么任务里——正在做什么事
3. 此前做过什么——事件前序
4. 卡在哪一步——具体的断裂点
5. 结果丢了什么——具体可感知的损失
```

### 追问 Prompt

```
这个痛点最常在什么时候发生？
是一整天都在，还是集中在某个时刻？
是只有他一个人遇到，还是周围人都有这个问题？
每次发生前，用户做了什么动作？
```

### 判定标准

- 五要素齐全的 ✓
- 只有"用户说太慢了"，没有具体场景的，需要继续补
- 场景描述太宽泛（如"上班的时候"）的，需要继续具体化

### 场景粒度

| 太粗 | 太细 | 适中 |
|------|------|------|
| "上班的时候" | "每天早上8点47分挤地铁时" | "早上通勤路上，单手操作手机回复消息时" |

## 第3步：阻碍识别

### 操作定义

阻碍是"为什么现在做不到/解决不了"。不是原因分析，是卡点的具象化。

### 追问 Prompt

```
用户现在想解决，他卡在哪里？
是工具不够好，还是流程有问题，还是没有支持？
如果给他一个魔法解决方案，他第一步还是迈不出去的原因是什么？
```

### 阻碍分类

| 类型 | 典型句式 | 例子 |
|------|----------|------|
| **能力阻碍** | 不知道怎么做 | 没有数据分析能力 |
| **工具阻碍** | 现有工具不支持 | 表格太复杂，用不起来 |
| **流程阻碍** | 流程里没有这个环节 | 会后没人负责跟进 |
| **资源阻碍** | 没有时间/人力/预算 | 手动整理太费时间 |
| **信息阻碍** | 不知道什么是对的 | 不知道转化率多少才算正常 |
| **心理阻碍** | 不想做/不敢做 | 怕改版影响现有数据 |

### 判定标准

- 能说清楚"是什么挡在路上"的 ✓
- 只说"因为不好用"或"因为没用"，没有具体的 ✓

## 第4步：代价估算

### 操作定义

代价是"这个阻碍不消除，会持续带来什么损失"。必须可感知、可量化、可影响行为。

### 代价写法规则

```
❌ 不能写：损失很大、很严重、影响效率
✓ 应该写：每漏一单损失X元、每天浪费X分钟、每月因为这个问题返工X次
✓ 应该写：被领导问"为什么又漏了"、在客户面前显得不专业
✓ 应该写：这个问题导致项目延期X天，直接影响交付
```

### 追问 Prompt

```
这个问题持续多久了？
每次发生，损失有多大？
如果不解决，一年下来会积累成什么问题？
他现在用什么方式临时应对？那个方式本身有什么代价？
```

### 代价分层

| 层级 | 可信度 | 例子 |
|------|--------|------|
| **直接损失** | ★★★★★ | 漏单、延期、返工、直接丢钱 |
| **间接损失** | ★★★★ | 信任度下降、协作摩擦增加 |
| **机会损失** | ★★★ | 本来可以做X，但没时间/精力 |
| **心理代价** | ★★ | 焦虑、自我怀疑、怕被问责 |

**注**：心理代价可信度低，必须配合前三者使用，不能单独做卖点驱动。

## 第5步：结果翻译

### 操作定义

把痛点（不要什么）和阻碍（卡在哪里）翻译成用户真正想要的结果状态。

### 核心追问

```
用户真正想要变成什么样？
不是"解决这个问题"，而是"变成什么状态"？
用户不要的是漏单，那他要的是什么？——可能是"所有待办都在一个地方"
用户不要的是返工，那他要的是什么？——可能是"第一次就做对"
```

### 结果翻译规则

```
❌ 不能写：提升效率、优化流程、改善体验
✓ 应该写：出门不用检查第二遍、开会就知道该干什么、下个月不用再返工
✓ 应该写：同行都觉得专业、领导问起来有数据支撑
```

### 判定标准

- 结果是可感知状态，不是功能描述 ✓
- 结果是用户能主动验证的，不是供应商承诺 ✓
- 结果能落回产品能力或服务能力，不能的先放进待验证池 ✓

### 待验证池

如果结果翻译不出产品能力，格式如下：

```
[待验证] 方向：XXX | 假设：YYY | 需要验证：Z
例：[待验证] 方向：减少焦虑 | 假设：自动提醒能减少遗漏 | 需要验证：用户漏跟进的频率
```

## 第6步：卖点表达

### 操作定义

把结果翻译成用户能听懂、愿意信、有画面感的卖点语言。

### 卖点表达三要素

```
1. 用户能感知——不写"智能"，写"不用你自己记"
2. 用户能验证——不写"高品质"，写"用三年不坏"
3. 用户能转述——用户能跟别人说清楚这个好在哪
```

### 卖点表达和结果的对应关系

| 结果 | 卖点表达方向 |
|------|-------------|
| 不用检查第二遍 | 让你放心出门 |
| 开会就知道干什么 | 会议结束就长出待办 |
| 下个月不用返工 | 第一次就做对 |
| 同行都觉得专业 | 拿出去别人问你用什么做的 |

### 卖点分类

| 类型 | 特征 | 适用场景 |
|------|------|----------|
| **功能型** | 能做到什么 | 新用户说服 |
| **信任型** | 证明可信赖 | 疑虑转化 |
| **效率型** | 节省多少 | 成本敏感用户 |
| **身份型** | 显得怎样 | 品牌溢价场景 |

## 证据恢复机制

当证据不足时，按以下顺序补：

```
第1优先：用户原话（直接引述，不加工）
第2优先：多条独立评论/工单/反馈（证明不是一次性情绪）
第3优先：反复出现的情境描述（证明是系统性问题）
第4优先：反复失败的动作记录（证明行为模式）
第5优先：竞品比较时的偏好表达（证明用户知道有更好方案）
第6优先：搜索词和问题词（证明用户主动寻找过）
```

### 证据缺口处理

```
[证据缺口] 类型：频率 | 现状：只有一条评论 | 填补方式：继续收集
[证据缺口] 类型：损失量化 | 现状：用户只说很麻烦 | 填补方式：结合行业均值估算
[证据缺口] 类型：行为验证 | 现状：用户说想要但没用过 | 填补方式：放进Aha时刻验证
```

**原则**：证据缺口不能通过想象填补，只能通过继续收集解决。标注清楚，比假装完整更诚实。

## 边界条件（什么时候不用/不适用）

### 硬边界（绝对不能用）

1. **没有真实痛点**：只有"我觉得可以更好"，没有具体抱怨
2. **痛点太弱**：偶发情绪，没有行动信号，用户没有真正在找解决方案
3. **痛点跟产品无关**：无法建立从痛点到产品能力的推理链
4. **强利益相关者干预**：甲方要求"必须体现XX卖点"，但用户痛点里没有这个

### 软边界（可以继续，但要额外小心）

1. **弱场景**：用户说"太慢了"，需要追问出具体场景才能继续
2. **模糊结果**：用户说"要好用"，需要翻译成具体可感知的结果
3. **多个痛点**：优先选最频繁、最痛、最能落回产品能力的那个
4. **跨角色场景**：老板说的痛点不代表用户痛点，需要验证

### 什么时候应该回退

```
场景还原不下去 → 回到第2步继续补场景，不要跳过
阻碍说不清楚 → 回到第3步继续追问，不要用"质量差"一笔带过
结果翻译不成产品能力 → 放进待验证池，不要强行套产品功能
证据太弱 → 继续收集，不硬推卖点
```

## 失败案例库

### 失败1：只写痛点，不写结果（最常见）

```
输入：用户说"每次项目一多，我就怕漏掉要跟进的事"
错误输出：
- 痛点：会后容易漏事（✓）
- 场景：项目多的时候（✓）
- 阻碍：信息分散（✓）
- 代价：遗漏跟进（✓）
- 卖点：防止漏跟进的系统（✗ 直接复述痛点，不是结果翻译）
正确输出：
- 结果：开完会就知道谁做什么、什么时候做
- 卖点：会议结束，自动长出待办清单
```

### 失败2：把愿望当卖点

```
错误输入：用户说"希望生活更好"
错误输出：让你拥有更好的生活（✗ 愿望，无法落地）
问题：结果翻译环节缺失
正确输出：
- 补场景：每天早上赶时间，单手操作
- 补结果：出门不用带超过三样东西
- 补卖点：一个背包解决所有通勤需求
```

### 失败3：跳过阻碍，直接给方案

```
输入：用户说"做报表太慢了"
错误输出：
- 痛点：报表慢
- 场景：月初
- 代价：耽误决策
- 卖点：自动生成报表（✗ 没有说清楚现在卡在哪里）
正确输出：
- 阻碍：现有系统导出的格式对不上 Excel，需要手动对齐（✗ 这个不说，方案就浮在上面）
- 结果：直接出能用来汇报的表
- 卖点：出的表直接能复制进 PPT
```

### 失败4：代价太抽象

```
错误：浪费大量时间、严重影响效率、造成不必要的麻烦
正确：
- 每月手动整理数据花8小时，够做一次完整复盘了
- 因为数据对不上，每次开会都要当场返工，所有人都看着
- 这个问题反复出现，每次都要重新理解逻辑，新人来了又要教一遍
```

### 失败5：场景太宽泛

```
错误：上班的时候、通勤路上
正确：周一早上 9 点部门会之前，需要把上周数据整理成 PPT，给 20 个人汇报
```

### 失败6：把一次反馈当长期规律

```
输入：一条评论说"界面太丑了"
错误：这个产品的用户普遍认为界面丑
问题：单条评论不能升格成长期规律
正确处理：标注[待验证：审美偏好]，继续收集其他评论或数据
```

### 失败7：把情绪当需求

```
输入：用户说"我真的受够了"
错误：用户需要情绪释放
问题：情绪是信号，不是需求本身
正确：追问"最受不的是什么？"找到底层的功能/业务/心理痛点
```

### 失败8：多个痛点混在一起

```
输入：用户同时抱怨速度慢、功能少、服务差
错误：把所有痛点放在一起写
正确：拆成独立链路，按频率×损失×产品匹配度排序，每个痛点单独跑一遍六步链
```

### 失败9：结果写成产品功能描述

```
错误结果：系统会自动提醒你
正确结果：不用再靠自己记，到了时间自动跳出来
背后的区别：系统会自动提醒（产品视角）vs 不用靠记（用户感知）
```

### 失败10：心理代价没有场景支撑

```
错误：用户会很焦虑，影响团队士气
正确：每次漏单之后都要在群里发红包道歉，久而久之不敢在群里说话
区别：前者是推断，后者是具体行为描述
```

## 交付骨架

完整输出必须包含：

```
1. 痛点清单（保留原话关键词）
2. 场景清单（五要素齐全）
3. 阻碍清单（说清楚卡在哪里）
4. 代价清单（可感知、可量化）
5. 期望结果（可感知、可验证、能落回产品）
6. 卖点表达（用户语言、有画面、可转述）
7. 证据缺口（诚实标注，不填补）
8. 待验证池（放回不去的假设）
```

## 新人校验

按顺序检查：

1. **第1关**：能不能把"痛点"翻成"结果"？——"漏水"→"不漏、放心带出门"
2. **第2关**：能不能把"阻碍"翻成"为什么现在做不到"？——"密封不好"→"普通杯盖一倒就开"
3. **第3关**：能不能把"代价"写成用户能感知的具体损失？——不是"很麻烦"，是"包里的文件都湿了"
4. **第4关**：能不能区分"功能型结果"和"身份型结果"？
5. **第5关**：能不能知道什么时候先补场景，什么时候先补原话？
6. **第6关**：能不能识别出哪些是证据缺口，而不是用假设填补？

## 绝对不能（红线）

1. **不能把痛点本身当卖点**。痛点是用户受的罪，卖点是他能得的好处。
2. **不能把愿望句当可交付结果**。"提升效率"不是结果，"每天省30分钟"才是。
3. **不能把一条抱怨直接扩写成完整策略**。单条反馈需要验证是否是规律。
4. **不能忽略场景和代价**。没有场景的痛点是情绪，没有代价的卖点是空话。
5. **不能在没有结果翻译的情况下结束**。结果翻译是连接痛点和卖点的唯一桥梁。
6. **不能用假设填补证据缺口**。诚实标注比虚假完整更有价值。
7. **不能让利益相关方意见覆盖用户真实痛点**。甲方说重要不等于用户真的痛。

## 交接判断

| 上一 Skill 产出 | 本 Skill 应该做什么 |
|----------------|-------------------|
| 只有原话/评论 | 跑完六步链，输出候选卖点 |
| 有痛点但缺场景 | 重点补第2步场景还原 |
| 有痛点有场景但不知道怎么说 | 重点走第5步结果翻译 |
| 候选卖点已足够 | 去正向卖点或结构选择 |
| 证据太弱 | 继续收集，不硬推卖点 |
