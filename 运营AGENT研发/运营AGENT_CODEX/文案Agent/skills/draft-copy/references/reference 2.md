# 初稿生成

## 适用场景

- 上游已给出卖点、用户和结构。
- 需要输出第一版可用草稿。
- 不要求一次写满，只要求先能用。

## 写作顺序

1. 结论 / 主张
2. 理由 / 解释
3. 证据 / 案例
4. 行动 / CTA

## 写作展开

### 结论 / 主张

- 先写用户一眼能懂的核心判断。
- 结论要和主卖点一致，不能先写方法论。

### 理由 / 解释

- 解释为什么这个主张成立。
- 如果用户认知低，先解释概念再解释价值。

### 证据 / 案例

- 优先放事实、数字、结果、对比。
- 没有强证据时，至少要有过程说明和边界。

### 行动 / CTA

- 只给一个主动作。
- 动作要和当前场景一致，不要同时催很多步。

## 不同长度怎么写

| 长度 | 重点 | 常见模块 |
|---|---|---|
| 短文案 | 先打中，再推动 | 标题、利益点、动作 |
| 中文案 | 先解释，再证明 | 问题、方案、证据、CTA |
| 长文案 | 先建立理解，再推进 | 场景、痛点、方案、证据、反对意见、CTA |

## 常见结构错位

- 短文案写成长说明书。
- 长文案只写口号，不给解释链。
- 该先证据的时候先下结论，导致不可信。
- CTA 太早出现，把用户劝退。

## 典型例子

### 短文案

- 标题先给最强利益点，再给一个动作。
- 不写复杂背景，不铺太多证据。

### 中文案

- 先说用户问题，再说方案和证据。
- 允许有少量解释，但不能离题。

### 长文案

- 先建立理解，再推进判断。
- 允许有场景、反对意见、证据和细节。

## 典型写法

### 1. 直接型

- 适合用户已经很接近行动。
- 先结论，再一句证据，再动作。

### 2. 解释型

- 适合用户还没完全理解。
- 先问题，再解释，再证据，再动作。

### 3. 说服型

- 适合用户需要一点思考时间。
- 先场景，再痛点，再方案，再证明。

## 交付样式

- 先给正文。
- 再给模块说明。
- 再给可删改段。
- 再给待确认信息。
- 如果缺口太多，先标缺口，不要伪装完整。

## 一条完整样例

### 输入

- 用户：项目负责人。
- 卖点：会议结束自动生成待办，减少漏项。
- 场景：产品详情页，中等阅读时间。

### 处理

1. 结论：这款工具能让项目会后的跟进更确定。
2. 理由：因为它会自动把会议内容变成待办。
3. 证据：支持录音转写、待办提取、多人协作、提醒同步。
4. 行动：去试用或看演示。

### 输出

- 初稿正文：会议开完，别再靠记忆。自动整理待办，让每个任务都有着落。
- 模块说明：前半句抓结果，后半句说功能支撑。
- 可删改段：多人协作、提醒同步、录音转写。
- 待确认信息：是否有真实案例、具体节省时长。

## 回退提示

- 如果卖点不够清楚，先回卖点 Skill。
- 如果场景不够清楚，先回场景 Skill。
- 如果结构不够清楚，先回结构 Skill。

## 失败案例

### 案例1：把缺口硬补成正文

- 原文：先用上，再说。
- 失败点：没有说清用户是谁、卖点是什么、为什么可信。

### 案例2：正文太平均

- 原文：功能、优势、案例、行动平均分配。
- 失败点：没有主线，用户抓不住重点。

### 案例3：CTA 太多

- 原文：先关注、再私信、再填表、再预约。
- 失败点：动作太多，转化链断掉。

## 边界条件

- 如果输入已经很完整，初稿可以更直接，不必额外解释太多。
- 如果输入不完整，初稿只负责把主线写出来，不负责补齐所有答案。
- 如果是短文案，初稿应该更像标题和动作组合，不必铺太多段落。

## 交接判断

- 卖点未定：先回卖点 Skill。
- 场景未定：先回场景 Skill。
- 结构未定：先回结构 Skill。

## 绝对不能

- 不能在卖点未定时写成稿。
- 不能在场景未定时乱写长度和 CTA。
- 不能把多种目标混成一个正文。
- 不能把没确认的信息伪装成已确认。
- 不能用口号代替正文。

## 生成原则

- 先能用，再优化。
- 一段只解决一个问题。
- 证据优先于形容词。
- 缺口要显式标注，不要偷补。

## 初稿检查点

1. 有没有明确主张。
2. 有没有把卖点写成用户能感知的话。
3. 有没有把证据放在该出现的位置。
4. 有没有只有一个主动作。
5. 有没有把没确认的信息单独列出来。

## 输出骨架

- 初稿正文
- 模块说明
- 可删改段
- 待确认信息

## 初稿生成补强

## 先确认四件事

1. 主写对象是否已定。
2. 主卖点是否已定。
3. 场景和长度是否已定。
4. 结构是否已定。

## 段落级写法

### 开头

- 只放一个主判断。
- 先让用户知道这是什么、为什么和我有关。

### 中段

- 每段只解决一个问题。
- 先理由，后证据，最后才是解释。

### 结尾

- 只保留一个主动作。
- 动作必须和当前场景一致。

## 平台约束

### 小红书

- 用场景和体验开头。
- 少抽象，多细节和对比。
- 标题和首段要先给感知点。

### 抖音

- 前 3 秒必须有钩子。
- 先冲突，再结果，再动作。

### 公众号

- 允许先解释，再推进。
- 但必须有明确主线，不能散。

### 详情页

- 首屏先判断，中段再补证据。
- 不要首屏讲太多背景。

## 初稿检查

- 有没有主张。
- 有没有把卖点写成人能感知的话。
- 有没有证据位置。
- 有没有只留一个动作。
- 有没有把没确认的信息单独列出。

## 常见错误

- 开头太长。
- 中段平均分配，导致没有主线。
- CTA 太多。
- 先写修辞，后写事实。
- 缺口被偷偷补成正文。



## 写作停止条件

- 如果用户未定，先回用户分层。
- 如果卖点未定，先回卖点提炼。
- 如果场景未定，先回场景判断。
- 如果结构未定，先回结构选择。
- 这 4 项没定，就不要硬写成稿。

## 证据展开顺序

1. 先写用户能感知的结果。
2. 再写理由。
3. 再写证据。
4. 最后给动作。

## 平台写法

### 小红书

- 开头先给场景和感受。
- 中段补对比和细节。
- 结尾动作要轻，不要像硬广。

### 抖音

- 前 3 秒先钩子。
- 先冲突，再结果，再动作。

### 公众号

- 允许先解释。
- 但必须在前半段把主线说清楚。

### 详情页

- 首屏先判断。
- 中段再补证据和 FAQ。
- 结尾再给动作和风险说明。

## 验收门禁

- 如果缺少关键输入，先回退到上一 Skill。
- 如果证据不足，不要伪装完整。
- 如果输出已经能直接进入下一步，就只给下一步，不额外加戏。
