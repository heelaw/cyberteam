# 结构选择

## 适用场景

- 卖点和用户已经清楚，但结构未定。
- 需要在多种结构中做选择，而不是机械套固定结构。

## 结构库

| 结构 | 适合什么 | 不适合什么 |
|---|---|---|
| AIDA | 先抓注意，再层层推进 | 用户已经很热、只差一句话 |
| 4P | 先问题，再放大，再承诺，再证明 | 没有清楚痛点时 |
| PAS | 痛点明确、需要快速共鸣 | 需要大量证据的长决策 |
| 直述式 | 短场景、强明确诉求 | 复杂价值、需要解释时 |
| 对比式 | 用户在比较方案 | 没有竞品或替代方案时 |
| 故事式 | 需要建立理解和认同 | 极短触点 |
| 清单式 | 需要快速扫描和执行 | 需要强情绪推动时 |

## 结构判别表

| 如果输入像这样 | 优先结构 | 关键区分点 | 不要误判成 |
|---|---|---|---|
| 用户还没意识到问题，只能先把问题讲出来 | 4P / 故事式 | 先搭场景和问题框架，再进说服 | PAS |
| 用户已经知道问题，但还不够被打动 | PAS / AIDA | 是否有明确痛点入口、是否需要钩子 | 直述式 |
| 用户已经在比较方案 | 对比式 | 是否能明确比较维度 | 4P / AIDA |
| 用户已经接近行动，只差一句推动 | 直述式 | 是否可以直接给结果和动作 | 故事式 |
| 信息点很多，用户只会扫读 | 清单式 | 是否每条都能独立成立 | 对比式 |
| 需要先建立认同，再慢慢进入判断 | 故事式 | 是否有真实变化过程 | 直述式 |

## 每种结构怎么用

### AIDA

- 先用注意力入口抓住用户。
- 再把兴趣和关联理由说出来。
- 再把用户想要的结果说清楚。
- 最后给动作，不要把 CTA 写得太软。

### 4P

- 先把用户问题讲出来。
- 再放大问题的代价或不舒服。
- 再给出明确承诺。
- 最后补证据，避免空承诺。

### PAS

- 适合痛点明显、共鸣要快的场景。
- 先痛点，后放大，再给方案。
- 如果用户还没意识到问题，先别用 PAS 硬推。

### 直述式

- 适合用户已经知道要什么。
- 直接说结果、好处和动作。
- 不适合长解释和复杂信任建立。

### 对比式

- 先列出用户正在比较的方案。
- 再说本方案在哪些维度更好。
- 一定要说明比较维度，不然容易变成泛泛攻击竞品。

### 故事式

- 先搭场景，再推进人物变化。
- 适合建立认同和理解。
- 故事必须服务卖点，不能为了讲故事而讲故事。

### 清单式

- 把信息拆成可扫读模块。
- 适合任务导向、信息较多但用户只想快速看懂的场景。
- 每一条都要可独立理解。

## 选择规则

1. 先看用户是否知道问题。
2. 再看用户是否需要解释或证据。
3. 再看阅读时长。
4. 再看是短触点还是长决策。
5. 最后确定主结构和备用结构。

## 决策树

1. 如果用户还不知道问题，优先用 `4P` 或 `故事式`。
2. 如果用户知道问题但还没被打动，优先用 `PAS` 或 `AIDA`。
3. 如果用户已经在比较方案，优先用 `对比式`。
4. 如果用户已经接近行动，优先用 `直述式`。
5. 如果信息很多但用户只会扫读，优先用 `清单式`。

## 结构内部怎么排

### AIDA

- 注意：抓住眼球。
- 兴趣：把和用户有关的点说出来。
- 欲望：把结果和收益说出来。
- 行动：只给一个动作。

### 4P

- 描绘：先把场景立住。
- 承诺：再说你能给什么。
- 证明：再给证据。
- 敦促：最后推进动作。

### PAS

- 痛点：先把问题摆出来。
- 放大：再说不解决的代价。
- 解决：最后给方案。

## 选型信号

- 需要认知重构时，结构必须先有解释链。
- 需要快速转化时，结构必须先有动作链。
- 需要建立信任时，结构必须先有证明链。

## 案例式判断

- 短促销页：通常选 `直述式` 或 `AIDA`。
- 产品介绍页：通常选 `4P`。
- 竞品比较页：通常选 `对比式`。
- 教育型长文：通常选 `故事式` 或 `4P`。

## 结构怎么落地

### AIDA 适合什么样的输入

- 用户已经被某个钩子吸住。
- 你手里有一个足够强的利益点。
- 你想把用户从看见推进到行动。

### 4P 适合什么样的输入

- 用户还没有完全相信你。
- 你需要先描述问题和场景，再讲承诺。
- 你手里有证据可以支撑承诺。

### PAS 适合什么样的输入

- 用户对问题本身有感知。
- 你要快速建立“我懂你”的感觉。
- 你不需要先讲很多背景。

### 对比式适合什么样的输入

- 用户已经有替代方案。
- 用户会问“为什么不是别的”。
- 你能清楚说出比较维度。

### 故事式适合什么样的输入

- 你需要带着用户慢慢理解。
- 你需要让用户先进入场景。
- 你有真实事件或变化过程可以讲。

### 清单式适合什么样的输入

- 信息点很多，但用户时间少。
- 你想让用户快速扫完并做动作。
- 每一条都能单独成立。

## 结构写法检查

1. 结构顺序有没有写反。
2. 每个模块有没有实际内容，不是只写标题。
3. 开头是不是和场景一致。
4. 结尾是不是只有一个动作。
5. 有没有结构和卖点不匹配的段落。

## 常见错误

- 先选喜欢的结构，再硬套内容。
- 把 AIDA 和 4P 混成一锅，但顺序不统一。
- 该解释时只写口号。
- 该推动时只写背景。

## 典型例子

### 例子1：AIDA

- 适合：海报标题、首屏钩子、短促销页。
- 顺序：注意 -> 兴趣 -> 欲望 -> 行动。

### 例子2：4P

- 适合：产品介绍页、销售页、需要理性说服的页面。
- 顺序：描绘 -> 承诺 -> 证明 -> 敦促。

### 例子3：PAS

- 适合：用户已经知道痛点的场景。
- 顺序：痛点 -> 放大 -> 解决。

### 例子4：对比式

- 适合：用户在比较替代方案的时候。
- 顺序：比较维度 -> 你更好的地方 -> 为什么更好。

### 例子5：清单式

- 适合：信息多但用户只想快速看懂。
- 顺序：条目化 -> 快速扫读 -> 一个动作。

## 失败案例

### 案例1：先有喜好再套结构

- 原文：我就想用 AIDA。
- 失败点：没有根据用户状态和场景判断。

### 案例2：结构和任务不匹配

- 原文：用户还不知道问题，却直接用 PAS。
- 失败点：用户没有痛点认知，先打痛点会失真。

### 案例3：结构顺序写反

- 原文：先证明，再讲问题，再讲承诺。
- 失败点：用户还没理解为什么要听，直接证明没有入口。

## 边界条件

- 如果场景非常短，优先选能最快传达一个动作的结构。
- 如果用户特别需要信任，优先选能先建立理解再推进的结构。
- 如果一个结构能用但不够强，可以保留备用结构，不要硬押单一方案。

## 交接判断

- 用户状态不清：先回 `detect-need-type` 或 `segment-users`。
- 证据不够：先回 `research-sellpoint-*`。
- 只缺结构：直接定主结构和备用结构。

## 绝对不能

- 不能先喜欢某个结构再硬套。
- 不能结构顺序写反。
- 不能用需要解释的任务硬上直述式。
- 不能用户还没认知问题就硬上 PAS。
- 不能只报结构名，不写为什么。

## 交付样式

- 先给主结构。
- 再给备用结构。
- 再给结构内部顺序。
- 最后给不用其他结构的原因。

## 一条完整样例

### 输入

- 用户：已经知道问题，但在比较几个协作工具。
- 场景：详情页，停留时间 1-3 分钟。
- 目标：让用户理解为什么这款更适合团队协作。

### 判断

- 主结构：对比式。
- 备用结构：4P。
- 原因：用户正在比较方案，需要明确比较维度和差异。

### 写法

- 先列比较维度：效率、接入成本、跟进确定性。
- 再说本方案更好的地方：自动生成待办、减少漏项、多人可见。
- 再给证据：会议后整理时间减少。
- 最后给动作：去看演示或试用。

### 输出

- 结构名称：对比式。
- 选择理由：用户在比较，且有明确替代方案。
- 写作顺序：比较维度 -> 优势 -> 证据 -> 动作。
- 备用结构：4P。
- 为什么不选其他结构：AIDA 太偏钩子，PAS 偏痛点，直述式证据不够稳。

## 回退提示

- 如果用户还没进入比较阶段，不要硬上对比式。
- 如果比较维度不清，先回卖点或用户分层。

## 输出要求再补一层

- 要能看出主结构和备用结构的差异。
- 要能看出为什么这个结构更适合当前输入。
- 要能看出每一段的职责，而不是只有一堆文案词。

## 常见误判

- 用户还没认知问题，却直接上 AIDA。
- 用户需要证据，却只写情绪和口号。
- 场景很短，却硬塞长结构。
- 结构名选对了，但写作顺序写反了。

## 输出要求

- 不只给结构名，要给选择原因。
- 要写明结构内部的模块顺序。
- 要写清备用结构和不用它的原因。

## 输出骨架

- 结构名称
- 选择理由
- 写作顺序
- 备用结构
- 为什么不选其他结构

## 结构选择补强

## 先问四个问题

1. 用户知不知道问题。
2. 用户是否需要解释。
3. 用户是否已经在比较。
4. 用户是否已经接近行动。

## 结构误配信号

- 用户不知道问题，却硬上 PAS。
- 用户已经接近行动，却写成长篇故事。
- 用户在比较，却不给比较维度。
- 用户只想快速看懂，却用了太重的解释链。

## 各结构的第一动作

### AIDA

- 先抓注意，再给关联，再给想要的结果，再给动作。
- 适合有钩子、有转化目标的短中触点。

### 4P

- 先立场景，再讲承诺，再给证明，再推进。
- 适合需要解释和证明的页面。

### PAS

- 先痛点，再放大，再解决。
- 适合用户已经感知到问题。

### 直述式

- 先结论，再理由，再动作。
- 适合强明确、低解释成本的场景。

### 对比式

- 先列比较维度，再说差异，再给理由。
- 适合用户已经在找替代方案。

### 故事式

- 先场景，再变化，再结果。
- 适合建立理解和认同。

### 清单式

- 先拆条目，再按优先级输出。
- 适合信息较多、用户偏扫读的场景。

## 选择顺序

1. 先定用户状态。
2. 再定说服目标。
3. 再定证据深度。
4. 再定主结构。
5. 最后留一个备用结构。

## 结构内部检查

- 开头是否和用户状态一致。
- 中间是否真的在推进，而不是重复背景。
- 结尾是否只保留一个主动作。
- 证据是否出现在该出现的位置。

## 备用原则

- 主结构负责当前最强目标。
- 备用结构负责当前最可能失配的风险。
- 不要为了“更全面”把两个结构完整混写。



## 判别表

| 结构 | 何时赢 | 何时输 |
|---|---|---|
| AIDA | 有钩子，想快速推进 | 用户还不理解问题 |
| 4P | 需要解释、证明、推进 | 没有证据时 |
| PAS | 用户已经感到痛点 | 用户没认知问题时 |
| 直述式 | 接近行动、低解释成本 | 需要长解释时 |
| 对比式 | 用户在比较方案 | 没有明确比较维度时 |
| 故事式 | 先建立理解和认同 | 极短触点 |
| 清单式 | 信息多、用户只扫读 | 需要强情绪推动时 |

## 不要用的情况

- 用户还没认知问题，不要硬上 PAS。
- 没有比较对象，不要硬上对比式。
- 需要证据但没有证据，不要硬上直述式。
- 只有一句口号，不要假装 AIDA 完整。

## 同输入不同目标

### 同样是协作工具

- 目标是唤起兴趣：AIDA。
- 目标是解释为什么有效：4P。
- 目标是比较竞品：对比式。
- 目标是快速促成试用：直述式。

### 同样是痛点明确

- 目标是让用户觉得“你懂我”：PAS。
- 目标是先讲场景再讲方案：4P。

## 结构选择检查

- 先判断目标。
- 再判断用户状态。
- 再判断证据深度。
- 最后才报结构名。

## 验收门禁

- 如果缺少关键输入，先回退到上一 Skill。
- 如果证据不足，不要伪装完整。
- 如果输出已经能直接进入下一步，就只给下一步，不额外加戏。
