# 结构选择器

## 作用

在结构之间做判断，不直接写初稿。

这个 Skill 负责把用户状态、证据深度、阅读窗口和说服目标翻译成可执行的结构选择结果。

## 启动前

- 先有用户状态和场景判断，再选结构。
- 如果卖点或证据还不清，先回上游。

## 判断顺序

判断顺序不是写作顺序，是决策顺序。必须按这个流程走完才能确定结构，不能跳步。

1. **Q1：用户是否知道自己的问题？**
   - 不知道 → 优先 4P 或故事式。先搭问题框架，再进说服。
   - 知道但没被说服 → 进入 Q2。
   - 已经接近行动 → 进入 Q3。

2. **Q2：用户是在比较方案，还是在考虑是否要解决？**
   - 在比较方案 → 优先对比式。必须有比较维度和本方案优势。
   - 在考虑要不要解决 → 优先 PAS 或 AIDA。先痛点共鸣，再给方案。

3. **Q3：用户已经接近行动了吗？**
   - 是 → 优先直述式。直接给结论、结果、动作。不铺背景。
   - 否 → 回到 Q2 或 Q1 重新判断。

4. **Q4：内容信息量很大，用户只会扫读？**
   - 是 → 优先清单式。每条独立可理解，最后一个动作收口。

> **快速决策树**
> ```
> 用户知道问题？
>   ├─ 否 → 4P 或 故事式
>   └─ 是 → 用户在比较方案？
>             ├─ 是 → 对比式
>             └─ 否 → 用户接近行动？
>                       ├─ 是 → 直述式
>                       └─ 否 → AIDA 或 PAS
> ```
> 遇到边界条件（内容多+用户扫读）：清单式。

## 结构库

| 结构 | 核心逻辑 | 入口条件 | 禁忌 |
|---|---|---|---|
| AIDA | 注意→兴趣→欲望→动作，层层递进 | 用户已有注意但未被吸引 | 用户已经很热时用AIDA显得绕 |
| 4P | 描绘→承诺→证明→敦促，问题框架驱动 | 用户不知道问题 | 没有清楚痛点时会显得假 |
| PAS | 痛点→放大→解决，快速共鸣 | 用户知道问题但没被说服 | 没有痛点时用PAS显得制造焦虑 |
| 直述式 | 结论→理由→动作，直接给判断 | 用户已接近行动或内容极短 | 用户还在认知早期时会显得急 |
| 对比式 | 维度→本方案优势→为什么重要 | 用户在比较多个方案 | 没有竞品参照时显得没对比 |
| 故事式 | 场景→变化→结果，建立认同 | 用户需要理解和共情 | 极短触点和已知信息不适用 |
| 清单式 | 信息模块→独立可读→一个动作收口 | 信息多+用户扫读 | 需要强说服/情绪推动时不适用 |

## SOP（执行步骤）

1. **输入检查**：用户状态 + 场景 + 证据深度 + 阅读窗口，四项检查是否完整。缺任何一项，先回上游。
2. **走决策树**：按Q1→Q2→Q3→Q4顺序走完快速决策树，记录每个节点的选择和理由。
3. **选主结构**：根据决策树结果确定主结构。记录结构名称和选择理由（必须具体到用户状态，不是"觉得AIDA更好"）。
4. **选备用结构**：给出当主结构不适用时的降级方案。常见降级：长结构→短结构，多信息→少信息。
5. **写排除理由**：至少排除2种结构，每种给出具体排除理由（不能只说"不适合"）。
6. **给写作顺序**：用三句话描述主结构和备用结构各自怎么写。
7. **平台适配**：检查平台长度是否支撑所选结构，如果平台很短，标注需要压缩哪些部分。

## 交付骨架（每项填写规范）

- **结构名称**：直接写结构名，如"主结构：AIDA；备用结构：直述式"。
- **选择理由**：必须包含"用户状态 + 场景 + 证据深度"三角，不能只说"因为AIDA很经典"。
  - 正确例子："用户知道问题但未被说服，场景是朋友圈广告（短触点），证据有数字但不够完整 → 选PAS，用痛点共鸣弥补证据缺口；备用直述式用于极短版本。"
  - 错误例子："因为这个更适合"。
- **写作顺序**：分主结构和备用结构各自给出三步以内的写作顺序。
  - 主结构AIDA写法：① 前3秒抓注意（数字/场景/冲突钩子）；② 30秒内推进到欲望（结果承诺）；③ 最后给一个动作。
- **备用结构**：必须给出具体切换条件，注明"当XXX时切换到YYY"。
  - 例子："当朋友圈版本压缩到15秒时，切换到直述式（结论→动作两段）；当发现用户在评论区比较方案时，切换到对比式"。
- **排除理由**：每种被排除的结构都要写明原因，格式："不选XXX，因为YYY（具体原因）"。
  - 正确例子："不选4P，因为用户已经知道问题，不需要再搭问题框架；不选故事式，因为触点只有30秒，没有叙事空间"。
- **平台适配提示**：根据所选平台，标注哪些部分要压缩、哪些要前置、入口是什么。
  - 例子："抖音：钩子必须前置3秒内；小红书：场景必须在前30字；海报：只保留主张+动作"。
- **证据深度提示**：根据证据现状，标注证据够不够、缺什么、要不要标注为待确认。
  - 例子："现有证据有数字但无案例，证据深度中等，建议把数字作为主要证据，案例标注待补充"。

## 新人校验

- 新人能不能先判断用户状态，再选结构。
- 新人能不能给出主结构和备用结构。
- 新人能不能说明为什么不选其他结构。
- 新人能不能知道平台很短时要先压入口。

## 绝对不能

- 不能把结构当固定模板套死。
- 不能只报结构名不报理由。
- 不能把主结构和备用结构混写成一团。
- 不能忽略需求类型和阅读时长。
- 不能让结构选择和平台预设冲突。
