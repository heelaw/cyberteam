# 自检清单

## 适用场景

- 文案已经写完，准备交付前最后检查。
- 需要避免漏掉用户、卖点、场景、结构或动作。
- 需要把风险显式标出来。
- 需要判断这份内容能不能直接交给新人继续处理。

## 七问

1. 写给谁？
2. 说了什么卖点？
3. 场景对不对？
4. 结构对不对？
5. 有没有明确动作？
6. 证据够不够？
7. 有没有夸大或硬承诺？

## 七问怎么查

### 1. 写给谁

- 是否已经选出主写对象。
- 是否混了多个互相冲突的人群。
- 是否把辅助对象误写成主对象。

### 2. 说了什么卖点

- 是否说的是用户能感知的结果，而不是功能名词。
- 是否把主卖点和辅助卖点混在一起。
- 是否把口号当卖点。

### 3. 场景对不对

- 长度、介质、阅读状态是否匹配。
- 是否在短场景里塞了长解释。
- 是否忽略平台预设。

### 4. 结构对不对

- 是否符合当前说服目标。
- 是否把需要证据的内容写成了纯情绪。
- 是否把结构顺序写反。

### 5. 有没有明确动作

- 是否知道用户下一步要干什么。
- CTA 是否具体、可执行、和场景匹配。
- 是否同时催了多个动作。

### 6. 证据够不够

- 是否有事实、案例、数据、过程或边界说明。
- 是否用空话代替证据。
- 是否把推断包装成事实。

### 7. 有没有夸大或硬承诺

- 是否写了无法保证的结果。
- 是否把条件、限制和适用范围说清楚。
- 是否出现绝对化表达。

## 通过规则（7/7 Gate）

- 主写对象清楚。
- 主卖点清楚。
- 场景清楚。
- 结构完整。
- 动作明确。
- 证据足够。
- 风险被标出。
- **7 项全部满足才可通过。**

## 评分法

- 每问 1 分，共 7 分。
- **7 分：直接通过。**
- 6 分及以下：回退补齐。

## 不通过时怎么改

- 用户不清：回 `segment-users`。
- 卖点不清：回 `research-sellpoint-forward` 或 `research-sellpoint-backward`。
- 场景不清：回 `judge-scenario`。
- 结构不清：回 `select-structure`。
- 文字太虚：回 `edit-copy`。
- 还不闭环：继续补证据，不要直接交付。

## 回退优先级

1. 对象不清先修对象。
2. 卖点不清先修卖点。
3. 场景不清先修场景。
4. 结构不清先修结构。
5. 动作不清先修动作。
6. 证据不清先补证据。
7. 风险不清先补风险。

## 输出格式要求

- 先给判断。
- 再给风险点。
- 再给修改建议。
- 最后给是否通过。

## 常见漏项

- 只检查总感觉，没有逐项核对。
- 只看文案好不好看，没有看证据和风险。
- 忘记检查 CTA 是否和场景匹配。
- 忘记检查是否夸大承诺。
- 忘记检查是否把辅助对象写成主对象。

## 典型例子

### 通过

- 主写对象清楚。
- 卖点是用户能感知的结果。
- 场景、结构、CTA 和证据都匹配。

### 不通过

- 用户写成“所有人”。
- 卖点只有功能名，没有结果。
- 场景是短触点，却写成长解释。
- CTA 一口气催三步。

## 失败案例

### 案例1：对象不清

- 原文：适合所有想提升效率的人。
- 问题：没有主写对象，无法判断切入口。

### 案例2：卖点不清

- 原文：这是一套很专业的方法。
- 问题：只有评价，没有用户感知结果。

### 案例3：场景不清

- 原文：适合各种场景使用。
- 问题：没有长度、介质、状态信息。

### 案例4：CTA 过多

- 原文：先看介绍，再加群，再填表，再预约。
- 问题：动作太多，用户不知道先做什么。

## 边界案例

- 如果文案是品牌认知型，卖点可以更轻，但对象和场景仍要明确。
- 如果文案是强转化型，证据和动作必须更强。
- 如果文案是长篇教育型，结构和解释必须更完整。

## 交付样式

- 先给是否通过。
- 再给失败原因或风险点。
- 再给回退路径。
- 再给下一步修改建议。

## 绝对不能

- 不能只给总评不逐项核对。
- 不能把风险藏起来。
- 不能把不能证明的东西判成通过。
- 不能忽略 CTA 和场景匹配。
- 不能把“感觉不错”当成通过标准。
- 不能把辅助对象误判为主对象。
