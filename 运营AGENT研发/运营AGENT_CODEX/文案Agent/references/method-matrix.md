# 文案方法矩阵

## 这份文件只保留不变量

细节放在各个 Skill 里，这里只写跨 Skill 的统一判断。

## 总链路

需求识别 -> 用户分层 -> 卖点提炼 -> 场景判断 / 说服模式 -> 结构选择 -> 初稿 -> 删改 -> 自检 -> 发布前验证

## 不变量

| 问题 | 只看这里 |
|---|---|
| 现在该不该写 | 先做需求识别 |
| 写给谁 | 先做用户分层 |
| 说什么 | 先做卖点提炼 |
| 怎么说 | 先做场景判断和说服模式 |
| 按什么顺序说 | 先做结构选择 |
| 先产出什么 | 先产出可用初稿 |
| 怎么变干净 | 再做删改 |
| 能不能交付 | 最后做自检与验证 |

## 核心规则

- 需求识别：必须先采证据，再分类。
- 用户分层：必须先找决策链，再找角色标签。
- 卖点提炼：有资料走正向，没资料走反向，不能硬编结论。
- 场景判断：平台和阅读窗口决定长度；说服模式决定入口。
- 结构选择：结构是顺序，不是模板名字。
- 初稿：一段只解决一个问题，只保留一个主动作。
- 删改：只改表达，不改事实，不新增卖点。
- 自检：按门禁做 pass / fail，不靠感觉。
- 发布前验证：高风险内容必须再测一次。

## 说服模式

- 不假思索：短触点、注意力短、动作先行。
- 细细琢磨：中长触点、解释和证据优先。
- 判断点放在场景后面，不要提前套死。

## 极端情况

- 高风险 -> 先边界后主张。
- 高争议 -> 先证据后动作。
- 高情绪 -> 先共鸣后解释。
- 高时效 -> 先结论后证据。

## 陌生概念处理

- 内部视角：换成用户会说的话。
- 专家视角：先解释再表达。
- 个人视角：别用圈内简称偷懒。

## 感知增强

- 让用户看到画面、比较和变化，而不是只有抽象结论。
- 句式短长交错，但不能为了花而丢掉主线。
- 真实细节优先于空泛修辞。
