# 运营AGENT_CODEX 记忆

## 当前结论

文案 Agent 不能只做目录收敛，必须做到“新人拿着 SOP 就能独立执行、判断、回退和验收”。

当前这套 `文案Agent` 已经具备可用的知识层，但还没有达到“自动化编排层可直接依赖”的程度。核心原因不是结构不清，而是执行层与验收层还不够硬。

## 这次暴露的问题

1. 早期版本只有骨架，没有足够的判断密度。
2. 每个 Skill 只写了“做什么”，没有写透“怎么做、何时停、何时回退、何为错误”。
3. 缺少完整样例，导致新人不知道好稿长什么样、坏稿长什么样。
4. 缺少失败案例和边界条件，导致遇到模糊输入时容易误判。
5. 缺少绝对不能做的红线，导致输出标准不够硬。
6. scripts 目前仍是占位层，不能承担真实自动化逻辑。
7. 部分 Skill 还缺平台预设、量化验收和超时兜底。

## 根因

第一性原理上，文案工作不是“把文字写出来”，而是“在信息不完整时完成判断，再把判断转成可交付文本”。

因此，一个可上岗的文案 Agent 必须同时具备：
- 输入分辨能力
- 证据采集能力
- 卖点收敛能力
- 场景判断能力
- 结构选择能力
- 初稿生成能力
- 删改优化能力
- 交付自检能力

如果这些能力只写成标题，没有写成具体 SOP，批量化时就会复制出一堆“看起来有结构、实际上不能执行”的 Skill。

更深一层的根因是：我前期把“知识完整”当成了“可上岗”，但这两者不是一回事。知识完整只说明知道什么是对的，可上岗还要求知道怎样在不完整输入、冲突输入、模糊输入下做判断。

## 以后必须遵守的标准

1. 每个 Skill 必须能独立完成一个最小 SOP 单元。
2. 每个 Skill 必须有：作用、SOP、输出、参考、考核。
3. 每个关键 Skill 必须补齐：典型例子、失败案例、边界条件、交接判断、绝对不能。
4. 每个 Agent 必须明确：岗位职责、适用范围、路由逻辑、回退规则。
5. 任何看起来像“总结”的内容，都要检查是否还能指导新人真实执行。
6. 如果 Skill 需要自动化，就必须有真实脚本；如果只是提示词型，就明确声明，不写空脚本。
7. 需要批量化时，必须先做单个 Skill 的完整演练，再复制到其他 Skill。

## 评估暴露的优先修正点

1. 脚本层要么补真实逻辑，要么明确降级为纯提示词型。
2. 根 Agent 需要补超时和兜底思路，避免模糊输入来回循环。
3. 场景层需要加入平台级预设，不能只停留在短/中/长抽象层。
4. assessment 要逐步从定性检查变成可打分、可门禁的验收标准。
5. 如果后续要接入 CyberTeam 调度，需要补 Agent 角色定义文件，而不是只保留 Skill。

## 递进式排错问题

1. 这个任务的输入到底是什么，是否已经足够清楚？
2. 现在缺的是事实、判断，还是表达？
3. 当前最该回退到哪个前置 Skill？
4. 这个 Skill 的输出，是否能直接被下游使用？
5. 如果给一个新人，他能不能靠这份 SOP 自己做对？
6. 这份内容里，什么是必须保留的判断，什么是可以删掉的修辞？
7. 是否已经写清“什么是好”“什么是绝对不能” ？

## 批量化方法

1. 先确定一个 Agent 的职责边界。
2. 再把这个 Agent 拆成最小 SOP 单元。
3. 每个单元补齐：输入样例、判断规则、反例、边界、回退、输出模板。
4. 再做一轮新人视角自检：是否能独立执行，是否能判错，是否能接到下一步。
5. 最后才批量复制到其他 Agent。

## 批量化前的提问顺序

1. 这个单元到底解决哪一个最小问题？
2. 输入是否足够，还是还需要上游补证据？
3. 输出能不能直接给下游用？
4. 遇到不完整输入时，回退到哪一层？
5. 有哪些情况绝对不能做？
6. 有没有完整样例证明它真的能跑？
7. 新人照着做，是否能分辨好坏？

## 当前工作方式

- 不再只做摘要式收敛。
- 改为按“输入 -> 判断 -> 输出 -> 反例 -> 边界 -> 回退”逐层加厚。
- 优先补最薄、最容易误判、最影响下游的 Skill。

## 经验教训

- 不能把“目录齐了”误判成“能力齐了”。
- 不能把“写了规则”误判成“能执行”。
- 不能把“补了案例”误判成“已可批量化”。
- 必须持续做新人视角自检：我拿到这份东西，能不能独立做对、做完、做稳。

## 本次完整复盘沉淀

### 1. 这次工作的真正目标

不是把一个 Agent“写出来”，而是把一个可上岗的文案数字员工做成能直接交付、能直接复核、能直接扩散的包。

判断标准只有一个：新人拿到后，能不能按文档把事情做完，且遇到不完整输入、冲突输入、边界输入时知道怎么回退、怎么补证据、怎么拒绝硬写。

### 2. 全流程 SOP 总表

#### 2.1 原始材料研读

目标：先读全局，再读局部，先提方法、流程、判断逻辑，再提案例和红线。

执行要点：
- 先做目录扫描，不要一上来就写。
- 每个章节至少抽出：输入、步骤、判断、案例、反例、红线。
- 图片、表格、口述里的隐含知识也要提取，不准只看正文标题。

QA：
- 是否覆盖了所有大章节。
- 是否把“方法论 / 流程 / 判断逻辑 / 案例 / 红线”分开记录。
- 是否存在只看标题不看正文的遗漏。

回退：
- 如果提炼不出动作和判断，就继续回原资料补读。

#### 2.2 知识提炼

目标：把“知道什么”变成“怎么做”。

执行要点：
- 所有知识点必须写成“看什么、怎么采、怎么判、怎么回退”。
- 不能只保留概念定义，必须保留决策动作。
- 每个判断点都要能让新人直接照做。

QA：
- 新人能不能照着做。
- 是否存在只有结论没有动作的内容。

回退：
- 只要讲不清如何执行，就说明提炼不够，要继续拆。

#### 2.3 最小单元拆分

目标：一个 Skill 只负责一个最小 SOP 单元。

执行要点：
- 以“输入状态”来拆，而不是以章节标题来拆。
- 需求、用户、卖点、场景、结构、初稿、删改、自检必须分开。
- 如果一个单元同时承担多个目标，就继续切。

QA：
- 这个单元是不是最小。
- 它是不是只服务一个决策点。
- 它和别的 Skill 是否重叠。

回退：
- 只要出现职责混叠，就继续切到更小颗粒度。

#### 2.4 Skill 编写

目标：让 Skill 既能被 AI 执行，也能被新人阅读后执行。

执行要点：
- 每个 Skill 固定写：作用、SOP、输出、参考、考核、红线。
- SKILL.md 保持精炼，只保留核心流程和边界。
- 细节下沉到 `references/reference.md` 和 `assessments/assessment.md`。
- 不能把大段教材直接塞进 SKILL.md。

QA：
- 是否只做一件事。
- 是否有明确输入和输出。
- 是否能和上下游接上。

回退：
- 如果 SKILL.md 开始变成讲义，就说明层级错了，要把内容下沉。

#### 2.5 Reference 展开

目标：把一件事讲透，既能指导执行，也能指导判断。

执行要点：
- 先写采什么，再写怎么采，再写怎么判。
- 每个判断点都要补例子、失败案例、边界条件。
- Reference 必须遵循渐进式披露，先给骨架，再给细节，再给案例。

QA：
- 是否讲清楚“看什么”。
- 是否讲清楚“怎么采”。
- 是否讲清楚“怎么判”。
- 是否讲清楚“什么是好、什么是坏、什么绝对不能”。

回退：
- 如果只有规则没有边界和反例，Reference 仍然太薄。

#### 2.6 Assessment 门禁

目标：把“差不多”改成“可判定”。

执行要点：
- 每个 Skill 都要能 pass / fail。
- 必须有明确门槛，不能只写“建议”。
- 高风险错误必须直接拦截，不能先放行再修。

QA：
- 是否能一眼判定通过或不通过。
- 是否有量化门槛。
- 是否能逼迫补缺，而不是鼓励糊弄。

回退：
- 只要门槛模糊，就继续补成可勾选条件。

#### 2.7 Script / 诊断

目标：脚本要么真有逻辑，要么明确声明为占位，不允许假装自动化。

执行要点：
- 如果脚本只是回显输入，就不能伪装成能力。
- 如果脚本要承担门禁，就必须输出诊断信息、缺口、红线、下一步。
- 空输入、脏输入、正常输入都要测。

QA：
- 是否真的执行了逻辑。
- 是否对坏输入有提示。
- 是否对缺口有结构化回传。

回退：
- 如果脚本不具备实际价值，就明确降级为纯提示词型，不装自动化。

#### 2.8 根 Agent 整合

目标：根 Agent 只做判断和分流，不吞掉子 Skill 细节。

执行要点：
- 根层只保留路由图、回退规则、总不变量、发布门禁。
- 不把大量局部 SOP 再重复一遍。
- 先判断阶段，再路由到对应 Skill。

QA：
- 是否存在越权写作。
- 是否存在重复解释。
- 是否存在无限回退。

回退：
- 只要根层开始堆知识，就说明职责错位，需要拆回子层。

#### 2.9 包级校验

目标：结构完整、禁词清零、链接可达、入口清楚。

执行要点：
- 先修链接，再修结构，再修表达密度。
- 检查目录是否有散落文件、重复职责、断链、死链。
- 做自动校验，也做人工抽查。

QA：
- 是否存在旧命名残留。
- 是否存在散落文件和重复职责。
- 是否能用一个索引读懂整个包。

回退：
- 任何一个基础校验不过，都不要谈扩散。

#### 2.10 扩散复用

目标：新章节必须继承同一套颗粒度和门禁。

执行要点：
- 先确定这个单元的最小 SOP。
- 再补齐输入、输出、证据、红线、回退。
- 先做一个完整演练，再批量复制。

QA：
- 新 Skill 与旧 Skill 的粒度、厚度、门禁是否一致。
- 新章节会不会把旧问题复制一遍。

回退：
- 只要没有完整样例，就不要批量扩散。

### 3. 这次最关键的经验教训

1. 不能把“目录齐了”误判成“能力齐了”。
2. 不能把“写了规则”误判成“能执行”。
3. 不能把“补了案例”误判成“已可批量化”。
4. 不能把“知识完整”误判成“可上岗”。
5. 任何看起来像总结的内容，都要反问：新人能不能照着做。

### 4. 为什么之前会变薄

根因不是不努力，而是当时把“摘要完整”当成了“操作完整”。

更准确地说，前期优先做了结构收敛，但没有同步把每个 Skill 的判断密度、失败案例、边界条件、回退路径加厚到能上岗的程度。结果就是看起来已经成体系，实际上还不能稳稳支撑新人执行。

### 5. 以后不能再犯的事

- 不能先写文案感的总结，再倒推 SOP。
- 不能把一个大流程塞进一个 Skill。
- 不能让 Reference 只剩概念，没有案例和边界。
- 不能让 Assessment 停在定性描述。
- 不能让脚本沦为摆设。
- 不能在没做包级校验前就宣布完成。

### 6. 下一轮批量化固定模板

以后每个新单元都按这个模板推进：

1. 先确定这个单元解决哪一个最小问题。
2. 再确认输入是否够，不够就回上游补证据。
3. 再写清输出能不能直接给下游用。
4. 再补齐例子、反例、边界、红线。
5. 再把门禁写成 pass / fail。
6. 再做新人视角测试。
7. 最后才允许复制扩散。

### 7. 本次项目级记忆结论

文案 Agent 的核心不是“会写”，而是“会判断并把判断落成 SOP”。

只要后续扩散继续遵守“最小单元、渐进披露、硬门禁、明确回退、可验证通过”这五条，就不会再回到一堆看起来完整、实际上不能上岗的状态。

## 8. 本次工作的完整方法论补丁

### 8.1 先把“厚度”当成第一质量指标

这次最重要的纠偏，是把“看起来结构完整”改成“新人真的能执行”。

因此以后判断任何 Skill 是否合格，不再只看有没有目录，而是看它是否同时具备：

1. 能不能独立执行。
2. 能不能独立判断。
3. 能不能独立回退。
4. 能不能独立验收。
5. 能不能给下游直接接。

只要少一项，这个 Skill 就不能算可上岗。

### 8.2 每个 Skill 的厚化顺序

以后任何 Skill 都按这个顺序厚化，不允许跳步：

1. 先写清这个单元只解决什么最小问题。
2. 再写输入不够时要补什么证据。
3. 再写具体动作怎么做，不写抽象口号。
4. 再写判断规则，不写模糊感觉。
5. 再写失败案例，告诉新人会错在哪。
6. 再写边界条件，告诉新人什么时候停。
7. 再写绝对不能，直接封死越界。
8. 最后写回退路径，告诉新人不通过时回哪一步。

### 8.3 文档层级的固定职责

- `SKILL.md`：只保留职责、流程、边界、回退、交接。
- `references/reference.md`：只放方法论、案例、反例、边界、判断密度。
- `assessments/assessment.md`：只放门禁、打分、通过/失败条件。
- `scripts/run.py`：只做真实逻辑、诊断或校验，不许假装自动化。
- `references/workflow-retrospective.md`：只放跨环节经验、作业卡和批量化规则。

任何文档只要开始越界，就要把内容往下沉一层，不能让根层吞掉细节。

### 8.4 新人视角的最终验收问题

以后审任何一个 Skill，都固定问这 8 个问题：

1. 我拿到它，知道第一步做什么吗？
2. 我知道输入不够时该去补什么吗？
3. 我知道什么算好、什么算坏吗？
4. 我知道哪些情况必须停吗？
5. 我知道失败后回到哪一层吗？
6. 我知道有没有反例可对照吗？
7. 我知道它能不能直接给下游接吗？
8. 我知道它是不是已经厚到能上岗了吗？

只要这 8 个问题里有一个答不上来，就继续补。

### 8.5 这次最关键的删除/合并原则

后续清理文档时，按下面顺序处理：

1. 先删重复入口。
2. 再合并同类内容。
3. 再把判断逻辑下沉到最小单元。
4. 再把例子、反例、边界补齐。
5. 最后跑一次包级校验。

如果一个文件既像索引又像讲义，就拆开，不要让一个文件承担两个角色。

### 8.6 批量化的标准操作卡

每个新章节以后都按下面作业卡推进：

1. 定义这个章节对应的岗位职责。
2. 找出它的最小 SOP 单元。
3. 确认它的输入、输出、证据和边界。
4. 为它补 3 类材料：典型案例、失败案例、绝对不能。
5. 把它写进 `SKILL.md` + `reference.md` + `assessment.md`。
6. 检查它是否会和别的 Skill 重叠。
7. 做新人视角测试。
8. 通过后再复制到下一个章节。

### 8.7 以后绝不能再发生的事

- 不能把摘要当 SOP。
- 不能把目录当能力。
- 不能把规则当执行。
- 不能把案例当完整。
- 不能把通过感觉当通过门禁。
- 不能把占位脚本当自动化。
- 不能把一个大流程塞进一个 Skill。

### 8.8 本次复盘的最终结论

这次真正学到的，不是如何把一个文案 Agent 写出来，而是如何把一个数字员工写到“新人能按文档做事”的程度。

后续所有扩散都要记住一句话：

**不是把内容做出来，而是把判断、流程、边界、回退和验收做出来。**
