## 3. 每个环节怎么做

### 3.1 任务识别

目标：先判断问题类型，不先急着写结论。

QA：

- 能否把用户问题改写成一句话。
- 能否明确主结论只走一条路径。
- 能否识别是否要拆分任务。

操作要点：

- 活动看目标、成本、传播、行为。
- 诊断看现象、时间、范围、口径、版本。
- 状态看渠道、功能、复购、活跃、趋势。

### 3.2 数据补齐

目标：只补会改变结论的关键信息。

QA：

- 补的是基准，不是闲聊信息。
- 补的是缺口，不是重复确认。
- 不该追问时要直接输出当前最佳判断。

操作要点：

- 先补时间范围和对标。
- 再补核心指标和影响范围。
- 再补能验证假设的最小数据。

### 3.3 最小 SOP 执行

目标：每个 Skill 只负责一个独立单元。

QA：

- 是否有清楚的输入 / 处理 / 输出。
- 是否有边界条件。
- 是否能给新人直接执行。

操作要点：

- 不把多个 SOP 混成一个大段。
- 不在根文档堆细节。
- 不让单个 Skill 承担跨域判断。

### 3.4 验收

目标：确认结果能交接、能复用、能继续往下做。

QA：

- 结论是否有基准。
- 是否写了风险和不确定性。
- 是否有 P0 / P1 / P2 动作。
- 是否给了下一步可执行项。

操作要点：

- 先验结构，再验结论，再验动作。
- 如果结论不稳，必须标明缺口。
- 如果脚本只是回显，就不能算通过。

### 3.5 质量门禁和异常处理

目标：防止“看起来齐了”但实际上不能上岗。

QA：

- 是否已经跑过包级校验。
- 是否存在残留的旧术语、旧路径、旧结构。
- 是否每个 Skill 都能独立回答：输入是什么、输出是什么、何时回退。
- 是否有可运行脚本而不是纯占位。

操作要点：

- 先查结构，再查内容，再查术语。
- 发现一个薄点，优先补这个薄点，而不是平均用力。
- 如果 validator 过了，也要再做人类视角检查：新人能不能照着做。
- 如果脚本不能输入、不能输出、不能解释，就不能算能力完成。

常见异常：

- 只补了标题，没有补 SOP。
- 只补了 SOP，没有补反例和边界。
- 只补了评估清单，没有补实际验收样例。
- 只补了脚本文件，没有补逻辑。

处理顺序：

1. 先修结构问题。
2. 再修逻辑薄弱点。
3. 再修样例和评估。
4. 最后才做跨 Agent 复制。

## 4. 这次暴露出的经验教训

1. 目录完整不等于能力完整。
2. 只写框架不等于能上岗。
3. 没有测试样例的 SOP，容易写成空话。
4. 没有脚本门禁的 Skill，容易制造自动化幻觉。
5. 参考资料必须足够厚，才能支撑渐进式披露。

## 5. 以后怎么避免再犯

1. 先写最小闭环，再扩展厚度。
2. 每个 Skill 必须配 `references`、`assess`、`evals`、`scripts`。
3. 每次改动后都跑包级校验。
4. 新窗口接手前，先读总接手文件和 memory，再读根包文件。
5. 任何新 Agent 都必须先定义岗位边界，再拆 Skill。

## 6. 批量化规则

1. 先完成一个 Agent 的全套标准。
2. 再把同一套结构复制到下一类 Agent。
3. 复制时只替换领域内容，不复制错误。
4. 新 Agent 也要走同样的 QA、脚本、评估、复盘流程。
