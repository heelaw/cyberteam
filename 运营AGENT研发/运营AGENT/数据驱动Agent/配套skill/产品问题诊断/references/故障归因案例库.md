# 故障归因案例库

## 1. 这份文件解决什么

这份文件只解决一个问题：让新人知道什么样的异常应该先排口径，什么样的异常应该先排链路，什么样的异常应该先止损。

它不是重复 `五步诊断详解.md`，而是补案例、补反例、补边界。

## 2. 先看什么

拿到诊断任务时，先看：

1. 指标是突然掉，还是连续掉。
2. 掉的是全局，还是单渠道、单人群、单功能。
3. 有没有版本、活动、配置、发布、外部事件。
4. 口径、埋点、延迟、漏数有没有问题。
5. 先做止损还是先做修复。

## 3. 异常类型总表

| 异常类型 | 典型表现 | 正确优先动作 | 不能做 |
|---|---|---|---|
| 口径异常 | 指标定义变了、口径混了 | 先停归因，先校口径 | 不能直接下业务根因 |
| 埋点异常 | 数据缺失、漏数、重复 | 先查埋点和采集链路 | 不能拿脏数据做结论 |
| 单点故障 | 某功能、某接口、某版本异常 | 先局部止损，再确认影响面 | 不能直接放大全局 |
| 策略异常 | 活动、规则、触达后掉量 | 先看策略生效时间和受影响人群 | 不能把运营动作忽略掉 |
| 市场异常 | 竞品、价格、政策、舆情变化 | 先看外部变量 | 不能只盯产品内部 |

## 4. 典型案例

### 4.1 突然下跌但口径变了

**信号**

- 指标今天明显变差。
- 但数据口径本周刚改过。

**正确结论**

- 先停业务归因，先确认口径变更。

**什么算好**

- 好是先把真假问题分开，而不是急着找业务原因。

**什么绝对不能**

- 不能拿新旧口径混算。

### 4.2 单渠道大跌

**信号**

- 全局看起来还行。
- 但某一个渠道突然掉得很厉害。

**正确结论**

- 先把这个渠道单独拆出来，不要一上来就说全局坏了。

**什么算好**

- 好是先定位到范围，再判断是否扩散。

**什么绝对不能**

- 不能把单渠道问题写成全站问题。

### 4.3 功能使用下降

**信号**

- 功能访问变少。
- 但转化链路本身没明显故障。

**正确结论**

- 可能不是功能坏了，而是用户意愿、场景或策略变了。

**什么算好**

- 好是先做假设链，而不是先编原因。

**什么绝对不能**

- 不能只看功能点击少了就断定功能有 bug。

### 4.4 发布后下跌

**信号**

- 新版本发布后立刻掉量。

**正确结论**

- 先看发布窗口、回滚记录、错误率、崩溃率、接口耗时。

**什么算好**

- 好是先看时间对齐，再看链路关联。

**什么绝对不能**

- 不能忽略发布事件。

### 4.5 活动触发的异常

**信号**

- 异常出现在活动开始后。
- 流量、转化、活跃都发生变化。

**正确结论**

- 先看活动人群、活动规则、活动触达和活动承接。

**什么算好**

- 好是把活动影响和产品问题拆开看。

**什么绝对不能**

- 不能把活动影响错判成产品故障。

## 5. 反例库

| 反例 | 为什么错 | 正确做法 |
|---|---|---|
| 看到下跌就说是功能坏了 | 没排口径 | 先排口径和埋点 |
| 看到一条曲线就说全局坏了 | 没看范围 | 先看时间、渠道、用户、功能 |
| 看到版本发布就默认是版本问题 | 没有证据 | 先看错误率和回滚 |
| 看到活动后变化就说一定是活动 | 混淆外部变量 | 先做时间和范围切分 |
| 只给结论不给动作 | 没法执行 | 必须写止损、修复、监控 |

## 6. 新人判断口诀

1. 先看是不是脏数据。
2. 再看是不是单点问题。
3. 再看是不是策略或活动影响。
4. 再看是不是外部变量。
5. 最后才写根因。
