# 实战归因手册

## 0. 这份手册的定位

这份手册只解决一件事：当指标异常出现时，如何把“看起来像原因的东西”一步步筛成“能验证、能止损、能恢复”的主因链。

它不是理论摘要，而是给新人直接照做的排查手册。先排数据，再排技术，再排产品，再排运营，再排外部因素。

---

## 1. 归因的基本原则

### 1.1 先定义现象，再谈原因

任何归因都必须先写清：

- 哪个指标异常。
- 异常有多大。
- 从什么时候开始。
- 影响了谁。
- 影响到了哪一层链路。

没有现象定义，就没有归因资格。

### 1.2 先排“假问题”

最常见的假问题只有四类：

- 数据口径变了。
- 埋点漏了或延迟了。
- 版本发布同步了变化。
- 局部异常被误判成全局异常。

如果这四类没排干净，后面的业务归因都不成立。

### 1.3 归因不是单选题

一个异常通常同时包含：

- 主因。
- 次因。
- 放大器。
- 干扰项。

主因负责解释“为什么掉”；次因负责解释“为什么掉得这么多”；放大器负责解释“为什么扩散”；干扰项负责解释“为什么看起来更乱”。

---

## 2. 输入清单与缺失处理

### 2.1 必须输入

| 类别 | 必填内容 | 为什么必须 |
|---|---|---|
| 现象 | 指标、幅度、起点、持续时间 | 决定异常边界 |
| 范围 | 渠道、用户、功能、版本 | 决定下钻顺序 |
| 背景 | 发布、活动、配置、外部事件 | 决定候选根因 |
| 口径 | 指标定义、埋点、延迟、对账方式 | 决定能否归因 |
| 对标 | 上期、同期、历史均值、行业均值 | 决定严重程度 |

### 2.2 缺失时怎么做

| 缺失项 | 处理方式 |
|---|---|
| 现象不清 | 先补现象，不进入归因 |
| 口径不清 | 先停在数据校验 |
| 范围不清 | 先做最小范围切分 |
| 背景不清 | 先列假设，不直接下根因 |
| 对标不清 | 先输出相对判断 + 不确定性 |

### 2.3 何时可以先出结论

只有在以下条件满足时，才允许先出“当前最佳判断”：

- 现象足够明确。
- 至少一个关键维度已完成切分。
- 至少一个高概率假设已被证据支持。
- 影响范围和止损动作都能说清。

## 3. 标准归因路径

### 第 1 步：现象压缩成一句话

模板：

> 指标 X 在时间 T 后，从 A 变到 B，影响了范围 R，严重程度为 S。

这一步的目标是把问题从散点变成对象。

### 第 2 步：先排数据问题

按这个顺序查：

1. 指标定义有没有变。
2. 埋点是否缺失或重复。
3. 数据是否延迟。
4. 是否新旧口径混用。
5. 是否对账结果和行为结果冲突。

判断规则：

- 数据结果和业务体感一致，继续下钻。
- 数据结果和业务体感不一致，优先查数据。

### 第 3 步：时间切片

先看三件事：

- 是单日跳水还是连续下滑。
- 是突然拐点还是慢性衰退。
- 是否与发布、活动、节日、竞品动作同步。

判断结果：

- 单日跳水通常优先查故障、发布、口径。
- 连续下滑通常优先查体验、供给、渠道、竞争。

### 第 4 步：渠道切片

先找跌得最重的渠道，再看该渠道内部。

查什么：

- 流量有没有变。
- 转化有没有变。
- 用户质量有没有变。
- CAC 有没有变。

常见结论：

- 某渠道单独掉，优先查渠道策略、素材、投放、接口。
- 全渠道一起掉，优先查产品、技术、市场或口径。

### 第 5 步：用户切片

优先看：

- 新用户。
- 老用户。
- 高价值用户。
- 沉默召回用户。

判断规则：

- 老用户掉得更厉害，通常不是拉新问题。
- 新用户掉得更厉害，通常是入口、承接、首屏或首单问题。
- 高价值用户掉得更厉害，通常是核心功能、价格、服务或竞品问题。

### 第 6 步：功能切片

查功能时不要只看使用率，要看漏斗：

入口 → 到达 → 使用 → 完成 → 转化

重点看：

- 哪一步流失最大。
- 哪一步在异常时间点开始恶化。
- 是否只在某端、某版本、某人群出现。

### 第 7 步：版本、配置、外部因素

最后再看：

- 版本发布。
- 配置变更。
- 灰度策略。
- 外部竞品动作。
- 节假日、政策、舆情。

如果前四步没跑完，不要先跳到外部因素。

## 4. 假设池怎么建

### 4.1 假设分级

| 等级 | 定义 | 处理方式 |
|---|---|---|
| 主因假设 | 最能解释现象的候选项 | 优先验证 |
| 次因假设 | 会放大问题的候选项 | 第二优先验证 |
| 干扰假设 | 可能干扰判断的项 | 快速排除 |

### 4.2 假设模板

每个假设必须包含：

- 现象连接点。
- 支持证据。
- 反证证据。
- 验证动作。
- 验证结果后的下一步。

### 4.3 排序公式

优先级 = 影响范围 × 发生概率 × 验证成本倒数。

先查“高影响、低成本、可快速验证”的项。

## 5. 常见故障模式

### 5.1 口径错觉

表面像业务问题，实际是：

- 分母变了。
- 分组方式变了。
- 时间窗变了。
- 去重逻辑变了。

### 5.2 局部异常放大全局

某个端、某个地区、某个渠道异常，被误判成全局崩塌。

处理方式：先局部确认，再看扩散范围。

### 5.3 版本更新误伤

发布后立刻出现异常，但没有对照发布日志。

处理方式：版本、灰度、回滚、错误码四件套一起查。

### 5.4 市场变化误当产品问题

竞品大促、价格下调、舆情变化导致用户流失，却被直接归因于产品体验。

处理方式：看外部变量是否在时间上同步。

---

## 6. 止损、修复、监控

### 6.1 立即止损

只做能马上减损的动作：

- 回滚版本。
- 关闭异常配置。
- 切换备用链路。
- 暂停异常渠道投放。
- 通知受影响用户。

### 6.2 根因修复

修复必须指向主因，不要只修表面。

### 6.3 监控回看

每次诊断都要写：

- 什么时候回看。
- 回看哪个指标。
- 什么结果算恢复。
- 什么结果算失败。

---

## 7. 新人自检

如果你是新人，拿着这份手册，必须能回答：

1. 我现在是在排数据、技术、产品、运营还是外部因素。
2. 我现在最该查的是哪一个维度。
3. 我现在最像的主因是什么。
4. 我现在还缺什么证据。
5. 我现在能不能先止损。

如果答不上来，说明还没进入可执行状态。

## 8. 绝对不能

- 不能先猜根因再找证据。
- 不能把一个维度异常直接扩大成全局问题。
- 不能不排数据问题就归业务问题。
- 不能没有止损动作。
- 不能没有回看点。
- 不能把“我觉得”写成结论。
