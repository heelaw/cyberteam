# 五步诊断详解

## 先定规则

诊断不是“看见跌了就猜”，而是顺着证据把问题从现象一路追到最可能根因。

### 诊断必须回答五件事

1. 问题从什么时候开始。
2. 影响了谁。
3. 问题集中在哪一层。
4. 最可能的根因是什么。
5. 怎么验证、怎么止损、怎么恢复。

### 1. 明确现象

先回答四个问题：

- 哪个指标异常。
- 异常幅度多大。
- 从什么时候开始。
- 影响到多少用户或多少业务量。

输出时必须给出一句话现象定义。

#### 严重程度分级

| 等级 | 含义 | 处理方式 |
|---|---|---|
| P0 | 核心链路受损、重大损失 | 立即止损 |
| P1 | 明显影响核心体验或核心指标 | 快速排查与修复 |
| P2 | 局部异常，需要修复但不致命 | 持续定位 |
| P3 | 轻微波动，可观察 | 监控趋势 |

### 2. 拆分维度

优先顺序：

1. 时间。
2. 渠道。
3. 用户。
4. 功能。
5. 版本 / 地域 / 设备 / 订单链路。

做法：先找最大跌幅维度，再看该维度内部是否继续分叉。

#### 每个维度要看的信号

- 时间：是突发还是连续。
- 渠道：是单渠道问题还是全渠道问题。
- 用户：是新用户、老用户还是高价值用户。
- 功能：是入口、核心路径还是支付/提交环节。
- 版本：是否与发版、灰度、配置变更同步。

### 3. 定位问题

把问题先归类到以下一种或多种：

- 产品问题：功能体验、流程设计、交互路径。
- 技术问题：接口、报错、性能、发布、稳定性。
- 运营问题：活动、触达、策略、内容。
- 市场问题：竞品、价格、口碑、需求变化。
- 数据问题：埋点、口径、延迟、漏数。

### 4. 提出假设

每个假设必须写全：

- 发生了什么。
- 为什么会发生。
- 哪些数据支持。
- 怎么验证。
- 验证通过后怎么修。

建议按“高概率 × 高影响 × 低验证成本”排序。

#### 假设写法模板

```markdown
- 假设：
- 支持证据：
- 反证可能：
- 验证动作：
- 修复动作：
- 优先级：
```

### 5. 验证解决

验证顺序建议：

1. 先看数据口径。
2. 再看版本和发布时间。
3. 再看渠道和用户分层。
4. 再看功能漏斗。
5. 最后看外部事件。

修复动作要分成：

- 立即止损。
- 根因修复。
- 后续预防。

### 6. 常见误判

#### 6.1 把表面波动当根因

指标跌了，不代表问题本身就是“转化差”。先看口径、版本和链路。

#### 6.2 把单渠道问题误判成全局问题

某一渠道异常，先局部确认，再判断是否全局。

#### 6.3 把趋势问题当突发事故

单日跳水和连续下跌不是一回事，处理方式也不同。

#### 6.4 把数据问题当业务问题

先排除埋点、延迟、重复、漏数，再做业务归因。

### 7. 诊断输出要求

- 现象必须量化。
- 根因必须可验证。
- 方案必须可执行。
- 风险必须明确。
- 不确定性必须说明。
- 需要明确什么时候回看。

### 8. 绝对不能

- 不能只给单一归因。
- 不能没验证就下结论。
- 不能把外部因素直接忽略。
- 不能忽略口径问题。
- 不能只说“再观察”。
- 不能把“现象”当成“根因”。

## 7. 新人执行顺序

1. 先定义现象和严重程度。
2. 再确认口径是否稳定。
3. 再拆时间、渠道、用户、功能。
4. 再列假设并排序。
5. 最后写止损、修复、监控。

## 8. 诊断前 QA

- 是否先排除数据问题。
- 是否看了版本、发布、活动、外部事件。
- 是否明确主因和次因。
- 是否知道要在哪个时间点回看。
- 是否有至少一个可以立即执行的止损动作。
