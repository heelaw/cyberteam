# 工作量估算指南

## 工作量估算维度

### 1. 开发工作量

| 等级 | 人天 | 描述 |
|------|------|------|
| XS | 0.5天 | 改动极小，如改个文案 |
| S | 1-2天 | 小改动，单一模块 |
| M | 3-5天 | 中等改动，涉及1-2个模块 |
| L | 1-2周 | 较大改动，涉及多个模块 |
| XL | 2-4周 | 大项目，需要跨团队 |
| XXL | 1月+ | 超大型项目 |

### 2. 技术复杂度

| 等级 | 描述 | 示例 |
|------|------|------|
| 低 | 熟悉技术，无风险 | 静态页面、简单CRUD |
| 中 | 需要研究，有一定风险 | 新技术集成、API对接 |
| 高 | 复杂技术，风险较高 | 高并发、算法实现 |
| 极高 | 前沿/未知技术，风险高 | AI集成、底层重构 |

### 3. 依赖关系

| 类型 | 描述 | 影响 |
|------|------|------|
| 无依赖 | 独立完成 | 低 |
| 内部依赖 | 依赖其他团队模块 | 中 |
| 外部依赖 | 依赖外部系统/服务 | 高 |
| 阻塞依赖 | 必须先完成才能继续 | 极高 |

### 4. 风险因素

| 风险类型 | 描述 | 评估 |
|---------|------|------|
| 技术风险 | 技术实现难度 | 高/中/低 |
| 人员风险 | 关键人员可用性 | 高/中/低 |
| 时间风险 | 交付时间紧迫性 | 高/中/低 |
| 外部风险 | 政策/市场变化 | 高/中/低 |

## 估算方法

### 方法1: 对比估算法

将新需求与已完成项目对比

```
已知: 用户登录模块 = 5人天
新需求: 用户注册模块（类似复杂度）
估算: 5人天
```

### 方法2: 三点估算法

```
乐观估计 (O): 3天
最可能估计 (M): 5天
悲观估计 (P): 10天

期望 = (O + 4M + P) / 6
期望 = (3 + 20 + 10) / 6 = 5.5天
```

### 方法3: 拆解估算法

```
功能A = 模块1(2天) + 模块2(3天) + 联调(1天) = 6天
功能B = 模块3(1天) + 模块4(2天) + 联调(1天) = 4天
总计 = 10天
```

## 常见错误

### 错误1: 乐观估算

```
❌ 错误: "这个简单，1天就能做完"
✓ 正确: 考虑各种边界情况和异常处理
```

### 错误2: 忽略测试

```
❌ 错误: 开发3天就够了
✓ 正确: 开发3天 + 测试1天 + 修复0.5天 = 4.5天
```

### 错误3: 忽略联调

```
❌ 错误: 我们模块3天能做完
✓ 正确: 我们3天 + 联调1天 + 其他团队配合1天 = 5天
```

### 错误4: 不留Buffer

```
❌ 错误: 精确估算正好10天
✓ 正确: 估算10天 + 20% Buffer = 12天
```
