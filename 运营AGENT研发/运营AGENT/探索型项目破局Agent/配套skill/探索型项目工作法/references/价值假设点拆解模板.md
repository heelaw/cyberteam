# 价值假设点拆解模板

## 模板说明

此模板用于将核心价值假设拆解为业务链条中的具体假设点。每个假设点都必须可验证。

## 拆解原则

1. **上游假设是下游假设的前提** - 必须先验证上游假设
2. **每个假设点都能用数据验证** - 不能验证的假设不是假设，是猜测
3. **假设之间有明确的依赖关系** - 标注清楚哪些假设必须先成立

## 假设点描述格式

每个假设点必须包含：

**格式**：
> 如果 [做了某事] ... 那么 [会出现某可观测结果] ...

**示例**：
> H1：如果我们在小红书发布精准的职场内容 ... 那么会对目标用户群体产生吸引力 ... 可观测结果：笔记互动率 > 5%

## 风险等级定义

| 风险等级 | 定义 | 应对策略 |
|---------|------|---------|
| 高 | 假设不成立，整个业务模型崩塌 | 优先验证，配置最多资源 |
| 中 | 假设不成立，业务模型需要调整 | 第二顺位验证 |
| 低 | 假设不成立，业务模型影响有限 | 最后验证，可用历史数据推断 |

## 最小验证闭环

**定义**：必须验证哪些核心假设点，才能证明整个业务模型成立？

| 最小闭环假设组合 | 验证目标 |
|-----------------|---------|
| H1 + H3 | 核心价值假设成立 |

**说明**：如果H1（用户获取）和H3（价值提供）成立，说明业务模型的基本逻辑成立。

## 拆解结果示例

**核心价值假设**：我们相信刚入职0-3年的互联网产品经理需要系统化的需求管理方法，因为他们经常因为需求优先级判断不准而被质疑。

| 假设点 | 位置 | 描述 | 风险 |
|-------|------|------|------|
| H1 | 用户获取 | 如果我们在知乎/小红书发布需求管理相关内容，那么目标用户会关注我们 | 高 |
| H2 | 用户激活 | 如果用户关注了我们，他们会添加我们的小助手并获取资料 | 中 |
| H3 | 价值提供 | 如果用户使用了需求优先级方法论，他们的日常工作会更高效 | 高 |
| H4 | 用户留存 | 如果方法论有效，用户会持续使用并推荐给同事 | 中 |
| H5 | 价值放大 | 如果用户量达到一定规模，我们可以推出付费课程 | 低 |
