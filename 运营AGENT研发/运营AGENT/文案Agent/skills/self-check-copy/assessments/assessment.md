# 考核标准

- 是否逐一分析七个方向。
- 是否每个方向至少有候选，而不是只写结论。
- 是否区分功能、收益、证据。
- 是否说明为什么选这个方向，而不是别的方向。
- 是否保留证据缺口，避免硬编。
- 是否能判断通过 / 回退。

## 量化门禁

- 七问必须全部逐项检查。
- 每问必须有明确判定。
- 必须指出至少 1 个具体问题或风险。
- 必须给出可执行修改建议。
- 必须明确是否通过。
- 如果有任何硬承诺或边界缺失，默认不通过。
- **必须 7 项全部为真才可通过。**

## 一票否决

- 只给“整体不错”之类的空评语。
- 不写风险点。
- 不给回退路径。
- 把硬承诺藏在措辞里。
- 把辅助对象当主对象。
- 把 6 分当成通过。
- 忽略 CTA 和场景匹配。

## 分数模型

| 项目 | 分值 |
|---|---|
| 对象 | 1 |
| 卖点 | 1 |
| 场景 | 1 |
| 结构 | 1 |
| 动作 | 1 |
| 证据 | 1 |
| 风险 | 1 |

- 7 分：通过。
- 4-6 分：回退修复。
- 0-3 分：不通过。

## 抽检要求

- 至少覆盖 1 个品牌认知型案例、1 个强转化型案例、1 个长篇教育型案例。
- 每个案例都必须写出“为什么通过 / 为什么不通过”，不能只贴结果。
- 遇到冲突证据时，必须说明优先级，不允许把冲突抹平。
- 任何一条红线命中都要直接回退，不允许先试着“修一修再说”。

## 通过判定

- **7 项全部为真且无硬承诺 → pass。**
- 4-6 项为真 → fail but repairable。
- 0-3 项为真 → fail。

## 新人能做的事

- 逐项核对七问并给出判定。
- 指出具体问题而不是说“还行”。
- 给出可执行修改建议。
- 明确说出通过还是不通过。
- 识别硬承诺和夸大表达。

## 交付标准

1. 先给判断（通过 / 回退）。
2. 再给风险点（逐项列出，不是泛泛而谈）。
3. 再给修改建议（可执行，不是“建议优化”这种空话）。
4. 最后给是否通过（明确）。

## 回退要求

- 通过：7 项全过，无硬承诺。
- 回退：给出具体回退路径（segment-users / research-sellpoint-forward / research-sellpoint-backward / judge-scenario / select-structure / edit-copy）。
