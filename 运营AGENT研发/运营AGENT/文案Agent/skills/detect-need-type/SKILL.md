# 需求类型判断器

## 作用

先把需求证据采全，再判断需求层级和下一步动作。

这一步不是猜，而是先把原话、行为、搜索、反馈、对话、对比、数据排好，再决定能不能进入下一步。

## 启动前

- 必须至少拿到 1 条原话或 1 条真实行为。
- 如果只有一句感受，没有目标和动作，不许直接定性。
- 如果原话和行为冲突，先保留冲突，不要硬合并。

## 执行原则

- 先证据，后判断。
- 先分方向，后合并结论。
- 先处理冲突，再给下一步。
- 追问有上限，不能无限循环。
- 证据不齐时，只能输出当前最佳判断和缺口。

## SOP

1. 按七个方向收集证据。
2. 记录每一向的原句、动作、比较点或指标，不先改写。
3. 判断目标是否已经被用户清晰说出。
4. 判断问题是否存在，但目标还不完整。
5. 判断用户是否还没有形成需求意识。
6. 处理冲突证据，保留优先级最高的一组。
7. 输出需求类型、依据、缺口和下一步动作。

## 七向最低采样

- 原话：至少保留 1 条可复述原句。
- 行为：至少保留 1 个真实动作链。
- 搜索：至少保留 1 个问题词、1 个比较词、1 个解决词。
- 反馈：至少保留 1 条高情绪表达和 1 条具体不满点。
- 对话：至少保留 1 轮追问链。
- 对比：至少保留 1 个默认替代方案和 1 个比较维度。
- 数据：至少保留 1 个指标趋势和 1 个解释假设。

## 裁决规则

1. 行为和数据同时一致时，优先级最高。
2. 行为和原话冲突时，优先信行为。
3. 原话和搜索一致、但行为缺失时，只能判为待验证的模糊需求。
4. 反馈和对比只能辅助判断，不能单独定性。
5. 只出现单点情绪、单条评论、单个异常时，不允许判为明确需求。
6. 只有搜索词没有行为，只能判为待验证的模糊或潜在需求。
7. 只有数据异常没有用户语境，只能作为线索，不能单独定性。

## 追问上限

- 最多追问 3 个问题。
- 最多补 2 轮证据。
- 超过上限仍不清楚时，输出当前最佳判断 + 最大缺口 + 下一步验证方向。

## 判定结果

- 明确需求：目标、对象、结果都说清了。
- 模糊需求：问题说清了，但目标、对象或结果不完整。
- 潜在需求：只有现象，没有需求意识。
- 混合信号：原话、行为、搜索彼此不一致，先保留冲突。

## 输出骨架

- 需求类型
- 七向证据地图
- 冲突证据
- 判断依据
- 最大缺口
- 下一步动作

## 新人校验

- 新人能不能区分”有人说”与”有人做”。
- 新人能不能知道缺哪一类证据就该去补哪一类证据。
- 新人能不能知道什么时候该继续追问，什么时候该停。
- 新人能不能知道什么情况下绝对不能直接写正文。

## 输入要求

启动本Skill前必须确认以下输入已获取：

| 输入项 | 最低要求 | 说明 |
|--------|----------|------|
| 原话或行为 | 至少1条 | 必须有用户原话或可观测行为之一，不能全靠推断 |
| 需求相关描述 | 至少1条 | 用户说的话、行为记录、反馈记录等 |

**禁止启动条件**（满足任一条不得启动）：
- 完全没有用户原话、行为、反馈、对话等真实信号
- 只有”老板说要做”、”我觉得应该”、”竞品这么做”等外部噪音
- 只有一个情绪词（如”烦”、”不好用”）而没有任何行为或反馈

**允许启动但需快速补证的条件**：
- 只有原话没有行为 → 先标记缺口，快速补1条行为证据
- 只有行为没有原话 → 先标记缺口，快速补1轮原话

## 好样例

### 样例1：明确需求（标准完成）

**输入**：
- 原话：”帮我把这条活动页改成能直接引导下单的版本”
- 行为：当前活动页跳出率78%，点击了活动入口但无人成交

**完整输出**：
```
需求类型：明确需求

七向证据地图：
- 原话✓ “帮我把这条活动页改成能直接引导下单的版本”（目标✓ 对象✓ 结果✓）
- 行为✓ 跳出率78%、点击活动入口但无人成交（具体数据）
- 搜索✗ 无
- 对话✗ 无
- 对比✗ 无
- 反馈✗ 无
- 数据✓ 跳出率78%、零成交（指标明确）

冲突证据：无

判断依据：
目标（改活动页）、对象（活动页）、结果（直接下单）三要素齐全
行为信号和数据信号一致，指向同一问题（活动页不促转化）
七向中4向有实质证据，达到4/7阈值

最大缺口：无重大缺口，原话+行为+数据已形成完整证据链

下一步动作：
→ 直接进入judge-scenario或select-structure
→ 如需补充，七向中可补对话（了解用户对”直接下单”的理解）
```

**为什么是好样例**：
- 证据覆盖充分（4/7向有实质内容）
- 判断依据引用了具体证据
- 没有过度推断
- 下一步动作明确

---

### 样例2：模糊需求（证据不足但处理正确）

**输入**：
- 原话：”这个文案我觉得不太对，但我说不出哪里不对”
- 行为：反复修改同一段落，改完又回退，没有一次定稿
- 反馈：之前给过”没有感觉”的评价

**完整输出**：
```
需求类型：模糊需求

七向证据地图：
- 原话✓ “这个文案我觉得不太对”（情绪✓ 目标✗ 对象✗ 结果✗）
- 行为✓ 反复修改同一段落，改完回退，从未定稿
- 反馈✓ “没有感觉”（高情绪 具体点✗）
- 其他方向：搜索✗ 对话✗ 对比✗ 数据✗

冲突证据：无

判断依据：
问题存在（文案有调整冲动）✓
但目标（改成什么样）、对象（具体哪一段）、结果（什么状态算改好）均不清晰
行为证据（反复回退）支持”不对”的感受，但不足以形成明确修改方向

最大缺口：
①具体哪一段/哪一句不对
②希望改成什么风格或方向
③”有感觉”的评判标准是什么

下一步动作：
→ 追问：”能说一下修改后和修改前比，你更不接受哪个版本？”
→ 2轮内补证后重新判断
```

**为什么是好样例**：
- 没有硬判为明确需求
- 明确列出了缺口清单
- 追问句式具体（不是泛泛问”哪里不对”）
- 保留了继续判断的空间

---

### 样例3：潜在需求（只有行为信号）

**输入**：
- 原话：无
- 行为：用户一直跳出、反复查看、收藏了竞品但没有收藏本品牌
- 搜索：搜了品类词、竞品词，但搜完离开
- 数据：本周UV持平，但转化率下降30%，竞品同期转化率上升

**完整输出**：
```
需求类型：潜在需求

七向证据地图：
- 行为✓ 跳出、反复查看、收藏竞品但未收藏本品牌
- 搜索✓ 品类词+竞品词，搜完离开
- 数据✓ 本品转化率-30%，竞品转化率上升
- 原话✗ 对话✗ 对比部分✓ 反馈✗

冲突证据：无

判断依据：
用户没有主动表达需求意识（无原话）
但行为（收藏竞品）和数据（转化率下降）形成一致信号
搜索词显示用户在主动比较，但尚未形成对本品牌的偏好

最大缺口：
①用户离开/不转化的具体原因（不知道）
②离开后去了哪里（竞品）③本品牌哪里没打中

下一步动作：
→ 先做认知唤醒，不进入正文写作
→ 文案方向：先建立”本品牌为什么值得考虑”的认知，而不是直接推转化
```

**为什么是好样例**：
- 正确识别为潜在需求而非模糊或明确
- 证据组合使用合理（行为+搜索+数据）
- 没有臆测原因
- 下一步方向清晰（认知唤醒先于转化）

## 坏样例

### 坏样例1：情绪误判为明确需求

**输入**：
- 原话：”这个页面让我感觉很烦躁”

**错误输出**：
```
需求类型：明确需求
目标：优化页面让用户不烦躁
```

**问题分析**：
- “烦躁”是情绪，不是目标
- 没有对象（哪一页）、没有结果（变成什么样）
- 违反裁决规则：情绪词不能当需求

**正确输出**：
```
需求类型：潜在需求（仅情绪，无目标）
缺口：①烦躁的具体原因②用户期望变成什么状态③障碍在流程哪一步
下一步：追问”能说一下哪一步让你最烦躁吗，当时你想做什么”
```

---

### 坏样例2：搜索词当结论

**输入**：
- 搜索词：”怎么写产品文案”

**错误输出**：
```
需求类型：明确需求
目标：学习产品文案写法
```

**问题分析**：
- 搜索词只说明用户在找答案，不代表用户真实目标
- 实际需求可能是：向上管理、证明自己、提升营销效能
- 违反绝对不能：不能把搜索词直接当最终目标

**正确输出**：
```
需求类型：模糊需求（搜索词方向待验证）
缺口：①为什么要写文案②是给谁看③希望达到什么效果
下一步：追问”现在是遇到了什么具体问题吗”
```

---

### 坏样例3：数据异常直接当写作指令

**输入**：
- 数据：Q3退款率上升15%

**错误输出**：
```
需求类型：明确需求
目标：修复退款率，写高可信度文案
```

**问题分析**：
- 退款率上升有N种原因：产品、价格、承诺过度、竞品、季节、渠道
- 没有证据指向”表达/文案”是根因
- 违反绝对不能：不能把数据异常直接当写作指令

**正确输出**：
```
需求类型：线索（数据异常）
缺口：①退款原因是否记录②退款用户画像是否清晰③退款发生在哪个环节
下一步：回溯退款原因标签，先确认环节再决定是否进入文案修复
```

---

### 坏样例4：证据不足强行结案

**输入**：
- 原话：”帮我优化一下”

**错误输出**：
```
需求类型：模糊需求（信息不足但强行给结论）
目标：优化文案
下一步：直接进入写作
```

**问题分析**：
- 只有1向证据（原话），其他6向全空
- 违反量化门禁：7向中缺少任意2向则直接不通过
- 违反追问上限：3轮仍不清必须输出当前判断+缺口

**正确输出**：
```
需求类型：信息不足，暂不结案
最大缺口：7向中缺少6向证据（原话✗ 行为✗ 搜索✗ 对话✗ 对比✗ 数据✗）
下一步验证方向：
①先确认是哪个具体问题（哪篇文案/哪个页面/哪个问题）
②再看用户对这个问题的感受和行为
③最后补搜索/对比/数据
建议：回退到context收集，不在此Skill继续打转
```

## 回退路径

| 当前状态 | 回退到 | 回退条件 |
|----------|--------|----------|
| 证据严重不足（只有1-2向） | context收集 | 7向中缺少6向以上，且追问3轮后仍无行为/数据证据 |
| 需求类型无法判断 | 上一Skill或context | 混合信号且无法确定优先级，或角色冲突无法裁决 |
| 输入全为外部噪音 | context收集 | 只有”老板说要做”、”竞品这么做”等，无用户信号 |
| B2B场景多人需求冲突 | segment-users | 发现不是单人决策，需要先分层再判断需求 |

**回退判断标准**：

1. **证据不足**（满足任一）：
   - 7向中只有1-2向有实质证据
   - 追问3轮后行为/数据仍为空
   - 只有情绪没有目标

2. **信号冲突无法裁决**（满足任一）：
   - 原话、行为、数据三方证据指向不同结论
   - 行为和原话完全相反且无法判断优先级

3. **需要先分层**：
   - B2B/B2C多人决策场景
   - 购买者≠使用者≠拍板者
   - 需求类型判断依赖角色判断

**回退时必须输出**：
- 当前最佳判断（即使不完整）
- 最大缺口清单
- 下一步验证方向
- 建议回退到哪个Skill

## 绝对不能

- 不能只靠一句话下结论。
- 不能把情绪词当需求。
- 不能把搜索词直接当最终目标。
- 不能把数据异常直接当写作指令。
- 不能忽略冲突证据。
- 不能在证据不齐时直接跳去写正文。
- 不能在7向中缺少2向以上时强行通过。
- 不能超过3轮追问上限后继续打转。
- 不能把”老板说”和”竞品这么做”当作用户需求证据。
