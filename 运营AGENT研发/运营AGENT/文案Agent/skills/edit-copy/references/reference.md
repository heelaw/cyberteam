# 删改优化

## 适用场景

- 已有初稿，需要更像人话、更像成交文案、更像业务文案。
- 需要压缩、强化或提高清晰度。
- 不需要重新发明卖点。

## 改写前先看 7 件事

1. 主卖点有没有丢。
2. 证据有没有丢。
3. 动作有没有丢。
4. 限制条件有没有丢。
5. 用户是否还是原来的用户。
6. 场景是否还是原来的场景。
7. 这次修改是删、补、调序，还是局部强化。

## 常见处理顺序

1. 删重复。
2. 删空话。
3. 删抽象。
4. 补证据。
5. 强动作。
6. 调顺序。

## 逐项怎么改

### 删重复

- 同一意思说了两次以上，保留最短最清楚的那句。
- 如果重复在不同段落，只保留一次。
- 如果重复是为了强调，保留最强一次即可，不要堆叠。

### 删空话

- 删除只有态度没有信息的词。
- 删除没有对象、没有结果、没有证据的形容。
- 删除“很厉害、很专业、很全面”这类不落地的套话。

### 删抽象

- 把“好、强、快、专业、极致”换成事实。
- 把概念改成场景、动作、结果。
- 把“全面方案”改成“具体步骤/机制/结果”。

### 补证据

- 没有证据的承诺要补出处。
- 没有数据就补例子，没有例子就补过程说明。
- 如果补不上证据，就把它留在待确认项里。

### 强动作

- 把“欢迎了解”换成具体动作。
- CTA 要和用户当前状态匹配，不能乱催。
- 如果用户还在理解阶段，动作先弱后强。

### 调顺序

- 先结论后解释，或先问题后方案，取决于场景。
- 如果用户只看标题，重要信息必须前置。
- 如果首段太软，优先把主张前移。

## 修改范围判断

- 词级：只改一个词，解决一个词的虚、空、假。
- 句级：只改一句，解决一句里的重复或不清楚。
- 句群级：只改一组相邻句子，调整主线和证据顺序。
- 段落级：只在结构错位时做，不能为了顺眼随便重排全文。

## 常见问题

| 问题 | 处理 |
|---|---|
| 太空 | 换成具体场景、数字、例子 |
| 太长 | 合并重复表达，保留主线 |
| 太虚 | 补事实、证据和边界 |
| 太像说明书 | 改成用户语言和用户视角 |
| 太软 | 把动作和结果前置 |
| 太乱 | 先定主线，再调句群顺序 |

## 典型例子

### 例子1：删空话

- 原句：这是一个非常专业、非常高效、非常值得选择的方案。
- 改法：改成具体结果、具体动作或具体证据。

### 例子2：删重复

- 原句：省时间、提效率、减少重复劳动、让你更快完成工作。
- 改法：保留最强的一句，其余并到证据或说明里。

### 例子3：补证据

- 原句：让你的转化率大幅提升。
- 改法：补数据、案例、前后对比、过程说明。

### 例子4：转场景

- 原句：我们提供全面、系统、完整的解决方案。
- 改法：改成用户会说的话，比如“你照着这套流程就能直接开工”。

## 改写节奏

1. 先看重复。
2. 再看空话。
3. 再看抽象词。
4. 再补证据和边界。
5. 最后调顺序和句式。

## 结果判断

- 如果删改后更短但更准，说明方向对。
- 如果删改后没有信息密度，只剩修辞，说明改偏了。
- 如果删改后动作更明确，说明优化有效。
- 如果删改后丢了事实，说明改坏了。

## 输出样式

- 先给修改版。
- 再列修改点。
- 再列保留信息。
- 再列还没解决的问题。

## 删改优先级

1. 先保留主卖点。
2. 再保留证据。
3. 再保留动作。
4. 最后才考虑修辞。

## 修改手法

### 1. 压缩

- 把多句合成一句。
- 把重复修饰删掉。

### 2. 强化

- 把模糊词换成可感知表达。
- 把结论前置。

### 3. 调序

- 把最重要的信息放到最先被看到的位置。
- 把解释和证据放到结论后面。

### 4. 转场景

- 用用户熟悉的场景替换内部表达。
- 用用户会说的话替换作者会说的话。

## 改写边界

- 不把删改做成重写。
- 不把一条文案改成另一条完全不同的文案。
- 不为了顺口而删掉关键事实。
- 不为了好看而加新承诺。
- 不把“删掉了”误写成“已经优化了”。

## 改写示例判断

- 如果删完后信息更少但更清楚，通常是对的。
- 如果删完后卖点不见了，通常删过头了。
- 如果改完只剩修辞，没有事实，通常改偏了。

## 典型例子

### 例子1：删掉空话但保留力度

- 原句：我们致力于打造高品质、高效率、全方位的解决方案。
- 改后：我们把用户最常卡住的 3 个步骤拆开，让你直接开始做。

### 例子2：删重复但不丢节奏

- 原句：更快、更省事、更高效、更轻松。
- 改后：更快完成，而且少走弯路。

### 例子3：补证据后再强化

- 原句：这个方案很好用。
- 改后：这个方案已经在 30 个项目里跑过，平均把交付时间缩短了 22%。

### 例子4：转成用户语言

- 原句：我们有完整的方法论和系统支持。
- 改后：你照着这套步骤走，不用自己试错。

## 失败案例

- 把“删改”做成“重写”，最后和原文不是一回事。
- 把所有形容词都删光，导致文案只剩骨架。
- 只改句子顺序，不解决空话和证据问题。
- 只追求短，不追求准。
- 为了顺口，把限制条件删掉。

## 边界案例

- 如果原文本来就很短，优先改准确度，不要硬压缩。
- 如果原文证据不足，先补缺口，再谈删减。
- 如果标题已经很强，正文可以更克制，不必再加修辞。
- 如果只是平台换版，不要改掉主卖点。

## 交付样式

- 先给修改版。
- 再列修改点。
- 再列保留信息。
- 再列还没解决的问题。
- 再列风险提示。

## 绝对不能

- 不能把删改做成重写。
- 不能删掉证据和动作。
- 不能把关键事实删没。
- 不能只改句子，不解决逻辑。
- 不能为了顺口而加新承诺。
- 不能把边界和限制改没。

## 常见误删

- 把证据删掉。
- 把限制条件删掉。
- 把动作删没了。
- 把用户能感知到的差异删成平庸表述。

## 修改原则

1. 不改变核心意思。
2. 不新增卖点。
3. 不把删改做成重写。
4. 每次只改一个问题群。
5. 只动最小必要范围。

## 输出骨架

- 修改版文案
- 修改点清单
- 保留信息
- 还没解决的问题
- 风险提示
