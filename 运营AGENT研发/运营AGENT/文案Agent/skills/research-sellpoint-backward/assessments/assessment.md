# 反向卖点考核

## 基础门禁

以下条件必须同时满足，缺一项就不通过：

1. 必须有至少 1 条用户原话或具体情境描述。
2. 必须有清楚的场景。
3. 必须有清楚的阻碍。
4. 必须有可感知的代价。
5. 必须有结果翻译。
6. 必须写出能落回产品能力的卖点表达。
7. 必须标出证据缺口或待验证项。

## 卖点成立条件

| 条件 | 通过标准 | 不通过表现 |
|---|---|---|
| 可回到产品能力 | 卖点背后有产品功能或服务支撑 | 卖点只是愿望句 |
| 用户能验证 | 用户自己能判断是否成立 | 只有供应商能解释 |
| 用户能转述 | 用户能用自己的话复述 | 只有专业术语能说 |
| 有场景承接 | 卖点和具体使用场景对得上 | 没有场景，只剩抽象表达 |

## 证据质量标准

| 层级 | 通过标准 | 不通过表现 |
|---|---|---|
| 原话 | 至少 1 条可追溯原话 | 全靠二次翻译 |
| 交叉验证 | 同一痛点最好有 2 条以上独立证据 | 只有单点情绪 |
| 频率说明 | 能说明高频、偶发或一次性 | 不知道这个问题是不是常见 |
| 行为验证 | 有实际使用、转化或替代行为 | 只有“想要”没有“做过” |

## 分项考核

### 1. 痛点

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 原话保留 | 保留用户原始表达 | 全部改写成你的总结 |
| 具体性 | 痛点里有动作或重复场景 | 只有“太麻烦” |
| 层次感 | 能区分功能/业务/心理/身份 | 不知道这是什么层级 |

### 2. 场景

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 五要素 | 至少写清谁、任务、卡点 | 只有“上班的时候” |
| 粒度 | 粒度适中，不太粗不太细 | 过粗或过细都不行 |
| 反复性 | 能看出是否重复发生 | 看不出是不是偶发 |

### 3. 阻碍

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 卡点明确 | 说清楚卡在哪一步 | 只写“效率低” |
| 阻碍分类 | 能判断是工具 / 流程 / 能力 / 资源 / 心理 | 全部混在一起 |
| 产品匹配 | 阻碍是产品可接住的 | 写了产品接不住的内容 |

### 4. 代价

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 可感知 | 具体到时间、金钱、信任、机会 | 只有“影响很大” |
| 有量级 | 有每次、每天、每月的量化 | 没有量级 |
| 有对象 | 知道是谁在承受代价 | 只写抽象受损 |

### 5. 结果翻译

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 状态化 | 写成用户能感知的状态 | 写成功能目标 |
| 可验证 | 用户自己能判断是否达成 | 只有你能判断 |
| 可落地 | 能落回产品能力 | 落不回产品 |

### 6. 卖点表达

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 用户语言 | 更像用户会说的话 | 更像产品宣传语 |
| 可转述 | 用户能转述给别人 | 只能你自己理解 |
| 可感知 | 一眼知道好在哪 | 只剩抽象形容词 |

## 一票否决

1. 只有抱怨，没有结果翻译。
2. 只有结果，没有阻碍。
3. 把愿望句当可交付卖点。
4. 没有场景，只写大家都需要。
5. 代价是抽象判断，不是可感知损失。
6. 跳过证据恢复，直接假设成立。
7. 卖点表达是产品功能描述，不是用户语言。

## 分级标准

### 合格

- 痛点、场景、阻碍、代价、结果、卖点六步都齐。
- 至少有 1 条原话或可追溯反馈。
- 至少标了 1 个证据缺口。

### 良好

在“合格”基础上，再满足：

- 场景写清五要素。
- 阻碍分类清楚。
- 代价有量级。
- 有至少 2 类证据来源。
- 卖点能自然转成用户语言。

### 优秀

在“良好”基础上，再满足：

- 每一步都写了失败案例。
- 说明了为什么不是别的阻碍、别的结果、别的卖点。
- 证据恢复过程清楚。
- 输出可以直接给下一步 Skill 使用。

## 新人实操题

### 题 1：痛点提炼

**输入**：用户说“每次项目一多我就怕漏事”。

**合格答案应该有**：

- 提炼出核心痛点，而不是只抄一句话。
- 保留“怕”这个情绪词。
- 区分“漏事”和“怕漏事”。

### 题 2：场景还原

**输入**：用户说“开会的时候经常记不住谁负责什么”。

**合格答案应该有**：

- 谁在说。
- 在做什么任务。
- 卡在哪一步。
- 不能只写“开会的时候”。

### 题 3：阻碍识别

**输入**：用户说“我想随时看到所有项目进度”。

**合格答案应该有**：

- 能区分信息阻碍、工具阻碍、流程阻碍、能力阻碍。
- 不能把“没有系统”当唯一答案。

### 题 4：代价估算

**输入**：用户说“我觉得进度管理很麻烦”。

**补充信息**：每月手工整理 6 小时；汇报前经常找数字；有时数据对不上。

**合格答案应该有**：

- 直接损失。
- 信任损失。
- 机会损失。

### 题 5：结果翻译

**输入**：痛点是“进度不知道问谁”，阻碍是“没有统一地方看到谁负责什么”。

**合格答案应该有**：

- 可感知状态。
- 用户能验证。
- 能落回产品能力。

### 题 6：卖点表达

**输入**：结果是“打开一个页面就知道所有人的进度”。

**合格答案应该有**：

- 不要写成“实时同步项目管理系统”。
- 要写成用户语言。
- 要写得能被转述。

## 交付判断

- 如果六步齐全、证据缺口清楚、卖点能回到产品能力，算通过。
- 如果只有痛点，没有阻碍、代价、结果，不通过。
- 如果只有结果，没有场景和阻碍，不通过。
- 如果像总结报告，但不能指导下一步，不通过。
