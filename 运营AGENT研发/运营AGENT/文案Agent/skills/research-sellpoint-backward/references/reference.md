# 参考资料

## 核心方法论

### 反向卖点的本质

用户购买不是因为了解产品，而是因为"受够了现在"。从"受够了"到"买这个"，中间有一条完整的心理路径：

```
痛点（我受不了）→ 场景（每次XX的时候）→ 阻碍（但XX做不到）→ 代价（所以我损失了XX）
↓                                                  ↑
  ← ← ← ← ← ← ← ← ← ← ← ← ← ← ←  ← ← ← ← ← ← ← ←
        ↑                                              ↓
     心理路径（我想变成XX）← ← ← ← ← 结果（变成XX）
              ↓
        卖点表达（正好能让我XX）
```

卖点能成立，是因为这条链的每一环都站得住脚。有一环断了，用户就会觉得"你说得挺好，但我还是不信"。

### 为什么"痛点+解决方案"不够

常见错误是：看到痛点，就直接给解决方案。但这两者之间隔着：

1. **场景验证**：这个痛点真的每次都发生吗？还是偶尔一次？
2. **阻碍验证**：用户是因为这个原因卡住的吗？还是另有原因？
3. **代价验证**：这个痛点不解决，用户实际损失了什么？损失有多大？
4. **结果验证**：用户想要的状态翻译成产品语言是什么？产品真的能接住吗？

跳过任何一步，都会导致卖点悬浮。

### 证据分层原理

| 证据层级 | 强度 | 说明 |
|----------|------|------|
| 直接引述 | ★★★★★ | 用户原话，最强证据 |
| 重复出现 | ★★★★ | 同一痛点在多条独立反馈中出现 |
| 行为验证 | ★★★ | 用户实际改变了行为（如从竞品转来） |
| 搜索词验证 | ★★ | 用户主动搜索过相关问题 |
| 推断/假设 | ★ | 需要继续验证，不能直接用 |

## 痛点采集渠道与质量排序

### 渠道质量排序（从高到低）

1. **用户原话录音/转录**：语气、停顿、反复强调的词，都是信号
2. **客服/售后工单**：有具体情境，不是抽象评价
3. **应用商店/电商差评**：可信度高，用户投入了时间写
4. **社媒评论**：需要交叉验证，情绪容易被放大
5. **访谈记录**：有追问空间，但需要过滤分析师偏见
6. **问卷开放题**：量大但浅，需要聚类分析
7. **搜索词数据**：证明用户主动找过，但不能说明结果
8. **竞品评论区**：用户说"不如XX"，是间接证据

### 各渠道采集要点

**差评采集标准**：
- 至少包含一个具体情境描述
- 有情绪词（怒、烦、失望、终于）
- 有具体损失或期望

**工单采集标准**：
- 有具体时间/场景
- 有业务影响描述
- 同一用户有多个相关工单更可信

**社媒评论采集标准**：
- 原创内容，非转发
- 有具体使用场景
- 评论互动数高，说明触动了其他人

## 追问技术

### 三层追问法

**第一层：补场景**（让痛点具体化）
```
问：能再说说是在什么情况下发生的吗？
问：这是从什么时候开始的问题？
问：每次都会这样吗？
```

**第二层：补代价**（让痛点可感知）
```
问：这个问题不解决，最直接影响什么？
问：你当时怎么处理的？
问：这个问题大概多久出现一次？
```

**第三层：补结果**（翻译成期望状态）
```
问：你当时最想变成什么样？
问：如果有一个方案能解决这个问题，你希望它能做到什么？
问：你怎么跟朋友描述这个问题？
```

### 典型追问场景

**场景1：用户只说"太慢了"**
```
追问1：什么操作的时候太慢了？
追问2：慢了大概多久？之前预期是多久？
追问3：每次都这么慢，还是有时候快？
追问4：慢的时候你最想砸键盘还是叹气接受？
```

**场景2：用户说"不好用"**
```
追问1：哪个具体功能让你觉得不好用？
追问2：不好用是指找不到功能，还是找到了但不会操作，还是操作结果不对？
追问3：这个问题出现的时候，你在做什么任务？
```

**场景3：用户说"太贵了"**
```
追问1：跟什么比觉得贵？
追问2：买的时候最犹豫的是什么？
追问3：如果不花钱，还有什么替代方案？
追问4：你觉得值在哪？差在哪？
```

## 完整案例

### 案例一：项目管理工具

**输入**：
- 用户原话："每次项目一多，我就怕漏掉要跟进的事。"
- 补充反馈："会后经常忘记谁负责什么。"

**六步还原**：

1. **痛点**：会后漏跟进（怕漏事）
2. **场景**：周一早上9点部门周会，20人参加，议题多，会后每人要执行不同任务
3. **阻碍**：现有工具（邮件/微信）的信息散在各处，没有统一待办落地机制；责任人是口头认领，没有系统追踪
4. **代价**：漏跟进的项要补火、延期项目被领导问"怎么会漏"、反复在群里@人确认进度
5. **期望结果**：开完会就知道谁做什么、什么时候交付什么，不需要靠记忆或反复追问
6. **卖点表达**：
   - 会议结束，自动长出每个人的待办清单
   - 不用再靠@确认"你记得吧？"
   - 到时间自动提醒，不用追

**证据缺口**：需要验证漏跟进的发生频率，以及"开完会自动生成"在现有技术下能否实现

### 案例二：保温杯

**输入**：
- 用户原话："杯子漏水，包里东西都湿了"
- 补充：换了几个杯子还是漏

**六步还原**：

1. **痛点**：杯子漏水
2. **场景**：冬天通勤，背双肩包，杯子放包侧袋，挤地铁时杯子倒了
3. **阻碍**：现有杯盖是螺旋盖，旋转时拧不紧会漏；而且倒置时压力会让水渗出来
4. **代价**：MacBook Pro 进水（直接损失）；文件袋里的合同湿了（高价值损失）；冬天没有热水喝（持续影响）
5. **期望结果**：把杯子扔包里不用管，倒了也不漏
6. **卖点表达**：
   - 倒置也不漏，往包里一塞就走
   - 不再用纸巾检查包里的东西
   - 装的热水8小时后还是烫的

### 案例三：男士袜子

**输入**：
- 用户原话："穿一天脚就臭，跑步更严重"
- 补充：同事说过他脚臭，很尴尬

**六步还原**：

1. **痛点**：脚臭、闷、磨脚
2. **场景**：久站开会（8小时）、跑步5公里以上、通勤走路多；下午脚开始不舒服
3. **阻碍**：普通棉袜不排汗，汗闷在鞋里；没有足弓支撑，长时间走路脚底酸
4. **代价**：下午不敢脱鞋、被人暗示脚臭后自我意识变强、运动后脚磨出水泡
5. **期望结果**：穿一整天脚不臭，运动后脚也不累
6. **卖点表达**：
   - 开完会敢脱鞋，不用担心味道
   - 跑步10公里脚底不起泡
   - 穿三年袜子不臭不变形

### 案例四：企业数据仪表盘

**输入**：
- 用户原话："我需要看数据的时候，找不到，想要的数据每次都要手工拼"
- 补充：管理层开会经常问数据，现场答不上来，很尴尬

**六步还原**：

1. **痛点**：数据找不到、拼不出来
2. **场景**：每周一管理层例会，汇报上周核心指标；每次都要提前一天让分析师准备数据，现场还经常对不上
3. **阻碍**：数据分散在4个系统里，没有统一出口；指标定义不统一，同一个词不同人理解不一样
4. **代价**：开会时现场被问住、靠记忆说数字事后被打脸、分析师成了人工传话筒
5. **期望结果**：随时打开就能看到管理层想看的所有指标，不需要等，不需要拼
6. **卖点表达**：
   - 打开一个页面，所有老板关心的数字都在
   - 开会不用等分析师，现场自己就能点出来
   - 数字永远对得上，不用再事后核对

## 常见误判清单

1. **把情绪当证据**：用户说"太垃圾了"不是证据，情绪背后的事实才是
2. **把抱怨当需求**：用户说"不要给我发短信"是抱怨，用户真正想要的是"让我知道但不打扰"
3. **把价格异议当唯一痛点**：价格敏感的根因可能是信任问题，也可能是价值感不够
4. **把竞品黑评当己方机会**：竞品被抱怨的功能，自家可能也有同样的问题
5. **把沉默当同意**：没人说不喜欢不代表都喜欢，可能是没有动力去说
6. **把新用户反馈当全部用户反馈**：高频使用用户和尝鲜用户的痛点完全不同
7. **把问卷数据当定性结论**：问卷告诉"70%用户觉得贵"，但不知道他们真正的心理价位和场景

## 术语对照表

| 术语 | 定义 | 常见混淆 |
|------|------|----------|
| 痛点 | 用户在具体场景里反复受的罪 | 与"痒点"混淆（痒点是想要但不急迫） |
| 场景 | 痛点发生时的具体情境 | 与"用户画像"混淆（画像是Who，场景是When+Where） |
| 阻碍 | 用户想解决但现在跨不过去的坎 | 与"原因"混淆（原因讲过去，阻碍讲现在） |
| 代价 | 阻碍不消除会持续带来的损失 | 与"痛点"本身混淆（痛点是现状，代价是后果） |
| 结果 | 用户真正想要变成的状态 | 与"解决方案"混淆（结果是用户语言，方案是产品语言） |
| 卖点 | 能让用户感知并愿意转述的表达 | 与"产品功能"混淆（功能是说什么，卖点是怎么说） |
| 证据缺口 | 目前缺失、无法自证的部分 | 与"待验证假设"区分（缺口是不知道，假设是有方向没验证） |

## 反推卖点的深挖手册

### 什么时候先停

- 痛点只有一句情绪，没有动作、场景或代价时，先停。
- 代价只有“很严重”，没有时间、金钱或信任损失时，先停。
- 场景太泛，只能写“上班的时候”时，先停。
- 结果无法落回产品能力时，先停。
- 任何一步卡住，都不要硬写卖点。

### 什么时候可以往下走

- 能说出具体谁在什么任务里卡住。
- 能说出卡点是什么，而不是只说“不好用”。
- 能说出不解决会损失什么。
- 能说出用户想变成什么状态。
- 能把结果翻译成一句用户愿意转述的话。

### 反推链路的断点判定

| 断点 | 常见表现 | 应对动作 |
|---|---|---|
| 痛点断 | 只有“烦”“麻烦”“不行” | 回到原话，补动作和情绪词 |
| 场景断 | 只有“开会的时候”“工作中” | 补 Who / What / When / Where / Why lost |
| 阻碍断 | 只有“效率低”“不好用” | 继续问卡在哪一步、卡给谁、卡多久 |
| 代价断 | 只有“影响很大” | 追问损失的具体量级 |
| 结果断 | 只有“想要更好” | 翻译成可感知状态 |
| 卖点断 | 只有产品术语 | 改成用户语言，再检查能否转述 |

### 证据补采顺序

1. 先补原话。
2. 再补行为。
3. 再补对比。
4. 再补数据。
5. 最后才考虑把假设放进待验证池。

### 反向卖点的优先级

- 先找高频痛点。
- 再找高损失痛点。
- 再找产品最能接住的痛点。
- 再找最容易被转述的表达。
- 再找最容易在当前场景成立的表达。

### 常见错位

- 把痛点写成“用户想要的功能”。
- 把代价写成品牌判断。
- 把结果写成抽象愿望。
- 把卖点写成产品说明。
- 把一次抱怨写成普遍结论。

### 额外样例一：流程卡点

- 原话：我不是不想做，就是每次做到一半就停住了。
- 场景：内容审核、改稿、发版前。
- 阻碍：没有统一标准，也不知道改到什么程度算过。
- 代价：反复返工，时间被吃掉，最后还要别人兜底。
- 结果：每次都能一次过，不用反复来回确认。
- 卖点：一眼知道该改什么、改到哪一步算完。

### 额外样例二：信任卡点

- 原话：我怕买了之后又踩坑。
- 场景：对比多个方案，最后准备下单前。
- 阻碍：之前被类似产品坑过，没办法判断真假。
- 代价：拖延、错失机会、信任感下降。
- 结果：不用赌，自己就能看出值不值。
- 卖点：把能验证的地方都摆出来，不让用户靠猜。

### 额外样例三：体面卡点

- 原话：我不想在老板面前显得很乱。
- 场景：周会、汇报、客户沟通。
- 阻碍：信息分散，临时拼凑，总会漏。
- 代价：显得不专业、解释成本高、压力大。
- 结果：汇报时说得清、拿得出、看起来稳。
- 卖点：让用户在关键场景里显得更专业。

## 反推输出骨架

- 核心痛点
- 具体场景
- 阻碍点
- 代价
- 结果翻译
- 可写卖点
- 待验证池
- 证据缺口

## 反推时最容易犯的错

- 先写卖点，再倒找理由。
- 用品牌视角代替用户视角。
- 把情绪当证据。
- 把愿望当结果。
- 把“可能有用”当“已经成立”。

## 反推卖点的复核问题

- 这是不是用户真的在受的罪。
- 这是不是重复发生的卡点。
- 这是不是产品能接住的层级。
- 这是不是用户愿意复述的话。
- 这是不是足够具体到能写进正文。
