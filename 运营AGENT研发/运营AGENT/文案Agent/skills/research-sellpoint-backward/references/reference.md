# 反向卖点参考资料

## 核心方法论

### 反向卖点的本质

用户不是先理解产品再购买，用户通常是先被一个现实问题折磨，再去找“能不能把这事解决掉”的方案。所以反向卖点的链路不是“功能 -> 卖点”，而是“受不了 -> 为什么受不了 -> 卡在哪 -> 代价是什么 -> 想变成什么 -> 产品能不能接住”。

```
痛点（我受不了）→ 场景（每次XX的时候）→ 阻碍（但XX做不到）→ 代价（所以我损失了XX）
                                       ↓
                                结果（我想变成XX）
                                       ↓
                             卖点表达（正好能帮我XX）
```

这条链只要断一环，卖点就会漂。

### 什么时候用这一步

- 只有用户痛点、评论、抱怨、需求描述，没有完整产品资料。
- 用户明确在说“不行”“太麻烦”“总是漏”“不敢”“怕出错”。
- 正向卖点证据很弱，需要先从用户语言倒推产品该怎么讲。

### 什么时候不要用

- 已经有足够完整的产品资料，优先走正向卖点。
- 连痛点原话都没有，只有模糊需求。
- 用户只给出一个想法，没有任何卡点、场景、代价信息。

## 输入证据质量排序

| 证据来源 | 质量 | 为什么 |
|---|---|---|
| 用户录音 / 逐字稿 | 最高 | 有原话、语气、重复词 |
| 客服 / 售后工单 | 很高 | 有具体问题和损失场景 |
| 差评 / 评论 | 很高 | 用户愿意主动表达真实不满 |
| 社媒评论 | 中高 | 情绪强，但要交叉验证 |
| 访谈记录 | 中高 | 可以追问，但要防分析师偏见 |
| 问卷开放题 | 中 | 量大但浅，需要聚类 |
| 搜索词 | 中低 | 说明有需求，不说明结果 |
| 竞品评论区 | 低到中 | 只能做间接参考 |
| 推断 / 假设 | 最低 | 只能进待验证池，不能直接当结论 |

## 三层追问法

### 第一层：补场景

目标：让痛点从一句情绪变成一个具体事件。

常用追问：

- 这事通常发生在什么时候？
- 是谁在做这件事？
- 这个问题最常出现在什么任务里？
- 是一次性事件还是反复发生？
- 发生前用户做过什么动作？

### 第二层：补阻碍

目标：让“为什么做不到”具体化。

常用追问：

- 用户卡在哪一步？
- 是工具不支持，还是流程没设计好？
- 是不会做，还是不敢做？
- 是信息不够，还是判断标准不清？
- 如果给他一个更好的工具，为什么现在还迈不过去？

### 第三层：补代价和结果

目标：把“受不了”翻译成“损失什么”和“想变成什么”。

常用追问：

- 这问题不解决，会损失什么？
- 每次发生，最直接的代价是什么？
- 一个月下来会积累成什么问题？
- 如果问题被解决，用户最想变成什么状态？
- 这状态能不能被产品真的接住？

## 六步链

```
第1步：痛点 → 第2步：场景 → 第3步：阻碍 → 第4步：代价 → 第5步：结果 → 第6步：卖点
```

任何一步缺失，都不能硬往下写。

### 第1步：痛点提炼

**做什么**

- 保留用户原话，先不翻译。
- 先抓最强烈的抱怨词、重复词、情绪词。
- 把多个痛点拆开，不要硬揉成一句。

**怎么看**

- 有动作、有重复场景、有情绪词，才算有效痛点。
- 只有“不好用”“太麻烦”，还不够。
- 把“想要”和“被卡住”分开看。

**典型误判**

- 把情绪当痛点本身。
- 把抽象抱怨当具体问题。
- 把一次情绪波动当普遍痛点。

### 第2步：场景还原

**做什么**

- 补齐谁、在什么任务里、何时、何地、前后发生了什么。
- 场景不是大背景，而是具体动作窗口。

**怎么看**

- 太粗：上班的时候。
- 太细：某天 10:47。
- 适中：周例会后整理任务、临时汇报前找数据、通勤中单手操作时。

**必须包含**

- Who：谁在用。
- What-task：在做什么任务。
- When：什么时候发生。
- Where：什么场景里发生。
- Broken point：卡在哪一步。
- Lost value：丢了什么。

### 第3步：阻碍识别

**做什么**

- 解释“为什么现在跨不过去”。
- 阻碍不是原因分析报告，而是眼前的卡点。

**阻碍类型**

| 类型 | 说明 | 例子 |
|---|---|---|
| 能力阻碍 | 不知道怎么做 | 不会做数据分析 |
| 工具阻碍 | 现有工具不支持 | 看不到汇总视图 |
| 流程阻碍 | 流程里没这一步 | 会后没人统一跟进 |
| 资源阻碍 | 没时间、没人、没预算 | 手工整理太耗人力 |
| 信息阻碍 | 不知道什么是对的 | 不知道标准在哪里 |
| 心理阻碍 | 不敢做、不想做 | 怕改错、怕背锅 |

**判定重点**

- 只要写“效率低”就不算阻碍。
- 只有“没有系统”也不够，要说清楚是系统的哪一层没接住。

### 第4步：代价估算

**做什么**

- 估算如果不解决，这个阻碍会持续造成什么损失。
- 代价必须具体、可感知、可量化。

**代价分层**

| 层级 | 可信度 | 例子 |
|---|---|---|
| 直接损失 | 最高 | 漏单、延期、返工、丢钱 |
| 间接损失 | 很高 | 协作摩擦、信任下降 |
| 机会损失 | 中高 | 本来可以做别的事但没空 |
| 心理代价 | 中低 | 焦虑、烦、怕被问责 |

**写法规则**

- 不能写“很严重”。
- 要写“一次损失多少”“每月浪费多少”“会被谁问什么”。
- 心理代价不能单独做主卖点，必须配合直接或间接损失。

### 第5步：结果翻译

**做什么**

- 把“不要什么”翻译成“想变成什么状态”。
- 这一步不是写方案，而是写用户语言。

**结果写法**

- 不要写：提升效率、优化流程、改善体验。
- 要写：开会就知道谁做什么、打开页面就看到所有数据、出门不用检查第二遍。

**判断重点**

- 结果必须是可感知状态。
- 结果必须用户自己能验证。
- 结果必须能落回产品能力。

**如果落不回去**

- 放进待验证池，不要硬写成结论。

### 第6步：卖点表达

**做什么**

- 把结果翻译成用户愿意听、愿意信、愿意转述的话。
- 语气要像用户说出来的，不像产品宣传稿。

**卖点三要素**

1. 用户能感知。
2. 用户能验证。
3. 用户能转述。

**卖点分类**

| 类型 | 特征 | 适用 |
|---|---|---|
| 功能型 | 能做什么 | 新用户理解 |
| 信任型 | 凭什么信 | 解决疑虑 |
| 效率型 | 少多少成本 | 成本敏感 |
| 身份型 | 让用户显得怎样 | 体面、专业、掌控感 |

## 证据恢复机制

证据不够时，按这个顺序补：

1. 原话。
2. 多条独立反馈。
3. 重复出现的情境。
4. 行为变化。
5. 竞品比较。
6. 搜索词和问题词。
7. 如果还是不够，就标待验证池。

## 证据缺口处理

格式建议：

- [证据缺口] 频率：只有 1 条原话，缺多来源印证。
- [证据缺口] 量化：只有“很麻烦”，缺具体损失。
- [证据缺口] 行为：知道用户想要，但还没看到实际使用或转化。

## 待验证池

凡是现在不能直接变成主卖点、但方向合理的内容，都放这里。

格式：

- [待验证] 方向：减少焦虑 | 假设：自动提醒能降低漏项 | 需要验证：漏跟进频率
- [待验证] 方向：体面感 | 假设：汇报时更稳能降低问责压力 | 需要验证：用户是否真在意这个场景

## 完整案例

### 案例 1：项目管理工具

**输入**

- 用户原话：“每次项目一多，我就怕漏掉要跟进的事。”
- 补充反馈：“会后经常忘记谁负责什么。”

**六步还原**

1. 痛点：漏跟进、怕漏事。
2. 场景：周会后，每个人都带着不同任务散开。
3. 阻碍：责任人只停留在口头认领，信息没有落到统一待办。
4. 代价：漏跟进、延期、被追问、反复@人。
5. 结果：开完会就知道谁做什么、什么时候交付。
6. 卖点：会议结束后自动长出待办，不用靠记忆。

**证据缺口**

- 需要验证漏跟进频率。
- 需要验证自动生成待办是否真能被现有能力接住。

### 案例 2：保温杯

**输入**

- 用户原话：“杯子漏水，包里东西都湿了。”

**六步还原**

1. 痛点：漏水。
2. 场景：通勤途中，杯子放在包侧袋里。
3. 阻碍：杯盖不够稳，倒置时会渗水。
4. 代价：包里的东西湿掉、文件受损、路上心烦。
5. 结果：扔包里不用管，倒了也不漏。
6. 卖点：倒置也不漏，往包里一塞就走。

### 案例 3：男士袜子

**输入**

- 用户原话：“穿一天脚就臭，跑步更严重。”

**六步还原**

1. 痛点：脚臭、闷、磨脚。
2. 场景：久站、通勤、跑步。
3. 阻碍：普通袜子不排汗，支撑不够。
4. 代价：不敢脱鞋、运动后起泡、被别人提醒脚臭。
5. 结果：穿一天脚不臭，跑步也不累。
6. 卖点：开完会敢脱鞋，不用担心味道。

## 常见误判清单

1. 把情绪当证据。
2. 把抱怨当需求。
3. 把价格异议当唯一痛点。
4. 把竞品黑评当己方机会。
5. 把沉默当同意。
6. 把新手反馈当全部用户反馈。
7. 把问卷结果当定性结论。

## 术语对照表

| 术语 | 定义 | 常见混淆 |
|---|---|---|
| 痛点 | 用户在具体场景里反复受的罪 | 和痒点混淆 |
| 场景 | 痛点发生的具体情境 | 和画像混淆 |
| 阻碍 | 现在卡住的坎 | 和原因混淆 |
| 代价 | 不解决会造成的损失 | 和痛点本身混淆 |
| 结果 | 用户想变成的状态 | 和解决方案混淆 |
| 卖点 | 用户能感知并愿意转述的表达 | 和产品功能混淆 |
| 证据缺口 | 目前缺失且不能自证的部分 | 和待验证假设混淆 |

## 反推卖点的深挖手册

### 什么时候先停

- 只有“烦”“麻烦”“不行”，没有动作和场景。
- 代价只有“严重”，没有损失量级。
- 场景只能写成“上班的时候”。
- 结果落不回产品能力。
- 任何一步卡住，都不要硬写卖点。

### 什么时候可以往下走

- 能说出具体谁在什么任务里卡住。
- 能说出卡点是什么。
- 能说出不解决会损失什么。
- 能说出用户想变成什么状态。
- 能把结果翻译成一句用户愿意转述的话。

### 反推链路断点判定

| 断点 | 表现 | 动作 |
|---|---|---|
| 痛点断 | 只有情绪词 | 回到原话补动作和情绪 |
| 场景断 | 只有宽泛描述 | 补 Who / What / When / Where |
| 阻碍断 | 只有效率低 | 继续问卡在哪一步 |
| 代价断 | 只有很严重 | 追问损失量级 |
| 结果断 | 只有想要更好 | 翻译成可感知状态 |
| 卖点断 | 只有产品术语 | 改成用户语言再看能否转述 |

## 反向输出骨架

- 核心痛点
- 具体场景
- 阻碍点
- 代价
- 结果翻译
- 可写卖点
- 待验证池
- 证据缺口
- 不能写进正文的点

## 绝对不能

- 不能先写卖点，再倒找理由。
- 不能用品牌视角代替用户视角。
- 不能把愿望当结果。
- 不能把卖点写成产品说明。
- 不能把一次抱怨写成普遍结论。
- 不能跳过场景、阻碍、代价直接收敛卖点。
