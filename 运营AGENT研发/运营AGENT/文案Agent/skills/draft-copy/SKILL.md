# 文案初稿生成器

## 作用

生成可用初稿，不伪装成最终稿。

这个 Skill 只负责把上游已定的信息写成第一版可读内容，不替代用户、卖点、场景和结构判断。

## 启动前必须确认

- 主写对象已经明确。
- 主卖点已经明确。
- 场景 / 触点已经明确。
- 结构已经明确。
- 至少有 1 类证据可用。
- 主动作已经明确。
- 需要保留的限制条件已经明确。

以上任意一项缺失，就先回上游 Skill，不进入正文。

## 生成顺序

生成顺序是内容组织的逻辑，不是写作时间顺序。实操时先在笔记里把六个要素都填好，再按顺序组装。

1. **主张**：只放最核心的判断句。判断句 = 有主语 + 有结论 + 用户能感知结果。口号不是主张。
   - 主张错误例子："专注、专业、匠心"（无主语无结果）
   - 主张正确例子："用了X，3个月内回款速度提升40%"（有主语有结果可感知）

2. **理由**：解释为什么这个主张成立。只写"因为..."结构，不写证据，不写动作。
   - 理由错误例子："因为我们很专业"（用结论当理由）
   - 理由正确例子："因为我们拆解了X个环节，发现回款慢的核心在Y节点"（有逻辑链路）

3. **证据**：证明理由和主张的事实。证据优先级：数据 > 案例 > 过程 > 形容词。
   - 证据错误例子："客户反馈很好"（无具体数据）
   - 证据正确例子："已服务XX个客户，平均回款周期从X天缩短到Y天"

4. **限制**：告诉用户适用条件和边界。不要把限制当负面信息，限制说清楚了信任感更强。
   - 必须写清的边界：适用行业/规模/场景、起效前提、不适用情况。
   - 不能写的边界：把限制删掉来让文案更顺。

5. **行动**：只保留一个主动作。动作必须具体到"做什么+在哪里做+什么时候做"。
   - 动作错误例子："欢迎了解更多信息"（太模糊）
   - 动作正确例子："扫码填写表单，顾问会在24小时内联系你"

6. **缺口**：把还没确认的信息单独列出。
   - 缺口不能硬补：没有数据就说"数据待补充"，不能编数字。
   - 缺口处理方式：标注"待与客户确认"，不让缺口流入正文。

## 写作骨架

### 短触点

- 主张：一句话给判断。
- 利益点：只保留一个用户结果。
- 证据：最多一个最强证据。
- 动作：只保留一个动作。

### 中触点

- 场景：先告诉用户现在为什么要看。
- 主张：再给核心判断。
- 理由：说明为什么成立。
- 证据：给 1-2 个可感知证据。
- 动作：收口到 1 个动作。

### 长触点

- 场景。
- 解释。
- 主张。
- 证据链。
- 反对意见。
- 限制条件。
- 动作。

## 句群规则

句群是初稿质量的基本单元。一段写得烂，整篇都救不回来。

- **一段只解决一个问题**：如果一段里同时出现了问题定义、原因分析、方案描述，就是三段内容压成了一段。拆开。
- **一段只保留一个主动作**：动作多了等于没有动作。如果一段里有"扫码"和"加微"两个动作，删掉一个。
- **需要解释时，把解释放在结论后面**：先给结论，用户不认同自然会走；先绕圈解释，用户根本没耐心看到结论。
  - 错误顺序："这是因为我们从X维度做了Y分析，所以能Z效果"（结论埋最后）
  - 正确顺序："Z效果（结论）。原因是X维度的Y分析（解释）。"（结论前置）
- **需要证据时，优先放事实而非形容词**：事实=数字+场景+过程；形容词=专业、高效、优质、领先。
  - 错误例子："我们的团队非常专业，服务过很多优质客户"
  - 正确例子："团队核心成员有X年行业经验，服务过XX家客户，YY%续约率"
- **如果内容太长，先拆成层次**：用数字序号或段落分隔，不要压成一坨。每层只推进一个信息点。
- **如果同一句里同时出现结论、证据和CTA，优先拆开**：三种信息混在一起，读者不知道该先消化哪个。
  - 错误例子："XX产品让你效率提升（结论），已帮助X用户（证据），马上扫码购买（CTA）"
  - 正确做法：拆成三句或三段，每段只推进一个信息类型。

## 平台化写法

每个平台有不同的入口逻辑和阅读节奏。初稿必须先服从平台的入口要求，再展开内部结构。

### 海报 / 广告位

- 入口：用户第一眼只看标题或主标语。
- 内部：只放一个主张 + 一个最强证据 + 一个动作。
- 写法：主张必须有判断、有结果感；不能是口号或形容词。
- 常见错：把"专业可靠值得信赖"当主张 → 改为"X年服务X个客户，XX%续约率"类事实主张。
- 字数：标题不超过12字，正文不超过20字。

### 抖音

- 入口：前3秒必须有可感知入口（数字/场景/冲突/疑问）。
- 内部结构：钩子 → 冲突/痛点 → 结果/证明 → 动作。
- 写法：对话感强，少用书面语；证据用过程感而非结论感。
- 常见错：上来就说结论"我们的产品很好" → 改为"你有没有遇到过XXX问题"。
- 时长适配：15秒/30秒/60秒版本的内容密度不同，初稿先写30秒基准版，标注可压缩点。

### 小红书

- 入口：封面标题先争眼球，正文前30字决定用户是否继续看。
- 内部结构：场景 → 体验 → 细节 → 对比 → 动作。
- 写法：少用"我们"，多用"我/你"；少用内部术语，多用用户语言。
- 常见错：用产品参数代替用户结果 → 改为"用了之后的变化"描述。
- 标签暗示：初稿要标注适合打哪类标签，便于后续SEO优化。

### 公众号

- 入口：标题决定打开率，前三段决定是否继续看。
- 内部结构：解释 → 证明 → 推进（或：主张 → 证据链 → 限制 → 动作）。
- 写法：允许完整论证，适合把限制条件和FAQ放到后段。
- 常见错：把全文写成产品说明书 → 改为"问题-方案-结果"三段式。
- 字数：公众号有阅读节奏感要求，初稿标注每段字数，便于压缩或展开。

### 详情页

- 入口：首屏必须在3秒内给出核心判断，否则用户直接划走。
- 内部结构：判断 → 证据 → 反对意见 → 限制 → 动作。
- 写法：每个证据段落必须有具体数字或可感知描述；反对意见段落必须诚实。
- 常见错：首屏说"我们致力于..." → 改为"针对X问题，我们的方案是X"。
- 移动端注意：详情页大量流量来自手机，首屏和第二屏必须能独立成立。

## 交付骨架

- 文案初稿
- 可删改段落
- 待确认信息
- 结构说明
- 风险 / 限制
- 主动作

## 新人校验

- 新人能不能看出这是第一版，不是最终稿。
- 新人能不能知道缺口应该单独列出来。
- 新人能不能知道一个稿子只保留一个主动作。
- 新人能不能按平台调整入口和展开方式。

## 绝对不能

- 不能在卖点、场景、结构不清时直接写成稿。
- 不能把口号当正文。
- 不能把缺口当作已完成内容。
- 不能在短触点里塞长解释。
- 不能写多个互相打架的 CTA。
- 不能把边界条件删掉来换顺口。
