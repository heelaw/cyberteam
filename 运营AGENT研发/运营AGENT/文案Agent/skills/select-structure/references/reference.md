# 结构选择

## 适用场景

- 卖点和用户已经清楚，但结构未定。
- 需要在多种结构中做选择，而不是机械套固定结构。
- 需要根据平台、阅读窗口、证据深度和用户状态做写法分流。

## 先看四个输入

1. 用户是否知道问题。
2. 用户是否在比较方案。
3. 用户是否需要解释或信任建立。
4. 用户距离行动有多近。

## 结构决策树

### 第 1 层：用户认知阶段

- 不知道问题 → `4P` 或 `故事式`。
- 知道问题但没被说服 → `PAS` 或 `AIDA`。
- 已经在比较方案 → `对比式`。
- 已经接近行动 → `直述式`。
- 信息很多但用户只会扫读 → `清单式`。

### 第 2 层：证据深度

- 证据弱但需要先唤醒 → 先用 `PAS` 或 `AIDA`。
- 证据强且对比明确 → `对比式`。
- 证据强且要完整解释 → `4P` 或长版故事式。
- 证据少但动作强 → `直述式` 要保守用。

### 第 3 层：平台限制

- 平台很短 → 先 `直述式` 或短版 `AIDA`。
- 平台允许展开 → `4P`、`故事式`、`PAS` 可展开。
- 平台要求扫描 → `清单式`。

## 结构库

| 结构 | 核心逻辑 | 入口条件 | 禁忌 |
|---|---|---|---|
| AIDA | 注意→兴趣→欲望→动作 | 用户已有注意但未被吸引 | 用户已经很热时用 AIDA 显得绕 |
| 4P | 描绘→承诺→证明→敦促 | 用户不知道问题 | 没有清楚痛点时会显得假 |
| PAS | 痛点→放大→解决 | 用户知道问题但没被说服 | 没有痛点时像在制造焦虑 |
| 直述式 | 结论→理由→动作 | 用户已接近行动或内容极短 | 复杂价值需要解释时不适合 |
| 对比式 | 维度→本方案优势→为什么重要 | 用户在比较多个方案 | 没有竞品参照时显得空 |
| 故事式 | 场景→变化→结果 | 用户需要理解和共情 | 极短触点不适用 |
| 清单式 | 信息模块→独立可读→一个动作收口 | 信息多且用户扫读 | 需要强情绪推动时不适合 |

## 每种结构怎么写

### AIDA

- 注意：抓住眼球。
- 兴趣：把和用户有关的点说出来。
- 欲望：把结果和收益说出来。
- 行动：只给一个动作。

### 4P

- 描绘：先把场景立住。
- 承诺：再说你能给什么。
- 证明：再给证据。
- 敦促：最后推进动作。

### PAS

- 痛点：先把问题摆出来。
- 放大：再说不解决的代价。
- 解决：最后给方案。

### 直述式

- 先结论，再理由，再动作。
- 适合短场景和强确定性内容。
- 不能堆背景。

### 对比式

- 先列比较维度。
- 再列本方案的优势。
- 再说明为什么这些维度重要。

### 故事式

- 先搭场景。
- 再推进变化。
- 再回到结果和动作。

### 清单式

- 把信息拆成可扫读模块。
- 每条都能独立理解。
- 最后只保留一个动作。

## 选择顺序

1. 先判断用户是否知道问题。
2. 再判断是不是在比较方案。
3. 再判断是不是接近行动。
4. 再看证据深度。
5. 再看平台长短。
6. 再定主结构和备用结构。

## 备用结构规则

- 主结构太长时，准备一个短版备用结构。
- 主结构需要对比但证据不足时，备用结构用 `直述式` 降级。
- 主结构是 `AIDA` 时，备用结构通常是 `直述式`。
- 主结构是 `故事式` 时，备用结构通常是 `PAS` 或 `清单式`。

## 排除理由模板

- 不选 `PAS`，因为用户还不知道问题，不适合先上痛点。
- 不选 `对比式`，因为没有明确竞品或替代方案。
- 不选 `故事式`，因为触点只有 3-8 秒，没有叙事空间。
- 不选 `4P`，因为用户已经接近行动，不需要再搭问题框架。

## 平台优先级

- 短触点先平台，再结构。
- 长触点先证据，再结构。
- 如果平台很短，结构必须服从长度，不许把内容撑爆。
- 如果平台允许展开，结构要把证据和反对意见放对位置。

## 典型例子

### 例子 1：新问题教育

- 用户还不知道问题。
- 选 `4P`。
- 先场景，再承诺，再证明，再动作。

### 例子 2：痛点共鸣

- 用户知道问题但还没被打动。
- 选 `PAS`。
- 先痛点，再放大，再解决。

### 例子 3：方案比较

- 用户在对比多个选项。
- 选 `对比式`。
- 先比较维度，再说本方案更好的地方。

### 例子 4：快速行动

- 用户已经准备下单。
- 选 `直述式`。
- 直接给结果和动作。

## 结构误选修正

### 场景 1：用户想知道问题，但内容被写成 PAS

- 现象：开头就放痛点，用户还没进入问题框架。
- 修正：换成 `4P` 或 `故事式`，先把问题和场景立住。

### 场景 2：用户已经比较方案，却写成故事式

- 现象：讲了很多经历，但没有比较维度。
- 修正：改成 `对比式`，先列维度再说优势。

### 场景 3：用户已经接近行动，却还在讲 AIDA

- 现象：用户已经想做了，却被拉去看完整铺垫。
- 修正：改成 `直述式`，先结论后动作。

### 场景 4：信息太多却不用清单式

- 现象：一堆信息压成一段，用户扫不过来。
- 修正：改成 `清单式`，每条独立成立。

## 结构选择的补判问题

- 用户现在是“要不要看”，还是“看哪个更好”，还是“现在就做”？
- 这次最需要的是共鸣、解释、比较，还是直接动作？
- 证据够不够支撑完整论证？
- 平台够不够承载这套结构？

## 结构输出模板

- 主结构：
- 备用结构：
- 选择理由：用户状态 + 说服目标 + 证据深度 + 平台
- 排除理由：至少 2 条
- 写作顺序：主结构三步，备用结构三步
- 压缩点：如果平台很短，删哪里、前置哪里


## 失败案例

### 案例 1：先喜欢结构再套

- 原文：我就想用 AIDA。
- 问题：没有依据，只是偏好。

### 案例 2：顺序写反

- 原文：先证明，再讲问题，再讲承诺。
- 问题：用户还没进入问题框架，证据没有入口。

### 案例 3：该比较却只讲口号

- 原文：我们更懂你。
- 问题：没有比较维度。

### 案例 4：该解释却只给动作

- 原文：马上试试。
- 问题：用户还没理解为什么要试。

## 边界条件

- 如果一个结构能用但不够强，可以保留备用结构。
- 如果平台限制极强，先保留最短入口，再决定是否补正文。
- 如果证据很强，但结构很弱，先换结构，不要硬抄顺序。
- 如果用户状态介于两种结构之间，优先选更短、更稳的那个。

## 输出骨架

- 结构名称
- 选择理由
- 写作顺序
- 备用结构
- 排除理由
- 平台适配提示
- 证据深度提示

## 红线

- 不能只报结构名不报理由。
- 不能把主结构和备用结构混写成一团。
- 不能忽略用户状态和证据深度。
- 不能让结构决定一切，忽略平台。
- 不能用户还没认知问题就上 `PAS`。
