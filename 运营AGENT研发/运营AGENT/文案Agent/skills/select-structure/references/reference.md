# 结构选择

## 适用场景

- 卖点和用户已经清楚，但结构未定。
- 需要在多种结构中做选择，而不是机械套固定结构。
- 需要根据平台、阅读窗口、证据深度和用户状态做写法分流。

## 先看四个输入

1. 用户是否知道问题。
2. 用户是否在比较方案。
3. 用户是否需要解释或信任建立。
4. 用户距离行动有多近。

## 结构库

| 结构 | 适合什么 | 不适合什么 |
|---|---|---|
| AIDA | 先抓注意，再层层推进 | 用户已经很热、只差一句话 |
| 4P | 先问题，再放大，再承诺，再证明 | 没有清楚痛点时 |
| PAS | 痛点明确、需要快速共鸣 | 需要大量证据的长决策 |
| 直述式 | 短场景、强明确诉求 | 复杂价值、需要解释时 |
| 对比式 | 用户在比较方案 | 没有竞品或替代方案时 |
| 故事式 | 需要建立理解和认同 | 极短触点 |
| 清单式 | 需要快速扫描和执行 | 需要强情绪推动时 |

## 选择逻辑

### 1. 用户还不知道问题

- 优先 `4P` 或 `故事式`。
- 先搭场景和问题框架，再进入说服。
- 不能一上来就用 `PAS`。

### 2. 用户知道问题但还没被打动

- 优先 `PAS` 或 `AIDA`。
- 如果需要入口，就先钩子；如果需要共鸣，就先痛点。

### 3. 用户正在比较方案

- 优先 `对比式`。
- 必须先写比较维度，再写本方案优势。
- 不能只写“我们更好”。

### 4. 用户已经接近行动

- 优先 `直述式`。
- 直接给结果、好处和动作。
- 不适合长解释和复杂铺垫。

### 5. 信息很多但用户只会扫读

- 优先 `清单式`。
- 每一条都要独立成立。
- 不要把段落写成流水账。

## 每种结构怎么写

### AIDA

- 注意：抓住眼球。
- 兴趣：把和用户有关的点说出来。
- 欲望：把结果和收益说出来。
- 行动：只给一个动作。

### 4P

- 描绘：先把场景立住。
- 承诺：再说你能给什么。
- 证明：再给证据。
- 敦促：最后推进动作。

### PAS

- 痛点：先把问题摆出来。
- 放大：再说不解决的代价。
- 解决：最后给方案。

### 直述式

- 先结论，再理由，再动作。
- 适合短场景和强确定性内容。
- 不能堆背景。

### 对比式

- 先列比较维度。
- 再列本方案的优势。
- 再说明为什么这些维度重要。

### 故事式

- 先搭场景。
- 再推进变化。
- 再回到结果和动作。

### 清单式

- 把信息拆成可扫读模块。
- 每条都能独立理解。
- 最后只保留一个动作。

## 平台优先级

- 短触点先平台，再结构。
- 长触点先证据，再结构。
- 如果平台很短，结构必须服从长度，不许把内容撑爆。
- 如果平台允许展开，结构要把证据和反对意见放对位置。

## 排除规则

- 不能先喜欢结构再硬套。
- 不能结构顺序写反。
- 不能用户还没认知问题就直接上 `PAS`。
- 不能用户在比较时还写成纯口号。
- 不能用户已接近行动却继续铺背景。

## 典型例子

### 例子1：新问题教育

- 用户还不知道问题。
- 选 `4P`。
- 先场景，再承诺，再证明，再动作。

### 例子2：痛点共鸣

- 用户知道问题但还没被打动。
- 选 `PAS`。
- 先痛点，再放大，再解决。

### 例子3：方案比较

- 用户在对比多个选项。
- 选 `对比式`。
- 先比较维度，再说本方案更好的地方。

### 例子4：快速行动

- 用户已经准备下单。
- 选 `直述式`。
- 直接给结果和动作。

## 失败案例

### 案例1：先喜欢结构再套

- 原文：我就想用 AIDA。
- 问题：没有依据，只是偏好。

### 案例2：顺序写反

- 原文：先证明，再讲问题，再讲承诺。
- 问题：用户还没进入问题框架，证据没有入口。

### 案例3：该比较却只讲口号

- 原文：我们更懂你。
- 问题：没有比较维度。

### 案例4：该解释却只给动作

- 原文：马上试试。
- 问题：用户还没理解为什么要试。

## 边界条件

- 如果一个结构能用但不够强，可以保留备用结构。
- 如果平台限制极强，先保留最短入口，再决定是否补正文。
- 如果证据很强，但结构很弱，先换结构，不要硬抄顺序。

## 交接判断

- 用户状态不清：回 `detect-need-type` 或 `segment-users`。
- 证据不够：回卖点 Skill。
- 场景不清：回 `judge-scenario`。
- 已经有结构但想优化：回这里重新做选择。

## 输出骨架

- 结构名称
- 选择理由
- 写作顺序
- 备用结构
- 排除理由
- 平台适配提示
- 证据深度提示

## 绝对不能

- 不能只报结构名不报理由。
- 不能把主结构和备用结构混写成一团。
- 不能忽略用户状态和证据深度。
- 不能让结构决定一切，忽略平台。
