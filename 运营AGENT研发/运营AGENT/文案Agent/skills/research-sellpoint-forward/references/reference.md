# 正向卖点提炼

## 适用场景

- 已有产品资料、功能说明、案例、演示或竞品信息。
- 需要先把“为什么值得看 / 值得买 / 值得点”提炼出来，再进入文案。
- 需要建立一个可复用的卖点池，而不是直接写成品文案。
- 需要判断：这个产品到底靠什么成立，哪些点能讲，哪些点只是装饰。

## 工作原则

1. 先列事实，再列候选。
2. 先补证据，再做承诺。
3. 先看差异，再看感知。
4. 先找主卖点，再找辅助卖点。
5. 先标缺口，再决定要不要继续往下收敛。
6. 七个方向必须全部过一遍，不能只挑顺手的写。
7. 如果某个方向证据很弱，只能降级为辅助候选或待验证项，不能硬升主卖点。
8. 不能把功能名、价格名、品牌口号直接当卖点。

## 七向采样总表

| 方向 | 你要看的不是 | 你真正要找的 | 常见证据 |
|---|---|---|---|
| 功能独特性 | 功能名称 | 和默认方案相比的差异 | PRD、demo、对比表、版本记录 |
| 效果承诺 | 目标口号 | 用后可见的变化 | 数据、案例、前后对比 |
| 解决问题 | 抽象麻烦 | 具体卡点、代价、受阻位置 | 原话、工单、评论、访谈 |
| 用户身份 | 职业标签 | 场景里的身份感、体面感、掌控感 | 成交记录、社群语言、访谈 |
| 价格优势 | 标价 | 综合成本、替代成本、单位成本 | 报价单、竞品页、薪资数据 |
| 服务保障 | 服务态度 | 范围、时效、边界、兑现机制 | SLA、FAQ、协议、承诺页 |
| 情感价值 | 空泛感受 | 可感知、可复述的情绪收益 | 用户故事、评论、品牌表达 |

## 每个方向怎么收集

### 1. 功能独特性

**要做什么**

- 拆功能结构：模块、流程、配置、规则、自动化方式。
- 找对比基准：竞品、人工、传统方式、默认工作流。
- 找差异类型：少一步、快一点、准一点、稳一点、自动一点。
- 找用户在意的理由：为什么这个差异会影响选择。

**怎么采**

- 先看产品资料里的功能清单，不要先写卖点。
- 对每个功能问三句：别人有没有？默认方案有没有？用户为什么在意？
- 如果功能本身不新，就看它的使用方式、效率、结果是否新。
- 如果功能太技术化，必须翻译成用户能感受到的结果。

**判断规则**

- 有明确对比，才算独特性。
- 用户感受不到，就不能放主卖点。
- 能被竞品轻易复制的，优先降级。

**不能做**

- 不能把“有 AI”“支持自动化”当成独特性。
- 不能把功能列表原样搬出来当卖点。
- 不能只写“我们更先进”却没有比较对象。

**小例子**

- 好：自动把会议录音在 3 分钟内转成待办，并同步到协作工具。
- 差：支持 AI 语音转写。
- 好：内置行业术语库，减少人工校对。
- 差：智能识别关键词。

### 2. 效果承诺

**要做什么**

- 找最终结果：效率、时间、成本、收入、转化、错误率。
- 找不到最终结果，就找中间结果：更快、更稳、更少出错。
- 再找不到，就找过程结果：少一步、少一次切换、少一次重做。
- 把“用了以后 X 变了”写出来，不写功能名。

**怎么采**

- 先看数据报表、案例数字、前后对比。
- 再看用户反馈中是否有“用了之后”“之前 vs 之后”的表达。
- 再看是否有时间范围、样本范围、场景范围。
- 任何承诺都要能回到证据，不然只能算候选。

**判断规则**

- 量化优先，但不是必须数字才成立；没有数字就必须标明是候选。
- 一次性案例不能直接变成普遍结论。
- 结果必须能被用户自己验证，而不是只能听你说。

**不能做**

- 不能把“目标”写成“结果”。
- 不能把“更高效”“更好”当结果。
- 不能拿跨行业数据直接当本产品数据。

**小例子**

- 好：把日报整理时间从 40 分钟压缩到 5 分钟。
- 差：提高工作效率。
- 好：使用 3 个月后留存率提升 22%。
- 差：让用户更满意。

### 3. 解决问题

**要做什么**

- 收集用户原话，先不翻译。
- 还原问题发生的场景：谁、做什么、卡在哪一步、损失什么。
- 把问题拆成三层：业务层、操作层、心理层。
- 找到这个问题的真实代价。
- 验证产品能接住哪一层，接不住的不要写进主卖点。

**怎么采**

- 优先用用户原话、客服工单、评论、访谈逐字稿。
- 同一痛点最好有多条独立来源。
- 把“想要”和“被卡住”分开。
- 把“麻烦”继续追问成具体动作和损失。

**判断规则**

- 必须有场景，不能只剩抽象抱怨。
- 必须有代价，不能只说“很严重”。
- 产品必须能接住至少一层阻碍，否则只能作为待验证项。

**不能做**

- 不能把用户情绪直接当证据。
- 不能把“用户想要”误判成“用户被卡住”。
- 不能把产品解决不了的问题硬写进去。

**小例子**

- 好：每次开完会都漏任务，散会后没人知道谁负责什么。
- 差：企业沟通效率低。
- 好：每月要花 6 小时手工拼数据，汇报前还经常对不上。
- 差：数据管理麻烦。

### 4. 用户身份

**要做什么**

- 找核心角色：谁使用、谁决策、谁受益。
- 找身份场景：用户在什么身份里使用产品。
- 找身份收益：体面、专业、掌控感、效率感、责任感。
- 找用户自己的表达方式，而不是用你的职业标签替代。

**怎么采**

- 看成交记录里的用户自述。
- 看社群、评论、访谈里用户怎么介绍自己。
- 找“我作为谁，需要什么结果”的句子。
- 多角色产品必须拆开写，不要混成一个身份。

**判断规则**

- 职业标签不等于身份。
- 身份卖点必须和产品能力匹配。
- 只有在场景里成立的身份，才值得进正文。

**不能做**

- 不能写“适合所有人”。
- 不能写“专为精英打造”这种空标签。
- 不能把身份词写得比产品能力还大。

**小例子**

- 好：给第一次带团队的管理者一套可直接照着做的方案。
- 差：适合中小企业主。
- 好：让用户在客户面前说得清、拿得出、显得稳。
- 差：提升个人品牌。

### 5. 价格优势

**要做什么**

- 先定比较对象：竞品、人工、外包、时间成本、试错成本。
- 再算综合成本：标价、周期、单位成本、隐性成本。
- 再看划算感是否明显。
- 再确认便宜是否建立在有效结果之上。

**怎么采**

- 读竞品公开价格页。
- 读替代方案的人力或时间成本数据。
- 读用户反馈中“原来花了多少”的描述。
- 读 ROI 计算器、报价单、套餐页。

**判断规则**

- 不是“便宜”就行，必须是“相对什么更划算”。
- 公平比较很重要，不能拿促销价对竞品正价。
- 便宜但没效果，不算真正优势。

**不能做**

- 不能把限时优惠当长期卖点。
- 不能只讲低价，不讲效果。
- 不能把估算数字写成事实。

**小例子**

- 好：同等效果下，比外包省 60% 预算。
- 差：限时 8 折。
- 好：一次付费，折算到每天不到 1 元。
- 差：价格实惠。

### 6. 服务保障

**要做什么**

- 拆服务要素：范围、时效、边界、支持方式、赔付或调整机制。
- 找用户默认会以为有、但产品没明说的承诺。
- 写清条件，不然就是过度承诺。
- 验证承诺是否真能兑现。

**怎么采**

- 看 SLA、FAQ、售后规则、培训说明、陪跑承诺。
- 看用户评论里最在意的支持点。
- 看团队是否真有能力按承诺执行。

**判断规则**

- 服务承诺必须可执行、可复核、可追责。
- 用户能感知到的服务，才适合进卖点。
- 只有“态度好”不够，必须有明确边界和动作。

**不能做**

- 不能把“服务很好”写成主卖点。
- 不能写超出实际能力的承诺。
- 不能只写合同条款，不写用户能感知的结果。

**小例子**

- 好：7 天内不满意可继续调整，直到通过为止。
- 差：服务很贴心。
- 好：响应时效、边界、责任人都写清楚。
- 差：我们会全程陪伴。

### 7. 情感价值

**要做什么**

- 找用户会转述、会晒、会认同的状态变化。
- 找具体情绪，不找空词。
- 把情绪落回场景。
- 看这个情绪是不是所有竞品都能写。

**怎么采**

- 看用户故事、评论、社群分享。
- 找用户原话里出现的安心、体面、松弛、掌控、被认可。
- 把“我感觉怎样”翻成能被看见的产品效果。

**判断规则**

- 情绪必须有产品路径支撑。
- 不能脱离场景，只写情绪口号。
- 如果所有竞品都能这么说，就不够独特。

**不能做**

- 不能把“高级、幸福、安心”当万能卖点。
- 不能只喊口号，不落场景。
- 不能把品牌自嗨当用户情绪。

**小例子**

- 好：开完会不用担心漏事。
- 差：更安心。
- 好：让用户在关键场景里显得更专业。
- 差：提升品牌温度。

## 选主卖点

1. 先选用户最在意、最容易立刻感知的点。
2. 再选证据最完整、最能讲清楚的点。
3. 再选能带动其他卖点一起成立的点。
4. 如果都差不多，优先选当前场景里最容易说清楚的那个。
5. 如果七个方向都弱，说明资料还不够，不要硬凑主卖点。

## 证据分级

| 层级 | 说明 | 用法 |
|---|---|---|
| ★★★★★ | 直接原话、直接数据、明确对比 | 可做主卖点核心证据 |
| ★★★★ | 多条独立反馈重复出现 | 可做辅助证据 |
| ★★★ | 行为验证、实际转化、实际使用 | 可做辅助或支撑证据 |
| ★★ | 搜索词、问题词、间接信号 | 只能做参考 |
| ★ | 个人推断、假设、猜测 | 只能进待验证池 |

## 证据恢复顺序

1. 找事实。
2. 找结果。
3. 找前后对比。
4. 找用户原话。
5. 找案例或数据。
6. 找边界条件。
7. 还不够，就放进待验证池，不要硬编。

## 输出骨架

- 产品画像
- 七方向候选清单
- 主卖点
- 辅助卖点
- 证据来源
- 证据缺口
- 待验证项
- 不能写进正文的点

## 一条完整样例

### 输入

- 产品：一款能自动整理会议纪要并生成待办的协作工具。
- 可见事实：支持录音转写、待办提取、多人协作、提醒同步。

### 处理

1. 功能独特性：自动把会议内容变成待办。
2. 效果承诺：减少会后整理时间。
3. 解决问题：解决“开完会没人知道要做什么”。
4. 用户身份：适合项目负责人、运营负责人、团队协作人。
5. 价格优势：比人工整理更省时间成本。
6. 服务保障：同步提醒、多人可见、可追溯。
7. 情感价值：让人觉得团队更有掌控感。

### 输出

- 主卖点：会议结束后自动生成待办，避免漏事。
- 辅助卖点：节省整理时间、提醒同步、多人协作。
- 证据缺口：缺少实际节省时长的数据和案例。

## 常见误判

- 把功能描述当卖点。
- 把优惠当卖点，但用户真正买的是结果。
- 把品牌自夸当卖点，但用户根本感知不到。
- 把一次性案例当长期结论。
- 把所有方向平均写一遍，最后没有主次。

## 边界条件

- 如果产品本身没有强差异，就把卖点转向结果、服务或场景。
- 如果证据不足，不要强行收敛成一个卖点，先保留多个候选。
- 如果不同卖点都成立，优先选当前场景里最容易被感知的那个。
- 如果证据链断了，就退回补证据，不要用想象补。

## 回退提示

- 功能事实太少，先补资料。
- 结果证据太少，先补数据或案例。
- 用户反馈太少，先补原话。
- 七向都弱，先回到信息收集，不要强行进入正文。

## 绝对不能

- 不能把功能描述直接当卖点。
- 不能把优惠当长期卖点。
- 不能把品牌口号当证据。
- 不能把一次案例当全部结论。
- 不能在证据不足时强行收敛成唯一卖点。
- 不能跳过七向采样直接给主卖点。
