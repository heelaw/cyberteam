# 用户分层器

## 作用

把“写给谁”拆成可执行的主写对象、辅助对象和不该写的人群。

这一步不是贴标签，而是先弄清谁出钱、谁使用、谁拍板、谁阻拦、谁影响。

## 启动前

- 先确认这是单人决策、买用一体、买用分离，还是委员会决策。
- 如果完全没有人群线索，先回去补原话、行为或对话。

## 判断顺序

1. 先找决策权，再找角色名。
2. 再找阶段、频率和意愿差异。
3. 再看价格敏感度和认知水平。
4. 最后确认主写对象、辅助对象和不该写的人群。

## SOP

1. 先找身份和角色差异。
2. 再找阶段、频率和意愿差异。
3. 识别价格敏感度和认知水平。
4. 判断谁是主写对象、谁是辅助对象。
5. 必要时拆出购买者、使用者、拍板者、阻拦者、影响者。
6. 处理单人产品、买用一体、买用分离、委员会决策和角色冲突。
7. 输出各层诉求、敏感点、主张重点和不该写的人群。

## 需要特别看

- 决策权是否和使用权分离。
- 价格敏感的人和真正拍板的人是不是同一个人。
- 用户会不会在不同阶段说不同的话。
- 一个群体里是否存在显著冲突。

## 输出骨架

- 用户分层表
- 主写对象
- 辅助对象
- 决策链角色
- 各层诉求
- 各层敏感点
- 不该写的人群

## 新人校验

- 新人能不能先找到决策链，再找角色标签。
- 新人能不能分清主对象、辅助对象、阻拦者。
- 新人能不能处理”一个人说了很多身份”的情况。
- 新人能不能知道什么时候该继续补角色，什么时候可以进入卖点。

## 输入要求

启动本Skill前必须确认以下输入已获取：

| 输入项 | 最低要求 | 说明 |
|--------|----------|------|
| 决策链线索 | 至少知道谁付款、谁使用之一 | B2B必须知道拍板人，B2C至少知道购买者 |
| 角色数量 | 至少2个角色线索 | 购买者≠使用者时必须分开 |

**禁止启动条件**（满足任一条不得启动）：
- 完全不知道任何角色信息
- 只有产品信息，没有用户线索
- 只有”面向所有人”这样的泛泛描述

**允许启动但需快速补证的条件**：
- 决策链不清晰 → 先标记缺口，快速补1条决策权证据
- 只有单一角色 → 先确认是否真的是单人决策

## 好样例

### 样例1：买用分离（B2B标准分层）

**输入**：
- 产品：企业培训软件
- 业务场景：负责人要推动项目，使用者每天执行，老板看结果，技术同事看接入成本

**完整输出**：
```
用户分层表：

| 角色 | 身份 | 核心诉求 | 敏感点 | 主/辅 |
|------|------|----------|--------|-------|
| 负责人 | 购买者+推动者 | 推动项目成功、证明投入产出 | 效率、结果可衡量 | 主写对象 |
| 执行者 | 使用者 | 好上手、不增加负担 | 上手成本、操作简便 | 辅助对象 |
| 老板 | 拍板者+预算审批 | 结果和风险可控 | 品牌可信度、投资回报 | 辅助对象 |
| 技术同事 | 阻拦者+接入者 | 安全、可接入现有系统 | 兼容性、技术风险 | 辅助对象 |

主写对象：负责人
选择理由：
- 最接近最终拍板（老板）
- 承担推动项目成功的责任
- 当前卖点（效率提升）最能打动他

辅助对象：执行者、老板、技术同事
分工：
- 执行者：在使用体验补充说明里兼顾
- 老板：在风险和结果证明里兼顾
- 技术同事：在技术可行性和接入成本里兼顾

不该写的人群：
- HR（不参与此次采购决策）
- 外部合作伙伴（无决策权）

各层诉求：
- 负责人：项目成功、效率提升、证明ROI
- 执行者：好用、不增加工作量、好上手
- 老板：品牌可信、风险可控、投资回报
- 技术同事：安全、兼容、不增加运维负担

各层敏感点：
- 负责人：时间成本、学习成本
- 执行者：操作复杂、上手难
- 老板：预算超支、效果达不到
- 技术同事：安全漏洞、集成问题
```

**为什么是好样例**：
- 识别出4类角色（购买者+使用者+拍板者+阻拦者）
- 主写对象选择理由清晰
- 各层诉求和敏感点具体可写
- 不该写的人群明确标注

### 样例2：买用一体（B2C单人决策）

**输入**：
- 产品：个人记账App
- 用户：自己用、自己决定买

**完整输出**：
```
用户分层表：

| 角色 | 身份 | 核心诉求 | 敏感点 | 主/辅 |
|------|------|----------|--------|-------|
| 用户 | 购买者+使用者+决策者 | 管好钱、不乱花 | 简便、不费时间、隐私 | 主写对象 |

主写对象：用户本人
选择理由：
- 唯一决策者，无其他角色干扰
- 买用一体，不需要兼顾他人

产品结构：买用一体（单人产品）

各层诉求：
- 核心：管好钱、不乱花
- 深层：安全感、成就感、自我掌控

各层敏感点：
- 操作复杂（不使用）
- 隐私暴露（不信任）
- 需要花时间学习（懒得用）

不该写的人群：
- 无（单人产品只有直接用户）
```

**为什么是好样例**：
- 正确识别为买用一体
- 无需分层但确认了单一角色
- 敏感点落到具体可写维度（操作/隐私/时间）

### 样例3：委员会决策（多层决策）

**输入**：
- 产品：B端客服系统
- 角色：采购总监（最终拍板）、客服主管（使用者+影响者）、IT负责人（技术把关+阻拦者）、CFO（预算审批）
- 背景：客服主管试用后认可，但IT担心集成，预算需要CFO审批

**完整输出**：
```
用户分层表：

| 角色 | 身份 | 核心诉求 | 敏感点 | 主/辅 |
|------|------|----------|--------|-------|
| 采购总监 | 拍板者 | 性价比、供应商可靠、团队满意 | ROI、供应商稳定性、风险 | 主写对象 |
| 客服主管 | 使用者+推动者 | 提升效率、团队好用 | 上手难度、是否增加工作量 | 辅助对象 |
| IT负责人 | 技术阻拦者 | 安全、可集成、不增加运维 | 技术风险、集成复杂度、安全认证 | 辅助对象 |
| CFO | 预算审批者 | 不超预算、决策合规 | 成本合理性、采购合规 | 辅助对象 |

主写对象：采购总监
选择理由：
- 最终签字拍板
- 需要平衡团队满意度和成本控制
- 当前卖点（效率提升+供应商可靠）最能打动他

辅助对象：客服主管、IT负责人、CFO
分工：
- 客服主管：在效果和上手难度补充说明里出现
- IT负责人：在技术集成和安全认证里出现
- CFO：在成本合理性和ROI证据里出现

决策链优先级：
1. 采购总监（最终拍板）
2. CFO（预算审批）
3. IT负责人（技术否决权）
4. 客服主管（使用者，无否决权但有推动权）

角色冲突处理：
- IT关注安全 vs 采购总监关注效率 → 正文先讲安全和稳定，补充说明里讲效率
- CFO关注成本 vs 客服主管关注功能 → ROI证据里同时回应

不该写的人群：
- 基层客服（不是决策链上的人）
- 外部客户（无决策权）
```

**为什么是好样例**：
- 识别出4类角色并明确决策链
- 主写对象选择理由具体
- 角色冲突处理有具体策略
- 决策链优先级清晰

## 坏样例

### 坏样例1：空泛标签

**输入**：
- 产品：某App

**错误输出**：
```
主写对象：25-35岁女性白领
辅助对象：无
```

**问题分析**：
- 25-35岁女性白领是画像标签，不是角色
- 没有决策权、使用权、拍板者等维度
- 这样的分层无法落到文案写作

**正确输出**：
```
主写对象：有孩子的职场妈妈（孩子教育类App）
选择理由：她既是购买决策者又是主要使用者

或（有具体场景时）：

主写对象：孩子刚上幼儿园的职场妈妈
核心诉求：省时、放心、性价比
敏感点：时间成本、孩子安全、内容质量
```

---

### 坏样例2：所有人都当主对象

**输入**：
- 产品：团队协作工具

**错误输出**：
```
主写对象：所有人
辅助对象：所有人
```

**问题分析**：
- 没有主次等于没有主写对象
- 文案不会聚焦，谁都不被打中
- 违反绝对不能：不能把所有人都写成主对象

**正确输出**：
```
主写对象：团队负责人/项目推动者
选择理由：承担推动项目成功的责任，最关心效率和结果

辅助对象：执行者、老板、技术同事
分工：各补各自关心的那一层，不抢主线
```

---

### 坏样例3：忽略决策权

**输入**：
- 产品：企业软件
- 背景：执行者说”很好用”，但拍板者觉得”不够有高度”

**错误输出**：
```
主写对象：执行者
理由：执行者每天用，最了解产品
```

**问题分析**：
- 执行者不是拍板者，满足执行者不等于成交
- 拍板者的”有高度”可能是否决点
- 违反绝对不能：不能忽略决策权

**正确输出**：
```
主写对象：拍板者（VP或总监）
选择理由：最终决策权在拍板者，执行者的认可不能替代拍板者判断

辅助对象：执行者
分工：在补充说明里体现执行者的正面体验，但正文优先满足拍板者对”高度”的需求

角色冲突处理：
- 拍板者要求”有高度”，执行者要求”好用”
- 正文先讲品牌高度和结果，补充说明里讲实际使用体验
```

---

### 坏样例4：不处理买用分离

**输入**：
- 产品：企业培训平台
- 背景：买单的是HR总监，使用的是一线员工

**错误输出**：
```
主写对象：HR总监
理由：买单的人

（实际写的时候全篇讲员工体验）
```

**问题分析**：
- 买用分离时，只写给使用者看会导致买单的人无感
- 或者反过来，只写给买单的人看会导致实际使用者不买账
- 需要明确分工

**正确输出**：
```
产品结构：买用分离

主写对象：HR总监
选择理由：采购决策权和预算控制者

辅助对象：使用者（一线员工）
分工：
- 正文（给HR总监）：讲招聘效率、管理合规、ROI
- 补充说明（给使用者）：讲上手简便、学习有趣

两者不混淆：不在正文中同时讲给两类人听
```

## 回退路径

| 当前状态 | 回退到 | 回退条件 |
|----------|--------|----------|
| 角色信息严重不足 | detect-need-type | 只有模糊描述”用户”，无任何角色线索 |
| 决策链完全不清 | detect-need-type | 不知道谁付款、谁使用、谁拍板 |
| 多人需求冲突无法裁决 | detect-need-type | 三方以上需求冲突，不知道该听谁的 |
| 单人产品但硬分多层 | 直接进入写作 | 产品天然只有一个用户，不需强行分层 |
| 主写对象已明确 | 直接进入卖点/Skill | 可以跳过此Skill直接下一步 |

**回退判断标准**：

1. **角色信息不足**（满足任一）：
   - 只有”用户”这样的泛泛描述
   - 不知道付款人、使用者、拍板者任何一方的线索
   - 完全无法判断决策链

2. **信号冲突无法裁决**（满足任一）：
   - 三方以上给出不同方向的需求
   - 无法判断谁是最终拍板者
   - 阻拦者的顾虑无法处理

3. **不需要分层的情况**：
   - 单人产品买用一体，且无其他影响者
   - 主写对象已经从上一Skill明确

**回退时必须输出**：
- 当前分层状态（即使不完整）
- 最大缺口（缺哪类角色/决策权证据）
- 下一步验证方向
- 建议回退到哪个Skill

```markdown
# 用户分层 - [日期]

[单人产品 / 买用一体 / 买用分离 / 委员会决策]

| 角色 | 身份 | 核心诉求 | 敏感点 | 主/辅 |
|------|------|----------|--------|-------|
| [角色名] | [身份标签] | [1-2个核心词] | [1-2个敏感词] | [主/辅] |
| ... | ... | ... | ... | ... |

## 主写对象
- **角色**：[角色名]
- **选择理由**：[为什么他是主写对象，引用决策链/影响力/痛点匹配]
- **核心诉求**：[1-2句话]
- **文案入口**：[从哪个点切入最能打动他]

## 辅助对象
| 角色 | 分工 | 出现在文案的哪个部分 |
|------|------|---------------------|
| [角色名] | [承接哪类疑虑/顾虑] | [正文/补充说明/证据/风险说明] |

1. [角色1] - [决策权重]
2. [角色2] - [决策权重]
3. ...

- **冲突点**：[哪两方关心什么]
- **处理策略**：[正文先讲什么，补充说明讲什么]

## 各层诉求
- **主写对象**：[核心诉求展开]
- **辅助对象**：[各辅助对象的诉求]

## 各层敏感点
- **主写对象**：[最怕什么]
- **辅助对象**：[各辅助对象的敏感点]

## 不该写的人群
- [人群1] - [为什么不该写]
- [人群2] - [为什么不该写]

## 交接判断
- 主写对象是否明确：[是/否]
- 是否可以进入写作：[是/否，需先确认XX]
- 是否需要回退：[是→回退到XX Skill / 否]
```

---

## 绝对不能

- 不能把人群写成空泛标签。
- 不能只按年龄、性别、职业分人。
- 不能把所有人都写成主对象。
- 不能替代卖点和场景判断。
- 不能忽略拍板者和阻拦者。
- 不能在角色信息不足时强行分层。
- 不能在单人产品上硬分多层角色。
- 不能混淆买用分离的分工（正文给谁看，补充说明给谁看）。
