# 痛点反推卖点四步法

## Skill概述

**输入输出格式**:
```json
{
  "输入": {
    "产品/服务名称": "string (必填)",
    "目标渠道": "string (选填: 电商平台/应用商店/社交媒体/客服记录/问卷调查/竞品评价)",
    "竞品名称": "string (选填)",
    "已收集的评价链接": "array (选填)"
  },
  "输出": {
    "痛点搜索": {
      "搜索渠道": ["string"],
      "收集数量": "string",
      "原始痛点": ["string"]
    },
    "痛点分类": {
      "功能类": [{"痛点": "string", "频次": number}],
      "体验类": [...],
      "效果类": [...],
      "服务类": [...],
      "价格类": [...],
      "情感类": [...]
    },
    "卖点提炼": {
      "痛点": "string",
      "对应方案": "string",
      "卖点表达": "string"
    },
    "卖点验证": {
      "主卖点": "string",
      "差异化程度": "高/中/低",
      "用户感知度": "高/中/低",
      "验证结论": "可选/不选"
    }
  }
}
```

**使用示例**:
```
用户: 帮我从用户评价中找卖点，产品是某在线英语课程

Skill:
痛点搜索:
- 搜索渠道: [应用商店, 电商平台, 社交媒体]
- 收集数量: "35条"
- 原始痛点: ["老师口音重", "学了没用", "太贵了", "没时间学", ...]

痛点分类:
- 功能类: [
    {"痛点": "学习效果不明显", "频次": 8},
    {"痛点": "课程体系不系统", "频次": 5}
  ]
- 体验类: [
    {"痛点": "老师口音重", "频次": 6},
    {"痛点": "课程安排不灵活", "频次": 4}
  ]
- 效果类: [
    {"痛点": "学了还是不会说", "频�": 10}
  ]
- 价格类: [
    {"痛点": "太贵了不值", "频次": 7}
  ]

卖点提炼:
- 痛点: "学了还是不会说" → 方案: "1对1外教实战" → 卖点: "每天开口说，不说不会退"
- 痛点: "老师口音重" → 方案: "欧美母语外教" → 卖点: "纯欧美外教，发音地道"

卖点验证:
- 主卖点: "每天开口说，不说不会退"
- 差异化程度: 高
- 用户感知度: 高
- 验证结论: 可选
```

**错误处理**:
- 未提供产品名称 → 提示"请告诉我产品名称"
- 无法找到足够评价 → 说明"样本不足，建议扩大搜索范围或提供已有评价"
- 痛点集中在某个方面 → 说明"这种情况说明该问题严重，但也可能意味着其他方面做得好"
- 卖点无法验证 → 提示"这个卖点虽然真实，但竞品也有，建议寻找更差异化的点"

**独特个性**:
- 像个产品侦探，善于从负面反馈中挖掘机会
- 强调"真实痛点"，拒绝编造和臆测
- 用数据和频次说话，客观分析问题严重程度
- 总是提醒"卖点必须有差异化，否则没有价值"

## 用户问题

"怎么从用户评价中找到卖点"

## 核心方法

### Step 1: 痛点搜索

**目标**: 全面收集用户负面反馈

**搜索渠道**:
1. **电商平台**: 京东/淘宝/天猫的追评、差评
2. **应用商店**: App Store、安卓应用市场的评价
3. **社交媒体**: 微博、小红书、知乎的相关讨论
4. **客服记录**: 投诉、退货、售后原因
5. **问卷调查**: 用户不满意的点
6. **竞品评价**: 竞品的差评也是参考

**搜索关键词**:
- 产品名 + 后悔/不推荐/避坑
- 产品名 + 差评/投诉/问题
- 产品名 + 怎么样/好用吗（看负面）

**输出**: 原始痛点清单

### Step 2: 痛点分类

**目标**: 将分散的痛点归类

**分类维度**:

| 类型 | 定义 | 示例 |
|------|------|------|
| 功能类 | 产品功能不能满足 | 功能太少、不好用 |
| 体验类 | 使用过程不顺畅 | 操作复杂、太慢 |
| 效果类 | 没有达到预期 | 没效果、不好使 |
| 服务类 | 售后体验不好 | 没人理、解决慢 |
| 价格类 | 觉得买贵了 | 不值、买亏了 |
| 情感类 | 心理不舒服 | 感觉被坑 |

**处理方法**:
1. 合并同类痛点
2. 统计出现频次
3. 标记严重程度

**输出**: 分类后的痛点列表

### Step 3: 卖点提炼

**目标**: 将痛点转化为卖点

**转化公式**:
```
痛点 → 卖点
（用户不满意的） → （我们解决得好的）

示例：
- 痛点：客服响应太慢 → 卖点：7x24小时秒回
- 痛点：操作太复杂 → 卖点：3步完成所有操作
- 痛点：效果不明显 → 卖点：无效全额退款
```

**提炼原则**:
1. 基于真实痛点（不是编造）
2. 我们确实能解决
3. 用户能感知到

**输出**: 痛点-卖点对照表

### Step 4: 卖点验证

**目标**: 确认卖点有竞争力

**验证方法**:

| 验证项 | 方法 |
|--------|------|
| 差异化 | 竞品是否也解决这个痛点 |
| 感知度 | 用户是否能感受到 |
| 可信度 | 能否给出证据 |
| 可持续 | 能否长期保持 |

**输出**: 验证后的卖点清单

## 工作流程

### Step 1: 痛点搜索

**输入**: 产品名称、搜索渠道限定
**处理**:
1. 在指定渠道搜索用户评价
2. 收集负面/中性评价
3. 整理原始痛点清单
**输出**: 原始痛点列表（20条以上）

### Step 2: 痛点分类

**输入**: 原始痛点列表
**处理**:
1. 去除水军/恶意评价
2. 按6大类型分类
3. 统计频次和严重程度
**输出**: 分类痛点表

### Step 3: 卖点提炼

**输入**: 分类痛点表
**处理**:
1. 每个痛点找对应解决方案
2. 将解决方案转化为卖点表达
3. 筛选有价值的卖点
**输出**: 痛点-卖点对照表

### Step 4: 卖点验证

**输入**: 痛点-卖点对照表
**处理**:
1. 验证差异化程度
2. 评估用户感知度
3. 确认可信度
4. 排除不可持续的卖点
**输出**: 最终卖点清单

## Critical Rules

1. **必须基于真实评价** - 不能编造痛点
2. **至少收集20条以上** - 样本太少不准确
3. **按分类维度整理** - 杂乱无章无法分析
4. **卖点必须有对应痛点** - 每个卖点都能回应某个痛点
5. **验证差异化** - 确认竞品没解决这个痛点
6. **禁止过度承诺** - 做不到的不要说

## 输入要求

用户需要提供:
1. 产品/服务名称（必填）
2. 目标渠道（选填，限定搜索范围）
3. 竞品名称（选填，用于对比）
4. 已收集的评价链接（选填）
