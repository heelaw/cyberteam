## 这次暴露的问题

1. 早期只搭了骨架，没有把执行层和验收层做硬。
2. 每个 Skill 只写“做什么”，没有把“怎么做、何时停、何时回退、何为错误”展开。
3. 缺少可复核样例，导致新人不知道什么叫好、什么叫坏。
4. 缺少边界条件和失败案例，遇到脏输入时容易误判。
5. 缺少严格的门禁，导致“差不多”会被误当成“已完成”。
6. scripts 层如果只有空壳，就会制造自动化幻觉。

## 根因

第一性原理上，数据驱动判断不是“看见一个指标就下结论”，而是“先定义问题，再找参照系，再拆维度，再验证假设，最后给动作”。

因此，一个可上岗的数据驱动 Agent 必须同时具备：

- 任务分类能力
- 数据充足度判断能力
- 参照系构建能力
- 多维拆解能力
- 假设排序能力
- 验证与止损能力
- 输出验收能力

如果这些能力只写成标题，没有写成具体 SOP，批量化时就会复制出一堆“看起来完整、实际上不能执行”的 Skill。

## 必须遵守的标准

1. 每个 Skill 必须只解决一个最小 SOP 单元。
2. 每个 Skill 必须有：作用、SOP、输出、参考、考核、脚本。
3. 每个关键 Skill 必须补齐：典型例子、失败案例、边界条件、交接判断、绝对不能。
4. 根 Agent 只做判断和路由，不吞掉子 Skill 的细节。
5. 如果脚本只是回显输入，就不能伪装成自动化能力。
6. 需要扩散时，必须先把一个 Skill 做厚，再复制到其他 Skill。

## 递进式排错问题

1. 这个任务到底属于活动、诊断还是状态评估？
2. 现在缺的是事实、判断，还是表达？
3. 参照系是否足够，还是还要补对比对象？
4. 当前最该回退到哪个前置 Skill？
5. 这个 Skill 的输出，是否能直接被下游使用？
6. 如果给一个新人，他能不能靠这份 SOP 自己做对？
7. 这份内容里，什么是必须保留的判断，什么是必须删掉的废话？

## 批量化方法

1. 先确定一个 Agent 的职责边界。
2. 再把这个 Agent 拆成最小 SOP 单元。
3. 每个单元补齐：输入样例、判断规则、反例、边界、回退、输出模板。
4. 再做一轮新人视角自检：是否能独立执行，是否能判错，是否能接到下一步。
5. 最后才批量复制到其它 Agent。

## 当前工作方式

- 不再只做摘要式收敛。
- 改为按“输入 -> 判断 -> 输出 -> 反例 -> 边界 -> 回退”逐层加厚。
- 优先补最薄、最容易误判、最影响下游的 Skill。

## 经验教训

- 不能把“目录齐了”误判成“能力齐了”。
- 不能把“写了规则”误判成“能执行”。
- 不能把“补了案例”误判成“已可批量化”。
- 必须持续做新人视角自检：我拿到这份东西，能不能独立做对、做完、做稳。
