## 根因

课程研发的第一性原理不是“先写 Skill 名称”，而是“先定义课程边界，再把输入拆成包，再把每个包做成可执行 Skill”。

因此，三个课程包都必须同时具备：

- 任务分类能力
- 输入输出映射
- 章节级拆解
- 依赖关系说明
- 验收与回退规则

如果这些能力只写成标题，没有写成具体 SOP，批量化时就会复制出一堆“看起来完整、实际上不能执行”的 Skill。

## 必须遵守的标准

1. 每个 Skill 必须只解决一个最小 SOP 单元。
2. 每个 Skill 必须有：作用、SOP、输出、参考、考核、脚本。
3. 每个关键 Skill 必须补齐：典型例子、失败案例、边界条件、交接判断、绝对不能。
4. 根 Agent 只做判断和路由，不吞掉子 Skill 的细节。
5. 如果脚本只是回显输入，就不能伪装成自动化能力。
6. 需要扩散时，必须先把一个 Skill 做厚，再复制到其他 Skill。

## 递进式排错问题

1. 这份内容属于入门、初阶还是高阶？
2. 现在缺的是课程边界、章节拆分，还是 Skill 骨架？
3. 参照系是否足够，还是还要补课程大纲与逐字稿？
4. 当前最该回退到哪个前置课程文件？
5. 这个 Skill 的输出，是否能直接被下游使用？
6. 如果给一个新人，他能不能靠这份 SOP 自己做对？
7. 这份内容里，什么是必须保留的判断，什么是必须删掉的废话？

## 批量化方法

1. 先确定一个课程包的职责边界。
2. 再把这个课程包拆成最小 SOP 单元。
3. 每个单元补齐：输入样例、判断规则、反例、边界、回退、输出模板。
4. 再做一轮新人视角自检：是否能独立执行，是否能判错，是否能接到下一步。
5. 最后才批量复制到其它 Agent。

## 当前工作方式

- 不再只做摘要式收敛。
- 改为按“输入 -> 判断 -> 输出 -> 反例 -> 边界 -> 回退”逐层加厚。
- 优先补最薄、最容易误判、最影响下游的 Skill。

## 经验教训

- 不能把“目录齐了”误判成“能力齐了”。
- 不能把“写了规则”误判成“能执行”。
- 不能把“补了案例”误判成“已可批量化”。
- 必须持续做新人视角自检：我拿到这份东西，能不能独立做对、做完、做稳。
