# Plan Mode 强制机制

## 概述

Plan Mode 是 Anthropic 2026 推荐的最佳实践，用于处理复杂任务的规划。本 Skill 提供了 Plan Mode 的强制触发机制和标准流程。

## 核心原则

### 何时必须进入 Plan Mode

**强制触发条件**（满足任一即触发）:
1. **任务步骤 >= 3**: 需要多步骤才能完成的任务
2. **架构决策**: 涉及系统架构、技术选型等重大决策
3. **Multi-Agent 协调**: 需要多个 Agent 协作完成
4. **不确定性**: 对实施方式不确定，需要探索
5. **高风险**: 任务失败会有重大影响

### Plan Mode 的价值

```
┌─────────────────────────────────────────────────────────┐
│                  为什么需要 Plan Mode                    │
├─────────────────────────────────────────────────────────┤
│  ✅ 避免遗漏关键步骤                                     │
│  ✅ 识别潜在风险和依赖                                   │
│  ✅ 确保资源合理分配                                     │
│  ✅ 提供清晰的执行路线图                                 │
│  ✅ 便于跟踪进度和调整                                   │
└─────────────────────────────────────────────────────────┘
```

## Plan Mode 标准流程

### Phase 1: 需求分析和确认

```yaml
目标明确:
  action: "清晰定义任务目标"
  questions:
    - "我们要达成什么目标？"
    - "成功的标准是什么？"
    - "有哪些约束条件？"

需求收集:
  action: "收集完整需求信息"
  sources:
    - "用户明确说明的需求"
    - "隐含的期望和假设"
    - "技术和业务约束"
    - "可用资源"

确认理解:
  action: "与用户确认理解"
  method: "复述关键点，征求确认"
```

### Phase 2: 制定详细实施计划

```yaml
任务分解:
  method: "将大任务分解为可执行的小步骤"
  principle: "每个步骤应该是独立、可验证的"

依赖分析:
  action: "识别步骤之间的依赖关系"
  questions:
    - "哪些步骤必须先完成？"
    - "哪些可以并行进行？"
    - "是否存在循环依赖？"

资源规划:
  action: "确定每步所需资源"
  items:
    - "时间估算"
    - "人力需求"
    - "工具和环境"
    - "外部依赖"

风险评估:
  action: "识别潜在风险"
  dimensions:
    - "技术风险"
    - "资源风险"
    - "时间风险"
    - "外部依赖风险"
```

### Phase 3: 识别潜在风险和依赖

```yaml
风险矩阵:
  high_impact_high_prob: "优先处理，制定缓解方案"
  high_impact_low_prob: "监控，准备应急预案"
  low_impact_high_prob: "优化流程，降低发生概率"
  low_impact_low_prob: "记录，定期回顾"

依赖管理:
  internal: "内部依赖，可以控制"
  external: "外部依赖，需要协调"
  blocking: "阻塞依赖，必须解决"
  optional: "可选依赖，可以延后"
```

### Phase 4: 制定验证标准

```yaml
成功标准:
  definition: "可量化的成功指标"
  examples:
    - "功能完整性：所有需求功能实现"
    - "质量标准：通过所有测试用例"
    - "性能指标：响应时间 < X ms"
    - "用户体验：用户满意度 > Y%"

验收标准:
  per_step: "每步有明确的完成标准"
  overall: "整体有清晰的验收条件"

检查点:
  frequency: "设置关键检查点"
  purpose: "及时发现问题，调整方向"
```

### Phase 5: 用户确认后执行

```yaml
确认机制:
  action: "向用户呈现计划，征求确认"
  format: |
    ## 执行计划
    ### 目标
    [清晰的目标描述]

    ### 步骤
    1. [步骤1] - [预估时间]
    2. [步骤2] - [预估时间]
    ...

    ### 风险和依赖
    - [风险1] - [缓解措施]
    - [依赖1] - [状态]

    ### 成功标准
    - [标准1]
    - [标准2]

    ### 预估时间和资源
    - 时间：[总预估]
    - 资源：[所需资源]

    请确认此计划是否可行？
```

## 常见误区

### 误区 1: Plan Mode 太浪费时间
**纠正**: 磨刀不误砍柴工，好的计划能避免返工

### 误区 2: 计划一成不变
**纠正**: 计划是动态的，根据执行情况调整

### 误区 3: 所有任务都要详细计划
**纠正**: 根据任务复杂度调整计划详细程度

### 误区 4: 计划完成后就不需要回顾
**纠正**: 定期回顾计划，及时调整方向
