# Success Metrics 标准模板

> **版本**: v1.0
> **创建时间**: 2026-03-10
> **灵感来源**: agency-agents 量化指标体系

---

## 为什么需要 Success Metrics

### 问题
当前 Skills 缺少量化成功标准，导致：
- ❌ 无法衡量输出质量
- ❌ 难以比较不同 Skills 的效果
- ❌ 缺少持续改进的基准

### 解决
每个 Skill 添加 Success Metrics 部分：
- ✅ 量化输出质量指标
- ✅ 定义业务结果指标
- ✅ 建立质量阈值

---

## 指标分类

### 1. 输出质量指标 (必需)

每个 Skill 必须包含至少 2 个输出质量指标。

#### 完整性指标
衡量输出是否完整覆盖所有必需部分。

```yaml
指标名称：完整性覆盖率
测量方法：实际覆盖部分数 / 必需部分数
目标值：100%
追踪方式：审核检查清单
```

**示例**:
```markdown
- **完整性**: 覆盖 AARRR 全部 5 个环节 (目标：100%)
```

#### 准确性指标
衡量输出内容的准确性。

```yaml
指标名称：数据准确率
测量方法：已验证数据数 / 总数据数
目标值：≥95%
追踪方式：来源交叉验证
```

**示例**:
```markdown
- **准确性**: 数据来源可验证 (目标：≥95%)
```

#### 可执行性指标
衡量输出是否可直接执行。

```yaml
指标名称：下一步明确度
测量方法：有明确下一步 / 无明确下一步
目标值：是
追踪方式：输出检查
```

**示例**:
```markdown
- **可执行性**: 每个策略有明确的下一步行动 (目标：是)
```

### 2. 业务结果指标 (如适用)

根据 Skill 类型选择适用指标。

#### 增长类 Skills

| 指标 | 定义 | 目标值 | 测量方式 |
|------|------|--------|---------|
| 用户增长率 | 月环比增长 | ≥20% | 数据分析 |
| K 因子 | 病毒传播系数 | ≥1.0 | (发出邀请数×转化率)/用户数 |
| CAC 回收期 | 获客成本回收时间 | <6 个月 | CAC/月利润 |
| LTV:CAC | 用户价值获客比 | ≥3:1 | LTV/CAC |
| 激活率 | 首周激活比例 | ≥60% | 完成激活事件用户/总用户 |
| 留存率 | D7/D30/D90 | 40%/20%/10% | 第 N 日活跃用户/初始用户 |

**示例**:
```markdown
### 业务结果指标
- 用户增长率：≥20% 月环比
- K 因子：≥1.0
- CAC 回收期：<6 个月
- LTV:CAC：≥3:1
- 激活率：≥60% 首周
- 留存率：D7≥40%, D30≥20%, D90≥10%
```

#### 营销类 Skills

| 指标 | 定义 | 目标值 | 测量方式 |
|------|------|--------|---------|
| 触达人数 | 内容曝光量 | ≥[目标] | 平台数据 |
| 转化率 | 点击→行动转化 | ≥[目标]% | 转化数/点击数 |
| ROI | 投入产出比 | ≥[目标]:1 | 收益/成本 |
| CTR | 点击率 | ≥2% | 点击数/曝光数 |
| 互动率 | 点赞评论分享 | ≥5% | 互动数/曝光数 |

**示例**:
```markdown
### 业务结果指标
- 触达人数：≥10,000 曝光/月
- 转化率：≥3% 点击→关注
- ROI：≥5:1
- CTR：≥2%
- 互动率：≥5%
```

#### 产品类 Skills

| 指标 | 定义 | 目标值 | 测量方式 |
|------|------|--------|---------|
| PMF 分数 | 用户留存意愿 | ≥40% | "非常失望"用户占比 |
| 留存率 | D7/D30 留存 | 40%/20% | 活跃用户/初始用户 |
| NPS | 净推荐值 | ≥30 | 推荐者% - 贬损者% |
| 功能使用率 | 核心功能使用 | ≥60% | 使用用户/总用户 |
| 任务完成率 | 核心任务完成 | ≥80% | 完成数/开始数 |

**示例**:
```markdown
### 业务结果指标
- PMF 分数：≥40%
- 留存率：D7≥40%, D30≥20%
- NPS：≥30
- 功能使用率：≥60%
```

#### 技术类 Skills

| 指标 | 定义 | 目标值 | 测量方式 |
|------|------|--------|---------|
| 代码覆盖率 | 测试覆盖代码 | ≥80% | 覆盖行数/总行数 |
| Bug 率 | 千行代码 Bug 数 | ≤1 | Bug 数/代码量 |
| 响应时间 | API p95 延迟 | <200ms | 监控系统 |
| 可用性 | 系统正常运行时间 | ≥99.9% | 正常时间/总时间 |
| 技术债务率 | 待优化代码占比 | ≤10% | 待优化/总代码 |

**示例**:
```markdown
### 业务结果指标
- 代码覆盖率：≥80%
- Bug 率：≤1/千行
- API 响应时间：p95 <200ms
- 可用性：≥99.9%
```

### 3. 质量阈值

定义最低可接受质量标准。

#### 阈值等级

| 等级 | 分数 | 说明 | 处理 |
|------|------|------|------|
| EXCELLENT | ≥0.95 | 超出预期 | 表彰/分享 |
| GOOD | ≥0.85 | 符合预期 | 通过 |
| NEEDS_WORK | <0.85 | 需要改进 | 返回修改 |

**示例**:
```markdown
### 质量阈值
- 最低可接受质量分数：0.85
- 通过标准：所有必需部分完整，数据已验证
- 优秀标准：≥0.95 分，有创新点
```

## 如何在 Skill 中添加

### Step 1: 确定适用指标

根据 Skill 类型选择：
- 增长类 → 增长指标
- 营销类 → 营销指标
- 产品类 → 产品指标
- 技术类 → 技术指标

### Step 2: 定义输出质量指标

至少 2 个：
1. 完整性指标
2. 准确性指标或可执行性指标

### Step 3: 定义业务结果指标

根据类型选择 2-4 个适用指标。

### Step 4: 定义质量阈值

明确通过标准。

### Step 5: 添加到 SKILL.md

在适当位置添加：
```markdown
## 📋 Success Metrics

### 输出质量指标
- [指标 1]: [量化标准]
- [指标 2]: [量化标准]

### 业务结果指标
- [指标 1]: [量化标准]
- [指标 2]: [量化标准]

### 质量阈值
- 最低可接受质量分数：[分数]
- 通过标准：[具体描述]
```

## 质量检查清单

在添加 Success Metrics 后，检查：

- [ ] 至少 2 个输出质量指标
- [ ] 至少 2 个业务结果指标（如适用）
- [ ] 所有指标可量化
- [ ] 有明确的质量阈值
- [ ] 指标可追踪和测量
- [ ] 目标值合理（不太高也不太低）

## 指标追踪方法

### 手动追踪
```markdown
# Success Metrics 追踪记录

| 日期 | 完整性 | 准确性 | 可执行性 | 备注 |
|------|--------|--------|----------|------|
| 3/10 | 100% | 95% | 是 | 首次追踪 |
| 3/17 | 100% | 98% | 是 | 改进 |
```

### 自动追踪
使用 skill-auditor 自动审核时记录指标。

---

## 持续改进

每月分析：
1. 哪个指标最难达成 → 调整目标或改进方法
2. 哪个 Skills 指标最好 → 学习经验
3. 哪些指标需要新增 → 补充指标
