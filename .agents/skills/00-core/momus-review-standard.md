## 📋 审核维度 (5 个维度，100 分)

### 1. 清晰度 (Clarity) - 25 分

**评估问题**:
- 每个任务是否指定 WHERE 查找实现细节？
- 任务描述是否无歧义？
- 技术术语是否已解释？

**评分标准**:
| 分数 | 标准 |
|------|------|
| 23-25 | 所有任务清晰，WHERE 明确，术语已解释 |
| 18-22 | 大部分任务清晰，1-2 处 WHERE 模糊 |
| 13-17 | 多处任务模糊，需要猜测 WHERE |
| 0-12 | 大部分任务不清晰，无法执行 |

**检查清单**:
- [ ] 每个任务有文件路径或模块名
- [ ] 每个任务有明确的输入/输出
- [ ] 专业术语有简短解释或链接
- [ ] 无模糊词汇（如"优化"、"改进"等无具体定义）

---

### 2. 可验证性 (Verification) - 25 分

**评估问题**:
- 验收标准是否具体可测量？
- 是否有明确的完成定义 (DoD)？
- 是否有测试/验证方法？

**评分标准**:
| 分数 | 标准 |
|------|------|
| 23-25 | 所有任务有量化验收标准，可自动化验证 |
| 18-22 | 大部分任务有验收标准，1-2 处模糊 |
| 13-17 | 多处任务验收标准不清晰 |
| 0-12 | 大部分任务无验收标准 |

**检查清单**:
- [ ] 每个任务有完成定义 (DoD)
- [ ] 验收标准可测量（数字、布尔值）
- [ ] 有测试方法或验证步骤
- [ ] 有预期结果描述

**好的验收标准示例**:
```markdown
✅ 好：API 响应时间 < 200ms (p95)
❌ 差：API 响应要快

✅ 好：单元测试覆盖率 ≥ 80%
❌ 差：添加足够的测试

✅ 好：所有 eslint 检查通过，0 错误 0 警告
❌ 差：修复代码问题
```

---

### 3. 上下文充分性 (Context) - 20 分

**评估问题**:
- 是否有足够上下文无需 >10% 猜测？
- 相关业务逻辑是否已说明？
- 依赖关系是否已识别？

**评分标准**:
| 分数 | 标准 |
|------|------|
| 18-20 | 上下文充分，无需猜测，依赖清晰 |
| 14-17 | 少量上下文缺失，<10% 需要猜测 |
| 10-13 | 多处上下文缺失，10-30% 需要猜测 |
| 0-9 | 上下文严重不足，>30% 需要猜测 |

**检查清单**:
- [ ] 业务背景已说明（为什么做这个）
- [ ] 用户场景已描述
- [ ] 依赖的外部系统已识别
- [ ] 相关代码位置已标注
- [ ] 历史决策/约束已说明

---

### 4. 完整性 (Completeness) - 20 分

**评估问题**:
- 是否覆盖所有必需部分？
- 边界情况是否已考虑？
- 异常处理是否已说明？

**评分标准**:
| 分数 | 标准 |
|------|------|
| 18-20 | 完整覆盖，边界情况清晰，异常处理完整 |
| 14-17 | 少量遗漏，1-2 个边界情况未考虑 |
| 10-13 | 多处遗漏，边界情况考虑不足 |
| 0-9 | 严重遗漏，关键部分缺失 |

**检查清单**:
- [ ] 需求分析部分完整
- [ ] 技术方案部分完整
- [ ] 测试策略部分完整
- [ ] 回滚/降级方案已考虑
- [ ] 监控/告警方案已考虑
- [ ] 边界情况（极端值、空值、并发）已考虑
- [ ] 异常处理（错误码、重试、降级）已说明

---

### 5. 大格局 (Big Picture) - 10 分

**评估问题**:
- 目的高度清晰？
- 与整体目标是否对齐？
- 长期影响是否已评估？

**评分标准**:
| 分数 | 标准 |
|------|------|
| 9-10 | 目标清晰，与整体对齐，长期影响已评估 |
| 7-8 | 目标清晰，1 项未考虑 |
| 5-6 | 目标模糊或对齐不足 |
| 0-4 | 目标不清晰，与整体脱节 |

**检查清单**:
- [ ] 核心目标一句话可描述
- [ ] 与产品/业务目标对齐
- [ ] 技术债务影响已评估
- [ ] 可维护性影响已评估
- [ ] 扩展性影响已评估

## 🎯 通过标准

### Momus 模式（严格审核）

用于：关键任务、核心业务逻辑、大规模应用

**必须同时满足**:
```yaml
总分要求: ≥ 95/100

分项要求:
  清晰度：≥ 23/25
  可验证性: ≥ 23/25
  上下文充分性：≥ 18/20
  完整性：≥ 18/20
  大格局：≥ 9/10

文件引用:
  - 100% 文件引用已验证
  - ≥ 80% 任务有明确参考来源
  - ≥ 90% 任务有具体验收标准

业务逻辑:
  - 0 任务需要假设业务逻辑
  - 0 关键红旗
```

### 标准模式（常规审核）

用于：常规任务、非核心功能

**必须同时满足**:
```yaml
总分要求: ≥ 85/100

分项要求:
  清晰度：≥ 18/25
  可验证性：≥ 18/25
  上下文充分性：≥ 14/20
  完整性：≥ 14/20
  大格局：≥ 7/10

文件引用:
  - ≥ 80% 文件引用已验证
  - ≥ 70% 任务有明确参考来源
  - ≥ 80% 任务有具体验收标准

业务逻辑:
  - ≤ 1 任务需要假设业务逻辑
  - 0 关键红旗
```

## 🔄 审核循环流程

```
┌─────────────────────────────────────────────────────────────┐
│                    Momus 审核流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  提交计划                                                   │
│     ↓                                                       │
│  预审核检查 (结构、Frontmatter、References)                  │
│     ↓                                                       │
│  深度内容审核 (5 维度评分)                                   │
│     ↓                                                       │
│  决策                                                       │
│  ├─ PASS (≥95 且无关键问题) → 批准执行                       │
│  └─ REJECTED (<95 或有关键问题) → 返回修改                   │
│         ↓                                                   │
│  修改后重新提交                                              │
│         ↓                                                   │
│  累计 3 次 REJECTED → 人工审核                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```markdown
| 上下文充分性 | X | 20 | [说明] |

| 大格局 | X | 10 | [说明] |

## 关键问题

### ❌ 问题 1: [问题标题]
**维度**: [维度名]

**描述**: [详细描述]
**位置**: [文件/章节]
**改进建议**: [具体建议]

### ❌ 问题 2: [问题标题]
...

## 必须修复项 (PASS 前必须完成)
- [ ] [修复项 1]
- [ ] [修复项 2]

## 建议修复项 (推荐但不强制)
- [ ] [建议项 1]
- [ ] [建议项 2]

## 累计 REJECTED 次数：X/3
**警告**: 累计 3 次 REJECTED 后将触发人工审核
```

### 案例 2：低质量计划（REJECTED）

**计划摘要**: 优化系统性能

**审核结果**:
```yaml
总分：72/100

分项:
  清晰度：13/25 - WHERE 模糊，多处需要猜测
  可验证性：13/25 - 验收标准不清晰
  上下文充分性：10/20 - 业务背景缺失
  完整性：19/20 - 技术方案完整
  大格局：9/10 - 目标清晰

文件引用:
  - 60% 文件引用已验证 ❌
  - 50% 任务有明确参考来源 ❌
  - 50% 任务有具体验收标准 ❌

业务逻辑:
  - 3 任务需要假设业务逻辑 ❌
  - 1 关键红旗 ❌

决策：REJECTED
```

**关键问题**:

1. **"优化 API 响应时间"** - WHERE 模糊
   - 问题：未指定优化哪些 API
   - 建议：列出具体 API 路径

2. **"提升数据库性能"** - 验收标准缺失
   - 问题：如何衡量"提升"？
   - 建议：定义具体指标（如 p95 < 100ms）

3. **"添加缓存"** - 业务逻辑假设
   - 问题：未说明缓存什么、为什么
   - 建议：分析热点数据，说明缓存策略

## 🛠️ 使用指南

### 何时启用 Momus 模式

**自动触发条件** (满足任一):
- 任务标记为 `critical: true`
- 涉及核心业务逻辑（支付、认证、资金）
- 预计工作量 > 8 小时
- 影响多个部门/模块
- 用户明确要求高准确度

**手动触发**:
```bash
# 在任务描述中添加
/momus-review
```

### 审核时长预期

| 计划复杂度 | 预计审核时间 |
|-----------|-------------|
| 简单 (<5 任务) | 2-5 分钟 |
| 中等 (5-15 任务) | 5-10 分钟 |
| 复杂 (>15 任务) | 10-20 分钟 |

### 常见拒绝原因

| 原因 | 占比 | 如何避免 |
|------|------|---------|
| WHERE 模糊 | 35% | 每个任务指定文件路径 |
| 验收标准不清 | 25% | 使用量化指标 |
| 业务逻辑假设 | 20% | 明确说明业务背景 |
| 边界情况遗漏 | 15% | 考虑极端值、空值、并发 |
| 其他 | 5% | - |
