> 版本：v1.0
> 日期：2026-03-18

---

## 一、项目概述

### 1.1 背景

将两门运营领域课程转化为可复用的 Skills：
- **运营初阶课**：64个逐字稿文件，涵盖6大模块
- **运营入门**：3个逐字稿文件，涵盖宏观认知、岗位理解、职业发展

### 1.2 目标

1. 每门课程转化为1-3个高质量 Skills
2. 每个 Skill 达到 90+ 评分
3. 通过三视角交叉验证确保方案完整性

## 四、Skills详细定义

### 4.1 P0-1: 运营岗位四要素拆解器

**基本信息**

| 字段 | 内容 |
|------|------|
| **Skill名称** | 运营岗位四要素拆解器 |
| **英文名** | Operation-JD-Analyzer |
| **版本** | v1.0 |
| **优先级** | P0-1（最高优先级） |

**核心方法论**

```
四要素 = 用户 + 目标 + 场景 + 手段

- 用户：服务谁（用户画像、需求、生命周期）
- 目标：达成什么（拉新/维系/转化/收入/供给）
- 场景：在哪里做（线上/线下、平台/私域）
- 手段：怎么做（内容/活动/产品/互动/数据）
```

**输入要求**

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 岗位JD文本 | text | 是 | 完整的职位描述内容 |
| 公司业务描述 | text | 否 | 公司主营业务描述 |
| 用户补充信息 | text | 否 | 任何补充信息 |

**输出格式**

```json
{
  "四要素拆解": {
    "用户": { "直接用户": "", "用户特征": [], "生命周期阶段": "" },
    "目标": { "核心KPI": [], "目标导向": "", "指标优先级": "" },
    "场景": { "主战场": "", "具体场景": [], "场景特征": "" },
    "手段": { "核心手段": [], "辅助手段": [], "技能要求": [] }
  },
  "导向判断": { "主要导向": "", "判断依据": "" },
  "面试准备建议": []
}
```

**评估标准**

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 要素完整性 | 25% | 四要素是否都分析到位 |
| 导向准确性 | 35% | 导向判断是否准确 |
| 洞察价值 | 25% | 洞察和建议是否有价值 |
| 可操作性 | 15% | 面试建议是否可执行 |

**Critical Rules**

*必须遵守：*
- 每个要素都必须有实质性分析，不能留空
- 导向判断必须有明确的关键词依据
- 面试建议必须与四要素分析结果一致

*禁止行为：*
- 不能凭空猜测用户特征
- 不能忽略JD中的关键信息
- 不能给出与岗位无关的建议

### 4.2 P0-2: 活动策划方案撰写

**基本信息**

| 字段 | 内容 |
|------|------|
| **Skill名称** | 活动策划方案撰写 |
| **英文名** | Event-Planner |
| **版本** | v1.0 |
| **优先级** | P0-2 |

**核心方法论**

```
活动策划方案 = 背景 + 目的 + 主题 + 时间 + 对象 + 玩法 + 推广 + 资源 + 风险 + 效果
```

**输入要求**

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 活动背景 | text | 是 | 为什么要做 |
| 活动目的 | text | 是 | 核心目标+量化指标 |
| 可用资源 | text | 是 | 预算、人力、技术 |
| 活动时间 | text | 否 | 预计时间范围 |
| 目标用户 | text | 否 | 用户群体描述 |

**输出格式**

```markdown
# [活动名称]

## 一、活动背景
## 二、活动目的
## 三、活动主题
## 四、活动时间
## 五、活动对象
## 六、活动玩法
## 七、活动推广
## 八、资源需求
## 九、风险预估
## 十、效果预估
```

**评估标准**

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 目标明确性 | 20% | 目的是否量化 |
| 逻辑完整性 | 20% | 10要素是否齐全 |
| 玩法可行性 | 20% | 规则是否清晰 |
| 资源匹配性 | 15% | 预算是否匹配 |
| 风险预案性 | 15% | 是否有风险识别 |
| 效果可衡量性 | 10% | 指标是否可量化 |

**Critical Rules**

*必须遵守：*
- 活动目的必须量化（不能只是"提升品牌"）
- 活动时间必须精确到日期
- 预算必须细化到项目
- 风险预估必须包含至少3个风险点

*禁止行为：*
- 目的模糊（如"提升品牌影响力"）
- 预算笼统（如"若干预算"）
- 忽略风险（不做风险预估）
- 玩法模糊（用户无法理解如何参与）

### 4.3 P0-3: 转化型文案写作

**基本信息**

| 字段 | 内容 |
|------|------|
| **Skill名称** | 转化型文案写作 |
| **英文名** | Conversion-Copywriting |
| **版本** | v1.0 |
| **优先级** | P0-3 |

**核心方法论**

**短文案5种诉求策略：**

| 策略 | 公式 | 示例 |
|------|------|------|
| 诉诸利益 | 利益点+行动 | "免费领取，限时3天" |
| 诉诸情感 | 情感共鸣+行动 | "送给妈妈最贴心的礼物" |
| 诉诸权威 | 权威背书+行动 | "专家推荐，畅销10万" |
| 诉诸数字 | 具体数字+行动 | "3天学会，99%学员推荐" |
| 诉诸氛围 | 场景描述+行动 | "人手必备的潮流好物" |

**中长文案结构模型：**

```
开头(10-20字)：吸引注意（提问/悬念/利益）
主体(80-100字)：传递价值（卖点+证据）
结尾(10-20字)：行动号召（紧迫感+行动）
```

**输入要求**

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 产品/服务信息 | text | 是 | 核心卖点 |
| 目标用户 | text | 是 | 用户群体 |
| 转化目标 | text | 是 | 用户需要做的动作 |
| 文案类型 | select | 是 | 短/中长/长文案 |
| 渠道特点 | text | 否 | 投放渠道要求 |
| 约束条件 | text | 否 | 字数限制等 |

**评估标准**

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 利益明确性 | 25% | 用户能清晰感知获得什么 |
| 行动明确性 | 20% | 用户知道该做什么 |
| 吸引力 | 20% | 能抓住用户注意力 |
| 可信度 | 15% | 数据、背书是否充分 |
| 差异化 | 10% | 与竞品有明显区别 |
| 渠道适配 | 10% | 符合投放渠道特点 |

**Critical Rules**

*必须遵守：*
- 利益点要具体可感知
- 行动号召要明确
- 数据要有依据
- 要考虑渠道特点

*禁止行为：*
- 利益模糊（"品质优良"）
- 行动模糊（"了解更多"）
- 夸大宣传（"全网第一"）
- 忽略渠道特点

### 4.4 P0-4: 运营复杂度分析器

**基本信息**

| 字段 | 内容 |
|------|------|
| **Skill名称** | 运营复杂度分析器 |
| **英文名** | Operation-Complexity-Analyzer |
| **版本** | v1.0 |
| **优先级** | P0-4 |

**核心方法论**

**三层递进模型：**

```
业务模式/业务逻辑（怎么赚钱）
    ↓ 决定
组织结构/部门划分（谁来负责什么）
    ↓ 决定
具体工作场景、目标、手段（运营的具体表现）
```

**输入要求**

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 公司业务描述 | text | 是 | 主营业务和商业模式 |
| 组织架构信息 | text | 否 | 部门设置 |
| 运营岗位列表 | list | 否 | 运营相关岗位名称 |

**输出格式**

```json
{
  "业务复杂度评估": {
    "等级": "高/中/低",
    "收入来源": [],
    "目标用户": [],
    "交付环节": 0
  },
  "组织架构特征": {
    "部门划分": "",
    "运营相关部门": [],
    "组织复杂度": ""
  },
  "运营工作全景": {
    "核心目标": [],
    "主要场景": [],
    "关键手段": []
  },
  "关键洞察": []
}
```

**评估标准**

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 完整性 | 30% | 是否覆盖业务、组织、运营三个层面 |
| 准确性 | 40% | 分析结论是否与实际情况一致 |
| 实用性 | 30% | 洞察是否有决策参考价值 |

### 4.5 P0-5: 社群裂变增长

**基本信息**

| 字段 | 内容 |
|------|------|
| **Skill名称** | 社群裂变增长 |
| **英文名** | Community-Fission-Growth |
| **版本** | v1.0 |
| **优先级** | P0-5 |

**核心方法论**

**裂变飞轮：**

```
种子用户 → 活动传播 → 新用户进群 → 再次传播
```

**裂变4要素设计：**

| 要素 | 设计要点 |
|------|----------|
| 诱饵 | 有吸引力、价值感知、获取成本低 |
| 任务 | 简单可行、门槛低 |
| 门槛 | 免费优先、流程简单 |
| 裂变物 | 海报/链接/小程序，需精美有冲击力 |

**输入要求**

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 种子用户 | text | 是 | 首批用户数量和质量 |
| 诱饵资源 | text | 是 | 奖励预算和类型 |
| 裂变目标 | text | 是 | 目标新增用户数、时间 |
| 社群基础 | text | 否 | 现有社群情况 |
| 承载能力 | text | 否 | 客服承接能力 |

**评估标准**

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 裂变系数 | 25% | 成功邀请人数/总参与人数 |
| 拉新成本 | 20% | 总成本/新增用户数 |
| 转化率 | 20% | 新用户进群后的转化表现 |
| 执行效率 | 15% | 问题响应速度 |
| 用户质量 | 10% | 新用户活跃度和留存 |
| ROI | 10% | 投入产出比 |

**Critical Rules**

*必须遵守：*
- 诱饵要有真实价值
- 任务流程要简单（不超过3步）
- 审核反馈要及时（不超过2小时）
- 要有应急预案

*禁止行为：*
- 诱饵虚假
- 任务复杂
- 审核延迟
- 忽略风险

### 4.6 P0-6: 职业发展路径规划器

**基本信息**

| 字段 | 内容 |
|------|------|
| **Skill名称** | 职业发展路径规划器 |
| **英文名** | Career-Path-Planner |
| **版本** | v1.0 |
| **优先级** | P0-6 |

**核心方法论**

**五级晋升模型：**

| 阶段 | 职位 | 时间 | 核心职责 | 关键能力 |
|------|------|------|----------|----------|
| L1 | 执行者 | 0-1年 | 完成具体任务 | 执行力、交付能力 |
| L2 | 业务骨干 | 1-2年 | 达成某项目标 | 单点技能、目标达成 |
| L3 | 项目负责人 | 2-3年 | 独立负责项目 | 项目管理、团队协作 |
| L4 | 中层管理 | 3-5年 | 管理部门/业务线 | 策略制定、团队管理 |
| L5 | 操盘手 | 5年+ | 独立业务线 | 战略思维、业务操盘 |

**输入要求**

| 输入项 | 类型 | 必需 | 说明 |
|--------|------|------|------|
| 当前职业阶段 | select | 是 | L1-L5 |
| 工作年限 | number | 是 | 从事运营总年限 |
| 当前核心技能 | list | 是 | 最多5项 |
| 职业发展方向 | select | 否 | 专家/操盘手/未确定 |
| 目标时间范围 | number | 否 | 规划年数，默认5年 |

**输出格式**

```json
{
  "当前状态评估": {
    "职业阶段": "",
    "核心竞争力": [],
    "能力短板": [],
    "评估说明": ""
  },
  "发展路线建议": {
    "推荐路线": "",
    "路线分析": "",
    "备选路线": ""
  },
  "发展规划": {
    "短期目标": { "目标": "", "核心任务": "", "关键成果": "" },
    "中期目标": { "目标": "", "核心任务": "", "关键成果": "" },
    "长期目标": { "目标": "", "核心任务": "", "关键成果": "" }
  },
  "技能强化计划": {
    "优先强化": [],
    "持续打磨": [],
    "学习建议": []
  },
  "行动建议": {}
}
```

**评估标准**

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 现状评估准确性 | 25% | 当前状态评估是否客观准确 |
| 路线建议合理性 | 25% | 发展路线是否适合用户情况 |
| 目标可达成性 | 25% | 发展目标是否可衡量 |
| 行动计划实用性 | 25% | 行动建议是否具体可执行 |

## 六、风险管理

### 6.1 潜在风险

| 风险 | 影响 | 对策 |
|------|------|------|
| 评估标准主观性强 | 评分可能不一致 | 增加量化指标，减少主观判断 |
| 案例覆盖不足 | Skill实用性降低 | 补充更多行业案例 |
| 与现有Skills冲突 | 功能重复 | 使用前进行冲突检查 |

### 6.2 质量保障

1. 每个Skill必须通过内部评审
2. 必须有至少2个测试用例
3. Critical Rules必须有明确的判定标准
