## 一、整体数据流

### 1.1 完整流程图

```
用户输入 (User Input)
        ↓
┌─────────────────┐
│   CEO接收层      │  ← 意图识别、初步解析
└────────┬────────┘
         ↓
┌─────────────────┐
│  问题分类器      │  ← 问题类型、复杂度、关键词
└────────┬────────┘
         ↓
┌─────────────────┐
│  思维模型路由器  │  ← 选择合适的思维模型组合
└────────┬────────┘
         ↓
┌─────────────────┐
│  CEO思维执行层   │  ← 调用思维模型、聚合结果
└────────┬────────┘
         ↓
┌─────────────────┐
│  问题定义输出    │  ← 结构化问题定义
└────────┬────────┘
         ↓
┌─────────────────┐
│  部门任务分配    │  ← 并行分配到6个部门
└────────┬────────┘
         ↓
┌─────────────────┐
│  部门并行执行    │  ← 各部门独立处理
└────────┬────────┘
         ↓
┌─────────────────┐
│  结果汇总层      │  ← CEO聚合各部门输出
└────────┬────────┘
         ↓
┌─────────────────┐
│  最终方案输出    │  ← 完整解决方案
└─────────────────┘
```

## 二、接口定义

### 2.1 用户 → CEO 接口

#### 用户输入格式

```json
{
  "type": "user_input",
  "version": "1.0",
  "timestamp": "2026-03-23T10:00:00Z",

  "content": {
    "goal": "用户目标的自然语言描述",
    "context": {
      "背景信息": "可省略，有则更准确"
    },
    "constraints": {
      "budget": "可选，数字或描述",
      "timeline": "可选，时间范围",
      "team_size": "可选，人数",
      "tech_stack": "可选，技术栈"
    }
  },

  "metadata": {
    "user_id": "可选，用户标识",
    "session_id": "会话ID",
    "priority": "normal/high/urgent"
  }
}
```

```json
{
  "type": "user_input",

    "goal": "做一个用户增长方案",

      "budget": 100000,

}
```

## 四、数据流控制

### 4.1 消息队列架构

```yaml
message_queue:
  inbound_queue:
    - 用户输入队列
    - 部门输出队列

  outbound_queue:
    - CEO分发队列
    - 部门任务队列

  priority_queues:
    - 高优先级队列 (urgent任务)
    - 普通优先级队列
    - 低优先级队列 (优化类任务)

  dead_letter_queue:
    - 失败消息存储
    - 人工处理标记
```

### 4.2 异步 vs 同步处理

```yaml
processing_modes:
  同步处理:
    适用场景:
      - 简单查询
      - 快速决策
      - 用户等待反馈
    超时设置: 30秒

  异步处理:
    适用场景:
      - 复杂分析
      - 多部门协作
      - 长任务执行
    超时设置: 5分钟
    轮询间隔: 30秒
```

## 五、错误处理

### 5.1 错误类型定义

```yaml
error_types:
  INPUT_ERROR:
    code: "E001"
    描述: "用户输入解析错误"
    处理: "请求用户重新输入"

  CLASSIFICATION_ERROR:
    code: "E002"
    描述: "问题分类失败"
    处理: "使用默认类型 + 人工确认"

  ROUTING_ERROR:
    code: "E003"
    描述: "思维模型路由失败"
    处理: "使用基础模型组合"

  DEPARTMENT_ERROR:
    code: "E004"
    描述: "部门执行失败"
    处理: "CEO重新分配或人工干预"

  TIMEOUT_ERROR:
    code: "E005"
    描述: "任务执行超时"
    处理: "触发降级策略或延长超时"

  SYSTEM_ERROR:
    code: "E999"
    描述: "系统级错误"
    处理: "记录日志 + 人工告警"
```

### 5.2 重试策略

```yaml
retry_strategy:
  max_retries: 3

  backoff:
    - attempt_1: "1秒"
    - attempt_2: "5秒"
    - attempt_3: "30秒"

  fallback:
    - 部门失败: "重新分配到其他部门或CEO执行"
    - 路由失败: "使用预设模型组合"
    - 分类失败: "使用默认类型"
```

## 六、会话状态管理

### 6.1 会话生命周期

```yaml
session_lifecycle:
  创建:
    触发: 用户新输入
    动作: 创建会话ID，初始化上下文

  进行中:
    状态: 任务执行中
    管理: 记录中间结果，支持断点恢复

  完成:
    触发: 所有部门输出完成
    动作: 汇总结果，生成最终方案

  超时:
    触发: 30分钟无活动
    动作: 保存状态，标记为待恢复
```

### 6.2 上下文存储

```yaml
context_storage:
  session_context:
    - 用户输入历史
    - 问题定义
    - 已分配的子问题
    - 部门输出缓存

  agent_context:
    - 各部门的中间状态
    - 依赖关系图
    - 阻塞信息

  artifacts:
    - 生成的文档
    - 图表和设计稿
    - 代码和配置文件
```

## 七、API端点汇总

| 端点 | 方法 | 用途 |
|------|------|------|
| `/input` | POST | 接收用户输入 |
