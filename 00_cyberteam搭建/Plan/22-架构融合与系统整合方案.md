## 一、三个核心问题分析

### 问题1：cyberteam-v2 的引擎架构 vs edict 的三省六部架构，哪个更适合？

**结论：两者互补，v3 采用「引擎注入 + 制度流转」双轨制。**

| 维度 | cyberteam-v2 引擎 | edict 三省六部 |
|------|-----------------|----------------|
| **核心机制** | 思维注入（100+专家模型库按需激活） | 制度化审批（状态机+权限矩阵） |
| **优势** | CEO智能强、意图理解深、灵活注入思维模型 | 流程可控、质量有保障、可观测性强 |
| **劣势** | 缺乏制度约束，Agent可自由协作（易产生假数据） | 智能不足，依赖Agent自身能力，审批流程机械 |
| **适用场景** | 复杂决策、战略规划、多维度分析 | 规范化执行、质量管控、团队协作 |

**融合思路**：
- 用 edict 的「制度外壳」包裹 cyberteam-v2 的「智能内核」
- 思维注入引擎负责「想清楚」，三省六部负责「做规范」
- CEO = 太子（分拣）+ 中书省（规划）+ 思维注入（智能增强）

### 问题2：CyberTeam v2.1 的运营专家层如何融入 edict 风格的架构？

**结论：运营专家作为「皇帝内阁顾问」，嵌入太子分拣和回奏环节。**

CyberTeam v2.1 的 21个Plan文档和 content-ops.md 等专家定义，本质上是「领域知识库」。融合方式：

- **内容运营专家** → 嵌入太子分拣环节（判断是否为内容运营类旨意）
- **增长运营专家** → 嵌入尚书省派发环节（提供增长策略建议）
- **用户运营专家** → 嵌入中书省规划环节（提供用户洞察）
- **品牌专家** → 嵌入礼部（品牌/文案执行）

运营专家不作为独立Agent存在，而是作为「专家知识注入模块」，在三省六部的关键环节提供领域知识支撑。

### 问题3：100个思考天团Agent如何作为 edict 六部的技能库？

**结论：六部各持一个「技能库子集」，尚书省负责按需调度。**

100个思考天团Agent按技能类型映射到六部：

| 六部 | 职能定位 | 映射的思考天团Agent |
|------|---------|-------------------|
| **兵部** | 代码实现官 | 工程实践、架构设计、系统思维、逆向思维 |
| **刑部** | 测试审查官 | 批判性思维、卡尼曼、确认偏差、第一性原理 |
| **礼部** | 文档编制官 | 六顶思考帽、精益创业、设计思维 |
| **户部** 数据分析官 | 概率思维、贝叶斯、边际成本、沉没成本 |
| **工部** | 基础设施官 | 系统思维、复杂性理论、迭代 |
| **吏部** | 人力资源官 | 邓巴数、彼得原理、帕金森定律、情绪调节 |

尚书省在派发任务时，从对应部门的技能库中选择最相关的思维Agent注入到执行Agent中。

### 3.2 中书省Agent = edict规划 + v2思维注入引擎

class ZhongshuAgent:
    """中书省Agent - 融合edict规划起草 + v2思维注入引擎"""

    def __init__(self):
        self.ceo_engine = CEOThinkingEngine(self.thinking_injector)
        self.thinking_injector = ThinkingInjector()

    def plan(self, task: Task) -> Plan:
        # 1. CEO思维引擎深度分析
        result = self.ceo_engine.decompose(task.requirement)

        # 2. 5W1H1Y深度拆解
        decomposition = result["decomposition"]

        # 3. MECE分类
        categories = result["mece"]["categories"]

        # 4. 激活相关思维专家
        selected_experts = result["thinking_context"]["selected_experts"]

        # 5. 生成执行计划
        execution_plan = result["execution_plan"]

        # 6. 提交门下省审议
        return Plan(
            decomposition=decomposition,
            mece=categories,
            experts=selected_experts,
            phases=execution_plan,
            todos=self._generate_todos(execution_plan)
        )

### 3.3 门下省Agent = edict审批 + v2质量门控

```python
class MenxiaAgent:
    """门下省Agent - 融合edict审议 + cyberteam-v2质量门控"""

    def __init__(self):
        self.quality_gate = QualityGateEngine()
        self.thinking_injector = ThinkingInjector()

    def review(self, plan: Plan) -> ReviewResult:
        # 1. 注入批判性思维专家
        ctx = self.thinking_injector.process(plan.description, agent="menxia")

        # 2. 三级质量门控评估
        qa_result = self.quality_gate.evaluate_all(plan.to_dict())

        # 3. 审查要点
        reviews = [
            self._check_feasibility(plan),      # 可行性
            self._check_completeness(plan),      # 完整性
            self._check_risk(plan),              # 风险
            self._check_cost_benefit(plan),      # 成本收益
        ]

        # 4. 决定：准奏 / 封驳
        if qa_result['passed'] and all(r['pass'] for r in reviews):
            return Approval(plan, suggestions=reviews)
        else:
            return Rejection(plan, reasons=reviews, round=self.review_round)
```

### 3.4 尚书省Agent = edict派发 + v2路由引擎

```python
class ShangshuAgent:
    """尚书省Agent - 融合edict派发 + v2路由引擎"""

    def __init__(self):
        self.routing_engine = RoutingEngine()
        self.six_bu_skills = load_six_bu_skills()  # 六部技能库

    def dispatch(self, approved_plan: Plan) -> DispatchResult:
        # 1. 路由引擎选择执行部门
        for phase in approved_plan.phases:
            dept = self.routing_engine.select_dept(phase, self.six_bu_skills)

            # 2. 从技能库注入思维模型
            skills = self.six_bu_skills.get(dept).select(phase.type)

            # 3. 派发给对应部门
            self._dispatch(dept, phase, skills)

        # 4. 监控并行执行
        # 5. 汇总结果，回奏皇上
        return self._summarize_and_report(approved_plan)

    def _select_six_bu(self, task_type: str) -> str:
        """根据任务类型选择六部之一"""
        mapping = {
            "code": "bingbu",     # 兵部
            "test": "xingbu",     # 刑部
            "doc": "libu",        # 礼部
            "data": "hubu",       # 户部
            "infra": "gongbu",   # 工部
            "hr": "libu_hr",     # 吏部
        }
        return mapping.get(task_type, "bingbu")
```

### 3.5 六部Agent = edict执行 + 思考天团技能库

```python
class SixBuAgent:
    """六部Agent基类 - 融合edict执行 + 100个思考天团技能"""

    def __init__(self, dept: str):
        self.dept = dept
        self.skills = load_thinking_agents(dept)  # 从100个中筛选
        self.thinking_injector = ThinkingInjector()

    def execute(self, task: SubTask) -> ExecutionResult:
        # 1. 从技能库注入相关思维模型
        ctx = self.thinking_injector.process(task.description, agent=self.dept)

        # 2. 选取最相关的3-5个思维专家
        relevant_skills = self.skills.select(task.type, top_k=5)

        # 3. 执行任务
        result = self._do_work(task, relevant_skills)

        # 4. 实时上报进展
        self._report_progress(task, result)

        return result
```

## 六、状态机与权限矩阵（继承edict，增强v2）

### 状态流转图

```
Pending → Taizi → Zhongshu → Menxia → Assigned → Doing → Review → Done
                  ↓              ↑
              (简单任务)      (封驳返回)
              Zhongshu
                  ↓
              (紧急任务)
              Assigned

状态说明：
- Pending: 皇上飞书消息到达
- Taizi: 太子分拣中
- Zhongshu: 中书省规划中
- Menxia: 门下省审议中（最多3轮）
- Assigned: 尚书省派发中
- Doing: 六部执行中（并行）
- Review: 尚书汇总审查中
- Done: 任务完成，回奏皇上
```

## 七、调度系统（继承edict，增强v2）

### 三阶段调度

```python
class Scheduler:
    """调度器 - 停滞检测 + 重试 + 升级 + 回滚"""

    STALL_THRESHOLD = 180  # 180秒无进展则触发

    def scan(self):
        """每60秒扫描一次"""
        for task in active_tasks():
            elapsed = now() - task.lastProgressAt

            if elapsed < self.STALL_THRESHOLD:
                continue

            # 阶段1: 重试
            if task.retryCount < task.maxRetry:
                self._retry(task)

            # 阶段2: 升级
            elif task.escalationLevel < 2:
                self._escalate(task)

            # 阶段3: 回滚
            elif task.autoRollback:
                self._rollback(task)

    def _retry(self, task):
        """重试：重新派发Agent"""
        task.retryCount += 1
        self._dispatch(task, task.state)

    def _escalate(self, task):
        """升级：唤醒上级协调"""
        task.escalationLevel += 1
        if task.escalationLevel == 1:
            self._wake_agent("menxia", f"任务{task.id}停滞，请介入")
        elif task.escalationLevel == 2:
            self._wake_agent("shangshu", f"任务{task.id}长期停滞，请企业级协调")

    def _rollback(self, task):
        """回滚：恢复到上一个稳定状态"""
        self._restore_snapshot(task)
        self._dispatch(task, task.snapshot.state)
```

### PUA监督机制集成

```python
class PUAIntegration:
    """PUA监督 - 注入到所有Agent"""

    def monitor(self, agent_id: str, context: dict):
        """监控Agent执行，触发PUA"""
        failure_count = context.get("failure_count", 0)

        if failure_count >= 2:
            pua_level = self._calculate_level(failure_count)
            pua_phrase = self._get_phrase(pua_level)
            self._inject_pua(agent_id, pua_phrase, level=pua_level)

        # 三条铁律
        self._enforce_rules(agent_id)

    def _calculate_level(self, failures: int) -> int:
        if failures >= 5: return 4  # L4: 毕业
        elif failures >= 4: return 3  # L3: 3.25
        elif failures >= 3: return 2  # L2: 拷问
        elif failures >= 2: return 1  # L1: 失望
        return 0
```

## 十三、外部集成与系统融合规则

> **目的**: 确保所有外部 Agent、Skill 和仓库在集成到 CyberTeam 系统时，必须遵循有机融合原则，而非简单的文件夹映射或物理堆叠。

### 规则一：外部集成原则（严禁映射）

**核心要求**: 项目中需要集成的外部 Agent 以及 Skill，**严禁使用映射关系**，必须**完全集成**到整个 CyberTeam 系统中。

**具体禁止**:
- 禁止创建 `external_agents/` 或 `legacy_skills/` 文件夹进行物理堆叠
- 禁止通过软链接、路径映射等方式"引用"外部组件
- 禁止保留原仓库的独立运行能力（必须融入系统）

**正确做法**:
- 将外部 Agent 的核心能力拆解，重写为 CyberTeam 的 Agent Definition
- 将外部 Skill 拆解为 Trigger + Workflow，融入现有 Skill 体系
- 确保所有组件通过 CyberTeam 的调用机制（CEO → 三省六部）统一调度

### 规则二：系统融合原则（有机融入五要素）

任何外部组件在并入 CyberTeam 系统前，必须完成以下**五个维度的融合分析**：

#### (a) 逻辑检查

在融合前，必须检查原 Agent、Skill 或仓库在融入后，CyberTeam 整体的运作逻辑是否自洽：

| 检查项 | 说明 |
|-------|------|
| 调用方式 | 原组件的调用入口是否对齐三省六部分层 |
| Agent 职责 | 是否需要定义新的 Agent 类型或复用现有 Agent |
| 状态机兼容 | 是否需要扩展现有状态流转 |
| 权限矩阵 | 是否需要新增权限或修改现有权限 |

#### (b) 数据流向

必须明确融合后的数据流：

| 数据流要素 | 必须回答 |
|-----------|---------|
| **Input** | 上游由谁提供信息？格式是什么？ |
| **Output** | 产出什么文件/内容？ |
| **下游传递** | 需要将什么传递给哪些部门或 Agent 继续工作？ |
| **反馈回路** | 结果如何回奏给皇上（用户）？ |

#### (c) 技能与逻辑

必须明确技能链及应用场景：

| 技能要素 | 必须明确 |
|---------|---------|
| **技能链** | 该组件的技能如何拆解？对应六部中的哪一部或多部？ |
| **应用场景** | 在何种场景下调用何种技能与逻辑？ |
| **触发条件** | 如何判断应该使用该组件？（太子分拣阶段） |
| **组合方式** | 是否需要与其他技能组合使用？ |

#### (d) 思维链

必须明确该组件解决问题的思维路径：

| 思维要素 | 必须明确 |
|---------|---------|
| **问题分析方式** | 该组件采用何种思维方式？（如：批判性思维、系统思维、第一性原理） |
| **决策逻辑** | 如何在多种方案中做出选择？ |
| **专家注入** | 需要激活哪些思维专家辅助？ |
| **质量保障** | 如何自我验证输出质量？ |

#### (e) 定位与资源

必须明确该组件在 CyberTeam 中的定位：

| 定位要素 | 必须明确 |
|---------|---------|
| **组织层级** | 管理层（六部三省会签）还是执行层（六部执行）？ |
| **中台工具** | 可调用哪些中台工具？（思维注入引擎、Dev-QA引擎、技能路由器等） |
| **资源清单** | 需要哪些资源支持？（Token预算、专家配额、并行数上限等） |
| **SLA 承诺** | 响应时间、吞吐量、质量指标的承诺是什么？ |

### 规则三：自动模式触发规则

**触发条件**: 在项目研发过程中，如果用户提到「自动模式」，则必须执行以下流程。

**触发后的动作**:

引用以下提示词，调用自主专家团解决上文中提出的所有问题，直至完成为止：

```
启动clawteam的Agent专家团，创建一个Agent团队，仔细检查所有问题并解决这些问题。
包括一些我没有发现的问题，Agent会仔细地去完成检查和解决。
启用不少于10个的专家团队进行并行处理所有问题。
其中使用PUA技能不断地push所有专家不断完成工作，
用goal-driven-main的伪代码形式，设立最终目标，
让整个团队持续发现问题解决问题，直到整个目标完成，
不死不休，使命必达！
```

**执行要求**:
1. **并行启动**: 至少 10 个专家 Agent 同时工作
2. **持续迭代**: 使用 goal-driven 循环，不断发现-解决-验证
3. **PUA 监督**: 全程使用 PUA 技能保持高能动性
4. **最终交付**: 使命必达，不达不休

**适用场景**:
- 用户明确说「自动模式」或「启动自动流程」
- 复杂多问题需要并行扫荡时
- 需要专家团深度介入的研发任务
