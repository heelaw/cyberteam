## 一、Dev-QA 循环实现方案

### 1.1 核心循环架构

基于 agency-agents 的 handoff-templates 设计，实现标准化 Dev-QA 循环：

```
┌─────────────────────────────────────────────────────────────────┐
│                        Dev-QA Loop                               │
│                                                                 │
│   ┌──────────┐    提交    ┌──────────┐    验证    ┌──────────┐ │
│   │ Developer │ ────────→│ Evidence  │ ────────→│  Pass?   │ │
│   │   Agent   │           │ Collector │           │          │ │
│   └──────────┘            └──────────┘            └────┬─────┘ │
│        ↑                                              │        │
│        │           ┌──────────┐                       │        │
│        └───────────│  Fix &   │                       │        │
│           修复      │ Resubmit │                       │        │
│                    └──────────┘                       │        │
│                                                    ┌──┴──┐    │
│                                                    │ Y/N │    │
│                                                    └──┬──┘    │
│                                                PASS   │   FAIL │
│                                                 ↓         ↓     │
│                                              完成    最多3次重试  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 循环流程定义

| 阶段 | 执行者 | 输出 | 触发条件 |
|------|--------|------|----------|
| **开发** | Developer Agent | 功能实现 | 任务分配 |
| **提交** | Developer Agent | QA Feedback Loop 文档 | 功能完成 |
| **验证** | Evidence Collector | PASS/FAIL 判定 | 收到提交 |
| **修复** | Developer Agent | 修复后的实现 | 收到 FAIL |
| **升级** | Agent Orchestrator | Escalation Report | 3次失败 |

### 1.3 关键实现机制

```yaml
dev_qa_loop:
  最大重试次数: 3
  重试超时: 30分钟/次
  失败升级条件: 3次重试均失败

  每次循环必须包含:
    - 具体问题描述
    - 期望 vs 实际对比
    - 修复指导
    - 文件修改清单
```

## 二、三级质量门控量化标准

### 2.1 门控层级定义

| 层级 | 名称 | 适用场景 | 量化指标 |
|------|------|----------|----------|
| **L1** | 完整性门控 | 所有输出 | 覆盖率 >= 95% |
| **L2** | 专业性门控 | 部门专项 | 专业评分 >= 80分 |
| **L3** | 可执行性门控 | 交付执行层 | 可执行度 >= 90% |

### 2.2 L1 完整性检查（量化指标）

```yaml
L1_量化检查项:
  KPI覆盖率:
    计算: 已覆盖KPI数 / 总KPI数 × 100%
    阈值: >= 95%

  功能覆盖率:
    计算: 已实现功能数 / 总功能需求数 × 100%
    阈值: >= 95%

  时间表完整性:
    检查项: 是否有明确里程碑
    阈值: 100%

  交付物完整性:
    检查项: 是否包含所有必需交付物
    阈值: 100%
```

### 2.3 L2 专业性检查（量化指标）

```yaml
L2_量化检查项:
  行业最佳实践符合度:
    计算: 符合项数 / 检查项总数 × 100%
    阈值: >= 80%

  数据支撑度:
    计算: 有数据支撑的论点 / 总论点 × 100%
    阈值: >= 80%

  风险识别覆盖率:
    计算: 已识别风险 / 已知风险 × 100%
    阈值: >= 90%

  备选方案完整度:
    检查: 是否有主方案 + 至少1个备选
    阈值: 是
```

### 2.4 L3 可执行性检查（量化指标）

```yaml
L3_量化检查项:
  歧义度:
    计算: 无歧义描述数 / 总描述数 × 100%
    阈值: >= 90%

  验收标准可量化率:
    计算: 可量化标准数 / 总标准数 × 100%
    阈值: >= 90%

  依赖关系清晰度:
    检查: 所有依赖是否明确标注
    阈值: 100%

  产出物完整性:
    检查: 是否包含所有必需产出物
    阈值: 100%
```

## 三、自动化质量检查流程设计

### 3.1 自动化检查架构

```
┌──────────────────────────────────────────────────────────────┐
│                    Auto-QA Pipeline                          │
│                                                              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │  Lint   │→ │ Schema  │→ │  L1     │→ │  L2     │       │
│  │ Check   │  │ Check   │  │ Auto    │  │ Auto    │       │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘       │
│       ↓            ↓            ↓            ↓              │
│  ┌──────────────────────────────────────────────────┐       │
│  │           Quality Report Generator               │       │
│  │  - 通过/失败判定                                   │       │
│  │  - 量化分数                                        │       │
│  │  - 问题清单                                       │       │
│  └──────────────────────────────────────────────────┘       │
│                          ↓                                    │
│              ┌─────────────────────┐                         │
│              │  Pass → 标记完成     │                         │
│              │  Fail → 生成修复指令  │                         │
│              └─────────────────────┘                         │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 检查规则配置

```yaml
auto_check_rules:
  L1_自动检查:
    - format_validation: JSON格式正确性
    - required_fields: 必填字段存在性
    - type_validation: 数据类型正确性
    - coverage_check: KPI/功能覆盖率

  L2_自动检查:
    - best_practice_scan: 最佳实践模式匹配
    - data_source_check: 数据来源标注检查
    - risk_pattern_scan: 风险模式识别
    - alternative_plan_check: 备选方案检查

  L3_自动检查:
    - ambiguity_detection: 模糊描述检测
    - quantifiability_check: 可量化标准检查
    - dependency_mapping: 依赖关系验证
    - deliverable_check: 交付物完整性
```

## 四、失败重试机制防无限循环设计

### 4.1 重试计数器设计

```yaml
retry_mechanism:
  最大重试次数: 3
  计数器存储: 任务元数据

  重试状态机:
    初始状态: ATTEMPT_0
    第1次失败: ATTEMPT_1 (允许重试)
    第2次失败: ATTEMPT_2 (允许重试)
    第3次失败: ATTEMPT_3 (升级处理)

  升级触发条件:
    - 重试次数 >= 3
    - 或者 单次重试超时 (30分钟)
    - 或者 检测到系统性问题
```

### 4.2 防止无限循环的保障机制

| 机制 | 实现方式 | 作用 |
|------|----------|------|
| **硬性上限** | 3次重试后强制升级 | 防止无限循环 |
| **超时保护** | 单次重试30分钟上限 | 防止单次卡死 |
| **问题累积** | 每次失败记录所有问题 | 避免重复相同问题 |
| **根因分析** | 升级前必须提供RCA | 防止治标不治本 |
| **人工介入** | 3次失败后CEO/Orchestrator处理 | 最终保障 |

### 4.3 升级路径定义

```
重试流程
    │
    ├── 尝试 1: 失败 → 记录问题 → 返回修复
    │
    ├── 尝试 2: 失败 → 记录问题 + 尝试不同方案 → 返回修复
    │
    ├── 尝试 3: 失败 → 生成 Escalation Report → 升级到 Orchestrator
    │
    └── 升级后决策:
          ├── 选项A: 重新分配 (给不同Agent)
          ├── 选项B: 分解任务 (拆分为子任务)
          ├── 选项C: 重新设计 (需要架构变更)
          ├── 选项D: 接受现状 (记录限制)
          └── 选项E: 推迟处理 (下个sprint)
```

### 4.4 循环终止条件

```yaml
termination_conditions:
  成功条件:
    - QA 验证通过 (PASS)

  失败终止条件:
    - 达到最大重试次数 (3次)
    - 达到总超时时间 (90分钟)
    - 检测到不可恢复错误
    - 人工决定终止

  终止后动作:
    - 生成完整的 Escalation Report
    - 记录所有尝试和失败原因
    - 提供决策建议清单
    - 通知相关责任人
```
