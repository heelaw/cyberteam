## 一、四大系统核心能力分析

### 1.1 cyberteam-v2 — 引擎系统

| 维度 | 核心能力 |
|------|---------|
| **思维注入** | `thinking_injector.py` (547行)，从 `expert_catalog.json` 动态加载100+思维专家，基于8类意图自动触发 |
| **三层架构** | CEO(5W1H1Y+MECE拆解) → 管理层(战略/产品/技术/设计总监) → 执行层(180个agency-agents) |
| **引擎矩阵** | ThinkingInjector + RoutingEngine + DevQAEngine + QualityGateEngine |
| **适配器** | ClawTeamAdapter + PUAAdapter + GoalDriverAdapter + AgencyAdapter |
| **质量保障** | 三级门控(L1完整性≥95%/L2专业度≥80/L3可执行性≥90) |
| **状态机** | 线性三层：CEO → Management → Execution |

**优势**: 引擎完整、意图分类成熟、适配器齐全、思维专家库丰富
**劣势**: 缺少制度化流程管控、协作模式偏线性、无看板可视化

### 1.2 CyberTeam v2.1 — 运营专家系统

| 维度 | 核心能力 |
|------|---------|
| **专家矩阵** | 8大自研运营专家(投资人/策略执行/用户激励/框架思维/规划/活动运营/新媒体/团队协作) |
| **部门体系** | 6个部门Agent(产品/运营/设计/工程/HR/财务) |
| **平台专业** | 10个中国平台Agent(小红书/抖音/微信/B站/百度SEO/私域/淘宝/京东/拼多多/快手) |
| **Skill库** | 30+自研Skill，覆盖运营/增长/内容/人力全流程 |
| **引擎系统** | Dev-QA-Loop + PUA-Power + Quality-Gate 三大引擎 |
| **工作流** | gstack技能链编排(office-hours→roadmap→plan→execute→review→qa→ship→retro) |

**优势**: 运营专业度极高、Skill可复用、中国平台覆盖全
**劣势**: 依赖gstack工具、自研Agent需外部Agent系统调用、缺少底层引擎

### 1.3 思考天团agents — 思维专家库

| 维度 | 核心能力 |
|------|---------|
| **规模** | 100个独立Agent，每个约295行 |
| **覆盖** | 决策(卡尼曼/贝叶斯/博弈论)、执行(WBS/GROW/OKR)、创新(六顶思考帽/TRIZ)、管理(McKinsey7S/Kotter)、专业(100种思维模型) |
| **结构** | 每个Agent含 `AGENT.md` + `evals/` + `references/` + `assess/` 四部分 |
| **评测体系** | 完整评估框架，evals/references/assess三分支 |

**优势**: 思维模型极其丰富、每个Agent有完整评测体系
**劣势**: 100个独立Agent无法自动组合、缺少引擎驱动、无法直接融入协作流程

### 1.4 edict 三省六部 — 制度化流程系统

| 维度 | 核心能力 |
|------|---------|
| **制度设计** | 帝国分权制衡：皇上→太子→中书省→门下省→尚书省→六部 |
| **状态机** | 9个状态(Pending→Taizi→Zhongshu→Menxia→Assigned→Doing/Next→Review→Done/Cancelled) |
| **权限矩阵** | 严格分层调用权限(太子只能调中书，中书只能调门下+尚书，不可越级) |
| **质量保障** | 门下省必审议，中书方案可封驳返回重做(最多3轮) |
| **可观测性** | React看板(10个面板) + 实时活动流(59条/任务) + Agent心跳检测 |
| **技术栈** | React看板(无后端依赖) + Python状态机 + JSONL日志 |

**优势**: 制度化质量保障最强、分权制衡严谨、可视化程度高、流程可干预
**劣势**: 缺少思维注入能力、运营专家专业度弱、Skill体系薄弱

## 二、融合架构设计

### 2.1 融合策略

| 来源 | 角色 | 融入方式 |
|------|------|---------|
| **edict 三省六部** | 流程骨架 | 作为**制度化协作框架**，替代v2的线性三层 |
| **cyberteam-v2 引擎** | 思维驱动 | ThinkingInjector + IntentClassifier 注入到所有Agent |
| **CyberTeam v2.1 运营** | 专业化执行 | 8大专家 + 6部门 + 10平台Agent 融入六部 |
| **思考天团** | 技能库 | 100个思维Agent作为Skills池，按需引用 |

**融合公式**: `CyberTeam v3 = edict制度骨架 + v2引擎驱动 + v2.1运营专业 + 天团思维库`

## 三、组件职责定义

### 3.1 三省六部职责重映射

| 部门 | 原职责 | v3融合职责 | 思维注入 |
|------|--------|-----------|---------|
| **太子** | 任务分拣、消息接入 | 5W1H1Y+MECE拆解、闲聊/旨意识别、intent分类 | 卡尼曼(判断偏差)、批判思维 |
| **中书省** | 方案起草、任务拆解 | 调用8大运营专家参与讨论、生成子任务清单 | 100思维专家按需注入 |
| **门下省** | 方案审议、封驳重做 | 三级质量门控、完整性和专业度审查 | 卡尼曼(决策审查)、McKinsey7S |
| **尚书省** | 派发执行、结果汇总 | 任务路由、六部协调、进度汇总回调 | GROW(目标追踪)、OKR(进度) |
| **吏部(HR)** | 人力资源 | 团队组建、Agent能力匹配、协作治理 | McKinsey7S、Kotter变革 |
| **户部(财务)** | 数据分析 | 成本估算、ROI评估、资源分配 | DCF、LTV/CAC、单位经济 |
| **兵部(工程)** | 代码实现 | 开发执行、架构设计、技术选型 | 12-Factor、系统设计、Tradeoff |
| **礼部(内容)** | 文档编制 | 内容创作、新媒体运营、品牌策略 | 六顶思考帽、精益创业 |
| **刑部(QA)** | 测试审查 | Dev-QA-Loop、端到端测试、安全审计 | Five Whys、鱼骨图 |
| **工部(DevOps)** | 基础设施 | CI/CD、部署、监控、日志 | 12-Factor、Terraform |

## 四、技术栈选择

### 4.1 技术栈总览

| 层级 | 技术选型 | 来源 | 理由 |
|------|---------|------|------|
| **协作编排** | OpenClaw / ClawTeam CLI | v2 | 104+Agent、任务管理、团队协作 |
| **流程框架** | edict三省六部状态机 | edict | 制度化保障、权限矩阵、封驳机制 |
| **思维注入** | v2 ThinkingInjector | v2 | 成熟意图分类、100+专家、自动触发 |
| **质量门控** | v2 QualityGateEngine + 门下省 | v2+edict | 三级门控+制度审议双重保障 |
| **前端看板** | React + TailwindCSS | edict | 10个面板、实时更新、零后端依赖 |
| **后端引擎** | Python 3.10+ | v2 | thinking_injector/routing/dev_qa 全Python |
| **持久化** | JSONL (flow_log + progress_log) | edict | 无数据库依赖、便于调试 |
| **Skills库** | Markdown格式 | v2.1 | 60+Skill可被Agent直接引用 |
| **评测体系** | 天团 evals/references/assess | 天团 | 100个Agent完整评测框架 |

## 六、关键技术决策

### 6.1 edict vs v2 流程骨架选择

| 对比维度 | edict三省六部 | v2线性三层 |
|---------|-------------|-----------|
| 质量保障 | **强** (门下省封驳) | 弱 (仅门控) |
| 可干预性 | **高** (任意节点) | 低 |
| 制度约束 | **严格** (权限矩阵) | 无 |
| 复杂度 | **高** (9状态+封驳循环) | 低 |
| 适合场景 | **复杂任务/正式交付** | 简单快速执行 |

**决策**: 选择 **edict三省六部** 作为v3骨架，但需简化封驳循环(最多2轮而非3轮)以平衡效率。

### 6.2 思维注入位置选择

| 方案 | 说明 | 优劣势 |
|------|------|--------|
| 全层注入 | 三省六部全部注入思维 | 思维一致性强，但开销大 |
| 核心层注入 | 太子+中书+门下注入，尚书/六部按需 | **推荐**: 平衡思维质量与效率 |

**决策**: 采用 **核心层注入** — 太子(5W1H1Y)、中书(按需选择)、门下(质量审查)、六部(按任务类型从思维库注入)。

### 6.3 思考天团100个Agent 融入路径

| 方案 | 说明 | 优劣势 |
|------|------|--------|
| 直接作为Agent调用 | 每个天团Agent独立运行 | 太重，100个Agent不可管理 |
| 转为Skill引用 | AGENT.md → SKILL.md，ThinkingInjector按需注入 | **推荐**: 与现有Skills体系一致 |
| 仅取评测框架 | 只用evals/references/assess | 丢失思维模型本身 |

**决策**: 采用 **Skill转化** — 将100个AGENT.md转换为SKILL.md格式，ThinkingInjector增强后支持从`thinking_library/`目录加载，IntentClassifier增加天团触发词。

### 6.4 v2.1运营专家定位

| 定位 | 融入方式 |
|------|---------|
| 中书省参与者 | 8大专家作为"幕僚"参与中书省方案讨论 |
| 六部专业支撑 | 10平台Agent作为兵部(工程)的平台执行专家 |
| Skills来源 | 60+自研Skill部署到`skills/`目录供全系统引用 |

## 七、总结

CyberTeam v3 通过以下融合实现架构升级:

1. **制度骨架**: edict三省六部的分权制衡 + 封驳机制 + 可视化看板
2. **思维驱动**: v2 ThinkingInjector 覆盖核心层 + 100个天团思维专家作为Skills池
3. **运营专业**: v2.1 8大专家(中书省幕僚) + 6部门 + 10平台Agent(六部执行)
4. **质量保障**: 门下省制度审议 + 三级QualityGateEngine 双重门控
5. **底层适配**: 4个适配器(ClawTeam/PUA/Goal/Agency) + gstack技能链 + edict观测

最终实现: **"一句话启动，制度化执行，思维专家全程护航，运营专业深度支撑"** 的AI模拟公司协作系统。
