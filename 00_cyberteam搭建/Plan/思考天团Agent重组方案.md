# 思考天团 Agent 重组方案

## 结论

最终形态建议采用 **Skill + 主持人 Agent + 少量核心专家 Agent + 长尾模型库** 的混合架构。

不要把 100 个思维模型压成一个大 Agent，也不要把它们全部做成平级 Agent。最佳方案是：

1. `思考天团` 作为入口 Skill，负责触发、分流和模式选择。
2. `思考天团主持人` 作为唯一协调 Agent，负责路由、编排和整合。
3. 保留 8-12 个高价值核心专家 Agent。
4. 其余模型全部降级为“思维模型卡 / 提示词片段 / 参考资料”。

## 为什么这样做

### 1) 不能做成一个大 Agent

大而全会把三件事混在一起：理解问题、执行推理、输出结果。这样虽然看起来统一，但会带来三个问题：

- 路由逻辑越来越重。
- 输出风格越来越散。
- 任意一个模型失真都会污染整套系统。

### 2) 不能做成 100 个独立 Agent

V2 已经扩展到 100 个模型，但其中大量是同一类能力的不同切面，真正应该独立运行的并不多。

- 100 个 Agent 的维护成本太高。
- 路由噪声会远大于收益。
- 用户也很难理解该用谁。

### 3) Skill 更适合作为入口

V2 已经明确把系统定义成“协调层 Skill”，这比把它当成单一专家更合理。入口层适合做：

- 意图分类
- 模式选择
- 专家编排
- 输出模板控制

## 对现有两版的判断

### V1

V1 的优点是小而清楚，14 个专家已经能覆盖大多数高频思维场景。它适合作为“核心骨架”。

但 V1 已经出现明显漂移，例如 `goldlin` / `kiss` 的文件内容和命名是反的，说明它不适合直接作为最终版本原样保留。

### V2

V2 的优点是覆盖面大，已经把决策、偏差、战略、系统、概率、组织、演化等扩展开了。

但 V2 的问题也很明显：

- 模型数量膨胀过快。
- 很多模型只是同族近义词。
- 许多模型适合做“能力模块”，不适合做“独立人格 Agent”。

## 重组原则

1. **一个能力只保留一个 canonical slot**。
2. **只有输出契约不同、失败模式不同，才值得独立成 Agent**。
3. **所有重复/近义/过细模型，统一下沉到模型库**。
4. **路由只看任务，不看“模型收藏”**。
5. **每个核心 Agent 必须有明确触发场景、输入格式、输出格式和边界**。

## 推荐架构

```text
用户问题
  ↓
思考天团 Skill
  ↓
主持人 Agent
  ├── 决策与偏差组
  ├── 问题定义与诊断组
  ├── 战略与竞争组
  ├── 系统与复杂性组
  ├── 概率与不确定性组
  ├── 执行与迭代组
  └── 组织与成长组
        ↓
   模型库 / 提示词片段
        ↓
     最终报告
```

## 核心分层

### 第一层：入口 Skill

- 负责触发词识别。
- 负责把用户问题归类到“决策 / 诊断 / 战略 / 创新 / 执行 / 复盘 / 组织”之一。
- 负责选择并行、辩论、链式三种模式之一。

### 第二层：主持人 Agent

- 负责标准化输入。
- 负责挑选最相关的 2-5 个核心专家。
- 负责整合观点，不做“堆砌式罗列”。

### 第三层：核心专家 Agent

建议保留这些独立 Agent：

- `kahneman`：决策与认知偏差
- `problem-reformulation` / `goldlin`：问题定义与重构
- `fivewhy`：根因追问
- `first-principle`：第一性原理
- `swot-tows`：战略竞争分析
- `systems-thinking`：系统与层次整合
- `probabilistic` / `bayesian`：概率与不确定性
- `grow` + `wbs`：目标实现与执行拆解
- `kiss`：复盘与简化
- `manager-leap`：组织与角色跃迁
- `ai-board`：重大决策评审

### 第四层：模型库

其余模型下沉为模型库，作为：

- Prompt 片段
- 触发标签
- 分析检查清单
- 参考资料

## 归并建议

### 应保留为独立 Agent

| 组件 | 理由 |
|------|------|
| `kahneman` | 决策场景高频且有独立输出风格 |
| `fivewhy` | 根因分析与其他模型边界清晰 |
| `problem-reformulation` | 问题重构是独立入口能力 |
| `first-principle` | 创新与重构强相关，识别度高 |
| `swot-tows` | 战略分析是稳定场景 |
| `systems-thinking` | 复杂系统整合需要独立视角 |
| `probabilistic` / `bayesian` | 不确定性判断需要专门方法 |
| `grow` / `wbs` | 目标到执行的链路足够明确 |
| `kiss` | 复盘是独立工作流 |
| `manager-leap` | 管理者成长类问题有稳定边界 |
| `ai-board` | 多角色决策需要模拟董事会风格 |

### 应合并

| 旧模型 | 建议归并到 |
|--------|-----------|
| `goldlin` / `kiss` 命名错位 | 统一校正为问题定义 / 复盘两类 |
| `mckinsey` + `critical-thinking` + `falsification-principle` | 结构化思维能力组 |
| `reverse-thinking` + `inversion` + `counterfactual` | 逆向/反事实推理组 |
| `six-hats` + `dialectical` | 多视角与辩证组 |
| `opportunity-cost` + `sunk-cost` + `pareto` | 资源权衡组 |
| `confirmation-bias` + `availability-heuristic` + `representativeness-heuristic` | 偏差识别组 |
| `occams-razor` + `hanlons-razor` + `maslows-hammer` | 简化与克制组 |
| `anti-fragile` + `second-order` + `systems-thinker` | 复杂性与韧性组 |

### 应降级为模型库

| 类别 | 处理方式 |
|------|----------|
| 行为偏差 | 作为偏差标签与检查清单 |
| 竞争理论 | 作为战略分支模块 |
| 组织/管理法则 | 作为管理层提示卡 |
| 认知启发式 | 作为路由辅助规则 |
| 心理学理论 | 作为输入约束与风险提示 |

## 迁移步骤

1. 先冻结现有命名，建立 canonical mapping。
2. 把 V1 的 14 个核心专家作为第一批骨架。
3. 从 V2 中挑选 8-12 个真正独立的专家补齐能力边界。
4. 剩余 70+ 模型全部转为模型库。
5. 再重写路由器，避免继续按“模型收藏”思路扩展。

## 最终判断

如果只选一个顶层形态，选 **Skill**。

如果要落地可用，必须是 **Skill + Agent + 模型库** 的混合体。

这才是既能控复杂度，又能保留思维深度的版本。
