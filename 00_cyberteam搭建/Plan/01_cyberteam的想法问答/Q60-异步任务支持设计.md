# Q60: 思考天团异步任务支持设计

**问题**: 用户离线时任务如何继续执行？任务完成后如何通知用户？用户如何获取结果？

---

## 一、异步任务场景

### 1.1 核心场景定义

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          异步任务触发场景                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  场景1: 用户离线                                                            │
│  ┌─────────┐    任务执行中    ┌─────────┐    任务完成     ┌─────────┐    │
│  │ 用户在线 │ ──▶ │    │ ──▶ │ 用户离线 │ ──▶ │  │ ──▶ │ 用户点击 │    │
│  │ 提交问题 │    │ 继续执行 │    │         │    │ 通知 │    │ 查看报告 │    │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    │
│                                                                             │
│  场景2: 长时间任务                                                          │
│  ┌─────────┐    长时间运行    ┌─────────┐    定期通知     ┌─────────┐        │
│  │ 用户提交 │ ──▶ │    │ ──▶ │ 用户离开 │ ──▶ │  │ ──▶ │ 用户查看 │        │
│  │ 复杂问题 │    │ 30分钟+ │    │ 电脑前   │    │ 进度 │    │ 中期报告 │        │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    │
│                                                                             │
│  场景3: 定时任务                                                            │
│  ┌─────────┐    等待触发     ┌─────────┐    自动执行     ┌─────────┐        │
│  │ 用户配置 │ ──▶ │    │ ──▶ │ 系统定时 │ ──▶ │  │ ──▶ │ 用户收到 │        │
│  │ 周报分析 │    │ 定时条件 │    │ 触发    │    │ 通知 │    │ 完整报告 │        │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 场景详细说明

| 场景 | 触发条件 | 用户状态 | 系统行为 |
|------|----------|----------|----------|
| **用户离线** | 用户提交问题后断开连接 | 离线 | 继续执行，完成后通知 |
| **长时间任务** | 任务预估时长 > 5分钟 | 可能在线/离线 | 定期进度通知，完成后通知 |
| **定时任务** | 用户设置定时触发 | 不在系统 | 到时自动执行，完成后通知 |
| **批量处理** | 用户提交多个问题 | 可能离线 | 队列执行，完成后汇总通知 |

### 1.3 任务类型与异步适配

| 任务类型 | 是否异步 | 异步原因 | 预计时长 |
|----------|----------|----------|----------|
| 简单问题分析 | 可选 | 用户离线 | 1-3分钟 |
| 复杂问题辩论 | 必须 | 时长 > 5分钟 | 10-30分钟 |
| 多专家交叉辩论 | 必须 | 多轮交互 | 30-60分钟 |
| 定时周报生成 | 必须 | 用户不在场 | 5-15分钟 |
| 批量问题处理 | 必须 | 任务队列 | 视任务数 |

### 2.2 通知类型与触发条件

| 通知类型 | 触发条件 | 通知内容 | 渠道 |
|----------|----------|----------|------|
| **任务开始** | 用户提交问题后立即 | 问题ID、预估时长、开始时间 | 可选 |
| **进度更新** | 每完成一个专家分析 | 当前进度、正在进行中的专家 | 推送 |
| **任务完成** | 所有专家完成分析 | 报告链接、报告摘要 | 邮件+推送 |
| **任务失败** | 执行过程中出错 | 错误原因、是否可重试 | 邮件+推送 |
| **辩论提醒** | 专家间需要用户输入 | 辩论主题、等待时间 | 推送 |
| **超时提醒** | 任务超时未完成 | 剩余预估时间、是否继续 | 推送 |

### 2.4 通知优先级

| 优先级 | 通知类型 | 渠道组合 | 打扰程度 |
|--------|----------|----------|----------|
| **P0 - 紧急** | 任务失败、辩论需要用户输入 | 邮件+推送+短信 | 高 |
| **P1 - 重要** | 任务完成 | 邮件+推送 | 中 |
| **P2 - 普通** | 进度更新 | 仅推送 | 低 |
| **P3 - 静默** | 定时任务开始 | 仅记录 | 无 |

### 3.1 结果交付方式

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            结果获取方式                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  方式1: 报告链接                                                            │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  邮件/推送中的链接 → 点击 → Web查看报告 → 下载PDF/分享               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  方式2: Webhook回调                                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  用户配置回调URL → 任务完成 → POST结果到回调URL → 用户系统处理        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  方式3: API拉取                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  用户调用 GET /api/tasks/{id}/result → 返回完整报告/摘要/结构化数据   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  方式4: 邮件附件                                                            │
│  ┌─────────────────────────────────────────────────────────────────────┐   │

│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 报告访问权限

| 访问方式 | 权限要求 | 超时时间 | 适用场景 |
|----------|----------|----------|----------|
| **临时链接** | 登录用户 | 24小时 | 邮件/推送中的链接 |
| **长期链接** | 登录用户+报告所有者 | 永久 | API/用户自己的报告列表 |
| **Webhook** | 已配置回调URL的用户 | 实时 | 自动化工作流集成 |
| **下载附件** | 接收邮件的用户 | 邮件有效期 | 离线查看 |

## 四、任务管理机制

### 4.1 任务状态机

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            任务状态机                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │

│                                          │                                 │
│                         ┌─────────────────┼─────────────────┐              │
│                         ▼                 ▼                 ▼               │
│                  ┌─────────┐       ┌─────────┐       ┌─────────┐          │
│           手动暂停 │ RUNNING │       │ PAUSED │       │ QUEUED  │          │

│                  └────┬────┘       └─────────┘       └─────────┘          │

│          ┌────────────┼────────────┐                                     │
│          ▼            ▼            ▼                                     │
│   ┌───────────┐ ┌───────────┐ ┌───────────┐                              │
│   │COMPLETED  │ │ FAILED    │ │ CANCELLED │                              │
│   │ 成功完成  │ │  执行失败  │ │  用户取消  │                              │

│                                                                             │
│                                                                             │
│  状态说明:                                                                 │
│  - CREATED:   任务刚创建，还未加入执行队列                                  │
│  - QUEUED:    已加入队列，等待调度器分配资源                               │
│  - RUNNING:   正在执行中                                                   │
│  - PAUSED:    暂停（用户手动暂停或系统资源不足）                          │
│  - COMPLETED: 成功完成                                                     │
│  - FAILED:    执行失败（可重试）                                           │
│  - CANCELLED: 用户主动取消                                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 4.2 任务管理操作

| 操作 | CLI命令 | API调用 | 触发条件 | 权限 |
|------|---------|---------|----------|------|
| **暂停** | `thinking-team pause q_001` | `POST /api/tasks/{id}/pause` | 用户手动/系统资源不足 | 任务所有者 |
| **恢复** | `thinking-team resume q_001` | `POST /api/tasks/{id}/resume` | 暂停后继续执行 | 任务所有者 |
| **取消** | `thinking-team cancel q_001` | `POST /api/tasks/{id}/cancel` | 用户主动取消 | 任务所有者 |
| **重试** | `thinking-team retry q_001` | `POST /api/tasks/{id}/retry` | 失败后可重试 | 任务所有者 |
| **查看** | `thinking-team status q_001` | `GET /api/tasks/{id}` | 任意时刻 | 任务所有者 |
| **删除** | `thinking-team delete q_001` | `DELETE /api/tasks/{id}` | 完成后可删除 | 任务所有者 |

### 4.4 暂停/恢复行为规则

| 操作 | 行为 | 适用场景 | 限制 |
|------|------|----------|------|
| **暂停** | 当前专家完成当前子任务后停止，不杀死进程 | 用户想稍后继续 | 不能暂停单个专家 |
| **恢复** | 从暂停位置继续执行，不重复已完成工作 | 用户回来继续 | 必须7天内恢复 |
| **取消** | 立即终止所有执行，清理临时文件 | 用户不再需要结果 | 不可恢复 |
| **重试** | 从失败节点重新执行，保留已完成部分 | 临时错误 | 最多3次 |

## 六、CLI操作与API

### 6.1 用户命令

```bash
# 1. 提交异步任务（用户离线模式）
$ thinking-team ask "如何提升DAU 10%？" --async --notify email,push
# → 创建 task.json，设置 notify 字段

# 2. 查看任务状态（包含异步状态）
$ thinking-team status q_001
# ← 读取 task.json，返回异步相关信息

# 3. 暂停任务
$ thinking-team pause q_001
# → 更新 task.json，状态 → PAUSED

# 4. 恢复任务
$ thinking-team resume q_001
# → 更新 task.json，状态 → RUNNING

# 5. 取消任务
$ thinking-team cancel q_001
# → 更新 task.json，状态 → CANCELLED

# 6. 重试失败任务
$ thinking-team retry q_001
# → 更新 task.json，重试计数+1

# 7. 配置Webhook
$ thinking-team webhook q_001 set https://example.com/callback
# → 创建 callback/webhook.json

# 8. 查看通知历史
$ thinking-team notify history q_001
# → 读取 notify/log.json
```

### 6.2 API接口

```bash
# 任务管理
POST   /api/tasks                  # 创建任务（支持异步）
GET    /api/tasks/{id}             # 获取任务状态
POST   /api/tasks/{id}/pause       # 暂停任务
POST   /api/tasks/{id}/resume      # 恢复任务
POST   /api/tasks/{id}/cancel      # 取消任务
POST   /api/tasks/{id}/retry       # 重试任务

# 通知配置
PUT    /api/tasks/{id}/notify      # 更新通知设置
GET    /api/tasks/{id}/notify      # 获取通知设置
POST   /api/tasks/{id}/webhook     # 配置Webhook

# 结果获取
GET    /api/tasks/{id}/result      # 获取任务结果
GET    /api/tasks/{id}/report      # 获取完整报告
GET    /api/tasks/{id}/pdf         # 下载PDF报告
```

### 7.1 兼容性说明

### 7.2 向后兼容性

- **异步任务**: 创建 `task.json`，不影响原有文件结构
- **混合模式**: 同时支持同步和异步，由用户决定

## 八、总结

### 8.1 异步任务处理流程

```
用户提交问题
      │
      ▼
是否异步模式？ ──否──▶ 同步执行（Q21原有流程）
      │
     是
      ▼
创建task.json（异步配置）
      │
      ▼
加入执行队列
      │
      ├──── 进度更新通知（每完成1个专家）
      │
      ▼
任务完成
      │
      ▼
生成报告 ──▶ 发送通知（邮件+推送）
      │
      ▼
用户获取结果
  ├── 点击链接查看报告
  ├── Webhook回调
  ├── API拉取
  └── 邮件附件
```

### 8.2 关键设计原则

| 原则 | 说明 |
|------|------|
| **用户知情** | 任务开始、进度、完成全程通知用户 |
| **结果可达** | 无论用户是否在线，都能获取结果 |
| **操作可逆** | 暂停可恢复，取消有确认 |
| **向后兼容** | 不改变Q21的同步执行流程 |
| **灵活通知** | 支持多渠道、用户可配置 |
