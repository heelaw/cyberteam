# Q61: 思考天团专家Agent版本管理设计

**问题**: 如何设计专家Agent的版本管理机制，支持版本追溯和回滚？

---

## 一、版本管理概述

### 1.1 设计背景

专家Agent（Expert Agent）是思考天团的核心组件，每个专家Agent由以下核心要素构成：

| 要素 | 说明 | 示例 |
|------|------|------|
| **Prompt/提示词** | 定义专家的行为模式、思维框架、输出风格 | Kahneman的系统性思考模式 |
| **参数配置** | 控制LLM调用参数、输出格式、工具权限 | temperature、max_tokens |
| **能力定义** | 专家擅长的思维方法、分析维度 | 第一性原理、5Why溯源 |
| **边界约束** | 明确专家不适合的场景 | Kahneman不处理纯技术问题 |

随着专家Agent的迭代优化，需要版本管理机制来：
- 追踪每次变更的内容
- 在新版本有问题时快速回滚
- 记录每个版本的适用场景

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **可追溯** | 每次变更都有完整记录，支持任意版本回溯 |
| **可回滚** | 一键切换到历史版本，确保线上稳定性 |
| **可对比** | 支持版本间差异对比，了解变更内容 |
| **可审计** | 版本变更有明确的原因、时间和责任人 |

### 1.3 与现有体系的关系

```

     │                      │
     │                      ├── 版本能力映射（v1.0=L2, v1.1=L2升级）
     │                      └── 版本升级路径记录

     │                    │
     │                    ├── 不同版本独立缓存

     │                    │
```

## 二、版本管理内容

### 2.1 版本管理的要素

专家Agent的版本管理包含以下四个维度：

```
专家Agent版本
│
├── 提示词版本 (Prompt Version)
│   ├── 核心指令 (Core Instructions)
│   ├── 思维框架 (Thinking Framework)
│   ├── 输出模板 (Output Template)
│   └── 示例库 (Examples)
│
├── 参数版本 (Parameters Version)
│   ├── LLM参数 (temperature, top_p, max_tokens)
│   ├── 输出格式 (format, length, style)
│   └── 工具权限 (tools, capabilities)
│
├── 能力版本 (Capabilities Version)
│   ├── 擅长场景 (strengths)
│   ├── 不擅长场景 (limitations)
│   └── 协作模式 (collaboration)
│
└── 边界版本 (Boundary Version)
    ├── 适用边界 (applicable_scope)
    ├── 禁忌场景 (forbidden_scope)
    └── 升级条件 (upgrade_criteria)
```

### 2.2 提示词版本 (Prompt Version)

**定义**: 专家的核心Prompt内容变更

| 子要素 | 说明 | 变更频率 |
|--------|------|----------|
| 核心指令 | 专家的角色定义、核心行为 | 低 |
| 思维框架 | 分析问题的思维方式 | 中 |
| 输出模板 | 结果输出格式规范 | 中 |
| 示例库 | 少样本学习示例 | 高 |

**版本标识**: `prompt.v1`, `prompt.v2`, ...

### 2.3 参数版本 (Parameters Version)

**定义**: 控制LLM行为的配置参数

| 参数 | 说明 | 示例值 |
|------|------|--------|
| temperature | 创造性程度 | 0.7 |
| top_p | 采样阈值 | 0.9 |
| max_tokens | 最大输出 | 2000 |
| presence_penalty | 重复惩罚 | 0.5 |
| frequency_penalty | 频率惩罚 | 0.5 |

**版本标识**: `params.v1`, `params.v2`, ...

### 2.4 能力版本 (Capabilities Version)

**定义**: 专家能力的量化描述

| 维度 | 说明 | 来自 |
|------|------|------|
| 擅长场景 | 专家适合的问题类型 | Q23能力边界 |
| 评级等级 | L1/L2/L3级别 | Q24能力评级 |
| 协作推荐 | 最佳协作专家 | Q24协作建议 |

**版本标识**: `capability.v1`, `capability.v2`, ...

### 2.5 边界版本 (Boundary Version)

**定义**: 专家的能力边界定义

| 边界类型 | 说明 |
|----------|------|
| 适用边界 | 什么场景适合使用该专家 |
| 禁忌边界 | 什么场景绝对不能使用 |
| 模糊边界 | 需要人工判断的场景 |

**版本标识**: `boundary.v1`, `boundary.v2`, ...

## 三、版本号命名规范

### 3.1 语义化版本号 (Semantic Versioning)

采用 `X.Y.Z` 三段式版本号：

```
{v}.{major}.{minor}

├── v: 版本前缀（固定）
│
├── major: 主版本号
│   ├── 含义: 重大变更，可能不兼容
│   ├── 触发条件:
│   │   ├── 核心Prompt架构调整
│   │   ├── 能力边界重大变化
│   │   └── 评级等级变化
│   └── 示例: v1.0.0 → v2.0.0
│
├── minor: 次版本号
│   ├── 含义: 新增功能，向后兼容
│   ├── 触发条件:
│   │   ├── 新增擅长场景
│   │   ├── Prompt优化（不破坏核心）
│   │   └── 参数微调
│   └── 示例: v1.0.0 → v1.1.0
│
└── patch: 补丁版本号
    ├── 含义: Bug修复，向后兼容
    ├── 触发条件:
    │   ├── 输出格式修复
    │   ├── 边界描述修正
    │   └── 示例补充
    └── 示例: v1.0.0 → v1.0.1
```

### 3.2 版本号示例

| 专家 | 当前版本 | 变更类型 | 新版本 | 变更说明 |
|------|----------|----------|--------|----------|
| Kahneman | v1.0.0 | patch | v1.0.1 | 修复输出格式问题 |
| Kahneman | v1.0.1 | minor | v1.1.0 | 新增决策场景支持 |
| Kahneman | v1.1.0 | major | v2.0.0 | 增加系统性思考框架 |
| FirstPrinciple | v1.0.0 | minor | v1.1.0 | 新增技术创新场景 |
| SixHats | v1.0.0 | patch | v1.0.1 | 补充创意帽示例 |

### 3.3 版本生命周期

```
版本生命周期
│
├── 开发版 (Development)
│   ├── 标识: -dev.N
│   ├── 示例: v1.1.0-dev.1
│   └── 用途: 内部测试
│
├── 测试版 (Beta)
│   ├── 标识: -beta.N
│   ├── 示例: v1.1.0-beta.1
│   └── 用途: 小范围试用
│
├── 稳定版 (Stable) ← 正式发布
│   ├── 标识: 无后缀
│   ├── 示例: v1.1.0
│   └── 用途: 正式环境使用
│
└── 废弃版 (Deprecated)
    ├── 标识: -deprecated
    ├── 示例: v1.0.0-deprecated
    └── 用途: 保留但不再推荐
```

### 3.4 版本命名规则

| 规则 | 说明 |
|------|------|
| 前缀 | 必须以 `v` 开头 |
| 分隔符 | 使用 `.` 分隔版本号 |
| 数字 | 仅使用阿拉伯数字 0-9 |
| 升级 | 版本号只能递增，不能重复 |
| 预发布 | 在正式版本后可加 `-beta.N` |

## 四、回滚机制设计

### 4.1 回滚类型

| 回滚类型 | 说明 | 触发场景 |
|----------|------|----------|
| **Prompt回滚** | 回滚提示词内容 | 输出质量下降、风格变化 |
| **参数回滚** | 回滚LLM参数配置 | 响应异常、输出不稳定 |
| **能力回滚** | 回滚能力定义变更 | 边界定义错误、评级偏差 |
| **全量回滚** | 回滚所有要素到历史版本 | 重大故障、全面回退 |

### 4.2 回滚流程

```
回滚执行流程
│
├── 1. 触发条件
│   ├── 手动触发: 人工确认回滚
│   │   └── 场景: 发现质量问题
│   │
│   └── 自动触发: 监控系统检测异常
│       └── 场景: 错误率升高、满意度下降
│
├── 2. 版本选择
│   ├── 指定版本: 回滚到指定版本
│   │   └── 命令: /expert rollback Kahneman v1.0.0
│   │
│   └── 自动选择: 回滚到上一个稳定版
│       └── 命令: /expert rollback Kahneman --previous
│
├── 3. 影响评估
│   ├── 缓存影响: 清除相关缓存
│   ├── 依赖影响: 通知相关专家
│   └── 用户影响: 记录切换日志
│
├── 4. 执行回滚
│   ├── 更新活跃版本指针
│   ├── 同步到所有实例
│   └── 验证回滚结果
│
└── 5. 验证确认
    ├── 功能验证: 测试核心功能
    ├── 质量验证: 检查输出质量
    └── 监控验证: 确认指标正常
```

### 4.3 回滚命令设计

| 命令 | 说明 | 示例 |
|------|------|------|
| `/expert rollback {name}` | 回滚到上一版本 | `/expert rollback Kahneman` |
| `/expert rollback {name} {version}` | 回滚到指定版本 | `/expert rollback Kahneman v1.0.0` |
| `/expert rollback {name} --list` | 查看可回滚版本 | `/expert rollback Kahneman --list` |
| `/expert rollback {name} --dry-run` | 模拟回滚（不执行） | `/expert rollback Kahneman --dry-run` |

### 4.4 回滚保护机制

| 保护措施 | 说明 |
|----------|------|
| **最小版本保护** | 禁止回滚到过于古老的版本（如超过5个主版本） |
| **强制确认** | 涉及主版本回滚需要二次确认 |
| **灰度回滚** | 先切换10%流量，确认无异常后再全量 |
| **快速回切** | 回滚后可立即回切到新版本 |
| **回滚日志** | 记录每次回滚的原因、操作人、时间 |

### 4.5 回滚场景示例

#### 场景1: Prompt问题回滚

```
场景: Kahneman v1.1.0 发布后，用户反馈输出变得冗长

分析:
- v1.1.0 变更: 增加了"详细分析"输出模板
- 问题: 新模板导致输出过长，影响用户体验
- 影响: 缓存命中率下降，满意度评分下降

处理:
1. 执行回滚: /expert rollback Kahneman v1.0.1
2. 验证: 检查输出长度恢复正常
3. 记录: 原因"输出模板导致过长"

结果:
- 活跃版本: v1.0.1
- v1.1.0 状态: deprecated
- 下一步: 优化模板后重新发布 v1.1.1
```

#### 场景2: 参数调整回滚

```
场景: FirstPrinciple v1.1.0 的 temperature 从 0.7 调整为 0.9 后，输出变得不稳定

分析:
- 参数变更: temperature: 0.7 → 0.9
- 问题: 创新性过度，逻辑性下降
- 影响: 用户投诉增加

处理:
1. 执行回滚: /expert rollback FirstPrinciple --previous
2. 验证: 检查输出稳定性恢复

结果:
- 活跃版本: v1.0.0
- 参数恢复: temperature: 0.7
```

## 六、版本对比与审计

### 6.1 版本对比功能

## 七、版本切换流程

### 7.1 发布流程

```
版本发布流程
│
├── 1. 开发准备
│   ├── 从当前版本创建分支
│   │   └── 命令: /expert branch Kahneman v1.2.0
│   │
│   └── 在新分支进行开发
│
├── 2. 内部测试
│   ├── 开发版测试
│   │   └── 命令: /expert test Kahneman --dev
│   │
│   └── 测试版验证
│       └── 命令: /expert test Kahneman --beta
│
├── 3. 发布审批
│   ├── 检查变更内容
│   │   └── 命令: /expert diff Kahneman
│   │
│   ├── 检查性能影响
│   │   └── 命令: /expert impact Kahneman v1.2.0
│   │
│   └── 审批确认
│
├── 4. 灰度发布
│   ├── 切换10%流量
│   │   └── 命令: /expert deploy Kahneman v1.2.0 --canary=10%
│   │
│   ├── 监控指标
│   │   └── 观察: 错误率、响应时间、满意度
│   │
│   └── 升量或回滚
│       ├── 指标正常 → 升到50%
│       └── 指标异常 → 回滚
│
└── 5. 全量发布
    ├── 切换全部流量
    │   └── 命令: /expert deploy Kahneman v1.2.0 --full
    │
    └── 更新元信息
        └── 自动更新 metadata.json
```

### 7.2 版本状态流转

```
版本状态流转
│
├── dev (开发中)
│   ├── 入口: 新建版本
│   ├── 出口: 测试通过 → beta
│   │
├── beta (测试中)
│   ├── 入口: dev通过测试
│   ├── 出口1: 验证通过 → stable
│   └── 出口2: 验证失败 → deprecated
│
├── stable (稳定版) ← 活跃版本
│   ├── 入口: beta验证通过
│   ├── 出口1: 发现问题 → deprecated
│   ├── 出口2: 新版本发布 → superseded
│   │
├── superseded (已废弃)
│   ├── 入口: 新版本发布
│   ├── 出口: 被完全取代 → archived
│   │
└── deprecated (不推荐)
    ├── 入口: 发现严重问题
    ├── 出口: 回滚后彻底废弃
    │
```

## 九、总结

### 9.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **向后兼容** | 次版本和补丁版本保持向后兼容 |
| **可追溯** | 每次变更都有完整记录 |
| **可回滚** | 支持一键回滚到任意版本 |
| **可对比** | 支持版本间差异对比 |
| **安全优先** | 回滚有保护机制，防止越滚越远 |

### 9.2 关键数据

| 指标 | 数值 |
|------|------|
| 版本要素 | 4个（Prompt/参数/能力/边界） |
| 版本号格式 | X.Y.Z 语义化版本 |
| 回滚类型 | 4种（Prompt/参数/能力/全量） |
| 状态数量 | 5个（dev/beta/stable/superseded/deprecated） |
| 存储位置 | versions/{version}/ 目录 |
