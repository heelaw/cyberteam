# 思考天团容错机制设计

**问题**: 如何设计思考天团系统的容错机制，保证系统在高可用性和数据一致性？

---

## 一、容错机制设计原则

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **高可用** | 单点故障不影响整体服务可用性 |
| **快速恢复** | 故障发生后快速恢复，不丢失进度 |
| **数据一致** | 保证任务状态和输出的一致性 |
| **可观测** | 故障可被实时监控和告警 |

## 二、单点故障类型定义

### 2.1 故障分类矩阵

| 故障类别 | 故障类型 | 严重程度 | 发生概率 | 影响范围 |
|----------|----------|----------|----------|----------|
| **LLM服务** | 主模型不可用 | P0 | 中 | 全局 |
| **LLM服务** | 模型响应超时 | P1 | 高 | 单任务 |
| **LLM服务** | 模型输出异常 | P1 | 中 | 单任务 |
| **专家Agent** | 专家崩溃/无响应 | P1 | 中 | 单任务 |
| **专家Agent** | 专家输出无效 | P2 | 低 | 单任务 |
| **文件系统** | 文件写入失败 | P1 | 中 | 单任务 |
| **文件系统** | 文件损坏/丢失 | P0 | 低 | 单任务 |
| **消息队列** | 消息丢失 | P1 | 低 | 单任务 |
| **主持人** | 主持人崩溃 | P0 | 低 | 全局 |
| **网络** | 网络中断 | P1 | 中 | 全局 |

## 三、降级策略设计

### 3.1 降级策略矩阵

| 故障类型 | 第一降级 | 第二降级 | 第三降级 | 极限降级 |
|----------|----------|----------|----------|----------|
| **主模型不可用** | 切换到备用模型 | 切换到低成本模型 | 返回缓存结果 | 告知用户稍后重试 |
| **模型响应超时** | 延长超时时间 | 切换模型 | 跳过该专家 | 返回部分结果 |
| **模型输出异常** | 重试1次 | 切换模型 | 使用默认值 | 跳过该专家 |
| **专家崩溃** | 等待恢复(10s) | 跳过该专家 | 替代专家 | 继续其他专家 |
| **专家输出无效** | 重试1次 | 使用默认输出 | 跳过该专家 | 继续其他专家 |
| **文件写入失败** | 内存缓存 | 重试3次 | 告警+继续 | 降级到内存模式 |
| **主持人崩溃** | 自动重启 | 恢复Checkpoint | 告警 | 暂停新任务 |

### 4.2 重试机制设计

#### 4.2.1 重试配置

```python
# 重试策略配置
RETRY_CONFIG = {
    "max_attempts": 3,
    "base_delay": 1,           # 基础延迟(秒)
    "max_delay": 30,           # 最大延迟(秒)
    "exponential_base": 2,     # 指数退避基数
    "jitter": True,            # 添加随机抖动

    # 按故障类型配置
    "by_error_type": {
        "timeout": {
            "max_attempts": 2,
            "strategy": "fallback_model",  # 超时直接切换模型
        },
        "api_error": {
            "max_attempts": 3,
            "strategy": "retry_with_backoff",
        },
        "validation_error": {
            "max_attempts": 1,
            "strategy": "skip_expert",      # 验证失败跳过
        },
        "system_error": {
            "max_attempts": 1,
            "strategy": "alert_and_pause",  # 系统错误告警暂停
        }
    }
}
```

### 4.3 回滚机制设计

#### 4.3.1 回滚触发条件

| 条件 | 描述 | 动作 |
|------|------|------|
| **连续失败** | 同一任务连续3次失败 | 回滚到上一个Checkpoint |
| **数据损坏** | 输出文件校验失败 | 回滚到上一个有效版本 |
| **专家死锁** | 打破死锁失败 | 回滚到辩论前状态 |
| **主持人崩溃** | 主持人无法恢复 | 回滚到任务初始状态 |

#### 4.3.2 回滚流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           回滚流程                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│   触发回滚 → 保存当前状态快照 → 加载目标版本 → 验证 → 继续执行           │
│                                                                          │
│   1. 保存当前状态快照                                                    │
│      - 保存位置: workspace/tasks/{id}/rollback/                         │
│      - 内容: board.json + 所有expert状态 + messages                     │
│      - 命名: rollback_{timestamp}.json                                  │
│                                                                          │
│   2. 加载目标版本                                                        │
│      - 优先: 最近的有效Checkpoint                                        │
│      - 备选: 任务初始状态                                                │
│                                                                          │
│   3. 验证一致性                                                          │
│      - Schema验证                                                        │
│      - 依赖检查                                                          │
│      - 状态验证                                                          │
│                                                                          │
│   4. 继续执行                                                            │
│      - 从回滚点重新开始                                                   │
│      - 跳过已完成的专家                                                   │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.4 告警机制

#### 4.4.1 告警分级 (与Q42/Q43一致)

| 级别 | 触发条件 | 通知方式 | 处理时限 |
|------|----------|----------|----------|
| **P0-紧急** | 主持人崩溃/主模型不可用 | 立即通知+自动处理 | <1min |
| **P1-高** | 专家连续失败/文件系统异常 | 用户+管理员 | <5min |
| **P2-中** | 单次失败/降级发生 | 用户 | <1hour |
| **P3-低** | 重试成功/降级成功 | 仅记录 | N/A |

        "notify": ["管理员", "值班人员"],
    },

## 九、总结

### 9.1 设计原则

1. **分层防御** - 预防→检测→处理→恢复，四层保护
2. **自动恢复** - 尽可能自动处理，减少人工介入
3. **降级有度** - 逐级降级，保证基本功能可用
4. **数据不丢** - Checkpoint + 回滚，保证一致性

### 9.2 关键机制

| 机制 | 说明 |
|------|------|
| **模型降级** | 3级备用模型，自动切换 |
| **专家跳过** | 专家失败时跳过，保证部分完成 |
| **Checkpoint** | 30秒保存，支持任务恢复 |
| **回滚** | 支持回滚到上一有效状态 |
| **告警** | P0-P3分级，及时通知 |

### 9.3 与现有架构集成

| 现有设计 | 容错增强 |
|----------|----------|
| Q20文件体系 | Checkpoint + 状态快照 |
| Q42异常检测 | 故障识别 → 故障处理 |
| Q43安全边界 | 降级策略 + 告警分级 |
