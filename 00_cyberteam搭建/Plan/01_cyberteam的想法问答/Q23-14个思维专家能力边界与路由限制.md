# Q23: 14个思维专家能力边界与路由限制

**问题**: 14个思维专家的能力边界在哪里？哪些问题不该路由给他们？

---

## 一、能力边界总览

### 1.1 专家能力矩阵

| 专家 | 核心能力 | 边界 | 不适合场景 |
|------|----------|------|------------|
| **Kahneman** | 认知偏差、决策心理 | 心理学范围 | 技术实现、代码开发 |
| **FirstPrinciple** | 本质重构、从零开始 | 创新类问题 | 常规运营、执行操作 |
| **SixHats** | 全方位分析 | 需要多视角的复杂问题 | 简单YES/NO问题 |
| **SWOTTows** | 竞争分析、战略规划 | 商业竞争场景 | 纯技术问题 |
| **FiveWhy** | 根因挖掘 | 问题的深层原因 | 已有明确答案 |
| **Goldlin** | 动机驱动、目的导向 | "为什么"类问题 | 纯执行类任务 |
| **GROW** | 目标达成、coaching | 目标设定与实现 | 无目标的情况 |
| **KISS** | 简化、避免过度 | 需要简化的方案 | 需要复杂权衡 |
| **McKinsey** | 结构化分析、MECE | 商业分析 | 创意发散类 |
| **AIboard** | AI趋势洞察 | AI/科技领域 | 传统行业分析 |
| **ReverseThinking** | 风险预判、漏洞发现 | 风险识别 | 纯正面评价 |
| **FiveDimension** | 全面评估 | 需要综合打分 | 单维度问题 |
| **WBS** | 任务分解、执行计划 | 落地执行 | 战略层面的问题 |
| **ManagerLeap** | 组织管理、领导力 | 组织与管理 | 技术实现问题 |

---

## 二、各专家详细边界

### 2.1 Kahneman（卡尼曼决策）

**能力**: 识别认知偏差、优化决策质量

```
✅ 适合
├── 用户决策困惑
├── 选择困难症
├── 风险评估
├── 偏见识别
└── 心理账户

❌ 不适合
├── 技术选型
├── 代码开发
├── 数据计算
└── 法律咨询
```

**边界说明**:
- 只处理心理学和决策科学相关
- 不提供具体技术方案
- 不做数学计算

---

### 2.2 FirstPrinciple（第一性原理）

**能力**: 从本质重构、突破性创新

```
✅ 适合
├── 需要突破常规
├── 成本结构重构
├── 从零开始设计
└── 创新方案

❌ 不适合
├── 常规运营问题
├── 按流程执行
├── 已有标准答案
└── 简单操作
```

**边界说明**:
- 不适合"照搬"类问题
- 需要创造性思维的场景
- 不处理例行事务

---

### 2.3 SixHats（六顶思考帽）

**能力**: 全方位、多角度分析

```
✅ 适合
├── 复杂问题分析
├── 需要不同视角
├── 全面评估
└── 避免盲点

❌ 不适合
├── 简单YES/NO问题
├── 紧急速决
├── 单点确认
└── 只需一面之词
```

**边界说明**:
- 需要多人视角的复杂性
- 不适合快速决策
- 不是简单判断

---

### 2.4 SWOTTows（SWOT分析）

**能力**: 竞争战略、态势分析

```
✅ 适合
├── 商业竞争分析
├── 战略规划
├── 市场定位
└── 竞品分析

❌ 不适合
├── 纯技术问题
├── 代码调试
├── 个人生活决策
└── 情感问题
```

**边界说明**:
- 专注于商业/战略层面
- 需要有"竞争对手"概念
- 不处理纯技术评估

---

### 2.5 FiveWhy（5Why溯源）

**能力**: 根因挖掘、问题诊断

```
✅ 适合
├── 问题根源分析
├── 故障诊断
├── 反复出现的问题
└── 需要"为什么"

❌ 不适合
├── 已有明确答案
├── 表面问题
├── 简单操作
└── 只需告知what
```

**边界说明**:
- 需要深入探究"为什么"
- 不处理"直接给答案"
- 不适合答案明确的问题

---

### 2.6 Goldlin（黄金圈法则）

**能力**: 动机驱动、目的导向

```
✅ 适合
├── 方向不明确
├── 缺乏动力
├── 需要找到"为什么"
└── 战略层面的"为什么"

❌ 不适合
├── 纯执行任务
├── 操作步骤
├── 技术实现
└── 有明确目标
```

**边界说明**:
- 专注于"为什么存在"
- 不处理"怎么做到"
- 需要有探索动机的需求

---

### 2.7 GROW（目标模型）

**能力**: 目标设定、路径规划

```
✅ 适合
├── 目标不清晰
├── 需要规划路径
├── 职业发展
└── 个人成长

❌ 不适合
├── 无目标的情况
├── 纯执行操作
├── 已有目标只需执行
└── 情感问题
```

**边界说明**:
- 需要有"目标"概念
- 不处理漫无目的的讨论
- 不适合纯执行任务

---

### 2.8 KISS（保持简单）

**能力**: 简化、避免过度

```
✅ 适合
├── 方案过于复杂
├── 需要简化
├── 避免过度设计
└── 追求MVP

❌ 不适合
├── 需要复杂权衡
├── 多方利益平衡
├── 需要全面考虑
└── 战略性问题
```

**边界说明**:
- 不适合需要复杂权衡
- 不适合多方利益平衡
- 不处理需要全面考虑的问题

---

### 2.9 McKinsey（麦肯锡方法）

**能力**: 结构化分析、MECE原则

```
✅ 适合
├── 商业分析
├── 问题结构化
├── 逻辑推理
└── 框架分析

❌ 不适合
├── 创意发散
├── 艺术创作
├── 情感表达
└── 主观判断
```

**边界说明**:
- 依赖逻辑和框架
- 不适合创意类
- 不适合主观判断

---

### 2.10 AIboard（AI趋势洞察）

**能力**: AI趋势、技术洞察

```
✅ 适合
├── AI选型
├── 技术趋势判断
├── AI应用场景
├── AI风险评估
└── 科技战略

❌ 不适合
├── 传统行业分析（非AI）
├── 纯商业运营
├── 人员管理
└── 财务问题
```

**边界说明**:
- 只处理AI/科技相关
- 不做通用商业分析
- 需要有技术元素

---

### 2.11 ReverseThinking（逆向思维）

**能力**: 风险预判、漏洞发现

```
✅ 适合
├── 风险识别
├── 漏洞发现
├── 失败模式分析
└── 危机预防

❌ 不适合
├── 需要正面推进
├── 只需正面评价
├── 创意鼓励
└── 激励团队
```

**边界说明**:
- 专注于负面/风险
- 不适合需要正能量的场景
- 不是"夸夸"型

---

### 2.12 FiveDimension（五维分析）

**能力**: 全面评估、多维度打分

```
✅ 适合
├── 需要综合评估
├── 多方案对比
├── 全面打分
└── 决策支持

❌ 不适合
├── 单维度问题
├── 简单选择
├── 只需YES/NO
└── 局部问题
```

**边界说明**:
- 需要多维度考量
- 不适合简单问题
- 不处理单点问题

---

### 2.13 WBS（任务分解）

**能力**: 工作分解、执行计划

```
✅ 适合
├── 执行计划制定
├── 任务拆解
├── 时间规划
└── 资源分配

❌ 不适合
├── 战略层面问题
├── 只需方向
├── 概念探讨
└── 纯分析
```

**边界说明**:
- 专注于落地执行
- 不处理战略方向
- 需要有执行需求

---

### 2.14 ManagerLeap（管理跃迁）

**能力**: 组织管理、领导力

```
✅ 适合
├── 团队管理问题
├── 组织变革
├── 领导力发展
├── 管理困境
└── 组织架构

❌ 不适合
├── 技术实现
├── 代码开发
├── 数据分析
└── 产品设计
```

**边界说明**:
- 只处理组织管理相关
- 不做技术决策
- 不处理个人成长（非管理）

## 三、路由决策矩阵

### 3.1 问题类型 → 专家映射

| 问题类型 | 推荐专家 | 禁用专家 |
|----------|----------|----------|
| **决策选择** | Kahneman, SixHats | AIboard, WBS |
| **创新突破** | FirstPrinciple, ReverseThinking | McKinsey, KISS |
| **问题诊断** | FiveWhy, ReverseThinking | GROW, Goldlin |
| **战略规划** | SWOTTows, McKinsey, FiveDimension | WBS, ManagerLeap |
| **目标设定** | GROW, Goldlin | ReverseThinking |
| **风险评估** | ReverseThinking, FiveDimension | Kahneman |
| **执行计划** | WBS, GROW | SixHats, McKinsey |
| **团队管理** | ManagerLeap, GROW | FirstPrinciple |
| **技术决策** | AIboard, FirstPrinciple | ManagerLeap, Kahneman |
| **创意发散** | SixHats, ReverseThinking | McKinsey, WBS |

### 3.2 简单问题路由

```
问题复杂度判断
    │
    ├── 简单问题 (30秒内有答案)
    │   └── ❌ 不需要专家，直接回答
    │
    ├── 中等复杂 (需分析)
    │   ├── Kahneman (决策)
    │   ├── FiveWhy (原因)
    │   └── GROW (目标)
    │
    └── 复杂问题 (需多角度)
        ├── SixHats (全面分析)
        ├── SWOTTows (战略)
        └── FiveDimension (评估)
```

## 四、绝对不要路由的情况

### 4.1 立即回答（不需要专家）

```
❌ 简单事实查询
   "今天天气如何？"
   "北京人口多少？"

❌ 简单操作
   "帮我打开文件"
   "执行这个命令"

❌ 闲聊
   "你好"
   "今天怎么样？"

❌ 只需YES/NO
   "这个方案可行吗？"
   → 直接回答YES/NO
```

### 4.2 其他Agent/工具处理

```
❌ 技术问题
   "代码报错" → 代码调试Agent
   "服务器挂了" → 运维Agent
   "帮我写个函数" → 编程Agent

❌ 外部信息
   "查一下最新新闻" → 搜索Agent
   "这个API怎么用" → 文档查询

❌ 执行操作
   "帮我发个邮件" → 工具调用
   "创建个文件" → 文件操作
```

### 4.3 超出系统能力

```
❌ 法律/医疗等专业
   "这个合同有风险吗？" → 需要律师
   "这个症状说明什么？" → 需要医生

❌ 实时信息
   "现在股价多少？" → 需要实时API
   "今天有什么新闻？" → 需要搜索

❌ 主观偏好
   "哪个颜色好看？" → 主观判断
   "你喜欢哪个？" → 无正确答案
```

## 五、路由决策流程

### 5.1 决策树

```
收到用户问题
    │
    ├── 是简单问题？
    │   ├── YES → 直接回答
    │   └── NO → 继续
    │
    ├── 需要专家分析？
    │   ├── NO → 工具调用/转交
    │   └── YES → 继续
    │
    ├── 意图识别
    │   │
    │   ├── 决策类 → Kahneman / SixHats
    │   ├── 创新类 → FirstPrinciple / ReverseThinking
    │   ├── 诊断类 → FiveWhy / ReverseThinking
    │   ├── 战略类 → SWOTTows / McKinsey / FiveDimension
    │   ├── 目标类 → GROW / Goldlin
    │   ├── 执行类 → WBS / GROW
    │   ├── 管理类 → ManagerLeap / GROW
    │   └── 技术类 → AIboard / FirstPrinciple
    │
    └── 选择协作模式
        ├── 单专家
        ├── 并行分析
        ├── 交叉辩论
        └── 链式分析
```

### 5.2 路由检查清单

```
收到问题 → 检查是否简单？
           │
           ├─ 是 → 直接回答，跳过专家
           │
           └─ 否 → 检查问题类型

问题类型检查：
□ 有"如何" → 可能需要GROW/WBS
□ 有"为什么" → 可能需要FiveWhy/Goldlin
□ 有"选择" → 可能需要Kahneman/SixHats
□ 有"风险" → 需要ReverseThinking
□ 有"创新" → 需要FirstPrinciple
□ 有"团队/管理" → 需要ManagerLeap
□ 有"AI/技术" → 需要AIboard
□ 有"分析/评估" → 需要McKinsey/FiveDimension

协作模式选择：
□ 多角度分析 → SixHats
□ 需要辩论 → 交叉模式
□ 链式推理 → 链式模式
□ 简单分析 → 单专家
```

## 六、总结

### 6.1 专家能力边界速查

| 专家 | 核心 | 边界 |
|------|------|------|
| Kahneman | 决策心理 | 不做技术 |
| FirstPrinciple | 创新重构 | 不做常规 |
| SixHats | 多视角 | 不做简单 |
| SWOTTows | 竞争分析 | 不做技术 |
| FiveWhy | 根因 | 不做表面 |
| Goldlin | 动机 | 不做执行 |
| GROW | 目标 | 不做无目标 |
| KISS | 简化 | 不做复杂 |
| McKinsey | 结构化 | 不做创意 |
| AIboard | AI趋势 | 不做传统 |
| ReverseThinking | 风险 | 不做正面 |
| FiveDimension | 评估 | 不做单点 |
| WBS | 执行 | 不做战略 |
| ManagerLeap | 管理 | 不做技术 |

### 6.2 快速判断

```
需要专家？ → 简单问题 → NO
          → 有明确答案 → NO
          → 需深入分析 → YES → 匹配专家

不该路由？
→ 简单问题 (30秒内)
→ 技术问题 (代码/运维)
→ 外部信息 (搜索/查询)
→ 专业问题 (法律/医疗)
→ 主观偏好 (无标准)
```
