## 一、反馈渠道设计

### 1.1 渠道清单

| 渠道类型 | 触发方式 | 反馈内容 | 优先级 |
|---------|---------|---------|--------|
| **星级评分** | 输出后自动提示 | 1-5星整体评分 | P0 |
| **有用/无用标记** | 快捷按钮 | 二元判断 | P0 |
| **文字评论** | 可选填写 | 具体意见 | P1 |
| **改进建议** | 专用入口 | 优化方向 | P1 |
| **问题报告** | 错误反馈 | 异常情况 | P2 |

### 1.2 交互设计

#### 1.2.1 输出后自动提示

```
═══════════════════════════════════════════
           【思考天团】分析报告
═══════════════════════════════════════════

[报告内容...]

───────────────────────────────────────────
反馈环节
───────────────────────────────────────────

本次分析对您有帮助吗？
👍 有用    👎 无用    ⭐ 评分 (1-5星)

您想补充反馈吗？（可选）
[输入您的意见...]
```

#### 1.2.2 快捷反馈按钮

| 按钮 | 语义 | 触发操作 |
|------|------|----------|
| 👍 | 有用 | 记录正面反馈 + 提升专家权重 |
| 👎 | 无用 | 记录负面反馈 + 触发分析流程 |
| ⭐ | 评分 | 记录评分 + 关联具体维度 |

#### 1.2.3 详细反馈入口

```
【详细反馈】（点击展开）

问题类型：
☐ 问题理解不准确
☐ 专家选择不合适
☐ 分析深度不足
☐ 结论不够清晰
☐ 建议缺乏可操作性
☐ 其他：________

改进建议：
[请输入您的建议...]

感谢您的反馈！
```

## 二、反馈分类体系

### 2.1 反馈类型矩阵

| 大类 | 子类 | 判定规则 | 影响范围 |
|------|------|----------|----------|
| **正面反馈** | 高度认可 | 评分=5星 或 连续3次"有用" | 增强专家权重 |
| **正面反馈** | 认可 | 评分=4星 | 维持当前状态 |
| **中性反馈** | 一般 | 评分=3星 | 标记关注 |
| **负面反馈** | 不满意 | 评分=1-2星 | 触发优化分析 |
| **负面反馈** | 无用 | 点击"无用"按钮 | 触发优化分析 |
| **建议反馈** | 改进建议 | 用户填写文字建议 | 人工审核 |

### 2.2 问题维度细分

| 维度 | 负面判定标准 | 优化方向 |
|------|-------------|----------|
| **问题理解** | "没理解我的问题"/"偏题了" | 改进意图识别/问题重构 |
| **专家选择** | "应该用XX专家"/"专家不对" | 调整路由规则/权重 |
| **分析深度** | "太浅"/"没分析到位" | 改进专家Prompt/增加追问 |
| **结论清晰度** | "结论模糊"/"不知道该怎么做" | 改进整合模板/增加结构化 |
| **可操作性** | "建议无法落地"/"太虚" | 增加具体化检查/案例库 |
| **时效性** | "太慢"/"等待太久" | 优化并行策略/缓存 |

## 三、反馈影响系统优化的机制

### 3.1 优化影响矩阵

| 反馈类型 | 调整专家权重 | 更新知识库 | 改进提示词 | 人工审核 |
|---------|-------------|-----------|-----------|----------|
| 专家选择不当 | ✅ 自动 | - | - | - |
| 分析深度不足 | - | - | ✅ 自动 | - |
| 结论不够清晰 | - | - | ✅ 自动 | - |
| 建议缺乏可操作性 | - | ✅ 案例 | ✅ 自动 | - |
| 复杂/新类型建议 | - | - | - | ✅ 必需 |

### 3.2 专家权重调整机制

#### 3.2.1 权重计算公式

```
新权重 = 当前权重 × (1 + 调整系数)

调整系数规则：
- 连续3次"有用"：+10%
- 评分5星：+5%
- 评分1-2星：-10%
- 点击"无用"：-15%
- 问题类型正确但效果差：-5%
```

#### 3.2.2 权重应用场景

| 场景 | 权重作用 |
|------|----------|
| 多专家选择 | 高权重专家优先被选中 |
| 并行数量决策 | 高权重专家可增加配额 |
| 结果整合 | 高权重专家观点权重增加 |
| 链式分析 | 高权重专家优先安排在关键步骤 |

#### 3.2.3 权重存储结构

```json
{
  "expert_weights": {
    "kahneman": 1.15,
    "first-principle": 1.05,
    "six-hats": 0.90,
    "swot-tows": 1.00,
    "fivewhy": 1.10,
    "goldlin": 1.00,
    "grow": 1.00,
    "kiss": 1.00,
    "mckinsey": 1.00,
    "ai-board": 1.00,
    "reverse-thinking": 1.05,
    "five-dimension": 1.00,
    "wbs": 1.00,
    "manager-leap": 1.00
  },

  "adjustment_history": [
    {
      "expert": "six-hats",
      "reason": "连续2次负面反馈",
      "adjustment": -0.10,
      "timestamp": "2026-03-20T10:00:00Z"
    }
  ]
}
```

### 3.3 知识库更新机制

#### 3.3.1 可更新的知识库

| 知识库类型 | 更新触发条件 | 更新内容 |
|-----------|-------------|----------|
| **案例库** | 负面反馈+建议 | 新增用户场景案例 |
| **模板库** | 负面反馈+具体建议 | 优化输出模板 |
| **规则库** | 路由错误累计≥3次 | 调整意图识别规则 |
| **检查清单** | 负面反馈+维度 | 增加检查项 |

#### 3.3.2 自动更新流程

```
用户反馈（可操作）
    ↓
判断是否可自动处理
    ↓
┌──────────────────┐
│ 是               │ 否
↓                  ↓
执行自动更新       加入人工审核队列
    ↓
记录变更日志
    ↓
验证更新效果
```

### 3.4 提示词改进机制

#### 3.4.1 提示词优化维度

| 维度 | 触发条件 | 改进方向 |
|------|---------|----------|
| 分析深度 | 评分"分析深度"≤2 | 增加追问要求/增加分析步骤 |
| 结论清晰 | 评分"结论"≤2 | 增加结构化模板/强制分层输出 |
| 可操作性 | 评分"可操作"≤2 | 增加"具体化"检查/增加案例引用 |
| 专家特色 | 专家效果差 | 强化专家特有视角的Prompt |

## 四、闭环流程设计

### 4.1 完整闭环流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        反馈闭环流程                              │
└─────────────────────────────────────────────────────────────────┘

     ┌─────────┐
     │ 1.收集  │  用户输出后自动提示反馈
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │ 2.分类  │  判定反馈类型和维度
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │ 3.分析  │  根因分析 + 确定优化方案
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │ 4.处理  │  执行优化（自动/人工）
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │ 5.反馈  │  记录变更，通知用户（可选）
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │ 6.验证  │  下次输出检查改进效果
     └────┬────┘
          │
          ▼
     ┌─────────┐
     │ 7.学习  │  更新优化策略，持续迭代
     └─────────┘
```

### 4.2 各阶段详细说明

#### 阶段1：反馈收集

| 动作 | 触发时机 | 收集内容 |
|------|---------|----------|
| 自动提示 | 输出完成后 | 评分、有用/无用 |
| 文字输入 | 用户主动 | 评论、建议 |
| 快捷按钮 | 用户点击 | 二元反馈 |
| 维度选择 | 展开反馈面板 | 具体问题维度 |

#### 阶段2：反馈分类

| 分类维度 | 判定逻辑 | 输出 |
|---------|---------|------|
| 情感分类 | 评分/按钮 | 正面/中性/负面 |
| 问题维度 | 维度选择/分析 | 理解/专家/深度/结论/操作 |
| 紧急程度 | 负面+新问题 | P0/P1/P2 |
| 可自动化 | 规则匹配 | 是/否 |

#### 阶段3：根因分析

```
负面反馈
    │
    ▼
问题归类（匹配已有维度？）
    │
    ├── 是 → 定位根因（匹配库）
    │        │
    │        └── 已有解决方案 → 标记待处理
    │
    └── 否 → 新问题识别
             │
             └── 加入人工审核队列
```

#### 阶段4：优化处理

| 类型 | 处理方式 | 执行者 |
|------|---------|--------|
| 权重调整 | 计算新权重 | 自动 |
| 提示词优化 | 替换/增强Prompt | 自动 |
| 知识库更新 | 追加案例/规则 | 自动/人工 |
| 复杂问题 | 人工设计新方案 | 人工 |

#### 阶段5：变更记录

```json
{
  "change_log": {
    "id": "chg_001",
    "type": "weight_adjustment",
    "target": "expert:six-hats",
    "before": 1.0,
    "after": 0.90,
    "reason": "连续2次负面反馈：专家选择不当",
    "trigger_feedback": "fb_20260320_001",
    "timestamp": "2026-03-20T10:05:00Z",
    "auto": true
  }
}
```

#### 阶段6：效果验证

| 验证方式 | 验证内容 | 判定标准 |
|---------|---------|----------|
| 追踪验证 | 下次同类问题输出 | 评分提升？ |
| A/B验证 | 对比新旧版本 | 指标提升？ |
| 用户确认 | 询问改进效果 | 用户认可？ |

### 4.3 场景示例

#### 场景1：用户给报告打3星

```
输入：
- 用户评分：3星（满分5星）
- 维度评分：理解=4, 专家=3, 深度=3, 结论=3, 操作=2

处理流程：
1. 收集：记录评分和各维度分数
2. 分类：判定为"中性反馈"，问题维度"可操作性"
3. 分析：可操作性分数低，需要增加具体化要求
4. 处理：
   - 自动：更新kahneman提示词，增加"具体化"检查项
   - 记录变更日志
5. 反馈：标记"已优化"
6. 验证：下次输出时检查是否改进

输出：
"感谢您的反馈！我们已记录您的评分，
如果您有具体的改进建议，欢迎告诉我们。"
```

#### 场景2：用户标记"无用"

```
输入：
- 用户点击：👎 无用
- 文字反馈："专家选择不太匹配，应该用kahneman分析决策问题"

处理流程：
1. 收集：记录"无用"标记 + 文字评论
2. 分类：判定为"负面反馈"，问题维度"专家选择"
3. 分析：评论明确指出专家选择问题
4. 处理：
   - 自动：six-hats权重 -15%
   - 自动：kahneman权重 +5%
   - 记录路由规则更新
5. 反馈：标记"已优化"
6. 验证：下次类似问题时检查是否选用kahneman

输出：
"感谢您的反馈！我们已调整专家选择策略，
kahneman专家将更优先用于决策类问题分析。"
```

#### 场景3：用户给出改进建议

```
输入：
- 用户评分：3星
- 改进建议："希望增加更多具体商业案例"

处理流程：
1. 收集：记录评分 + 建议文字
2. 分类：判定为"建议反馈"
3. 分析：需要更新案例库
4. 处理：加入人工审核队列
5. 反馈：告知"已记录，将人工审核"
6. 验证：人工审核后更新案例库

输出：
"感谢您的建议！我们的团队将审核您的建议，
并考虑在后续版本中增加相关案例。"
```

## 五、与现有系统的整合

### 5.1 与Success Metrics的对应

已有的SKILL.md中的Success Metrics：

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 问题理解准确率 | ≥90% | 用户确认率 |
| 专家选择匹配度 | ≥85% | 用户反馈 |
| 整合结论清晰度 | ≥90% | 用户评分 |
| 可操作性评分 | ≥4.5/5 | 行动建议采纳率 |

**反馈闭环整合方式**：

| Success Metrics | 反馈数据源 | 优化机制 |
|-----------------|-----------|----------|
| 问题理解准确率 | "问题理解"维度评分 | 改进意图识别 |
| 专家选择匹配度 | "专家选择"维度评分 + 评论 | 权重调整 |
| 整合结论清晰度 | "结论清晰"维度评分 | 改进整合模板 |
| 可操作性评分 | "可操作性"维度评分 | 改进Prompt + 案例库 |

### 5.2 数据流向

```

    │                         │
    │                         ├──→ 权重管理模块

    │                         ├──→ 知识库更新模块
    │                         │
    │                         └──→ Prompt优化模块

    └──→ Success Metrics 更新
```
