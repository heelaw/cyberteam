# 思考天团输出质量评估机制设计

**问题**: 如何设计思考天团（多专家Agent协作系统）的输出质量评估机制？

---

## 一、评估目标与原则

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **可量化** | 质量可数字化衡量，避免主观模糊 |
| **可追溯** | 评估结果可存档，支持历史查询 |
| **可优化** | 评估结果可指导后续改进 |
| **多维度** | 从多个角度全面评估输出质量 |

### 1.2 评估对象

评估对象为思考天团的最终输出产物：
- 用户问题分析报告
- 多专家分析结果整合
- 辩论/讨论过程记录
- 最终决策建议

### 1.3 与系统监控的关系

| 系统监控（Q41） | 输出质量评估 |
|----------------|-------------|
| 关注执行过程 | 关注输出结果 |
| 实时指标 | 事后评估 |
| Agent表现 | 内容质量 |

---

## 二、量化指标体系

### 2.1 指标维度设计

基于Q41中COL_OUTPUT_QUALITY的设计，扩展为六大维度：

```
输出质量 = 完整性(20%) + 准确性(20%) + 深度(15%) + 可执行性(20%) + 一致性(10%) + 用户价值(15%)
```

| 维度 | 权重 | 说明 |
|------|------|------|
| **完整性** | 20% | 报告是否包含所有必要部分 |
| **准确性** | 20% | 事实、数据、引用是否准确 |
| **深度** | 15% | 分析是否有足够深度，非表面 |
| **可执行性** | 20% | 建议是否可落地执行 |
| **一致性** | 10% | 多专家观点是否一致整合 |
| **用户价值** | 15% | 对用户的实际帮助程度 |

### 2.2 详细指标定义

#### 2.2.1 完整性（Completeness）

| 指标ID | 指标名称 | 计算方式 | 目标值 |
|--------|---------|---------|--------|
| CMP_SECTION | 章节完整率 | 已有章节数 / 必需章节数 | >= 90% |
| CMP_EXPERT | 专家覆盖数 | 参与专家数 / 应参与专家数 | >= 70% |
| CMP_ASPECT | 视角覆盖度 | 已覆盖视角数 / 必需视角数 | >= 80% |

**必需章节清单**：
1. 问题定义 - 明确用户核心问题
2. 背景分析 - 问题产生的上下文
3. 专家分析 - 各专家的观点输出
4. 辩论/讨论过程 - 多方观点碰撞
5. 决策建议 - 最终建议
6. 风险提示 - 潜在风险与应对

**必需视角**（根据问题类型动态调整）：
- 用户视角 - 从用户角度分析
- 业务视角 - 从业务目标角度
- 风险视角 - 识别潜在风险
- 机会视角 - 发现潜在机会

#### 2.2.2 准确性（Accuracy）

| 指标ID | 指标名称 | 计算方式 | 目标值 |
|--------|---------|---------|--------|
| ACC_FACT | 事实准确率 | 准确事实数 / 总事实数 | >= 95% |
| ACC_DATA | 数据准确率 | 准确数据数 / 总数据数 | >= 90% |
| ACC_SOURCE | 引用准确率 | 准确引用数 / 总引用数 | >= 85% |
| ACC_LOGIC | 逻辑自洽率 | 自洽观点数 / 总观点数 | >= 90% |

**评估方式**：
```python
def evaluate_accuracy(report: Report) -> dict:
    facts = extract_facts(report)
    data_points = extract_data(facts)
    sources = extract_sources(report)

    accurate_facts = fact_checker.verify(facts)
    accurate_data = data_verifier.verify(data_points)
    accurate_sources = source_verifier.verify(sources)

    return {
        "acc_fact": len(accurate_facts) / len(facts) if facts else 1.0,
        "acc_data": len(accurate_data) / len(data_points) if data_points else 1.0,
        "acc_source": len(accurate_sources) / len(sources) if sources else 1.0,
        "acc_logic": check_logical_consistency(report)
    }
```

#### 2.2.3 深度（Depth）

| 指标ID | 指标名称 | 计算方式 | 目标值 |
|--------|---------|---------|--------|
| DPT_ANALYSIS | 分析深度 | 嵌套层级数 / 基准层级数 | >= 3层 |
| DPT_CAUSE | 归因深度 | 根因追溯层级 | >= 2层 |
| DPT_IMPACT | 影响分析深度 | 影响链长度 | >= 3步 |
| DPT_NUANCE | 细微差异识别 | 识别的细微差异点数量 | >= 3个 |

**评估方式**：
```python
def evaluate_depth(report: Report) -> dict:
    # 分析深度：是否有递进式分析
    analysis_depth = count_nesting_levels(report.analysis)

    # 归因深度：是否追溯到根因
    cause_depth = trace_to_root_cause(report.problem)

    # 影响分析：是否分析多级影响
    impact_depth = analyze_impact_chain(report.situation)

    # 细微差异：是否识别出容易忽略的差异
    nuance_count = identify_nuances(report.subject)

    return {
        "dpt_analysis": analysis_depth,
        "dpt_cause": cause_depth,
        "dpt_impact": impact_depth,
        "dpt_nuance": nuance_count,
        "depth_score": weighted_average([analysis_depth, cause_depth, impact_depth, nuance_count])
    }
```

#### 2.2.4 可执行性（Actionability）

| 指标ID | 指标名称 | 计算方式 | 目标值 |
|--------|---------|---------|--------|
| ACT_SPECIFIC | 具体程度 | 可量化指标数 / 建议总数 | >= 60% |
| ACT_OWNER | 责任明确 | 有明确负责人的建议数 / 总建议数 | >= 80% |
| ACT_TIMELINE | 时间明确 | 有明确时间的建议数 / 总建议数 | >= 70% |
| ACT_METRIC | 可度量 | 有成功指标的建议数 / 总建议数 | >= 60% |

**评估方式**：
```python
def evaluate_actionability(report: Report) -> dict:
    suggestions = report.extract_suggestions()

    specific = sum(1 for s in suggestions if s.is_specific())
    has_owner = sum(1 for s in suggestions if s.owner is not None)
    has_timeline = sum(1 for s in suggestions if s.timeline is not None)
    has_metric = sum(1 for s in suggestions if s.success_metric is not None)

    total = len(suggestions)

    return {
        "act_specific": specific / total if total > 0 else 0,
        "act_owner": has_owner / total if total > 0 else 0,
        "act_timeline": has_timeline / total if total > 0 else 0,
        "act_metric": has_metric / total if total > 0 else 0,
        "actionability_score": weighted_average([specific, has_owner, has_timeline, has_metric])
    }
```

**可执行性评估示例**：

| 建议 | 具体程度 | 责任人 | 时间 | 指标 | 评分 |
|------|---------|--------|------|------|------|
| "提升DAU" | ❌ 无量化 | ❌ 无 | ❌ 无 | ❌ 无 | 0% |
| "Q2提升DAU到100万" | ✅ 量化 | ❌ 无 | ✅ Q2 | ❌ 无 | 50% |
| "通过push优化，Q2将DAU从80万提升到100万，负责@产品，指标：DAU≥100万" | ✅ | ✅ | ✅ | ✅ | 100% |

#### 2.2.5 一致性（Consistency）

| 指标ID | 指标名称 | 计算方式 | 目标值 |
|--------|---------|---------|--------|
| CON_VIEWPOINT | 观点一致性 | 一致观点对数 / 总观点对数 | >= 80% |
| CON_TERM | 术语一致性 | 一致术语数 / 总术语数 | >= 90% |
| CON_LOGIC | 逻辑一致性 | 逻辑自洽的建议数 / 总建议数 | >= 85% |
| CON_CONFLICT | 冲突解决率 | 已解决冲突数 / 总冲突数 | >= 90% |

**评估方式**：
```python
def evaluate_consistency(report: Report) -> dict:
    # 观点一致性
    viewpoints = report.extract_viewpoints()
    consistent_pairs = count_consistent_pairs(viewpoints)

    # 术语一致性
    terms = report.extract_terms()
    consistent_terms = check_term_consistency(terms)

    # 冲突检测与解决
    conflicts = detect_conflicts(report.expert_outputs)
    resolved = sum(1 for c in conflicts if c.resolved)

    return {
        "con_viewpoint": consistent_pairs / total_pairs if total_pairs > 0 else 1.0,
        "con_term": consistent_terms / len(terms) if terms else 1.0,
        "con_logic": check_overall_logic(report),
        "con_conflict": resolved / len(conflicts) if conflicts else 1.0
    }
```

#### 2.2.6 用户价值（User Value）

| 指标ID | 指标名称 | 计算方式 | 目标值 |
|--------|---------|---------|--------|
| VAL_RELEVANCE | 相关性 | 与用户问题的相关程度 | >= 85% |
| VAL_NOVELTY | 新颖性 | 新观点/新视角占比 | >= 30% |
| VAL_PRACTICAL | 实用性 | 用户可直接应用的内容占比 | >= 70% |
| VAL_SATISFACTION | 用户满意度 | 用户反馈评分 | >= 4.0/5.0 |

**评估方式**：
```python
def evaluate_user_value(report: Report, user_question: str) -> dict:
    # 相关性：报告是否回答了用户问题
    relevance = calculate_relevance(report, user_question)

    # 新颖性：是否提供了新视角
    novelty = calculate_novelty(report)

    # 实用性：可直接使用的建议占比
    practical = calculate_practical_utility(report)

    # 用户满意度（如有反馈）
    satisfaction = report.user_feedback.score if report.user_feedback else None

    return {
        "val_relevance": relevance,
        "val_novelty": novelty,
        "val_practical": practical,
        "val_satisfaction": satisfaction
    }
```

### 2.3 综合评分计算

```python
def calculate_output_quality_score(
    completeness: dict,
    accuracy: dict,
    depth: dict,
    actionability: dict,
    consistency: dict,
    user_value: dict
) -> dict:
    """计算综合输出质量分数"""

    # 各维度加权平均
    dimension_scores = {
        "completeness": weighted_average([
            completeness["cmp_section"],
            completeness["cmp_expert"],
            completeness["cmp_aspect"]
        ]),
        "accuracy": weighted_average([
            accuracy["acc_fact"],
            accuracy["acc_data"],
            accuracy["acc_source"],
            accuracy["acc_logic"]
        ]),
        "depth": weighted_average([
            depth["dpt_analysis"],
            depth["dpt_cause"],
            depth["dpt_impact"],
            depth["dpt_nuance"]
        ]),
        "actionability": weighted_average([
            actionability["act_specific"],
            actionability["act_owner"],
            actionability["act_timeline"],
            actionability["act_metric"]
        ]),
        "consistency": weighted_average([
            consistency["con_viewpoint"],
            consistency["con_term"],
            consistency["con_logic"],
            consistency["con_conflict"]
        ]),
        "user_value": weighted_average([
            user_value["val_relevance"],
            user_value["val_novelty"],
            user_value["val_practical"]
        ])
    }

    # 综合评分
    weights = {
        "completeness": 0.20,
        "accuracy": 0.20,
        "depth": 0.15,
        "actionability": 0.20,
        "consistency": 0.10,
        "user_value": 0.15
    }

    total_score = sum(
        dimension_scores[dim] * weights[dim]
        for dim in weights
    )

    return {
        "dimensions": dimension_scores,
        "total_score": round(total_score * 100, 1),  # 转换为百分制
        "grade": score_to_grade(total_score),
        "timestamp": datetime.now().isoformat()
    }

def score_to_grade(score: float) -> str:
    """分数转等级"""
    if score >= 0.9:
        return "A+"
    elif score >= 0.85:
        return "A"
    elif score >= 0.8:
        return "A-"
    elif score >= 0.75:
        return "B+"
    elif score >= 0.7:
        return "B"
    elif score >= 0.65:
        return "B-"
    elif score >= 0.6:
        return "C+"
    elif score >= 0.55:
        return "C"
    elif score >= 0.5:
        return "C-"
    else:
        return "D"
```

## 三、评估主体设计

### 3.1 评估主体类型

| 评估主体 | 适用维度 | 优点 | 缺点 |
|---------|---------|------|------|
| **LLM自评** | 全部维度 | 快速、大规模、可重复 | 可能存在偏差 |
| **用户评价** | 用户价值、一致性 | 最真实反馈 | 主观性强、回收难 |
| **专家互评** | 准确性、深度 | 专业、客观 | 成本高、周期长 |
| **规则检查** | 完整性、可执行性 | 客观、可自动化 | 只能检查结构 |

### 3.2 评估流程设计

```
用户提交问题
      ↓
思考天团生成报告
      ↓
┌─────────────────────────────────────────┐
│         多维度评估流程                   │
├─────────────────────────────────────────┤
│  1. 规则检查 → 完整性 + 可执行性         │
│  2. LLM评估 → 准确性 + 深度 + 一致性     │
│  3. 用户评价 → 用户价值                  │
│  4. 综合评分 → 输出质量分                │
└─────────────────────────────────────────┘
      ↓
生成评估报告
      ↓
存储评估结果
      ↓
影响后续优化
```

### 3.3 各评估主体职责

#### 3.3.1 LLM自评

**评估时机**: 输出完成后自动触发

**评估内容**:
- 准确性评估（事实核查、逻辑检验）
- 深度评估（分析层级、归因深度）
- 一致性评估（观点冲突检测）

**提示词设计**:
```markdown
你是一个专业的质量评估专家。请对以下思考天团输出进行评估：

## 输出内容
{output_content}

## 用户问题
{user_question}

## 评估维度

### 1. 准确性评估
- 事实准确率：检查输出中的事实陈述是否准确
- 逻辑自洽：检查论证过程是否存在逻辑矛盾
- 给出具体问题列表（如有）

### 2. 深度评估
- 分析层级：是否有递进式深度分析
- 归因深度：是否追溯到根本原因
- 细微差异：是否识别出容易被忽略的细节

### 3. 一致性评估
- 观点一致性：多专家观点是否协调一致
- 冲突检测：是否存在未解决的矛盾

请以JSON格式输出评估结果：
{
  "accuracy": {
    "fact_accuracy": 0.0-1.0,
    "logic_consistency": 0.0-1.0,
    "issues": ["问题1", "问题2"]
  },
  "depth": {
    "analysis_depth": 0.0-1.0,
    "cause_depth": 0.0-1.0,
    "nuance_identification": 0.0-1.0
  },
  "consistency": {
    "viewpoint_consistency": 0.0-1.0,
    "conflicts": ["冲突1", "冲突2"]
  }
}
```

#### 3.3.2 用户评价

**评估时机**: 用户收到输出后

**评估方式**:
- 满意度评分（1-5分）
- 维度评价（有用性、可执行性、准确性）
- 开放式反馈

**评估表单**:
```
# 输出质量评价

   [ ] 完全相关 [ ] 大部分相关 [ ] 部分相关 [ ] 不太相关

_________________________________________
```

#### 3.3.3 规则检查（自动化）

**评估时机**: 输出生成时同步执行

**检查内容**:
- 完整性：必要章节是否齐全
- 可执行性：建议是否包含SMART要素
- 格式规范性：是否符合输出模板

**检查脚本**:
```python
def automated_check(report: Report) -> dict:
    """自动化规则检查"""

    # 完整性检查
    required_sections = ["问题定义", "背景分析", "专家分析", "辩论过程", "决策建议", "风险提示"]
    present_sections = [s for s in required_sections if s in report.sections]
    completeness = len(present_sections) / len(required_sections)

    # 可执行性检查
    suggestions = report.extract_suggestions()
    actionable = sum(1 for s in suggestions if s.is_actionable())
    actionability = actionable / len(suggestions) if suggestions else 0

    # 格式检查
    format_valid = validate_format(report)

    return {
        "completeness": completeness,
        "actionability": actionability,
        "format_valid": format_valid,
        "issues": collect_issues(report)
    }
```

## 四、评分标准

### 4.1 分数等级定义

| 等级 | 综合分数 | 说明 | 行动建议 |
|------|---------|------|---------|
| **A+** | 90-100 | 卓越 | 优秀案例，可作为标杆 |
| **A** | 85-89 | 优秀 | 略作优化即可 |
| **A-** | 80-84 | 良好 | 有小问题需改进 |
| **B+** | 75-79 | 较好 | 有明显问题需改进 |
| **B** | 70-74 | 中等 | 需要较大改进 |
| **B-** | 65-69 | 及格 | 勉强可用，需改进 |
| **C+** | 60-64 | 边缘 | 存在明显不足 |
| **C** | 55-59 | 不及格 | 需要重新生成 |
| **C-** | 50-54 | 较差 | 质量较差 |
| **D** | <50 | 很差 | 需要重新设计 |

### 4.2 各维度评分标准

#### 4.2.1 完整性评分标准

| 等级 | 分值 | 标准 |
|------|------|------|

| C | 60-74 | 包含4个必需章节 + 专家覆盖>=50% |

### 4.3 及格线设定

| 场景 | 及格线 | 说明 |
|------|--------|------|
| **日常使用** | >=70分 (B) | 可接受的输出质量 |
| **重要决策** | >=80分 (A-) | 需要高质量输出 |
| **模板生成** | >=60分 (C+) | 最低质量底线 |
| **低于60分** | - | 需要重新生成 |

## 六、评估结果的应用

### 6.1 质量门禁

```
输出生成 → 自动评估 → 分数判定
                            │
           ┌────────────────┼────────────────┐
           ↓                ↓                ↓
        >=80分          70-80分          <70分
           ↓                ↓                ↓
       直接输出        提示优化          重新生成
```

| 分数区间 | 处理方式 |
|---------|---------|
| >= 80分 | 直接输出，可标记为高质量案例 |
| 70-80分 | 提示具体改进点，用户确认后输出 |
| < 70分 | 自动触发重新生成流程 |

### 6.2 优化触发

#### 6.2.1 自动优化规则

```python
def suggest_improvements(evaluation: Evaluation) -> list:
    """根据评估结果建议改进"""

    suggestions = []

    # 完整性问题
    if evaluation.dimensions.completeness.score < 80:
        missing = find_missing_sections(evaluation)
        suggestions.append(f"补充缺失章节: {missing}")

    # 可执行性问题
    if evaluation.dimensions.actionability.score < 70:
        weak_suggestions = find_weak_suggestions(evaluation)
        suggestions.append(f"增强以下建议的可执行性: {weak_suggestions}")

    # 深度问题
    if evaluation.dimensions.depth.score < 70:
        suggestions.append("增加分析深度，追溯到根本原因")

    # 一致性问题
    if evaluation.dimensions.consistency.score < 80:
        conflicts = find_unresolved_conflicts(evaluation)
        suggestions.append(f"解决以下观点冲突: {conflicts}")

    return suggestions
```

```
评估结果 → 识别问题 → 生成改进建议 → 用户确认 → 重新生成 → 再次评估
```
