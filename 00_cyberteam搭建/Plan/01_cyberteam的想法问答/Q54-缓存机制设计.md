# Q54: 思考天团缓存机制设计

**问题**: 如何设计思考天团的缓存机制，避免重复计算并实现专家分析结果复用？

---

## 一、设计目标与原则

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| **结果复用** | 相同或相似问题直接返回缓存结果，避免重复LLM调用 |
| **成本降低** | 通过缓存减少Token消耗，优化Q53的Token成本 |
| **响应加速** | 缓存命中时直接返回，大幅降低响应时间 |
| **一致性保证** | 缓存更新时确保用户获得最新结果 |

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **与Q53对齐** | 复用Token消耗优化机制的存储体系 |
| **分层缓存** | 按缓存内容类型采用不同策略 |
| **质量优先** | 缓存不影响输出质量，必要时主动失效 |
| **可观测** | 缓存命中/未命中需可追踪 |

### 1.3 与现有体系的关系

```
Q53 Token优化 ──────► 缓存减少Token消耗
     │                    │
     │                    ├── 意图分析缓存 → 减少LLM调用
     │                    ├── 专家分析缓存 → 最大Token节省
     │                    └── 路由决策缓存 → 减少判断开销

Q52 容错机制 ──────► 缓存失效保护
     │                    │
     │                    ├── 缓存不可用时降级到实时计算
     │                    ├── 过期缓存的安全处理
     │                    └── 缓存回源保护

Q20 文件体系 ──────► 缓存数据存储
     │                    │
     │                    ├── workspace/cache/ (缓存数据)
     │                    ├── metadata/ (缓存索引)
     │                    └── cache_policy.json (策略配置)
```

## 三、缓存策略设计

### 3.1 分层缓存策略

根据缓存内容的特性，采用分层策略：

```
┌─────────────────────────────────────────────────────────┐
│                    缓存策略总览                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  意图分析缓存 ──► LRU + TTL (7天)                       │
│       │                                                  │
│       ├── 最近最少使用：保留最近1000个问题意图            │
│       └── 固定TTL：7天后强制刷新                         │
│                                                         │
│  专家分析缓存 ──► 版本化LRU + 主动失效                  │
│       │                                                  │
│       ├── LRU：保留最近500个专家分析                     │
│       ├── 版本号：专家Prompt更新时失效                   │
│       └── 关键词变化：问题核心词变化时失效               │
│                                                         │
│  问题模板缓存 ──► TTL + 事件触发                        │
│       │                                                  │
│       ├── 固定TTL：30天                                 │
│       └── 事件触发：模板库更新时刷新                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 缓存策略详细设计

#### 3.2.1 意图分析缓存策略

| 属性 | 值 | 说明 |
|------|-----|------|
| **策略类型** | LRU + TTL | 结合访问频率和时间 |
| **最大容量** | 1000条 | 超过后删除最旧项 |
| **TTL** | 7天 | 时间过期后重新计算 |
| **存储位置** | workspace/cache/intent/ | 按Q20文件体系 |
| **键格式** | `intent:{question_hash}` | 问题哈希作为键 |

**问题哈希生成**:
```python
def generate_question_hash(question: str) -> str:
    """生成问题唯一哈希"""
    # 标准化问题文本（去除空格、标点）
    normalized = normalize_text(question)
    # 取前128字符 + 关键词哈希
    key_text = normalized[:128]
    return hash_sha256(key_text)[:16]
```

#### 3.2.2 专家分析缓存策略

| 属性 | 值 | 说明 |
|------|-----|------|
| **策略类型** | 版本化LRU + 主动失效 | 核心缓存，重点保护 |
| **最大容量** | 500条/专家 | 每个专家保留最近500条 |
| **默认TTL** | 30天 | 长时间有效 |
| **版本管理** | Expert Version | Prompt变更时失效 |
| **键格式** | `expert:{expert_name}:{question_hash}` | 专家+问题组合键 |

**版本化失效机制**:
```yaml
版本失效触发条件:
  - 专家Prompt修改 (version++)
  - 专家配置变更
  - 系统升级公告

主动失效流程:
  1. 检测到版本变化
  2. 标记旧版本缓存为stale
  3. 新请求触发重新计算
  4. 旧缓存在后台清理
```

#### 3.2.3 问题模板缓存策略

| 属性 | 值 | 说明 |
|------|-----|------|
| **策略类型** | TTL + 事件触发 | 中频更新 |
| **最大容量** | 200条 | 模板数量有限 |
| **TTL** | 30天 | 月度刷新 |
| **事件触发** | 模板库更新时刷新 | 保持模板时效性 |

### 3.3 缓存容量规划

| 缓存类型 | 最大容量 | 平均大小 | 总存储估算 |
|----------|----------|----------|------------|
| 意图分析 | 1000条 | 2KB | 2MB |
| 专家分析 | 500条/专家 × 14专家 | 8KB | 56MB |
| 问题模板 | 200条 | 4KB | 0.8MB |
| **总计** | - | - | **~60MB** |

## 四、缓存更新机制

### 4.1 更新策略分类

| 更新方式 | 适用场景 | 触发条件 |
|----------|----------|----------|
| **被动失效** | TTL过期 | 时间到达预设阈值 |
| **主动刷新** | 版本变更 | 专家Prompt/配置更新 |
| **被动刷新** | 缓存未命中 | 新问题进入系统 |
| **预热加载** | 系统启动 | 启动时加载热点数据 |

### 4.2 被动失效（TTL过期）

```
TTL过期检查流程:

  用户请求
      │
      ▼
┌─────────────────┐
│ 检查缓存是否存在 │
└────────┬────────┘
         │
    ┌────┴────┐
    │ 存在?   │
    └────┬────┘
     Yes │ No
     ┌───┴───┐
     ▼       ▼
┌─────────┐ ┌─────────┐
│检查TTL  │ │ 实时计算│
└────┬────┘ │  + 写入│
     │      │   缓存  │
┌────┴────┐ └─────────┘
│ 过期?   │
└────┬────┘
 Yes │ No
 ┌───┴───┐
 ▼       ▼
失效  返回缓存
+重算
```

**TTL配置建议**:
- 意图分析: 7天（问题类型相对稳定）
- 专家分析: 30天（分析结果可复用较长时间）
- 问题模板: 30天（月度更新即可）

### 4.3 主动刷新（版本失效）

```yaml
主动失效触发条件:
  
  1. 专家Prompt版本变更
     - 文件: experts/{expert_name}/prompt.md
     - 监控: md5变化
     - 动作: 标记该专家所有缓存为stale
  
  2. 专家配置变更
     - 文件: experts/{expert_name}/config.yaml
     - 监控: version字段变化
     - 动作: 标记该专家所有缓存为stale
  
  3. 系统升级公告
     - 手动触发: 管理员发布刷新公告
     - 动作: 标记所有缓存为stale

主动刷新流程:
  1. 检测到变更事件
  2. 更新版本号 (v1.0 → v1.1)
  3. 广播失效通知
  4. 后台异步清理旧版本缓存
  5. 新请求触发重新计算
```

### 5.1 命中验证流程

```

      用户问题输入
            │
            ▼
    生成问题哈希
    (标准化+哈希)
            │
            ▼
    ┌───────────────┐
    │ 查询缓存索引   │
    └───────┬───────┘
            │
    ┌───────┴───────┐
    │   命中?       │
    └───────┬───────┘
     Yes   │   No
     ┌─────┴─────┐
     ▼           ▼
┌─────────┐ ┌─────────┐
│ 验证缓存 │ │ 实时计算│
│ 有效性   │ │ + 写入  │
└────┬────┘ └─────────┘
     │
┌────┴────┐
│ 有效?   │
└────┬────┘
 Yes │ No
 ┌───┴───┐
 ▼       ▼
返回缓存 失效+重算
+记录命中
```

### 5.2 验证维度

| 验证维度 | 检查内容 | 失效条件 |
|----------|----------|----------|
| **TTL有效性** | 当前时间 < 过期时间 | 已过期 |
| **版本一致性** | 缓存version == 当前version | 版本不匹配 |
| **哈希匹配** | 问题哈希 == 缓存键 | 哈希变化 |
| **完整性检查** | 缓存数据无损坏 | 数据损坏 |
| **质量评分** | 缓存结果质量评分 | 评分过低 |

### 5.3 缓存命中判定逻辑

```python
def verify_cache_hit(question: str, cache_entry: dict) -> dict:
    """验证缓存是否有效命中"""
    
    result = {
        "hit": False,
        "reason": None,
        "data": None
    }
    
    # 1. TTL检查
    if is_expired(cache_entry["expires_at"]):
        result["reason"] = "ttl_expired"
        return result
    
    # 2. 版本检查
    if cache_entry.get("version") != get_current_version():
        result["reason"] = "version_mismatch"
        return result
    
    # 3. 哈希匹配检查
    question_hash = generate_question_hash(question)
    if question_hash != cache_entry["question_hash"]:
        result["reason"] = "hash_mismatch"
        return result
    
    # 4. 完整性检查
    if not validate_cache_integrity(cache_entry):
        result["reason"] = "data_corrupted"
        return result
    
    # 5. 质量评分检查（可选）
    if cache_entry.get("quality_score", 1.0) < MIN_QUALITY_THRESHOLD:
        result["reason"] = "low_quality"
        return result
    
    # 所有检查通过
    result["hit"] = True
    result["data"] = cache_entry["data"]
    return result
```

### 5.4 相似问题匹配

对于非精确匹配的情况，支持相似问题召回：

```yaml
相似匹配策略:
  
  1. 关键词重叠匹配
     - 提取问题关键词
     - 与缓存问题关键词对比
     - 重叠度 > 80% 视为相似
  
  2. 向量相似度匹配
     - 使用问题Embedding
     - 计算余弦相似度
     - 相似度 > 0.9 可复用
  
  3. 模板匹配
     - 匹配问题模板
     - 模板匹配时复用专家组合

相似匹配降级:
  - 相似匹配作为缓存未命中的补充
  - 需要用户确认后才返回
  - 确认后写入缓存
```

## 八、缓存指标与监控

### 8.1 核心指标

| 指标 | 说明 | 目标 |
|------|------|------|
