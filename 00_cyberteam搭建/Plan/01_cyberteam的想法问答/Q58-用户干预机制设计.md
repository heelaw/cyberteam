# Q58: 思考天团用户干预机制设计

**问题**: 如何设计思考天团中的用户干预机制，允许用户在协作过程中进行干预（如指定专家、暂停任务、修改目标）？

---

## 一、设计目标与原则

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| **用户控制感** | 赋予用户对任务执行的全程控制权 |
| **实时干预能力** | 支持任务执行过程中的动态调整 |
| **清晰边界** | 明确用户能干预和不能干预的范围 |
| **一致性保持** | 与现有系统设计保持一致 |

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **最小阻力干预** | 干预操作应简单直观，降低用户学习成本 |
| **可逆优先** | 优先支持可逆的干预操作 |
| **即时反馈** | 干预后立即反馈执行结果 |
| **上下文感知** | 干预需考虑当前任务状态 |
| **与Q56保持一致** | 干预可能影响ROI，干预行为本身可度量 |

## 二、用户可以干预什么？

### 2.1 干预类型矩阵

| 干预类型 | 描述 | 示例 | 可逆性 |
|----------|------|------|--------|
| **专家选择** | 指定、跳过或重新选择专家 | "跳过Kahneman" | ✅ 可逆 |
| **目标修改** | 调整任务目标或期望 | "目标改为提升20%" | ✅ 可逆 |
| **任务控制** | 暂停、继续、终止任务 | "暂停任务" | ✅ 可逆 |
| **输出格式** | 调整输出详细程度或格式 | "只输出要点" | ✅ 可逆 |
| **协作模式** | 切换并行/串行/辩论模式 | "改成辩论模式" | ✅ 可逆 |
| **优先级调整** | 调整专家调用优先级 | "战略专家优先" | ✅ 可逆 |

### 2.2 干预意图解析

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│                    干预意图解析器                         │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │ 关键词检测                                        │   │
│  ├── "跳过" / "不要" / "去掉" → 排除专家            │
│  ├── "只保留" / "只要" → 限定专家                    │
│  ├── "暂停" / "停止" → 任务控制                     │
│  ├── "继续" / "恢复" → 任务继续                      │

│                          │                             │

│  ┌─────────────────────────────────────────────────┐   │
```

## 三、干预的时机

### 3.1 干预时机分类

| 时机 | 说明 | 可用干预类型 |
|------|------|--------------|
| **任务前** | 用户发起任务后、专家执行前 | 全部类型 |
| **任务中** | 专家执行过程中 | 部分类型（有限制） |
| **任务后** | 任务完成后 | 仅限查看历史 |

### 3.2 任务阶段与干预权限

```
任务生命周期
─────────────────────────────────────────────────────────────

[任务创建] ──► [意图分析] ──► [专家选择] ──► [执行前]
     │             │             │             │
     │             │             │             │
   用户干预      用户干预      用户干预      用户干预
   (全部类型)   (全部类型)   (全部类型)   (全部类型)

           │
           ▼
    ┌─────────────────────────────────────┐
    │           专家执行阶段                │
    │                                     │
    │  [专家1]  [专家2]  [专家3]  ...    │
    │    │        │        │              │
    │    │        │        │              │
    │  用户干预  用户干预  用户干预        │
    │  (有限制)  (有限制)  (有限制)        │
    │                                     │
    │  可干预: 暂停、目标修改、输出调整     │
    │  不可: 跳过已完成专家                │
    └─────────────────────────────────────┘
           │
           ▼
     [结果整合] ──► [最终输出]
           │             │
           │             │
         用户干预       用户干预
         (有限制)      (仅查看)

─────────────────────────────────────────────────────────────
```

### 3.3 各阶段干预详细说明

#### 阶段1: 任务创建后

```
状态: 等待意图分析
干预权限: ✅ 全部

可执行操作:
├── 修改问题描述
├── 指定专家
├── 排除专家
├── 设置目标
├── 选择协作模式
└── 取消任务
```

#### 阶段2: 意图分析中

```
状态: 系统正在分析问题类型
干预权限: ✅ 全部

可执行操作:
├── 修改问题
├── 指定/排除专家
├── 设置目标
├── 取消任务
└── 等待分析完成
```

#### 阶段3: 专家选择后（执行前）

```
状态: 已确定调用哪些专家
干预权限: ✅ 全部

可执行操作:
├── 排除特定专家
├── 添加专家
├── 调整专家顺序
├── 修改目标
├── 切换协作模式
└── 取消任务
```

#### 阶段4: 专家执行中

```
状态: 专家正在并行/串行执行
干预权限: ⚠️ 有限

可执行操作:
├── 暂停任务 ← 等待当前专家完成
├── 修改目标 ← 下一步专家生效
├── 调整输出 ← 后续生效
└── 终止任务 ← 放弃后续执行

不可操作:
├── 跳过当前正在执行的专家
├── 修改已完成专家的输出
└── 改变已确定的执行顺序
```

#### 阶段5: 结果整合中

```
状态: 正在整合专家观点
干预权限: ⚠️ 有限

可执行操作:
├── 调整输出详细程度
├── 要求补充分析
└── 终止任务（跳过整合）
```

#### 阶段6: 任务完成后

```
状态: 输出已生成
干预权限: ❌ 仅查看

可执行操作:
├── 查看历史干预记录
├── 发起新任务（基于原任务）
└── 反馈干预效果
```

## 四、干预如何生效？

### 4.1 干预优先级体系

```
┌─────────────────────────────────────────────────────────────┐
│                      干预优先级                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  P0 (最高): 用户显式终止       ─────► 立即停止所有执行    │
│                                                             │
│  P1: 用户显式指定专家         ─────► 强制调用指定专家      │
│                                                             │
│  P2: 用户显式排除专家         ─────► 从列表中移除          │
│                                                             │
│  P3: 用户暂停任务            ─────► 等待当前步完成后暂停  │
│                                                             │
│  P4: 用户修改目标            ─────► 下一步生效            │
│                                                             │
│  P5: 用户调整输出            ─────► 剩余步骤生效         │
│                                                             │
│  P6: 系统自动决策            ─────► 无用户干预时默认行为  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 生效方式分类

| 生效方式 | 说明 | 适用场景 |
|----------|------|----------|
| **立即执行** | 干预立即生效，停止当前执行 | P0终止、P1指定专家 |
| **排队执行** | 干预进入队列，等待当前步完成 | P3暂停、P4目标修改 |
| **下一步生效** | 当前步完成后生效 | P4目标修改 |
| **剩余步骤生效** | 当前步完成后，后续步骤生效 | P5输出调整 |

### 4.3 干预执行流程

```
用户干预输入
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│                    干预验证器                             │
│                                                         │
│  1. 检查干预时机是否合法                                  │
│  2. 检查干预内容是否有效                                  │
│  3. 检查干预优先级                                       │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │ 验证结果                                          │   │
│  ├── 合法 + 有效 → 执行干预                          │   │
│  ├── 合法 + 无效 → 提示用户                          │   │
│  └── 非法 → 拒绝干预 + 说明原因                      │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
    │
    ├──► 干预执行
    │       │
    │       ▼
    │   ┌─────────────────────────────────────────────┐
    │   │ 干预生效方式                                 │
    │   │                                             │
    │   ├── 立即执行: 修改expert_queue立即生效         │
    │   ├── 排队执行: 加入intervention_queue          │
    │   ├── 下一步生效: 修改next_step_config          │
    │   └── 剩余生效: 修改remaining_config            │
    │   └─────────────────────────────────────────────┘
    │       │
    │       ▼
    │   ┌─────────────────────────────────────────────┐
    │   │ 反馈用户                                     │
    │   │                                             │
    │   ├── 成功: "已跳过Kahneman专家"                │
    │   ├── 等待: "任务将在当前步完成后暂停"           │
    │   └── 失败: "该干预无法执行，原因..."           │
    │   └─────────────────────────────────────────────┘
    │
    └──► 拒绝干预 → 提示原因
```

### 4.4 干预队列管理

```python
# 干预队列管理器
class InterventionQueue:
    """干预队列管理器"""

    def __init__(self):
        self.queue = []  # 干预队列
        self.executing = None  # 正在执行的干预

    PRIORITY_MAP = {
        "terminate": 0,      # P0 终止
        "include_expert": 1, # P1 指定专家
        "exclude_expert": 2, # P2 排除专家
        "pause": 3,         # P3 暂停
        "modify_goal": 4,    # P4 修改目标
        "modify_output": 5,  # P5 调整输出
    }

    def add_intervention(self, intervention: dict) -> bool:
        """添加干预到队列"""

        # 1. 验证干预合法性
        if not self.validate_intervention(intervention):
            return False

        # 2. 获取优先级
        priority = self.PRIORITY_MAP.get(
            intervention["type"],
            99
        )

        # 3. 处理高优先级干预
        if priority == 0:  # 终止任务
            self.queue.clear()
            self.queue.insert(0, intervention)
            return True

        # 4. 按优先级插入队列
        inserted = False
        for i, item in enumerate(self.queue):
            if priority < self.PRIORITY_MAP.get(item["type"], 99):
                self.queue.insert(i, intervention)
                inserted = True
                break

        if not inserted:
            self.queue.append(intervention)

        return True

    def get_next_intervention(self) -> dict:
        """获取下一个待执行的干预"""
        if not self.queue:
            return None

        # P0 立即执行
        if self.queue[0]["type"] == "terminate":
            return self.queue.pop(0)

        # 其他按队列执行
        return self.queue[0]

    def execute_immediately(self, intervention: dict) -> bool:
        """立即执行干预（高优先级）"""
        self.executing = intervention
        # 执行干预逻辑
        # ...
        self.executing = None
        return True
```

## 五、干预的边界

### 5.1 边界定义

```
┌─────────────────────────────────────────────────────────────────┐
│                        干预边界                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────┐    ┌─────────────────────┐            │
│  │     可干预区域       │    │    不可干预区域      │            │
│  ├─────────────────────┤    ├─────────────────────┤            │
│  │                     │    │                     │            │
│  │ • 专家选择           │    │ • 已完成专家的结果   │            │
│  │ • 任务目标           │    │ • 专家的思考过程     │            │
│  │ • 执行模式           │    │ • 系统决策逻辑       │            │
│  │ • 暂停/继续          │    │ • 跨任务上下文       │            │
│  │ • 输出格式           │    │ • 历史已存储数据     │            │
│  │ • 执行顺序           │    │ • Token消耗计算      │            │
│  │                     │    │                     │            │
│  └─────────────────────┘    └─────────────────────┘            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 具体边界规则

| 边界类型 | 规则 | 示例 |
|----------|------|------|
| **已完成不可逆** | 已完成的专家分析结果不可修改 | "修改kahneman的分析结果" → ❌ 拒绝 |
| **思考过程不可干预** | 专家的推理过程不可干预 | "让kahneman用系统1思考" → ❌ 拒绝 |
| **系统逻辑不可干预** | 路由逻辑、成本计算不可干预 | "降低token消耗" → ❌ 拒绝 |
| **跨任务不可干预** | 历史任务数据不可修改 | "修改上周的任务目标" → ❌ 拒绝 |
| **最低限度执行** | 至少保留一个专家 | "排除所有专家" → ❌ 拒绝 |
| **最大专家数限制** | 干预后专家数不超过上限 | "添加20个专家" → ⚠️ 提示限制 |

### 5.3 边界验证示例

```python
# 干预边界验证器
class InterventionBoundaryValidator:
    """干预边界验证器"""

    def validate(self, intervention: dict, task_state: dict) -> dict:
        """验证干预是否在边界内"""

        intervention_type = intervention["type"]
        task_status = task_state["status"]
        completed_experts = task_state.get("completed_experts", [])
        current_experts = task_state.get("expert_queue", [])

        # 规则1: 任务完成后仅可查看
        if task_status == "completed":
            if intervention_type in ["exclude_expert", "include_expert",
                                      "modify_goal", "pause", "terminate"]:
                return {
                    "valid": False,
                    "reason": "任务已完成，仅可查看历史记录"
                }

        # 规则2: 不可跳过已完成的专家
        if intervention_type == "exclude_expert":
            excluded = intervention.get("expert_id")
            if excluded in completed_experts:
                return {
                    "valid": False,
                    "reason": f"专家 {excluded} 已完成分析，不可跳过"
                }

        # 规则3: 至少保留一个专家
        if intervention_type == "exclude_expert":
            remaining = [e for e in current_experts
                        if e != intervention.get("expert_id")]
            if len(remaining) == 0:
                return {
                    "valid": False,
                    "reason": "至少需要保留一位专家"
                }

        # 规则4: 专家数量上限
        MAX_EXPERTS = 14
        if intervention_type == "include_expert":
            new_count = len(current_experts) + 1
            if new_count > MAX_EXPERTS:
                return {
                    "valid": False,
                    "reason": f"专家数量不能超过{MAX_EXPERTS}位"
                }

        # 规则5: 执行中不可改变已完成专家
        if task_status == "executing":
            if intervention_type == "include_expert":
                # 添加新专家只能在当前步完成后
                return {
                    "valid": True,
                    "execute_mode": "next_step"  # 下一步生效
                }

        return {"valid": True}


# 边界规则检查
BOUNDARY_RULES = {
    "task_completed": {
        "allowed": [],
        "message": "任务已完成，无法干预"
    },
    "expert_completed": {
        "allowed": [],
        "message": "该专家已完成分析，无法跳过"
    },
    "min_experts": {
        "min": 1,
        "message": "至少需要保留一位专家"
    },
    "max_experts": {
        "max": 14,
        "message": "专家数量不能超过14位"
    }
}
```

### 8.3 与Q56 ROI度量的关系

```

```

## 九、错误处理

### 9.1 干预错误处理

| 错误类型 | 错误原因 | 处理方式 | 用户反馈 |
|----------|----------|----------|----------|
| INVALID_TIME | 干预时机不对 | 拒绝执行 | "该操作在当前阶段无法执行" |
| INVALID_TARGET | 干预目标无效 | 拒绝执行 | "指定的专家不存在" |
| BOUNDARY_VIOLATION | 违反边界规则 | 拒绝执行 | "已完成专家无法跳过" |
| QUEUE_FULL | 干预队列满 | 拒绝执行 | "干预队列已满，请稍后再试" |
| PARSE_ERROR | 解析失败 | 要求澄清 | "请明确说明要排除哪个专家" |

### 9.2 错误处理示例

```python
# 干预错误处理器
class InterventionErrorHandler:
    """干预错误处理器"""

    ERROR_MESSAGES = {
        "INVALID_TIME": {
            "task_completed": "任务已完成，无法干预",
            "expert_executing": "该专家正在执行中，完成后可跳过"
        },
        "INVALID_TARGET": {
            "expert_not_found": "专家 {expert_id} 不存在",
            "goal_invalid": "目标格式不正确，请使用具体描述"
        },
        "BOUNDARY_VIOLATION": {
            "expert_completed": "专家 {expert_id} 已完成分析，无法跳过",
            "min_experts": "至少需要保留一位专家",
            "max_experts": "专家数量不能超过14位"
        },
        "QUEUE_FULL": "干预队列已满（最多5个），请稍后再试",
        "PARSE_ERROR": "无法理解您的干预意图，请更具体地描述"
    }

    def handle_error(self, error: dict) -> str:
        """生成用户友好的错误消息"""

        error_type = error.get("type")
        error_detail = error.get("detail", {})

        template = self.ERROR_MESSAGES.get(error_type, {})

        if isinstance(template, dict):
            # 格式化错误消息
            key = error_detail.get("reason", "")
            message = template.get(key, "未知错误")
            return message.format(**error_detail)
        else:
            return template
```

## 十一、总结

### 11.1 设计要点

| 要点 | 说明 |
|------|------|
| **干预类型** | 6种：专家选择、目标修改、任务控制、输出格式、协作模式、优先级调整 |
| **干预时机** | 3个阶段：任务前（全部）、任务中（有限）、任务后（仅查看） |
| **生效方式** | 4种：立即执行、排队执行、下一步生效、剩余步骤生效 |
| **边界规则** | 6条边界：已完成不可逆、思考过程不可干预、系统逻辑不可干预等 |
| **优先级体系** | P0-P6，终止任务最高优先级 |

### 11.2 与系统一致性
