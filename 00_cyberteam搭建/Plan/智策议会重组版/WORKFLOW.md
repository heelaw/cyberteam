# 智策议会 工作流

## 1. 输入

用户输入自然语言问题，附带背景、约束和希望看的角度。

优先识别三件事：

- 这个问题到底有没有决策价值。
- 用户想要的是分析、建议，还是直接方案。
- 用策源议会能不能明显比不用更好。

## 2. 预处理

先做三件事：

- 提炼主问题。
- 识别隐含约束。
- 判断问题是否需要多个专家协作。
- 判断 `with / without` 的差异。

## 3. 分类

优先判断属于哪类主任务：

- 决策选择
- 原因诊断
- 目标实现
- 战略规划
- 创新突破
- 多角度分析
- 复盘总结
- 方法提炼
- 管理跃迁
- 不确定性判断

如果问题同时命中多个类别，以最接近最终交付的那一类为主。

## 4. 路由

- 决策选择 → `kahneman`
- 原因诊断 → `fivewhy` + `problem-reformulation`
- 目标实现 → `grow` + `wbs`
- 战略规划 → `swot-tows`
- 创新突破 → `first-principle`
- 多角度分析 → `systems-thinking` + `ai-board`
- 复盘总结 → `kiss`
- 方法提炼 → `problem-reformulation` + `systems-thinking`
- 管理跃迁 → `manager-leap`
- 不确定性判断 → `probabilistic`

## 5. 协作模式

### 并行分析

适合需要多个角度的场景。选 2-5 个核心专家并行输出，再由主持人整合。

### 交叉辩论

适合重大决策。先组织支持与反对两组观点，再由主持人裁决。

### 链式分析

适合问题诊断和执行路径问题。前一步结果必须成为下一步输入。

## 6. 整合

主持人必须把专家输出压缩为：

- 问题理解
- 值不值得用议会
- with / without 差异
- 共识
- 分歧
- 结论
- 行动建议

整合时默认采用作者化表达：先抛总判断，再按观察点展开，最后回到趋势和行动，不输出碎片化会议记录。要把价值点、预期和边界讲清楚。

## 7. 失败回退

- 如果分类不清，先走 `problem-reformulation`。
- 如果信息不足，先补问题定义，不要盲目展开。
- 如果专家输出过散，主持人只保留最能推动结论的部分。

## 8. 交付标准

- 不堆砌。
- 不空泛。
- 不回避结论。
- 不把长尾模型当主角。
