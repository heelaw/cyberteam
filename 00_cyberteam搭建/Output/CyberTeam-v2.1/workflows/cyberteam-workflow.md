# CyberTeam v2.1 × gstack 完整工作流

## 技能链编排

```
用户输入
    │
    ▼
┌─────────────────┐
│ /office-hours   │ ← 产品定义 (6个强制性问题)
└────────┬────────┘
         │ [设计文档]
         ▼
┌─────────────────┐
│ /gsd-roadmap   │ ← 路线图规划
└────────┬────────┘
         │ [路线图]
         ▼
┌─────────────────┐
│ /gsd-plan       │ ← 规格制定 (工程/CEO/设计审查)
└────────┬────────┘
         │ [规格文档]
         ▼
┌─────────────────┐
│ /gsd-execute   │ ← 开发执行
└────────┬────────┘
         │ [代码]
         ▼
┌─────────────────┐
│ /gsd-review     │ ← 代码审查 (安全 + 质量)
└────────┬────────┘
         │ [审查报告]
         ▼
┌─────────────────┐
│ /gsd-qa        │ ← QA测试 (Playwright + E2E)
└────────┬────────┘
         │ [测试报告]
         ▼
┌─────────────────┐
│ /gsd-ship       │ ← 部署发布
└────────┬────────┘
         │ [发布]
         ▼
┌─────────────────┐
│ /gsd-retro     │ ← 回顾改进 (每周)
└─────────────────┘
```

## 各环节职责

### 1. /office-hours
- 角色: YC合作伙伴
- 输入: 用户想法
- 输出: 设计文档
- 关键问题: Q1-Q6 (需求现实/现状/用户/最小 wedge/观察/未来)

### 2. /gsd-roadmap
- 角色: 项目路线图规划
- 输入: 设计文档
- 输出: 分阶段路线图
- 内容: 阶段划分/需求映射/成功标准

### 3. /gsd-plan
- 角色: 规格制定
- 输入: 路线图
- 输出: 详细规格文档
- 审查: 工程审查 + CEO审查 + 设计审查

### 4. /gsd-execute
- 角色: 开发执行
- 输入: 规格文档
- 输出: 功能代码
- 遵循: Boil the Lake原则

### 5. /gsd-review
- 角色: 高级工程师
- 输入: 代码
- 输出: 审查报告
- 覆盖: 安全 + 质量 + 性能

### 6. /gsd-qa
- 角色: QA负责人
- 输入: 代码 + 规格
- 输出: 测试报告 + 回归测试
- 工具: Playwright持久浏览器

### 7. /gsd-ship
- 角色: 发布工程师
- 输入: 测试报告
- 输出: 生产部署
- 步骤: 同步 → 测试 → 推送 → PR

### 8. /gsd-retro
- 角色: 工程经理
- 频率: 每周
- 输出: 改进建议

## 文档流转

```
office-hours → roadmap → plan → execute → review → qa → ship → retro
     │           │        │        │        │      │      │
     ▼           ▼        ▼        ▼        ▼      ▼      ▼
  [设计]    [路线图]  [规格]   [代码]   [报告]  [报告] [发布]
```
