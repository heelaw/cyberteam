## 核心框架

### 框架一：进度管理三环模型

```
         ┌─────────────────────────────────────────┐
         │              进度管理                    │
         ├─────────────┬─────────────┬──────────────┤
         │   计划环    │   执行环    │   控制环     │
         ├─────────────┼─────────────┼──────────────┤
         │ 制定基线    │ 实施任务    │ 监控偏差     │
         │ 分解任务    │ 记录进展    │ 分析原因     │
         │ 设定里程碑  │ 产出成果    │ 调整计划     │
         └─────────────┴─────────────┴──────────────┘
```

**三环关系**：
- 计划环在前，确定"做什么、何时做"
- 执行环在中，实际"做"的过程
- 控制环在后，确保"做对方向"

### 框架二：进度状态四象限

| 状态 | 描述 | 应对策略 |
|------|------|----------|
| **正常** | 按计划推进 | 保持监控 |
| **轻微偏差** | 进度滞后<10% | 加强资源 |
| **严重偏差** | 进度滞后>10% | 紧急干预 |
| **失控** | 目标无法达成 | 重新规划 |

### 框架三：进度预警机制

```
进度健康度 = (实际进度 / 计划进度) × 100%

┌─────────┬─────────┬─────────┬─────────┐
│ 绿灯    │ 黄灯    │ 橙灯    │ 红灯    │
│ ≥95%    │ 85-95%  │ 70-85%  │ <70%    │
├─────────┼─────────┼─────────┼─────────┤
│ 正常运行 │ 关注跟进 │ 预警干预 │ 紧急处理 │
└─────────┴─────────┴─────────┴─────────┘
```

## 执行步骤

### Step 1: 制定进度基线

**目的**：建立进度测量的标准参照

**操作流程**：

1. **明确项目范围与目标**
   
   ```
   项目名称：[项目名]
   最终目标：[量化目标]
   时间窗口：[开始日期] - [结束日期]
   总工期：[X]个工作日
   ```

2. **建立工作分解结构（WBS）**
   
   ```
   1.0    项目总任务                   总工期：[X]天
   1.1    阶段一：[名称]               工期：[X]天
   1.1.1  任务1.1.1                   工期：[X]天
   1.1.2  任务1.1.2                   工期：[X]天
   1.2    阶段二：[名称]               工期：[X]天
   1.2.1  任务1.2.1                   工期：[X]天
   ...
   ```

3. **设定关键里程碑**
   
   | 里程碑 | 计划完成日 | 交付物 | 责任人 |
   |--------|------------|--------|--------|
   | M1：方案确认 | 04-05 | 方案文档v1.0 | @张三 |
   | M2：设计完成 | 04-12 | 设计稿v2.0 | @李四 |
   | M3：开发完成 | 04-25 | 可测试版本 | @王五 |
   | M4：测试通过 | 05-03 | 测试报告 | @赵六 |
   | M5：上线发布 | 05-10 | 正式版本 | @张三 |

4. **绘制基线甘特图**

   ```
   任务            │ W1 │ W2 │ W3 │ W4 │ W5 │ W6 │
   ────────────────┼────┼────┼────┼────┼────┼────┤
   方案设计        │████│    │    │    │    │    │
   评审确认        │████│    │    │    │    │    │
   详细设计        │    │████│████│    │    │    │
   开发执行        │    │    │████│████│████│    │
   测试验证        │    │    │    │    │████│████│
   上线部署        │    │    │    │    │    │████│
   
   里程碑：  M1      M2       M3        M4      M5
   ```

### Step 2: 建立进度追踪机制

**目的**：实时掌握项目实际进展

**操作流程**：

1. **选择追踪粒度**
   
   | 项目周期 | 追踪频率 | 报告周期 |
   |----------|----------|----------|
   | <1周 | 每日 | 每日 |
   | 1-4周 | 每2日 | 每周 |
   | 1-3月 | 每周 | 每周 |
   | >3月 | 每周 | 每月 |

2. **建立进展报告模板**

   ```
   【周进度报告】第X周（03/18-03/24）
   
   一、整体进度
   ├─ 计划完成：45%
   ├─ 实际完成：42%
   └─ 偏差：-3%（滞后）
   
   二、本周完成情况
   ├─ [任务A] ✓ 已完成
   ├─ [任务B] ✓ 已完成
   └─ [任务C] ○ 进行中（80%）
   
   三、下周计划
   ├─ [任务D] 开始执行
   ├─ [任务E] 继续推进
   └─ [任务F] 准备启动
   
   四、风险预警
   ├─ 风险1：[描述] - 影响：高 - 应对：[措施]
   └─ 风险2：[描述] - 影响：中 - 应对：[措施]
   
   五、资源需求
   └─ [具体需求]
   ```

3. **收集进展数据**
   
   - 任务完成百分比
   - 消耗工时
   - 里程碑达成情况
   - 阻塞问题列表

---

### Step 3: 分析进度偏差

**目的**：识别问题根源，制定纠偏策略

**偏差分析模型**：

```
进度偏差 = 实际进度 - 计划进度
          = (AC / PV) - 1  （AC=实际完成，PV=计划完成）

偏差分类：
├─ 进度超前：偏差 > 0 → 可喜，评估是否可持续
├─ 进度一致：偏差 ≈ 0 → 正常
├─ 进度滞后：偏差 < 0 → 需分析原因并纠偏
```

**根因分析四问**：

| 问题 | 具体问句 | 可能的答案 |
|------|----------|------------|
| 发生了什么？ | 具体哪些任务延期？ | [具体描述] |
| 什么时候发生的？ | 从哪个节点开始滞后？ | [时间点] |
| 影响范围多大？ | 对总工期的影响？ | [天数/百分比] |
| 根本原因是什么？ | 为什么延期？ | [原因分类] |

**原因分类表**：

| 类别 | 典型原因 | 应对方向 |
|------|----------|----------|
| 资源不足 | 人力不够、资金不到位 | 争取资源 |
| 需求变更 | 范围蔓延、需求不清 | 控制变更 |
| 依赖阻塞 | 外部依赖、协作延迟 | 协调推进 |
| 能力问题 | 技术难度、经验不足 | 培训/外包 |
| 计划问题 | 估算不准、遗漏任务 | 优化方法 |

---

### Step 4: 采取纠正措施

**目的**：消除进度偏差，确保目标达成

**纠偏策略矩阵**：

| 偏差程度 | 策略 | 手段 |
|----------|------|------|
| 轻微（<5%） | 加强执行 | 增加工时、加班 |
| 中度（5-15%） | 优化方法 | 并行推进、简化流程 |
| 严重（>15%） | 重新规划 | 删减范围、延长时间 |
| 失控（>30%） | 止损决策 | 放弃/缩减/延期 |

**常用纠偏手段**：

1. **压缩工期法**
   - 赶工（Crashing）：增加资源缩短工期，成本增加
   - 并行执行：将串行任务改为并行
   - 快速跟进：部分重叠执行原本顺序的任务

2. **资源调配法**
   - 从容项目借调人力
   - 外包部分任务
   - 临时招聘/外包

3. **范围调整法**
   - 删除低优先级功能
   - 降低质量标准
   - 推迟非核心交付

**纠偏决策表**：

| 方案 | 效果 | 代价 | 风险 | 推荐 |
|------|------|------|------|------|
| 方案A：加班赶工 | +5天 | 成本+20% | 质量下降 | ✓ |
| 方案B：外包非核心 | +7天 | 成本+15% | 沟通成本 | ○ |
| 方案C：删减功能 | +10天 | 无额外成本 | 影响价值 | △ |

**目的**：统一认知，争取支持，协调资源

**沟通机制设计**：

| 场景 | 受众 | 内容重点 | 频率 | 形式 |
|------|------|----------|------|------|
| 例行汇报 | 管理层 | 整体/里程碑 | 周 | 邮件/会议 |
| 风险预警 | 管理层 | 风险/决策需求 | 随时 | 专项报告 |
| 协作同步 | 相关部门 | 依赖/接口 | 周 | 站会 |
| 执行指导 | 执行团队 | 任务/资源 | 日/周 | 群/面对面 |

## 使用示例

### 完整场景：季度运营活动进度管理

**背景**：
- 项目：618大促活动
- 总工期：60天（4月15日-6月15日）
- 团队：运营10人 + 技术5人

**Step 1: 建立进度基线**

关键里程碑：
- M1（04/20）：活动方案审批通过
- M2（05/01）：商品招商完成
- M3（05/15）：技术开发完成
- M4（05/25）：预热传播启动
- M5（06/18）：大促正式开启
- M6（06/25）：活动收尾复盘

**Step 2: 建立追踪机制**
- 每日站会：15分钟，同步进展
- 周报：每周一发送上周进展
- 例会：每周三进度评审会

**Step 3-4: 偏差分析与纠偏**

第5周检查点：
- 计划进度：50%
- 实际进度：42%
- 偏差：-8%（严重）

原因分析：
- 第三方系统对接延期（外部依赖）
- 运营素材制作人力不足（资源问题）

纠偏措施：
- 派专人跟进第三方，快速锁定方案
- 从其他项目借调2名设计支援
- 调整部分非核心功能延后

**Step 5: 沟通汇报**

向管理层汇报：
```
618活动进度预警

当前状态：🟠 橙色预警

1. 进度偏差：较计划滞后8天
2. 主要原因：第三方接口延期（不可抗力）+ 资源紧张

3. 已采取措施：
   - 派专人跟进第三方接口
   - 借调设计资源支援
   - 调整部分功能优先级

4. 预计影响：
   - M3里程碑可能延期2-3天
   - 不影响核心活动（618当天）

5. 决策需求：
   - 是否批准追加5万预算用于赶工？
   - 是否接受部分功能延后上线？
```
