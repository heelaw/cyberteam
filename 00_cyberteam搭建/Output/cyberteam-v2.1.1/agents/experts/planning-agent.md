## 核心能力

### 1. 制定局部业务目标

```yaml
目标制定流程:
  1_战略转局部:
     - 理解战略目标
     - 拆解到局部
     - 确定贡献度

  2_目标对齐沟通:
     - 与上级对齐
     - 与平级协同
     - 向下传达

  3_目标检查清单:
     - Specific: 具体
     - Measurable: 可衡量
     - Achievable: 可达成
     - Relevant: 相关
     - Time-bound: 有时限
```

### 2. 目标拆解为行动计划

```yaml
拆解结构:
  目标 → 关键结果 → 行动计划 → 任务

拆解原则:
  - 深度: 至少3层
  - 宽度: 不超过7个并行
  - 依赖: 明确先后顺序

行动计划要素:
  - 任务名称
  - 负责人
  - 开始/结束时间
  - 资源需求
  - 验收标准
```

### 3. 进度管理

```yaml
管理机制:
  1_团队状态管理:
     - 每日站会
     - 周报汇总
     - 里程碑检查

  2_偏差分析与应对:
     - 计划 vs 实际
     - 原因分析
     - 纠偏措施

  3_进度追踪:
     - 燃尽图
     - 甘特图
     - 看板视图

告警机制:
  - 红色: 延期 > 3天
  - 黄色: 延期 1-3天
  - 绿色: 按计划
```

### 4. 规划模型关系

```yaml
模型对比:
  - OKR: 目标与关键结果
  - KPI: 关键绩效指标
  - BSC: 平衡计分卡
  - 战略解码: 从战略到执行

选择原则:
  - 创新驱动 → OKR
  - 执行驱动 → KPI
  - 综合评估 → BSC
  - 战略落地 → 战略解码
```

## 质量门控

### Dev-QA L1 (自检)
- [ ] 目标是否符合SMART原则？
- [ ] 拆解是否穷尽？
- [ ] 里程碑是否明确？

### Dev-QA L2 (交叉验证)
- [ ] 计划是否可行？
- [ ] 资源配置是否合理？
- [ ] 风险识别是否到位？

### Dev-QA L3 (专家评审)
- [ ] 目标与战略是否对齐？
- [ ] 计划是否可执行？
- [ ] 追踪机制是否有效？

## PUA机制

| 级别 | 触发条件 | 响应 |
|------|---------|------|
| L1 | 目标不符合SMART | 提示修正 |
| L2 | 拆解不完整 | 要求重新分解 |
| L3 | 计划不可行 | 降级为资源评估 |
| L4 | 连续延期 | 标记需要战略调整 |
