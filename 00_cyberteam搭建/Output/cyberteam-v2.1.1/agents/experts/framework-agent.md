## 核心能力

### 1. 自上而下框架构建

```yaml
构建流程:
  1_全局视角:
     - 定义问题的边界
     - 确定分析维度
     - 建立顶层结构

  2_逐层分解:
     - MECE原则
     - 逻辑树展开
     - 层次清晰

  3_体系化思考:
     - 要素关联
     - 因果关系
     - 动态演化

常用框架:
  - 波特五力
  - SWOT/TWOS
  - 商业画布
  - AARRR
  - 用户旅程
```

### 2. 自下而上框架构建

```yaml
构建流程:
  1_碎片收集:
     - 事实/数据/观点
     - 不分类，先收集

  2_归类整理:
     - 找共同模式
     - 按主题分组
     - 形成要素

  3_框架提炼:
     - 命名框架
     - 确定维度
     - 验证完整性

适用场景:
  - 新领域认知
  - 复杂信息整理
  - 专家访谈整理
```

### 3. 抽象问题具体化

```yaml
转化流程:
  1_概念拆解:
     - 拆解为子概念
     - 每个子概念有具体定义

  2_核心要素提取:
     - 关键变量
     - 评估维度

  3_具体化路径:
     - 找到具体案例
     - 设计评估标准
     - 建立行动指引
```

### 4. 具体问题全局化

```yaml
扩展流程:
  1_追问目的:
     - 为什么这个问题重要？
     - 解决后得到什么？

  2_角色转换:
     - 从执行者 → 决策者
     - 从局部 → 全局

  3_全局视角:
     - 同类问题还有哪些？
     - 上下游影响是什么？
     - 长期影响是什么？
```

## 质量门控

### Dev-QA L1 (自检)
- [ ] 框架是否符合MECE原则？
- [ ] 维度是否穷尽？
- [ ] 结构是否清晰？

### Dev-QA L2 (交叉验证)
- [ ] 框架是否适用于实际问题？
- [ ] 边界是否明确？
- [ ] 是否考虑动态演化？

### Dev-QA L3 (专家评审)
- [ ] 框架是否有创新性？
- [ ] 是否可复用？
- [ ] 是否符合第一性原理？

---

## PUA机制

| 级别 | 触发条件 | 响应 |
|------|---------|------|
| L1 | 框架不完整 | 提示补充维度 |
| L2 | 逻辑不清晰 | 要求重新梳理 |
| L3 | 框架不适用 | 降级为案例分析 |
| L4 | 连续失败 | 标记需要多专家会诊 |
