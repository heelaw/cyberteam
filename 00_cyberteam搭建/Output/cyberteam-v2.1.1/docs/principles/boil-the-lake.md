# Boil the Lake 完整性原则

> AI时代，完整性的边际成本接近于零。

## 核心原则

### 1. 湖 vs 海洋
- **湖**: 可煮沸的——100%测试覆盖、完整功能实现、所有边缘用例
- **海洋**: 不可煮沸的——重写整个系统、无限范围的需求

### 2. 工作量估算表

| 任务类型 | 人工团队 | CC+gstack | 压缩比 |
|----------|----------|-----------|--------|
| 样板/脚手架 | 2天 | 15分钟 | ~100x |
| 测试编写 | 1天 | 15分钟 | ~50x |
| 功能实现 | 1周 | 30分钟 | ~30x |
| Bug修复+回归测试 | 4小时 | 15分钟 | ~20x |
| 架构/设计 | 2天 | 4小时 | ~5x |
| 研究/探索 | 1天 | 3小时 | ~3x |

### 3. 反模式清单
- ~~"选择B——它覆盖了90%的价值，代码更少"~~
- ~~"我们可以跳过边缘用例处理来节省时间"~~
- ~~"把测试覆盖推迟到后续PR"~~
- ~~"差不多就行"~~

### 4. 正确的做法
- 如果选项A是完整实现(全部边缘用例、100%覆盖)而B是捷径——**永远选择A**
- 湖的delta(80行vs150行)在CC+gstack面前毫无意义
- 完整性边际成本在AI时代接近零

## 在CyberTeam中的应用

### Agent工作标准
所有CyberTeam Agent必须遵循:
1. 完整实现功能，不留TODO
2. 完整测试覆盖，不跳过边缘用例
3. 完整文档，不留"待补充"
4. 完整验证，不止步于"看起来可以"

### Dev-QA集成
- L1完整性门控 ≥95% 正是Boil the Lake的量化体现
- 每个交付物必须通过完整性检查
