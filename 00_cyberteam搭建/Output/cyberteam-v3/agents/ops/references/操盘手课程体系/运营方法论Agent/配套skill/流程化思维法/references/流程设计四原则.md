# 流程设计四原则

## 原则一：单一入口出口

### 定义
每个流程必须有明确的起点（开始符号○）和终点（结束符号○）。

### 为什么重要
- 便于追踪和复盘
- 避免流程陷入死循环
- 便于计算流程效率

### 常见错误

```
错误示例：
○ → □ → □ → ○
    ↑       ↓
    ←←←←←←←

问题：步骤2和步骤3形成循环
```

```
正确示例：
○ → □ → □ → ◇是 → ○
    ↓否
  □ → ○
```

### 应用检查
- [ ] 有明确的开始符号○
- [ ] 有明确的结束符号○
- [ ] 无死循环
- [ ] 每个路径都能到达结束

---

## 原则二：步骤不可逆

### 定义
流程的方向是清晰的，不会出现步骤之间的循环依赖。

### 为什么重要
- 保证流程的逻辑性
- 便于定位问题
- 提高执行效率

### 常见错误

```
错误示例：
□ 步骤A（依赖步骤C的输出）
□ 步骤B
□ 步骤C（依赖步骤A的输出）

问题：A依赖C，C依赖A，循环依赖
```

```
正确示例：
□ 步骤A
□ 步骤B（依赖A的输出）
□ 步骤C（依赖B的输出）
```

### 应用检查
- [ ] 无循环依赖
- [ ] 步骤顺序清晰
- [ ] 每步的输入输出明确

---

## 原则三：角色责任明确

### 定义
流程中的每个步骤必须有明确的责任人以确保可执行性。

### 为什么重要
- 避免责任推诿
- 便于问题追溯
- 提高执行效率

### 标注方式

| 标注位置 | 格式 | 示例 |
|----------|------|------|
| 步骤下方 | 【责任人】 | □ 填写订单【客服】 |
| 步骤右侧 | /负责人 | □ 审核订单/运营专员 |
| 使用泳道图 | 按角色分泳道 | 市场/运营/技术 |

### 常见错误

```
错误示例：
□ 处理用户投诉
□ 发送补偿
□ 记录处理结果

问题：三个步骤分别是谁负责？
```

```
正确示例：
□ 处理用户投诉【客服】
□ 发送补偿【运营】
□ 记录处理结果【客服】
```

### 应用检查
- [ ] 每个步骤有明确责任人
- [ ] 责任人名称具体（非"团队"）
- [ ] 跨部门接口清晰

---

## 原则四：关键节点可控

### 定义
影响目标达成的关键步骤必须可量化、可优化。

### 为什么重要
- 确保流程可控
- 便于优化迭代
- 有量化标准

### 关键节点分类

| 类型 | 定义 | 示例 | 衡量指标 |
|------|------|------|----------|
| 瓶颈节点 | 拖慢整体流程的步骤 | 人工审核 | 平均等待时长 |
| 放大节点 | 优化后效果最明显的步骤 | 落地页优化 | 转化率提升 |
| 风险节点 | 容易出错的步骤 | 支付环节 | 失败率 |

### 标注方式

```
□ 人工审核【瓶颈】【日均处理量200单，当前等待4小时】
□ 落地页加载【放大】【当前3秒，优化目标1.5秒】
□ 支付确认【风险】【当前失败率5%，目标<2%】
```

### 应用检查
- [ ] 关键节点已标注
- [ ] 有量化指标
- [ ] 优化目标明确

---

## 四原则综合检查清单

### 完整性检查
- [ ] 单一入口：流程有明确的开始和结束
- [ ] 不可逆：步骤顺序清晰，无循环依赖
- [ ] 责任明确：每个步骤有具体责任人
- [ ] 关键可控：关键节点有量化指标

### 复杂度检查
- [ ] 单流程步骤≤7步
- [ ] 超过7步已拆分为子流程
- [ ] 避免不必要的分支

### 实用性检查
- [ ] 可执行：步骤描述是具体动作
- [ ] 可追踪：每个步骤可检查完成状态
- [ ] 可优化：关键节点有优化空间
