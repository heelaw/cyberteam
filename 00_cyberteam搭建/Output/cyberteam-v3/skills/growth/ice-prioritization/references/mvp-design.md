# MVP实验设计方法

## MVP思维核心

**原则**: 用最小成本验证假设，在验证前不要过度投入

### 两个关键问题

**问题1：能不能证明/证伪假设？**
- 如果可以通过数据分析验证，不需要开发
- 这是成本最低的方式

**问题2：能不能开发简化版？**
- 如果不能做完整版本，有没有简化方案
- 简化版能否提供可信结论

---

## MVP验证策略

### 策略1：数据分析验证（成本0）

**适用场景**: 假设可以通过现有数据分析证明/证伪

**案例**:
- 假设：用户不喜欢某个功能入口
- 验证方法：分析该入口的点击率和跳出率
- 如果点击率<5%且跳出率高 → 假设成立

**执行步骤**:
1. 明确假设
2. 设计分析维度
3. 提取数据
4. 得出结论

---

### 策略2：用户调研验证（成本低）

**适用场景**: 需要了解用户心理、需求、障碍

**方法**:
- 问卷调查：量化用户偏好
- 用户访谈：深入了解原因
- 卡片分类：了解用户认知

**案例**:
- 假设：用户需要个性化推荐
- 验证方法：用户访谈了解是否真的需要
- 如果80%用户表示需要 → 假设有数据支持

---

### 策略3：简化版验证（成本中）

**适用场景**: 需要产品功能验证，但可以简化

**方法**:
- 功能简化：只做核心功能
- 用户简化：只针对特定用户群
- 体验简化：只做用户路径的一部分

**案例：个性化注册流程**

**完整版（成本高）**:
- 开发全新的个性化注册流程
- 需要工程、设计、数据资源

**MVP版（成本低）**:
- 选择单一用户群（如25岁女性）
- 只为这个群体做个性化
- 验证后再开发完整版

---

### 策略4：页面测试验证（成本低）

**适用场景**: 验证页面设计、文案、布局

**方法**:
- Lading Page测试：不同页面版本对比
- 文案测试：不同文案方案对比
- 灰度发布：小流量测试

**工具**:
- Optimizely
- VWO
- Google Optimize

---

## MVP设计检查清单

### 必须满足

- [ ] 能够回答假设是否正确
- [ ] 改动量最小
- [ ] 结论可信（不能太简化导致无法判断）

### 尽量满足

- [ ] 用户覆盖面足够
- [ ] 数据收集可行
- [ ] 可以快速迭代

---

## 常见MVP模式

### 模式1：单功能MVP

**原理**: 只做一个功能点验证

**案例**:
- 想做社交功能 → 先只做分享到微信
- 想做推荐系统 → 先只做"猜你喜欢"模块

### 模式2：单用户群MVP

**原理**: 只针对一个细分用户群

**案例**:
- 想做B2B产品 → 先只服务大公司
- 想做个性化 → 先只针对某一类用户

### 模式3：单路径MVP

**原理**: 只打通一个完整用户路径

**案例**:
- 想做电商平台 → 先只做单一品类
- 想做内容平台 → 先只做单一内容形式

### 模式4：时间限制MVP

**原理**: 功能有时间限制或次数限制

**案例**:
- 想做会员体系 → 先做限时试用
- 想做AI功能 → 先限制每日次数

---

## 避免的MVP陷阱

### 陷阱1：过分简化

**问题**: 简化到无法得出结论

**案例**:
- 想验证"社交分享能提升留存"
- MVP版本：只显示分享按钮
- 问题：分享按钮显示但没有引导分享
- 无法验证用户是否真的会分享

**正确做法**: 确保MVP能完整回答假设

---

### 陷阱2：忽视负面结果

**问题**: MVP验证失败但继续投入

**案例**:
- MVP验证：用户不愿意分享
- 应该：放弃或大幅修改
- 错误：继续开发完整版希望改善

---

### 陷阱3：MVP变成永久产品

**问题**: MVP验证成功后直接上线

**案例**:
- MVP验证：用户确实需要某个功能
- 错误：直接上线MVP版本
- 正确：MVP验证成功后继续开发完整版

---

## MVP→完整版路径

```
假设生成 → MVP验证 → 决策点 → 完整版开发 → 正式上线
                    ↓
              验证失败？
              → 放弃或修改假设
```

**关键决策点**:
- MVP验证成功 → 开发完整版
- MVP验证失败 → 放弃或修改假设
- 结果不确定 → 做更多探索
