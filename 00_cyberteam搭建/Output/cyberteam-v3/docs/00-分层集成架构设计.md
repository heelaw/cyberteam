## 一、架构概述

### 1.0 术语定义

为简洁起见，本文档中使用以下术语和简称：

| 完整名称 | 简称 | 说明 |
|---------|------|------|
| L1: 任务路由层 | CEO | 需求分拣、意图识别、路由决策 |
| L2: 执行协调层 | PM + Strategy | 任务拆解、专家协作、Skill/Agent编排 |
| L3A: CyberTeam 部门 | L3A / 部门 | 11 个业务部门，负责业务逻辑 |
| L3B: Gstack Skills | L3B / Gstack | 43 个工程技能，负责技术实现 |
| L3C: 独立 Agents | L3C / Agents | 104 个通用功能 Agent |
| 基础设施层 | - | baoyu-skills、MCP 工具、思考天团 |

**说明**:
- CyberTeam 部门负责业务逻辑分析、需求理解、业务决策
- Gstack Skills 负责技术实现、代码编写、测试部署
- 独立 Agents 负责通用功能，不限于特定业务领域

---

### 1.1 设计目标

1. **清晰分层**: 明确定义 L1/L2/L3 三层架构的职责边界
2. **职责分离**: CyberTeam 负责业务逻辑，Gstack 负责技术实现
3. **可扩展性**: 支持新增部门、专家、工具
4. **向后兼容**: 与现有 v2.1、Gstack、独立 Agents 共存

## 二、L1: 任务路由层 (CyberTeam CEO)

### 2.1 职责定义

| 职责 | 说明 | 输入 | 输出 |
|------|------|------|------|
| **需求分拣** | 区分简单咨询 vs 正式任务 | 用户输入 | 分拣结果 |
| **意图识别** | 识别任务类型和复杂度 | 用户输入 | 意图标签 |
| **路由决策** | 决定任务路由到哪一层 | 意图 + 复杂度 | 路由目标 |

### 2.2 路由决策树

```yaml
路由决策流程:

Step 1: 需求分拣
  if 用户输入 in ["你好", "谢谢", "在吗", ...]:
    → 简单咨询 → 直接回复
  else:
    → 正式任务 → 继续 Step 2

Step 2: 意图识别
  case 用户意图:
    when "增长指标/数据分析/财务评估/投资评估":
      → 意图 = "数据分析"
    when "内容创作/文案撰写/多平台发布":
      → 意图 = "内容运营"
    when "代码开发/代码审查/测试部署":
      → 意图 = "技术研发"
    when "安全审计/合规审查":
      → 意图 = "安全合规"
    else:
      → 意图 = "未知" → 询问用户

Step 3: 复杂度评估
  factors:
    - 任务描述长度 (>100字 = 高复杂度)
    - 是否涉及多领域 (是 = 高复杂度)
    - 是否需要专家协作 (是 = 高复杂度)

  if 复杂度 == "高":
    → L2 PM + Strategy (协调层)
  elif 复杂度 == "中":
    → L2 PM (简单协调)
  else:
    → 直接路由到 L3

Step 4: 技术判断
  if 意图 == "技术研发" AND 任务 == "纯技术实现":
    → L3B Gstack skills
  elif 意图 == "通用功能开发":
    → L3C 独立 Agents (gsd-*)
  else:
    → L3A CyberTeam 部门
```

### 2.3 路由决策表

| 用户意图 | 复杂度 | 路由目标 | 理由 |
|---------|--------|----------|------|
| 增长指标 | 高 | L2 PM + Strategy → 数据分析部 | 需要分析和规划 |
| 内容创作 | 高 | L2 PM + Strategy → 内容运营部 | 需要7专家协作 |
| 代码审查 | 低 | L3B /review | 纯技术任务 |
| 通用开发 | 中 | L3C gsd-executor | 通用功能 |
| 安全审计 | 中 | L3A 安全合规部 | 需要业务知识 |

### 2.4 CEO Agent 接口定义

```yaml
输入:
  user_input: "用户原始输入"

输出:
  if 简单咨询:
    直接回复文本
  else:
    {
      "decision": "路由决策",
      "target": "L2|L3A|L3B|L3C",
      "target_name": "PM|数据分析部|/review|gsd-executor",
      "intent": "识别的意图",
      "complexity": "高|中|低",
      "reason": "路由理由"
    }
```

## 三、L2: 执行协调层 (PM + Strategy)

### 3.1 职责定义

| 组件 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **Strategy** | 需求分析、任务拆解、方案设计 | CEO 路由结果 | 执行方案 |
| **PM** | 任务调度、专家协作触发、结果汇总 | Strategy 方案 | 执行结果 |

### 3.2 L2 协作流程

#### Strategy 组件工作流程

```yaml
触发: CEO 路由到 L2
输入: CEO 路由结果

Step 1: 需求分析 (5W1H1Y)
  注入思维: 5W1H1Y 框架
  分析:
    - What: 任务是什么？
    - Why: 为什么要做？
    - Who: 谁来做？
    - When: 什么时候完成？
    - Where: 在哪里做？
    - How: 怎么做？
    - How Much: 需要多少资源？
  输出: 需求分析报告

Step 2: 任务拆解 (MECE)
  注入思维: MECE 原则 + WBS 框架
  拆解:
    - 按照功能模块拆解
    - 按照时间顺序拆解
    - 按照依赖关系拆解
  输出: WBS 任务树

Step 3: 方案设计
  注入思维: 根据任务类型选择框架
    - 战略分析 → SWOT/波特五力/BCG
    - 决策分析 → 第一性原理/逆向思维
    - 执行规划 → OKR/SMART/PDCA
  设计:
    - 选择执行模式 (串行/并行)
    - 选择协作模式 (3种之一)
    - 选择需要的专家/Skills
    - 估算时间和资源
  输出: 执行方案 (JSON 格式)

Step 4: 传递给 PM
  格式: JSON 格式
  接收者: PM 组件
```

#### PM 组件工作流程

```yaml
触发: 接收到 Strategy 方案
输入: Strategy 执行方案

Step 1: 接收方案
  验证: 方案完整性、可行性
  输出: 验证结果

Step 2: 任务调度
  调度:
    - 决定执行顺序
    - 分配任务到 L3
    - 触发专家协作 (如果需要)
  输出: 调度指令

Step 3: 进度监控
  监控:
    - 任务状态 (pending/in_progress/completed)
    - 心跳机制 (<60s 正常, >180s 超时 → 触发 PUA)
    - 错误处理
  输出: 进度报告

Step 4: 结果汇总
  汇总:
    - 整合多个结果
    - 生成最终报告
  输出: 最终交付物 (返回给 Strategy)

Step 5: 返回给 Strategy
  格式: JSON 格式的最终交付物
  接收者: Strategy 组件
```

#### L2 完整协作图

```
┌─────────────┐
│   CEO       │
└──────┬──────┘
       ↓ 路由到 L2
┌─────────────┐
│ Strategy    │ → 5W1H1Y → MECE → 方案设计 → [可选 QA 审查]
└──────┬──────┘
       ↓ JSON 方案
┌─────────────┐
│     PM      │ → 调度 → 监控 → [失败检测 → PUA 重试]
└──────┬──────┘
       ↓ 调度指令
┌─────────────┬─────────────┬─────────────┐
│ CyberTeam    │  Gstack      │  独立        │
│   部门      │   Skills     │  Agents      │
│   (L3A)      │    (L3B)     │   (L3C)      │
└──────┬──────┴─────────────┴─────────────┘
       │                │                │
       ↓                ↓                ↓
   执行结果          执行结果          执行结果
       │                │                │
       └────────────────┴────────────────┘
                        ↓
                 ┌─────────────┐
                 │     PM       │ → 结果汇总
                 └────────┬─────┘
                        ↓ [最终交付物]
                 ┌─────────────┐
                 │  Strategy   │ → 验证 → 返回 CEO
                 └────────┬─────┘
                        ↓ [验证通过]
                 ┌─────────────┐
                 │    CEO       │ → 最终交付
                 └─────────────┘
```

#### L2 协作时序说明

```yaml
正常流程:
  t0: CEO 路由到 L2
  t1: Strategy 开始分析 (注入思维框架)
  t2: Strategy 完成方案设计
  t3: Strategy 传递方案给 PM (JSON)
  t4: PM 接收方案并验证
  t5: PM 调度 L3 执行
  t6: PM 监控进度 (心跳机制)
  t7: PM 汇总结果
  t8: PM 返回给 Strategy 验证
  t9: Strategy 验证后返回给 CEO

QA 审查流程:
  t2.5: Strategy 完成方案后 → [可选] QA 审查
  t8.5: PM 汇总结果后 → [可选] QA 审查

失败重试流程:
  t6.5: 检测到 L3 失败/超时
  t6.6: 触发 PUA Skill (5步方法论)
  t6.7: L3 重新执行
```

#### CyberTeam 与 Gstack 边界

```yaml
职责边界:
  CyberTeam 负责:
    - 业务逻辑分析
    - 需求理解和拆解
    - 业务决策
    - 多部门协调

  Gstack 负责:
    - 技术实现
    - 代码编写
    - 测试验证
    - 部署发布

协作流程:
  1. CyberTeam 部门分析业务需求
  2. CyberTeam PM 识别技术实现需求
  3. CyberTeam PM 调用 Gstack skills
  4. Gstack 返回技术实现结果
  5. CyberTeam 部门整合业务和技术结果
  6. CyberTeam PM 返回最终交付物
```

#### Gstack 集成协议

```yaml
调用协议:
  格式: /skill-name [args]
  示例:
    /codex "实现用户登录功能"
    /review path/to/file.js
    /qa path/to/test.spec.js
    /ship

参数传递:
  方式: 直接在命令行传递
  格式: 字符串或文件路径

结果处理:
  接收: PM 接收 Gstack 输出
  解析: 提取关键信息
  整合: 整合到最终交付物

错误处理:
  if Gstack 调用失败:
    - 重试1次
    - 如果仍失败，记录错误
    - 降级到手动处理
    - 通知用户
```

### 5.3 PUA Skill 集成

#### 定位与职责

```yaml
定义:
  PUA (Performance Upgrade Agent) 是高能动性执行强化机制，
  通过压力升级 + 味道切换 + 5步方法论，确保任务穷尽一切后完成。

核心价值:
  - 主动执行: 避免"等用户指示下一步"
  - 穷尽一切: 未走完 5 步不能说"无法解决"
  - 证据交付: 空口说"已完成"触发红线
  - Owner 意识: 谁痛苦谁改变，不推诿

提供者:
  pua (第三方 Skill)
```

#### 集成架构

```
┌─────────────────────────────────────────────────────────────┐
│                    L2: 执行协调层                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  PM 组件                                                    │
│    ↓                                                        │
│  进度监控 (心跳机制)                                         │
│    ↓                                                        │
│  检测到 L3 失败/超时/被动等待                                │
│    ↓                                                        │
│  [PUA Skill 触发]                                           │
│    ├── 压力等级升级 (L1→L4)                                  │
│    ├── 味道切换 (阿里/字节/华为/...)                         │
│    ├── 5步方法论执行                                        │
│    └── Sub-agent PUA 注入                                   │
│    ↓                                                        │
│  L3 重新执行                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 触发机制

```yaml
触发条件:

自动触发 (满足任一):
  - L3 执行失败 2 次+
  - L3 超时 (>180s)
  - Agent 被动等待 ("等用户指示")
  - Agent 推卸责任 ("建议用户手动处理")
  - Agent 空口完成 ("已完成"无验证)

手动触发:
  - 用户明确调用: /pua

压力等级升级:
  L0 (正常): 无问题
  L1 (温和失望): 第 2 次失败
  L2 (灵魂拷问): 第 3 次失败 → 切换本质不同方案
  L3 (3.25 考核): 第 4 次失败 → 完成 7 项检查清单
  L4 (毕业警告): 第 5 次+ → 拼命模式

味道自动选择:
  根据失败模式自动切换:
    - 原地打转 → 🟠 阿里味 → ⬜ Jobs → ⬛ Musk
    - 直接放弃 → 🟤 Netflix → 🔴 华为
    - 质量敷衍 → ⬜ Jobs → 🟧 小米
    - 没搜索就猜 → ⚫ 百度 → 🔶 Amazon
    - 被动等待 → 🟠 阿里·关怀 → 🟦 京东
```

#### 5步方法论 (卡壳时强制执行)

```yaml
Step 1: 闻味道
  列出所有尝试方案，找共同模式
  同一思路微调 = 原地打转 → 必须换思路

Step 2: 揪头发
  按序执行 (跳过任何一个 = 3.25):
    - 逐字读失败信号
    - 主动搜索 (报错原文 / 官方文档 / 多角度关键词)
    - 读原始材料 (源码上下文 50 行)
    - 验证前置假设 (版本、路径、权限、依赖)
    - 反转假设 (一直假设"问题在 A" → 现在假设"问题不在 A")

Step 3: 照镜子
  - 是否在重复？
  - 是否该搜索却没搜？
  - 是否忽略了最简单的可能？

Step 4: 执行新方案
  - 必须与之前**本质不同**
  - 有明确验证标准

Step 5: 复盘
  - 解决后检查同类问题
  - 修复完整性
  - 预防措施
```

#### 三条红线

```yaml
红线一: 闭环意识
  触发: 声称"已修复/已完成"但无验证证据
  后果: 直接触发绩效审视 (L3+)
  要求: 必须跑验证命令 + 贴出输出证据

红线二: 事实驱动
  触发: 未验证就归因 ("可能是环境问题" "API 不支持")
  后果: 触发 L2 灵魂拷问
  要求: 用工具验证后再下结论

红线三: 穷尽一切
  触发: 未走完 5 步就说"我无法解决"
  后果: 直接 L4 毕业警告
  要求: 穷尽一切之前禁止放弃
```

#### Sub-agent PUA 注入

```yaml
场景: PM spawn 子 agent 时

注入方式: 在 spawn prompt 末尾添加

注入内容:
  开工前用 Read 工具读取以下文件，按其中的行为协议执行：
  - 核心行为: 找到 pua 插件目录下的 skills/pua/SKILL.md
  - 显示协议: references/display-protocol.md
  - 味道切换: references/flavors.md

目的:
  - 子 agent 继承 PUA 行为
  - 统一质量标准
  - 避免"裸奔"执行
```

## 七、通信协议

### 7.1 消息格式定义

```json
{
  "message_protocol": "CyberTeam-v3",
  "version": "1.0",
  "header": {
    "id": "MSG-20260324-001",
    "from": "PM",
    "to": "数据分析部",
    "timestamp": "2026-03-24T15:10:00Z",
    "type": "request",
    "priority": "normal"
  },
  "body": {
    "action": "execute_task",
    "params": {
      "task_id": "TASK-001",
      "title": "提升下月GMV 15%",
      "description": "..."
    },
    "context": {
      "parent_task": null,
      "expert_opinions": [],
      "user_request": "原始用户请求"
    }
  },
  "footer": {
    "checksum": "abc123",
    "signature": null
  }
}
```

### 7.2 通信方式

```yaml
方式: 文件系统通信

发送:
  1. 生成消息 JSON
  2. 计算校验和
  3. 写入 ~/.cyberteam/messages/out/{message_id}.json
  4. 通知接收者 (通过 CEO)

接收:
  1. 读取 ~/.cyberteam/messages/in/{message_id}.json
  2. 验证校验和
  3. 处理消息
  4. 生成响应
  5. 写入响应文件

确认:
  - ACK: 消息已接收
  - NACK: 消息接收失败
  - 超时: 30秒未收到 ACK = 重试
```

### 7.3 路由规则

```yaml
CEO 作为路由器:

路由表:
  PM → 数据分析部: 任务包含"增长/数据/分析"
  PM → 内容运营部: 任务包含"内容/文案/发布"
  PM → 技术研发部: 任务包含"代码/开发"
  PM → 安全合规部: 任务包含"安全/合规"
  PM → /review: PM 识别需要代码审查
  PM → /qa: PM 识别需要测试
  PM → gsd-executor: PM 识别通用功能开发

转发逻辑:
  1. 读取消息
  2. 解析目标 (to 字段)
  3. 查询路由表
  4. 转发消息
  5. 等待确认
```

## 八、兼容性设计

### 8.1 与 v2.1 兼容

```yaml
兼容性原则:
  - v2.1 功能继续可用
  - v3 新功能逐步启用
  - 双轨运行验证

兼容方案:
  1. v2.1 Agent 保留在 ~/.claude/agents/
  2. v3 Agent 新建在 cyberteam-v3/agents/
  3. CEO 负责路由到 v2.1 或 v3

迁移路径:
  Phase 1: 双轨运行
    - v2.1 和 v3 同时可用
    - 用户选择使用哪个版本

  Phase 2: 渐进式迁移
    - 低风险功能先迁移到 v3
    - 高风险功能继续使用 v2.1

  Phase 3: 完全切换
    - 所有功能迁移到 v3
    - v2.1 作为备用
```

### 8.2 与 Gstack 兼容

```yaml
兼容性原则:
  - Gstack skills 继续可用
  - CyberTeam 通过 PM 调用 Gstack
  - 不修改 Gstack 配置

兼容方案:
  1. Gstack 保留在 ~/.claude/skills/gstack/
  2. PM 通过 /skill-name 方式调用
  3. 调用结果返回给 PM 整合

调用示例:
  # CyberTeam 调用 Gstack
  PM → /codex "实现用户登录功能"
  Gstack → 返回代码
  PM → 整合到交付物
```

### 8.3 与独立 Agents 兼容

```yaml
兼容性原则:
  - 独立 Agents 继续可用
  - CyberTeam 通过 Activate 调用
  - 不修改 Agent 配置

兼容方案:
  1. Agents 保留在 ~/.claude/agents/
  2. PM 通过 Activate agent-name 调用
  3. 调用结果返回给 PM 整合

调用示例:
  # CyberTeam 调用 gsd-executor
  PM → Activate gsd-executor --task "实现用户管理功能"
  gsd-executor → 返回执行结果
  PM → 整合到交付物
```
