# LOOP循环验证机制 (MVP 版本)

> **版本**: v1.0-MVP
> **创建日期**: 2026-03-24
> **状态**: MVP 实现

## 一、MVP 设计原则

### 1.1 核心目标

```yaml
LOOP 目标:
  - 发现问题并不断验证所有细节
  - 启用 LOOP 循环去跑、去验证
  - 最小单元执行与迭代
  - 专家团审核与决策
```

### 1.2 MVP 范围

| 功能 | MVP | 未来 |
|------|-----|------|
| **循环流程** | 执行→检查→修复→再验证 | +自动触发 |
| **检查团队** | PM 兼任 | +并行检查团队 |
| **问题追踪** | 简单记录 | +问题管理系统 |
| **终止条件** | 基础规则 | +智能判断 |

## 二、LOOP 循环流程

### 2.1 基础循环

```
┌─────────────────────────────────────────────────────────────────┐
│                       LOOP 循环验证流程                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  用户需求                                                         │
│    ↓                                                             │
│  ┌─────────────┐                                                  │
│  │ 执行单元     │  → 最小单元执行                              │
│  └──────┬──────┘                                                  │
│         │                                                         │
│         ↓                                                         │
│  ┌─────────────┐                                                  │
│  │ 检查单元     │  → PM 兼任检查 + 快速验证                       │
│  └──────┬──────┘                                                  │
│         │                                                         │
│         ↓                                                         │
│  有问题?                                                         │
│    ├─── 是 → ┌─────────────┐                                      │
│  │        │ 修复单元     │ → 针对问题修复                             │
│  │        └──────┬──────┘                                      │
│  │               │                                              │
│  │               └────────────────────────                      │
│  │                                                          │
│  └─── 否 → ┌─────────────┐                                      │
│           │ 验证单元     │ → 专家团审核                             │
│           └──────┬──────┘                                      │
│                  │                                              │
│                  ↓                                              │
│           ┌─────────────┐                                      │
│           │ 交付单元     │ → 最终交付                              │
│           └─────────────┘                                      │
│                  │                                              │
│                  ↓                                              │
│              用户满意？                                           │
│                ├─── 是 → 完成                                   │
│                └─── 否 → 继续循环                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 循环阶段详解

#### 阶段1: 执行单元 (最小单元执行)

```yaml
目标: 完成最小可验证单元

执行原则:
  - 拆解到最小单元
  - 快速执行
  - 快速验证

执行示例:
  任务: "创作小红书内容"
  最小单元:
    - 单元1: 用户画像分析
    - 单元2: 文案撰写
    - 单元3: 配图生成
    - 单元4: 发布执行
```

#### 阶段2: 检查单元 (PM 兼任检查)

```yaml
检查类型:
  进度检查:
    - 是否按时完成
    - 是否阻塞依赖
    - 是否超时

  质量检查:
    - 是否符合规范
    - 是否有明显错误
    - 是否达到标准

  问题识别:
    - 是否有遗漏
    - 是否有错误
    - 是否有优化空间

检查方式:
  - 自动检查: 格式、规范
  - 人工检查: 质量、创意
  - 快速验证: 核心指标
```

#### 阶段3: 修复单元 (针对问题修复)

```yaml
修复类型:
  格式修复:
    - 文件命名
    - 元数据
    - 排版格式

  内容修复:
    - 错误修正
    - 逻辑优化
    - 内容补充

  技术修复:
    - 工具调用
    - API 调用
    - 系统错误

修复原则:
  - 最小化修复: 只修复必要部分
  - 快速修复: 不拖延执行
  - 验证修复: 修复后立即验证
```

#### 阶段4: 验证单元 (专家团审核)

```yaml
验证类型:
  功能验证:
    - 是否完成目标
    - 是否满足需求
    - 是否可交付

  质量验证:
    - 质量评分是否达标
    - 是否符合标准
    - 是否无风险

  效果验证:
    - 用户满意度
    - 传播效果预期
    - 成本效益分析

审核方式:
  - 自动审核: 基础指标
  - 专家审核: 质量/创意
  - 用户审核: 最终确认
```

## 三、问题追踪系统

### 3.1 问题分类

```yaml
问题类型:
  格式问题:
    - 文件命名错误
    - 元数据缺失
    - 排版不规范

  内容问题:
    - 错误信息
    - 逻辑混乱
    - 内容缺失

  技术问题:
    - 工具调用失败
    - API 调用错误
    - 系统异常

  流程问题:
    - 进度延迟
    - 依赖阻塞
    - 资源不足
```

### 3.2 问题记录

```json
{
  "issue_id": "ISSUE-20260324-001",
  "task_id": "TASK-20260324-001",
  "unit": "文案撰写",

  "issue": {
    "type": "内容问题",
    "level": "P1",
    "description": "标题不够吸引",
    "location": "第1段",
    "suggestion": "使用对比法优化标题"
  },

  "status": "open",
  "assigned_to": "文案撰写专家",
  "priority": "high",

  "timeline": {
    "discovered_at": "2026-03-24T15:20:00Z",
    "assigned_at": "2026-03-24T15:25:00Z",
    "resolved_at": null,
    "resolved_by": null
  },

  "context": {
    "current_title": "AI写作工具介绍",
    "suggested_title": "用AI写作后，我每天多出2小时"
  }
}
```

### 3.3 问题状态

```yaml
状态流转:
  open → in_progress → resolved → verified

  open: 问题已发现
  in_progress: 问题正在处理
  resolved: 问题已解决
  verified: 解决方案已验证

优先级:
  urgent: 立即处理
  high: 优先处理
  normal: 正常处理
  low: 空闲处理
```

## 四、循环终止条件

### 4.1 终止条件

```yaml
成功终止:
  - 用户满意
  - 专家团通过
  - 所有问题解决
  - 质量达标

失败终止:
  - 循环次数 > 最大限制
  - 时间 > 超时限制
  - 成本 > 预算限制
  - 用户明确取消
```

### 4.2 循环限制

```yaml
保护层:
  单次修复: 3 次重试
  整体循环: 10 次
  单元超时: 30 分钟
  整体超时: 2 小时
```

## 五、检查清单

### 5.1 执行前检查

```yaml
检查项:
  - [ ] 需求是否清晰
  - [ ] 目标是否可验证
  - [ ] 资源是否充足
  - [ ] 时间是否合理
  - [ ] 风险是否识别

检查方式:
  - PM 自查
  - Strategy 审核
```

### 5.2 执行中检查

```yaml
检查项:
  - [ ] 进度是否正常
  - [ ] 是否遇到阻塞
  - [ ] 是否需要帮助
  - [ ] 是否需要调整

检查方式:
  - PM 监控
  - 心跳机制
```

### 5.3 执行后检查

```yaml
检查项:
  - [ ] 目标是否达成
  - [ ] 质量是否达标
  - [ ] 用户是否满意
  - [ ] 是否有遗留问题

检查方式:
  - 自动检查: 基础指标
```

## 六、快速验证

### 6.1 验证原则

```yaml
快速验证原则:
  - 验证核心指标
  - 不追求完美
  - 快速迭代
  - 数据驱动
```

### 6.2 验证指标

```yaml
核心指标:
  功能性: 是否完成功能
  质量: 是否达到标准
  时效: 是否按时完成
  成本: 是否在预算内

验证方式:
  - 自动测试: 功能测试
  - 人工审核: 质量测试
  - 用户反馈: 满意度
```

## 七、专家团审核

### 7.1 审核流程

```yaml
审核时机:
  - 单元完成后
  - 修复后验证
  - 最终交付前

审核方式:
  - PM 初审
  - 相关专家复审
  - 用户终审
```

### 7.2 审核标准

```yaml
审核标准:
  目标达成:
    - [ ] 是否达成原定目标
    - [ ] 交付物可验证

  质量标准:
    - [ ] 质量评分 ≥ 80 分
    - [ ] 无 P0/P1 问题
    - [ ] 符合平台规范

  用户体验:
    - [ ] 用户满意度 ≥ 4/5
    - [ ] 无明显体验问题
```

## 八、执行示例

### 8.1 完整 LOOP 示例

```yaml
场景: 创作小红书内容

Loop 1:
  执行: 用户画像分析
  检查: PM 检查 - 画像清晰 ✅
  验证: 通过
  状态: 继续

Loop 2:
  执行: 文案撰写
  检查: PM 检查 - 标题不够吸引 ⚠️
  修复: 优化标题
  验证: 通过
  状态: 继续

Loop 3:
  执行: 配图生成
  检查: PM 检查 - 配图数量不足 ⚠️
  修复: 增加配图
  验证: 通过
  状态: 继续

Loop 4:
  执行: 发布准备
  检查: PM 检查 - 格式正确 ✅
  验证: 专家团审核
  状态: 通过

最终交付
```
