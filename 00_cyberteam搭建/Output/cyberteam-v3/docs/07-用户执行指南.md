## 一、快速开始

### 1.1 系统简介

CyberTeam v3 是一个企业级 AI 协作系统，模拟真实公司运作：

```yaml
核心特点:
  - 三层架构: L1路由 → L2协调 → L3执行
  - 并行协作: 多专家同时工作
  - 自主决策: 减少80%的用户询问
  - 质量保证: QA三阶段模型

能力范围:
  - 数据分析与增长诊断
  - 内容创作与多平台发布
  - 技术开发与代码审查
  - 安全审计与合规审查
  - 团队管理与人力资源
```

### 1.2 基础使用流程

```
┌─────────────────────────────────────────────────────────────┐
│                    使用流程概览                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 描述需求                                                 │
│     ↓                                                      │
│  2. CEO 自动路由 (L1)                                        │
│     ↓                                                      │
│  3. Strategy 规划 + PM 协调 (L2)                            │
```

## 二、任务类型与路由

### 2.1 任务类型分类

| 类型 | 关键词 | 路由目标 | 典型任务 |
|------|--------|----------|----------|
| **数据分析** | 增长、数据、分析、财务、投资、评估 | 数据分析部 | 增长诊断、财务分析、竞品调研 |
| **内容运营** | 内容、文案、文章、发布、小红书、公众号 | 内容运营部 | 文案创作、多平台发布 |
| **技术研发** | 代码、开发、功能、架构 | 技术研发部 + Gstack | 功能开发、代码审查 |
| **安全合规** | 安全、审计、合规、漏洞 | 安全合规部 | 安全扫描、合规审查 |
| **团队管理** | 团队、管理、招聘、绩效 | 人力资源部 | 团队诊断、绩效评估 |
| **通用任务** | 规划、执行、验证 | 独立 Agents | 通用功能开发 |

### 2.2 复杂度分级

```yaml
低复杂度 (直接执行):
  - 描述: 单一任务，明确目标
  - 示例: "审查这段代码"、"生成一张封面图"
  - 处理: CEO → 直接路由到 L3

中复杂度 (简单协调):
  - 描述: 需要一定拆解，但范围明确
  - 示例: "写一篇公众号文章"、"部署这个功能"
  - 处理: CEO → L2 PM → L3

高复杂度 (完整协调):
  - 描述: 多步骤、多部门、需要专家协作
  - 示例: "提升下月GMV 15%"、"品牌升级"
  - 处理: CEO → L2 PM + Strategy → 专家协作 → L3
```

### 2.3 路由决策示例

| 用户输入 | 意图识别 | 复杂度 | 路由决策 |
|---------|---------|--------|----------|
| "你好" | 简单咨询 | - | 直接回复 |
| "审查这段代码" | 技术研发 | 低 | → L3B /review |
| "写一篇小红书" | 内容运营 | 中 | → L2 PM → L3A 内容运营部 |
| "提升下月GMV" | 数据分析 | 高 | → L2 PM + Strategy → L3A 数据分析部 |
| "开发用户登录" | 技术研发 | 高 | → L2 PM + Strategy → L3B Gstack |

## 三、使用模式

### 3.1 模式概览

```yaml
模式1: 直接任务模式
  适用: 单一明确任务
  特点: 快速执行，无需规划
  示例: "审查这段代码"、"生成封面图"

模式2: 标准协作模式
  适用: 中等复杂度任务
  特点: PM 协调，部门执行
  示例: "写一篇文章"、"部署功能"

模式3: 专家协作模式
  适用: 高复杂度、多领域任务
  特点: Strategy 规划 + 专家协作
  示例: "品牌升级"、"增长策略"

模式4: 自助服务模式
  适用: 探索性、研究性任务
  特点: 提供框架，用户主导
  示例: "帮我分析这个市场"
```

### 3.2 直接任务模式

```yaml
使用场景:
  - 代码审查
  - 图像生成
  - 文档格式化
  - 简单查询

用户操作:
  1. 直接描述任务
  2. 系统立即执行
  3. 返回结果

示例:
  用户: "审查 src/auth.js 文件"
  系统:
    - CEO 识别为代码审查
    - 路由到 L3B /review
    - 执行代码审查
    - 返回审查报告
```

### 3.3 标准协作模式

```yaml
使用场景:
  - 内容创作
  - 功能开发
  - 活动策划
  - 常规分析

用户操作:
  1. 描述任务和需求
  2. Strategy 拆解任务
  3. PM 协调执行
  4. 返回完整结果

示例:
  用户: "写一篇关于AI写作的小红书"
  系统:
    - CEO 识别为内容创作
    - 路由到 L2 PM + Strategy
    - Strategy 拆解: 画像 → 文案 → 配图 → 发布
    - PM 调度内容运营部
    - 执行 7 专家协作
    - QA 审核内容
    - 返回完整内容 + 配图
```

### 3.4 专家协作模式

```yaml
使用场景:
  - 品牌升级
  - 增长策略
  - 重大决策
  - 复杂问题

用户操作:
  1. 描述复杂任务
  2. Strategy 深度分析
  3. 触发多专家协作
  4. 专家讨论辩论
  5. 整合综合方案
  6. 返回决策建议

示例:
  用户: "帮我制定下季度增长策略"
  系统:
    - CEO 识别为高复杂度任务
    - 路由到 L2 PM + Strategy
    - Strategy 进行 5W1H1Y + MECE 拆解
    - PM 触发专家协作:
      * 数据分析部: 增长诊断
      * 内容运营部: 内容策略
      * 商务拓展部: 市场机会
    - 专家讨论整合
    - 返回综合增长方案
```

### 3.5 自助服务模式

```yaml
使用场景:
  - 市场研究
  - 竞品分析
  - 行业调研
  - 知识探索

用户操作:
  1. 描述研究主题
  2. 系统提供分析框架
  3. 用户选择方向
  4. 系统执行分析

    - CEO 识别为研究任务
    - 路由到 L2 Strategy

      1. 市场规模分析
      2. 竞品格局分析

      4. 发展趋势分析
    - 用户选择分析维度
    - 执行分析
    - 返回研究报告
```

## 五、完整端到端示例

### 5.1 示例一：完整内容创作流程

```yaml

────────────────────────────────────────────

  What: 推广文章创作

  Who: 目标用户群体

  Where: 公众号 + 小红书
  How: 7专家协作 + baoyu-skills
  How Much: 1篇文章 + 配图 + 多平台发布

任务拆解 (MECE):
  1. 用户画像分析
  2. 产品卖点提炼
  3. 文案创作
  4. 视觉设计
  5. 多平台发布

方案设计:
  协作模式: 模式1 (并行创意风暴)
  执行模式: 并行执行
  预计时间: 30分钟

方案输出 (JSON):

    "mode": "parallel",

      "baoyu-cover-image",
```

### 5.3 LOOP 循环验证

```yaml
功能说明:
  执行 → 检查 → 修复 → 再验证循环

循环阶段:
  阶段1: 执行单元
    - 最小单元执行
    - 快速完成

  阶段2: 检查单元
    - PM 兼任检查
    - 快速验证

  阶段3: 修复单元
    - 针对问题修复
    - 快速迭代

  阶段4: 验证单元
    - 专家团审核
    - 最终确认

终止条件:
  成功: 用户满意、专家通过、问题解决
  失败: 循环次数>10、时间>2小时、用户取消
```

### 5.4 跨部门协作指南

```yaml
协作场景:
  需要多个部门协同完成任务

典型场景:
  - 品牌升级: 内容运营部 + 设计创意部
  - 产品发布: 技术研发部 + 运维部署部 + 内容运营部
  - 安全审计: 技术研发部 + 安全合规部
  - 数据驱动决策: 数据分析部 + 战略规划部

协作流程:

Step 1: 需求分析 (Strategy)
  → 识别涉及的部门
  → 定义部门间的依赖关系
  → 确定协作模式

Step 2: 任务分配 (PM)
  → 拆解任务到各部门
  → 定义输入输出
  → 设置时间节点

Step 3: 并行/串行执行 (各部门)
  → 按依赖关系执行
  → 定期同步进度
  → 处理跨部门问题

Step 4: 结果整合 (PM)
  → 汇总各部门产出
  → 验证整体一致性
  → 生成最终交付物

协作模式矩阵:

┌──────────────────────────────────────────────────────────┐
│ 部门组合          │ 协作模式    │ 典型任务               │
├──────────────────────────────────────────────────────────┤
│ 内容运营 + 设计    │ 串行评审    │ 文案 + 配图            │
│ 技术研发 + 安全    │ 并行执行    │ 开发 + 安全审查        │
│ 数据分析 + 战略    │ 紧密协作    │ 分析 + 决策            │
│ 多部门 (3+)        │ PM 协调     │ 复杂项目              │
└──────────────────────────────────────────────────────────┘

协作最佳实践:

1. 明确接口定义
   - 部门间传递的数据格式
   - 输入输出的质量标准
   - 接口变更的通知机制

2. 建立同步机制
   - 定期进度同步 (每 30 分钟)
   - 关键节点通知
   - 异常及时上报

3. 定义升级路径
   - 部门内无法解决 → 升级到 PM
   - PM 无法协调 → 升级到 CEO
   - CEO 无法决策 → 升级到用户

4. 保持沟通透明
   - 共享任务状态
   - 共享问题和风险
   - 共享决策依据

协作示例:

场景: 开发新功能并发布宣传内容

涉及部门:
  - 技术研发部 (功能开发)
  - 运维部署部 (部署上线)
  - 内容运营部 (宣传内容)
  - 设计创意部 (宣传素材)

协作流程:

  t0: Strategy 规划
    → 识别 4 个部门需要协作
    → 定义依赖关系: 技术研发 → 运维部署 → 内容运营
    → 设计创意部与技术研发部并行

  t1: PM 分配任务
    → 技术研发: 开发功能 (预计 2 小时)
    → 设计创意: 设计素材 (预计 1 小时)
    → 运维部署: 准备部署 (等待功能完成)
    → 内容运营: 准备文案 (等待功能信息)

  t2: 并行执行
    → 技术研发: 开发中...
    → 设计创意: 设计中...

  t3: 技术研发完成
    → 产出: 功能代码 + 文档
    → 通知: 运维部署、内容运营

  t4: 运维部署执行
    → 部署功能
    → 产出: 部署状态 + 访问链接

  t5: 内容运营 + 设计创意协作
    → 整合: 功能 + 素材 + 链接
    → 产出: 宣推内容

  t6: 最终交付
    → 技术成果: 功能上线
    → 宣传成果: 多平台内容发布

协作风险与应对:

风险 1: 部门间沟通不畅
  应对: PM 主动协调，建立共享状态

风险 2: 依赖任务延期
  应对: 预留缓冲时间，准备降级方案

风险 3: 输出质量不一致
  应对: 统一质量标准，QA 终审把关

风险 4: 责任边界模糊
  应对: 明确责任划分，建立争议解决机制
```

## 七、故障排除

### 7.1 常见问题

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| 任务无响应 | 任务阻塞、超时 | 检查心跳、重新调度 |
| 结果不满意 | 需求不清、执行偏差 | 重新描述需求、触发 LOOP |
| 系统报错 | 工具调用失败、API 错误 | 检查错误日志、降级处理 |
| 质量不达标 | 标准不清、执行偏差 | QA 审核反馈、触发修复 |

### 7.2 错误代码

```yaml
ERR-001: 需求不清晰
  解决: 重新描述需求，提供更多上下文

ERR-002: 任务超时
  解决: 检查任务状态，重新调度或降级

ERR-003: 工具调用失败
  解决: 检查工具配置，尝试备用方案

ERR-004: 质量不达标
  解决: 根据 QA 反馈进行修复

ERR-005: 资源不足
  解决: 调整任务范围，分批执行
```

### 7.4 完整错误处理示例

#### 示例 1: 技能调用失败

```yaml
场景: 调用 baoyu-cover-image 生成封面图失败

错误输出:
  {
    "skill": "baoyu-cover-image",
    "status": "error",
    "error": "API_KEY_NOT_FOUND",
    "message": "OpenAI API key not configured"
  }

处理步骤:

Step 1: 检查错误类型
  错误代码: API_KEY_NOT_FOUND
  严重程度: 阻塞性错误

Step 2: 查阅错误代码说明
  → ERR-003: 工具调用失败

Step 3: 尝试自动恢复
  方案 A: 检查配置文件
  方案 B: 使用备用技能

Step 4: 用户通知
  系统提示:
    "封面图生成失败：OpenAI API key 未配置。
     请选择：
     1. 配置 API key 后重试
     2. 使用本地图片作为封面
     3. 跳过封面图，仅发布文字"

Step 5: 用户选择后继续执行
  选项 1: 提供配置指导
  选项 2: 调整执行计划
  选项 3: 继续执行
```

#### 示例 2: L3 执行超时

```yaml
场景: 数据分析部执行任务超过 180 秒

监控输出:
  {
    "task_id": "TASK-001",
    "department": "数据分析部",
    "status": "timeout",
    "duration": 185,
    "heartbeat": "lost"
  }

处理步骤:

Step 1: PM 检测到超时
  心跳机制: 超过 180s 无响应
  触发: 超时告警

Step 2: 尝试恢复
  方案 A: 发送心跳探测
  方案 B: 检查任务日志

Step 3: 触发 PUA Skill
  压力等级: L1 (温和失望)
  操作: 重新调度任务

Step 4: 再次超时
  压力等级升级: L2 (灵魂拷问)
  操作:
    - 切换本质不同的方案
    - 询问用户是否简化需求

Step 5: 用户介入
  系统提示:
    "数据分析任务执行时间过长，可能需要简化分析范围。
     请选择：
     1. 简化需求（仅分析核心指标）
     2. 延长超时时间
     3. 取消任务"

Step 6: 根据用户选择继续
```

#### 示例 3: QA 质量不达标

```yaml
场景: 内容创作完成，QA 审查不通过

QA 输出:
  {
    "task_id": "TASK-002",
    "qa_score": 65,
    "status": "failed",
    "issues": [
      "标题吸引力不足 (评分: 6/10)",
      "配图风格不一致 (评分: 5/10)",
      "缺少行动号召 (评分: 4/10)"
    ]
  }

处理步骤:

Step 1: QA 触发 LOOP 循环
  检查 → 修复 → 验证

Step 2: 内容运营部接收反馈
  分析问题:
    - 标题: 缺乏吸引力
    - 配图: 风格不统一
    - CTA: 缺失

Step 3: 执行修复
  修复 1: 重写标题 (增加吸引力)
  修复 2: 重新生成配图 (统一风格)
  修复 3: 添加行动号召

Step 4: 重新 QA 验证
  输出:
    {
      "qa_score": 88,
      "status": "passed",
      "improvements": [
        "标题吸引力提升 (评分: 9/10)",
        "配图风格统一 (评分: 8/10)",
        "行动号召清晰 (评分: 9/10)"
      ]
    }

Step 5: 交付给用户
  最终交付物包含:
    - 初始版本 + QA 反馈
    - 修复后版本 + QA 验证
    - 改进说明
```

#### 示例 4: 用户需求变更

```yaml
场景: 任务执行中，用户需求发生变更

用户输入:
  "等等，我改主意了。这篇文章我想换个角度写。"

处理步骤:

Step 1: CEO 接收变更请求
  评估: 是否可以中断当前任务

Step 2: PM 检查任务状态
  状态: in_progress (40% 完成)
  决策: 可以中断，保留已完成部分

Step 3: Strategy 重新规划
  新需求: 换角度写文章
  复用: 用户画像、产品卖点 (可复用)
  重做: 文案创作、配图生成 (需要重做)

Step 4: 用户确认
  系统提示:
    "需求变更评估：
     - 已完成部分: 用户画像、产品卖点 (可复用)
     - 需要重做: 文案创作、配图生成
     - 额外时间: 约 10 分钟
     是否确认变更？"

Step 5: 用户确认后执行
  更新任务计划
  复用已完成成果
  执行新需求

Step 6: 交付最终结果
  包含变更说明
```
