# 并行质量检查机制 (MVP 版本)

> **版本**: v1.0-MVP
> **创建日期**: 2026-03-24
> **状态**: MVP 实现

---

## 一、MVP 设计原则

### 1.1 核心改变

| 维度 | 旧流程 | 新流程 (MVP) |
|------|--------|--------------|
| **QA介入时机** | 所有阶段 | 仅交付阶段 |
| **QA标准** | 一刀切 | 分阶段标准 |
| **评审模式** | 单一QA | (未来)多专家并行 |
| **争议解决** | 无明确机制 | (未来)专家会诊 |

### 1.2 MVP 范围

```yaml
MVP 实现:
  - QA 三阶段模型
  - QA 介入时机规则
  - 分阶段评分标准
  - PASS/FAIL 判定

未来迭代:
  - 多专家并行评审
  - 专家会诊流程
  - 自动化程度提升
```

## 二、QA 三阶段模型

### 2.1 阶段划分

```
┌─────────────────────────────────────────────────────────────┐
│                    CyberTeam 工作流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │  方案规划阶段  │ → │  内容创作阶段  │ → │  交付验收阶段  │  │
│  │  (Strategy)  │    │  (Content)   │    │  (Delivery)  │  │
│  └─────┬────┘    └─────┬────┘    └─────┬────┘  │
│         │               │               │         │
│    质量自查              质量自查             QA审查     │
│    (Strategy)           (Content)          (强制介入)   │
│                         (PM监控)                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 阶段规则

| 阶段 | QA介入 | 审查类型 | 说明 |
|------|--------|----------|------|
| **方案规划阶段** | ❌ 不介入 | - | Strategy 部门自查 |
| **内容创作阶段** | ⚠️ 可选介入 | 质量咨询 | 仅主动申请时介入 |
| **交付验收阶段** | ✅ 强制介入 | 质量门控 | 必须通过才能交付 |

## 三、分阶段评分标准

### 3.1 方案阶段标准 (Strategy 自查)

```yaml
负责部门: 战略规划部

自查清单:
  目标定义:
    - [ ] 目标是否符合 SMART 原则
    - [ ] 目标是否可量化验证
    - [ ] 目标是否有时间限制

  任务拆解:
    - [ ] 是否符合 MECE 原则
    - [ ] 是否覆盖所有必要环节
    - [ ] 是否有依赖关系遗漏

  资源评估:
    - [ ] 时间线是否合理
    - [ ] 预算/人力是否明确
    - [ ] 风险识别是否充分

输出要求:
  - 5W1H1Y 分析完整
  - WBS 任务树符合 MECE
  - 每个任务有 Skills 调用
  - 交付物可验证
```

### 3.2 创作阶段标准 (Content 自查 + PM 监控)

```yaml
负责部门: 内容运营部 + 项目管理部

自查清单:
  内容质量:
    - [ ] 内容是否符合选题定位
    - [ ] 内容是否符合调性要求
    - [ ] 格式是否规范

  进度监控:
    - [ ] 心跳机制正常 (<60s)
    - [ ] 无超时任务
    - [ ] 无阻塞依赖

  技术规范:
    - [ ] 文件命名规范
    - [ ] 元数据完整
    - [ ] 版本控制规范

输出要求:
  - 内容符合平台规范
  - 无敏感内容
  - 技术指标达标
```

### 3.3 交付阶段标准 (QA 强制审查)

```yaml
负责部门: 质量审核部

质量门控 (5级评分):

合规性 (30分):
  - [ ] 无法律风险
  - [ ] 无平台违规
  - [ ] 无敏感内容
  - [ ] 广告法合规

准确性 (25分):
  - [ ] 事实准确
  - [ ] 数据可靠
  - [ ] 逻辑严密
  - [ ] 引用准确

传播性 (25分):
  - [ ] 标题吸引
  - [ ] 有情绪价值
  - [ ] 有传播点
  - [ ] 情绪适度

完整性 (20分):
  - [ ] 要素齐全
  - [ ] 结构完整
  - [ ] 体验流畅
  - [ ] 行动引导

总分合格线: ≥ 70 分
单项否决线: 任一维度 < 5 分直接打回
```

## 四、QA 介入时机

### 4.1 介入决策树

```yaml
决策流程:

任务进入
    ↓
任务类型判断
    ↓
  方案规划任务?
    ↓ 是
  Strategy 自查 → 不介入
    ↓ 否
  内容创作任务?
    ↓ 是
  Content 自查 + PM 监控 → 可选介入
    ↓ 否
  交付验收任务?
    ↓ 是
  QA 强制审查 → 必须介入
```

### 4.2 可选介入触发

```yaml
创作阶段可选介入触发条件:
  - Content 部门主动申请
  - PM 识别质量问题
  - 用户明确要求审核

介入方式:
  - 提供质量咨询
  - 给出改进建议
  - 不做强制性要求
```

## 五、PASS/FAIL 判定

### 5.1 判定标准

```yaml
优秀通过 (PASS):
  总分 ≥ 90 分
  所有维度 ≥ 8 分
  无 P0/P1 问题
  → 直接交付

正常通过 (PASS):
  总分 ≥ 80 分
  所有维度 ≥ 7 分
  无 P0 问题，P1 问题 ≤ 2 个
  → 直接交付

有条件通过 (CONDITIONAL):
  总分 ≥ 70 分
  合规性 = 30 分
  无 P0 问题，P1 问题 ≤ 3 个
  → 修复后交付

不通过 (FAIL):
  总分 < 70 分
  或合规性 < 30 分
  或存在 P0 问题
  → 打回修改
```

### 5.2 问题分级

```yaml
P0 问题 (阻断):
  - 法律风险: 虚假宣传、侵权、敏感内容
  - 平台风险: 严重违规，可能被处罚
  - 处理: 必须修复，否则拒绝

P1 问题 (重要):
  - 数据错误: 数据不准确或来源不明
  - 逻辑错误: 逻辑混乱或前后矛盾
  - 传播问题: 标题党、情绪过度
  - 处理: 建议修复，影响质量

P2 问题 (优化):
  - 表达优化: 措辞可以更精准
  - 结构优化: 结构可以更清晰
  - 体验优化: 体验可以更流畅
  - 处理: 可选优化，提升质量
```

## 六、QA 反馈格式

### 6.1 标准反馈格式

```markdown
## QA 反馈报告

**任务ID**: TASK-20260324-001
**审查时间**: 2026-03-24T15:00:00Z
**审查人**: 质量审核部

### 评审结果

| 维度 | 得分 | 状态 |
|------|------|------|
| 目标达成 | 8/10 | ✅ |
| 内容质量 | 6/10 | ⚠️ |
| 交付规范 | 9/10 | ✅ |
| 风险控制 | 7/10 | ✅ |
| 用户体验 | 5/10 | ❌ |

**总分**: 70/100
**结论**: 有条件通过

### 问题清单

#### 🔴 阻断问题 (P0)

无

#### 🟡 重要问题 (P1)

1. **[用户体验]** 交互流程不清晰
   - 位置: 第3章第2节
   - 影响: 用户无法完成核心操作
   - 建议: 增加操作引导

#### 🟢 优化建议 (P2)

2. **[交付规范]** 元数据不完整
   - 建议: 补充 tags 和 category

### 改进期限

请在 **2026-03-24T18:00:00Z** 前完成修复并重新提交。
```

## 七、QA 循环控制

### 7.1 循环限制

```yaml
保护层:
  单次修复: 3 次重试
  L1 检查: 5 次修复循环
  L2 检查: 3 次修复循环
  L3 检查: 2 次修复循环
  全链路: 10 次总循环
```

### 7.2 升级路径

```
QA 审查失败
    ↓
修复 + 重试 (最多 3 次)
    ↓
仍然失败？
    ↓ 是
┌─────────────────────────┐
│   升级决策树            │
├─────────────────────────┤
│ P0/P1 问题 → 升级技术负责人│
│ P2 问题 → 升级 PM 确认    │
│ 争议问题 → 专家会诊      │
│ 超时问题 → 升级 CEO       │
└─────────────────────────┘
```

## 八、测试验证

### 8.1 功能测试

```yaml
测试场景 1: 合格内容
  输入: 符合所有标准的内容
  预期: 直接通过

测试场景 2: 有条件通过
  输入: 有小问题但整体良好的内容
  预期: 有条件通过，给出优化建议

测试场景 3: 不合格内容
  输入: 有严重问题的内容
  预期: 打回修改
```

### 8.2 效果验证

```yaml
验证指标:
  - QA 效率: 平均审查时间 < 30 分钟
  - 一次通过率: ≥ 60%
  - 平均循环次数: ≤ 1.5 次
  - 用户满意度: ≥ 4.5/5

验证方法:
  - 对比测试: 新流程 vs 旧流程
  - 用户反馈: 收集用户意见
  - 数据分析: 分析 QA 数据
```

## 九、使用示例

### 9.1 完整流程示例

```yaml
场景: 内容创作完成，提交 QA 审核

Step 1: 提交审核
  输入: 完整文案内容
  触发: PM 提交到 QA

Step 2: QA 审查
  - 合规性审核: 30/30 分 ✅
  - 准确性审核: 23/25 分 ✅
  - 传播性审核: 20/25 分 ✅
  - 完整性审核: 12/20 分 ⚠️

Step 3: 生成报告
  总分: 85/100
  结论: 有条件通过
  问题: 配图不足、缺少互动引导

Step 4: 反馈给 Content
  Content 接收反馈
  Content 进行优化
  Content 重新提交

Step 5: 二次审查
  - 完整性审核: 18/20 分 ✅
  - 总分: 91/100
  结论: 通过

Step 6: 交付
  QA 批准交付
  PM 交付给用户
```
