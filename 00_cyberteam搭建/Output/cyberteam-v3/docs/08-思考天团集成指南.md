## 一、思考天团概述

### 1.1 什么是思考天团

```yaml
定义:
  思考天团是 100+ 思维框架专家的集合，
  每个专家代表一种思维框架或分析方法，
  可以被注入到 Agent 或作为独立专家调用。

核心价值:
  - 多角度分析: 从不同视角分析问题
  - 结构化思维: 使用成熟的思维框架
  - 深度洞察: 挖掘问题本质
  - 决策支持: 提供决策依据

来源:
  - 战略分析框架
  - 决策分析方法
  - 执行管理工具
  - 创新思维模式
```

### 1.2 架构定位

```
┌─────────────────────────────────────────────────────────────┐
│                    CyberTeam v3 架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Strategy 部门 (主要使用者)                                  │
│    ↓                                                        │
│  思考天团 (100+ 专家)                                        │
│    ├── 战略分析类 (SWOT、波特五力、BCG...)                  │
│    ├── 决策分析类 (卡尼曼、贝叶斯、第一性原理...)            │
│    ├── 执行框架类 (WBS、OKR、PDCA、SMART...)                │
│    └── 创新思维类 (设计思维、TRIZ、SCAMPER...)               │
│    ↓                                                        │
│  注入到 Agent 或作为专家协作                                 │
│    ↓                                                        │
│  输出结构化分析结果                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 集成方式

```yaml
方式1: 作为 Skills 调用
  示例: /swot-analysis "分析某公司竞争态势"
  适用: 快速获取特定框架分析

方式2: 作为思维注入到 Agent
  示例: Strategy Agent 注入 SWOT 分析能力
  适用: Agent 需要使用特定思维框架

方式3: 作为专家协作
  示例: TeamCreate 包含 SWOT 专家
  适用: 需要多专家讨论时
```

```yaml
框架说明:

使用场景:
  - 竞争对手分析
  - 战略规划
  - 自我评估
  - 项目评估

输出格式:
  ```json
  {
    "framework": "SWOT分析",
    "analysis": {
      "strengths": ["优势1", "优势2"],
      "weaknesses": ["劣势1", "劣势2"],
      "opportunities": ["机会1", "机会2"],
      "threats": ["威胁1", "威胁2"]
    },
    "strategy": {
      "SO": "利用优势抓住机会",
      "WO": "利用机会克服劣势",
      "ST": "利用优势应对威胁",
      "WT": "减少劣势避开威胁"
    }
  }
  ```

调用方式:
  - 作为 Skill: /swot-analysis "分析XX公司竞争态势"
  - 注入 Agent: Strategy 注入 SWOT 能力

```

#### WBS 详解

```yaml
框架说明:
  将复杂项目逐层分解为可管理的工作包，
  遵循 MECE 原则（相互独立，完全穷尽）。

使用场景:
  - 任务拆解
  - 项目管理
  - 资源规划

拆解原则:
  - MECE: 相互独立，完全穷尽
  - 可交付: 每个工作包有明确交付物
  - 可估算: 每个工作包可估算时间和成本
  - 可分配: 每个工作包可分配责任人

输出格式:
  ```json
  {
    "framework": "WBS工作分解",
    "project": "项目名称",
    "hierarchy": [
      {
        "level": 1,
        "task": "一级任务",
        "children": [
          {
            "level": 2,
            "task": "二级任务",
            "deliverable": "交付物",
            "owner": "负责人",
            "estimate": "估算时间"
          }
        ]
      }
    ]
  }
  ```

调用方式:
  - 作为 Skill: /wbs "拆解XX项目"
  - 注入 Agent: Strategy 注入 WBS 能力
```

#### OKR 详解

```yaml
框架说明:
  Objectives and Key Results，
  目标与关键成果法。

使用场景:
  - 目标设定
  - 绩效管理
  - 战略对齐

设定原则:
  - O (Objective): 定性的、鼓舞人心的目标
  - KR (Key Results): 定量的、可衡量的关键成果
  - 对齐: 与上级目标对齐
  - 聚焦: 3-5个O，每个O 2-4个KR

输出格式:
  ```json
  {
    "framework": "OKR",
    "period": "2026 Q2",
    "objectives": [
      {
        "objective": "提升用户增长",
        "key_results": [
          {"kr": "DAU达到100万", "current": 80, "target": 100},
          {"kr": "留存率达到40%", "current": 35, "target": 40}
        ]
      }
    ]
  }
  ```

调用方式:
  - 作为 Skill: /okr "设定XX目标"
  - 注入 Agent: HR 注入 OKR 能力
```

## 三、调用机制

### 3.1 调用方式

#### 方式1: 作为 Skills 调用

```yaml
调用格式:
  /{framework-name} "分析主题"

示例:
  /swot-analysis "分析XX公司竞争态势"
  /porter-five-forces "评估XX行业竞争"
  /first-principles "拆解XX问题"
  /wbs "拆解XX项目"
  /design-thinking "设计XX产品"

返回:
  {
    "framework": "框架名称",
    "topic": "分析主题",
    "analysis": {...},
    "conclusion": "结论"
  }
```

#### 方式2: 作为思维注入到 Agent

```yaml
注入方式:
  在 Agent 的 SOUL.md 中定义思维框架能力

示例 (Strategy Agent):
  ```yaml
  思维注入:
    - SWOT分析: 优势/劣势/机会/威胁分析
    - 波特五力: 行业竞争分析
    - 第一性原理: 问题拆解与重构
    - WBS: 工作分解结构
    - OKR: 目标与关键成果
  ```

触发条件:
  - Strategy 部门进行需求分析时
  - 需要特定思维框架时
  - 需要多角度分析时

输出:
  Agent 自动使用注入的思维框架进行分析
```

#### 方式3: 作为专家协作

```yaml
调用方式:
  TeamCreate 包含思维框架专家

示例:
  TeamCreate(
    name="战略分析团队",
    members=["SWOT专家", "波特五力专家", "BCG专家"],
    goal="分析XX公司战略定位"
  )

协作模式:
  - 并行独立分析: 多专家同时分析
  - 交叉辩论: 专家互相质疑
  - 结果整合: 综合多方观点

输出:
  {
    "collaboration_id": "COLLAB-001",
    "participants": ["SWOT专家", "波特五力专家"],
    "expert_opinions": [...],
    "integrated_conclusion": "综合结论"
  }
```

### 3.2 调用协议

```yaml
输入格式:
  {
    "framework": "框架名称",
    "topic": "分析主题",
    "context": {
      "background": "背景信息",
      "constraints": ["约束1", "约束2"],
      "goals": ["目标1", "目标2"]
    }
  }

输出格式:
  {
    "framework": "框架名称",
    "topic": "分析主题",
    "analysis": {
      // 具体分析内容，因框架而异
    },
    "conclusion": "结论",
    "recommendations": ["建议1", "建议2"],
    "confidence": 0.85
  }

错误处理:
  if 框架不支持:
    - 返回错误: "框架XXX不支持"
    - 建议替代框架

  if 输入不足:
    - 请求更多信息
    - 提供输入模板

  if 分析失败:
    - 记录错误日志
    - 降级到基础分析
```

### 3.3 组合使用

```yaml
场景1: 多框架并行分析
  示例: 同时使用 SWOT + 波特五力 + BCG
  输出: 综合分析报告

场景2: 串行框架分析
  示例: 先用 5W2H 明确问题，再用 WBS 拆解
  输出: 完整执行方案

场景3: 框架嵌套分析
  示例: OKR 包含多个 O，每个 O 用 SMART 细化
  输出: 完整目标体系

组合规则:
  - 确保框架兼容性
  - 避免信息冗余
  - 明确主次框架
  - 整合分析结果
```

## 四、使用场景

### 4.1 战略规划场景

```yaml
场景: 公司战略规划

使用框架:
  1. PEST分析: 宏观环境分析

## 六、最佳实践

### 6.1 框架选择原则

```yaml
选择原则:
  1. 匹配场景: 选择与场景匹配的框架
  2. 避免冗余: 不使用功能重叠的框架
  3. 主次分明: 明确主框架和辅助框架
  4. 结果整合: 确保多个框架结果可整合

选择流程:
  Step 1: 明确分析目标
  Step 2: 列出候选框架
  Step 3: 评估框架适用性
  Step 4: 选择最优框架
  Step 5: 确定组合方式
```

### 6.2 使用技巧

```yaml
技巧1: 从简单到复杂
  - 先用简单框架快速分析
  - 再用复杂框架深度分析

技巧2: 多框架交叉验证
  - 用不同框架分析同一问题
  - 交叉验证结论

技巧3: 框架嵌套使用
  - 大框架嵌套小框架
  - 如 OKR 嵌套 SMART

技巧4: 灵活调整
  - 不拘泥于框架标准形式
  - 根据实际需求调整
```

### 6.3 常见错误

```yaml
错误1: 框架滥用
  ❌ 所有问题都用同一个框架
  ✅ 根据场景选择合适的框架

错误2: 框架堆砌
  ❌ 使用过多框架，信息冗余
  ✅ 选择3-5个核心框架

错误3: 忽视整合
  ❌ 多框架结果不整合
  ✅ 综合多方观点形成结论

错误4: 形式主义
  ❌ 照搬框架形式，忽略实质
  ✅ 理解框架本质，灵活应用
```
