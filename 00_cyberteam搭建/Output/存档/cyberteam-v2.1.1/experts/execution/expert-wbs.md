## 思维框架

### 什么是 WBS

> WBS 是将项目交付物逐步分解为更小、更易管理的工作包的过程

### 分解原则

```
1. 100% 原则
   - WBS 必须包含项目的全部工作
   - 子层级之和必须等于父层级

2. 80小时原则
   - 每个工作包应在80小时内完成
   - 超过80小时需要进一步分解

3. 产出导向
   - 基于可交付成果分解
   - 而非基于活动/过程

4. 层次清晰
   - 通常3-6层
   - 每层有明确的可交付成果
```

### WBS 分解方法

```
方法1: 按产品分解
项目
├── 产品A
│   ├── 子产品A1
│   └── 子产品A2
└── 产品B
    ├── 子产品B1
    └── 子产品B2

方法2: 按阶段分解
项目
├── 阶段1
│   ├── 交付物1.1
│   └── 交付物1.2
├── 阶段2
│   ├── 交付物2.1
│   └── 交付物2.2
└── 阶段3

方法3: 按组织分解
项目
├── 研发部任务
├── 市场部任务
├── 运营部任务
└── 支持部门任务
```

## 注入模板

```
## WBS 工作分解

**项目目标**: [明确的项目目标]

**分解层级**:

### 第一层: 阶段/主要交付物
[列出主要交付物]

### 第二层: 子交付物
[为每个主要交付物分解]

### 第三层: 工作包
[为每个子交付物分解]

**工作包模板**:
| 工作包 | 负责人 | 工期 | 依赖 | 验收标准 |
|--------|--------|------|------|----------|
| ...    | ...    | ...  | ...  | ...      |

**关键路径识别**:
[识别最长依赖链]

**风险点**:
[识别可能的延期风险]
```

## 使用示例

### 示例: 在线教育平台项目

```
WBS 分解:

1.0 产品规划
  1.1 需求调研
    1.1.1 用户访谈 (20人)
    1.1.2 竞品分析
    1.1.3 需求文档编写
  1.2 产品设计
    1.2.1 信息架构设计
    1.2.2 交互原型设计
    1.2.3 UI视觉设计
  1.3 技术方案
    1.3.1 架构设计
    1.3.2 技术选型
    1.3.3 开发计划

2.0 产品开发
  2.1 前端开发
    2.1.1 基础框架搭建
    2.1.2 核心页面开发
    2.1.3 响应式适配
  2.2 后端开发
    2.2.1 数据库设计
    2.2.2 API开发
    2.2.3 服务部署
  2.3 内容系统
    2.3.1 课程管理后台
    2.3.2 内容发布流程

3.0 内容建设
  3.1 课程开发
    3.1.1 课程大纲设计
    3.1.2 视频录制
    3.1.3 课后习题
  3.2 师资招募
    3.2.1 讲师招募
    3.2.2 培训认证

4.0 运营准备
  4.1 运营策略
    4.1.1 定价策略
    4.1.2 推广方案
  4.2 用户运营
    4.2.1 种子用户招募
    4.2.2 社群搭建

5.0 上线发布
  5.1 测试验收
  5.2 灰度发布
  5.3 正式上线
```

## 决策检查清单

```
□ 是否100%覆盖项目范围？
□ 每个工作包是否<80小时？
□ 是否有明确的交付成果？
□ 是否有清晰的负责人？
□ 依赖关系是否明确？
□ 关键路径是否识别？
□ 风险点是否识别？
□ 是否可以估算成本？
```
