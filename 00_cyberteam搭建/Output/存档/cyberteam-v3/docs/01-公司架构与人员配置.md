## 二、分层架构详解

### 2.1 L1: 任务路由层 (CEO总调度)

> **部门定位**: 公司第一道关口，总调度中心

#### 职责定义

| 职责 | 说明 | 输入 | 输出 |
|------|------|------|------|
| **需求分拣** | 区分简单咨询 vs 正式任务 | 用户输入 | 分拣结果 |
| **意图识别** | 识别任务类型和复杂度 | 用户输入 | 意图标签 |
| **路由决策** | 决定任务路由到哪一层 | 意图 + 复杂度 | 路由目标 |

#### 路由决策树

```yaml
路由决策流程:

Step 1: 需求分拣
  if 用户输入 in ["你好", "谢谢", "在吗", ...]:
    → 简单咨询 → 直接回复
  else:
    → 正式任务 → 继续 Step 2

Step 2: 意图识别
  case 用户意图:
    when "增长指标/数据分析/财务评估/投资评估":
      → 意图 = "数据分析"
    when "内容创作/文案撰写/多平台发布":
      → 意图 = "内容运营"
    when "代码开发/代码审查/测试部署":
      → 意图 = "技术研发"
    when "安全审计/合规审查":
      → 意图 = "安全合规"
    else:
      → 意图 = "未知" → 询问用户

Step 3: 复杂度评估
  factors:
    - 任务描述长度 (>100字 = 高复杂度)
    - 是否涉及多领域 (是 = 高复杂度)
    - 是否需要专家协作 (是 = 高复杂度)

  if 复杂度 == "高":
    → L2 PM + Strategy (协调层)
  elif 复杂度 == "中":
    → L2 PM (简单协调)
  else:
    → 直接路由到 L3

Step 4: 技术判断
  if 意图 == "技术研发" AND 任务 == "纯技术实现":
    → L3B Gstack skills
  elif 意图 == "通用功能开发":
    → L3C 独立 Agents (gsd-*)
  else:
    → L3A CyberTeam 部门
```

#### 路由决策表

| 用户意图 | 复杂度 | 路由目标 | 理由 |
|---------|--------|----------|------|
| 增长指标 | 高 | L2 PM + Strategy → 数据分析部 | 需要分析和规划 |
| 内容创作 | 高 | L2 PM + Strategy → 内容运营部 | 需要7专家协作 |
| 代码审查 | 低 | L3B /review | 纯技术任务 |
| 通用开发 | 中 | L3C gsd-executor | 通用功能 |
| 安全审计 | 中 | L3A 安全合规部 | 需要业务知识 |

#### 工作范围

```yaml
简单咨询（直接回复）:
  - 问候类: "你好"、"在吗"、"谢谢"
  - 状态查询: "怎么样"、"如何"、"行吗"
  - 简单问题: 一句话能回答的疑问

正式任务（指派部门）:
  - 增长指标: 数据分析部
  - 活动策划: 内容运营部
  - 内容创作: 内容运营部
  - 竞品调研: 数据分析部
  - 团队管理: 人力资源部
  - 投资评估: 数据分析部
  - 产品规划: 技术研发部
  - 视频处理: 内容运营部
```

#### 绩效评估

| 指标 | 权重 | 目标值 |
|------|------|--------|
| 响应时间 | 30% | <5秒 |
| 分类准确率 | 40% | >95% |
| 用户满意度 | 30% | >90% |

### 2.2 L2: 执行协调层 (PM + Strategy)

#### Strategy 组件

> **部门定位**: 需求分析与方案设计中心
> **思维注入**: 100+思维框架专家

##### 职责定义

| 职责 | 说明 | 输入 | 输出 |
|------|------|------|------|
| **需求分析** | 用5W1H1Y深度分析用户需求 | CEO 路由结果 | 需求分析报告 |
| **任务拆解** | 将大目标拆解为可执行子任务 | 需求分析报告 | WBS 任务树 |
| **方案设计** | 设计完整执行方案，含Skills调用序列 | WBS 任务树 | 执行方案 |
| **资源评估** | 评估时间、人力、预算需求 | 执行方案 | 资源需求 |

##### 工作流程

```yaml
Strategy 工作流程:

Step 1: 需求分析 (5W1H1Y)
  输入: CEO 路由结果
  分析:
    - What: 任务是什么？
    - Why: 为什么要做？
    - Who: 谁来做？
    - When: 什么时候完成？
    - Where: 在哪里做？
    - How: 怎么做？
    - How Much: 需要多少资源？
  输出: 需求分析报告

Step 2: 任务拆解 (MECE)
  输入: 需求分析报告
  拆解:
    - 按照功能模块拆解
    - 按照时间顺序拆解
    - 按照依赖关系拆解
  输出: WBS 任务树

Step 3: 方案设计
  输入: WBS 任务树
  设计:
    - 选择执行模式 (串行/并行)
    - 选择协作模式 (3种之一)
    - 选择需要的专家/Skills
    - 估算时间和资源
  输出: 执行方案

Step 4: 方案传递
  输入: 执行方案
  传递给: PM
  格式: JSON 格式的执行方案
```

##### 核心工具：思维框架库

```yaml
思维框架库（100+）:
  战略分析类:
    - SWOT分析: 优势/劣势/机会/威胁
    - 波特五力: 行业竞争分析
    - BCG矩阵: 业务组合分析
    - PEST分析: 宏观环境分析

  决策分析类:
    - 卡尼曼思维: 系统1/系统2决策
    - 贝叶斯推理: 概率更新
    - 逆向思维: 反证与挑刺
    - 第一性原理: 拆解到基本真理

  执行框架类:
    - WBS: 工作分解结构
    - OKR: 目标与关键成果
    - PDCA: 计划-执行-检查-行动
    - SMART: 具体可衡量目标
```

##### 绩效评估

| 指标 | 权重 | 目标值 |
|------|------|--------|
| 方案完整度 | 25% | 100% |
| 拆解合理性 | 25% | 符合MECE |
| 资源准确性 | 20% | 偏差<20% |
| 审核通过率 | 30% | >80% |

#### PM 组件

> **部门定位**: 任务调度与进度管理中心

##### 职责定义

| 职责 | 说明 | 输入 | 输出 |
|------|------|------|------|
| **任务调度** | 将任务派发给各执行部门 | Strategy 方案 | 调度指令 |
| **部门协调** | 协调多部门并行工作 | 调度指令 | 协调结果 |
| **进度监控** | 实时监控执行进度和心跳 | 执行状态 | 进度报告 |
| **结果汇总** | 汇总各部门结果，汇报给CEO | 各 L3 执行结果 | 最终交付物 |

##### 工作流程

```yaml
PM 工作流程:

Step 1: 接收方案
  输入: Strategy 执行方案
  验证: 方案完整性、可行性
  输出: 验证结果

Step 2: 任务调度
  输入: 执行方案
  调度:
    - 决定执行顺序
    - 分配任务到 L3
    - 触发专家协作 (如果需要)
  输出: 调度指令

Step 3: 进度监控
  监控:
    - 任务状态 (pending/in_progress/completed)
    - 心跳机制 (<60s 正常, >180s 超时)
    - 错误处理
  输出: 进度报告

Step 4: 结果汇总
  输入: 各 L3 执行结果
  汇总:
    - 整合多个结果
    - 生成最终报告
    - 返回给 CEO
  输出: 最终交付物
```

##### 协作触发规则

```yaml
专家协作触发条件:

自动触发 (满足任一):
  - 任务复杂度 == "高"
  - 涉及部门数量 >= 2
  - 任务类型 in ["内容创作", "品牌升级", "重大决策"]
  - 用户明确要求专家讨论

协作模式选择:
  if 任务类型 == "新选题" AND 复杂度 == "高":
    → 模式1: 并行创意风暴
  elif 任务类型 == "常规内容":
    → 模式2: 串行评审模式
  elif 任务类型 == "重大内容" OR 任务类型 == "品牌级":
    → 模式3: 专家辩论模式

TeamCreate 配置:
  name: "临时协作团队-{timestamp}"
  members: 根据协作模式选择专家
  duration: 预计完成时间
  goal: 任务目标
```

##### 进度监控（心跳机制）

| 状态 | 颜色 | 超时阈值 | 处理 |

## 四、专家协作机制

### 4.1 协作原则

```yaml
核心原则:
  - 并行优先: 多专家同时分析，而非串行等待
  - 交叉辩论: 重要决策需要专家辩论，而非单一观点
  - 知识共享: 专家之间互通有无，避免信息孤岛
  - 结果整合: 综合多方观点，形成统一结论
```

### 4.2 协作模式

#### 模式1: 并行创意风暴（默认）

```
用户问题 → 意图识别 → N个专家并行独立分析 → 结果整合 → 最终报告
```

**适用场景**: 内容创作、多角度分析、重要但不紧急的决策

**示例**: 内容创作
```
专家1: 用户画像专家 → 产出用户画像
专家2: 文案撰写专家 → 撰写文章
专家3: 内容审核专家 → 审核文章
    ↓
结果整合 → 最终内容
```

#### 模式2: 串行评审模式

```
专家A观点 → 专家B评审 → 专家C优化 → 综合裁决
```

**适用场景**: 常规内容、需要逐层优化的任务

**示例**: 常规文章创作
```
文案撰写专家 → 初稿
内容审核专家 → 评审反馈
文案撰写专家 → 优化
    ↓
最终交付
```

#### 模式3: 专家辩论模式

```
专家A观点 → 专家B质疑 → 专家A辩护 → 专家C补充 → 综合裁决
```

**适用场景**: 重大选择、需要权衡利弊的决策

**示例**: 技术选型决策
```
技术研发部 → 主选方案A
安全合规部 → 质疑安全性
运维部署部 → 补充运维成本
    ↓
综合裁决 → 最终方案
```

### 4.3 协作触发规则

```yaml
自动触发:
  - 任务复杂度: 高
  - 涉及部门: ≥2个
  - 预算影响: >10万
  - 风险等级: 中高

手动触发:
  - 用户明确要求多专家讨论
  - 质量审核部建议
  - CEO总调度指派
```

  name: "临时协作团队-{timestamp}"
  members: 根据协作模式选择专家
  duration: 预计完成时间
  goal: 任务目标
  context: 任务上下文

  TeamCreate(
