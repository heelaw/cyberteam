## 一、核心问题诊断

### 1.1 现有QA流程的问题

**问题1: QA介入时机错误**
- ❌ 当前：QA在方案阶段就介入审查
- ✅ 应该：QA在内容创作完成后进行审查

**问题2: 所有阶段都要求QA**
- ❌ 当前：每个阶段都要QA审批
- ✅ 应该：仅关键节点需要QA把关

**问题3: QA标准一刀切**
- ❌ 当前：方案阶段和交付阶段用同一标准
- ✅ 应该：不同阶段有不同标准

## 二、新QA流程设计

### 2.1 QA介入时机（三阶段模型）

```
┌─────────────────────────────────────────────────────────────┐
│                    CyberTeam 工作流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │  方案规划阶段  │ → │  内容创作阶段  │ → │  交付验收阶段  │  │
│  │  (Strategy)  │    │  (Content)   │    │  (Delivery)  │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│         ↓                    ↓                    ↓         │
│    质量自查              质量自查             QA审查        │
│    (Strategy)           (Content)          (强制介入)       │
│                         (PM监控)                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 QA介入时机规则

| 阶段 | QA介入 | 审查类型 | 说明 |
|------|--------|----------|------|
| **方案规划阶段** | ❌ 不介入 | - | Strategy部门自查 |
| **内容创作阶段** | ⚠️ 可选介入 | 质量咨询 | 仅主动申请时介入 |
| **交付验收阶段** | ✅ 强制介入 | 质量门控 | 必须通过才能交付 |

## 三、不同阶段的QA标准

### 3.1 方案阶段标准（Strategy自查）

**负责部门**: 战略规划部

**自查清单**:
```yaml
方案阶段自查:
  目标定义:
    - 目标是否符合SMART原则
    - 目标是否可量化验证
  任务拆解:
    - 是否符合MECE原则
    - 是否覆盖所有必要环节
  资源评估:
    - 时间线是否合理
    - 预算/人力是否明确
  风险识别:
    - 是否识别主要风险
    - 是否有应急预案
```

**输出要求**:
- 5W1H1Y分析完整
- 任务拆解符合MECE
- 每个任务有Skills调用
- 交付物可验证

### 3.2 创作阶段标准（Content自查+PM监控）

**负责部门**: 内容运营部 + 项目管理部

**自查清单**:
```yaml
创作阶段自查:
  内容质量:
    - 内容是否符合选题定位
    - 内容是否符合调性要求
    - 格式是否规范
  进度监控:
    - 心跳机制正常(<60s)
    - 无超时任务
    - 无阻塞依赖
  技术规范:
    - 文件命名规范
    - 元数据完整
    - 版本控制规范
```

**输出要求**:
- 内容符合平台规范
- 无敏感内容
- 技术指标达标

### 3.3 交付阶段标准（QA强制审查）

**负责部门**: 质量审核部

**质量门控（5级评分）**:

| 维度 | 权重 | 评分标准 | 不合格阈值 |
|------|------|----------|-----------|
| **目标达成** | 25% | 是否达成原定目标 | <6分 |
| **内容质量** | 25% | 内容专业度/可读性 | <6分 |
| **交付规范** | 20% | 格式/元数据/版本 | <5分 |
| **风险控制** | 15% | 无重大风险遗留 | <5分 |
| **用户体验** | 15% | 用户可用性 | <6分 |

**总分合格线**: ≥70分
**单项否决线**: 任一维度 <5分直接打回

## 四、多专家并行评审机制

### 4.1 并行评审架构

```
┌─────────────────────────────────────────────────────────────┐
│                   多专家并行评审系统                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│     ┌──────────┐    ┌──────────┐    ┌──────────┐          │
│     │  内容专家  │    │  技术专家  │    │  安全专家  │          │
│     │ (Content) │    │(Technical)│    │(Security) │          │
│     └─────┬────┘    └─────┬────┘    └─────┬────┘          │
│           │               │               │                │
│           └───────────────┼───────────────┘                │
│                          ↓                                │
│                   ┌──────────────┐                        │
│                   │  QA整合器     │                        │
│                   │ (Integrator) │                        │
│                   └──────┬───────┘                        │
│                          ↓                                │
│                   ┌──────────────┐                        │
│                   │  最终报告     │                        │
│                   └──────────────┘                        │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 专家分工

| 专家 | 审查维度 | 标准 |
|------|----------|------|
| **内容专家** | 专业性/可读性/逻辑性 | 内容质量评分 |
| **技术专家** | 格式/元数据/版本 | 技术规范评分 |
| **安全专家** | 敏感内容/合规性 | 风险控制评分 |

### 4.3 并行评审流程

```python
class ParallelReview:
    """并行评审系统"""

    EXPERTS = {
        "content": ContentExpert,
        "technical": TechnicalExpert,
        "security": SecurityExpert,
    }

    TIMEOUT = 300  # 5分钟

    async def review(self, artifact):
        """并行评审"""

        # 并行调用所有专家
        results = await asyncio.gather(
            *[
                expert.review(artifact)
                for expert in self.EXPERTS.values()
            ],
            timeout=self.TIMEOUT
        )

        # 整合结果
        return self.integrate(results)

    def integrate(self, results):
        """整合专家意见"""
        return {
            "overall_score": self._calc_score(results),
            "expert_opinions": results,
            "blockers": self._find_blockers(results),
            "suggestions": self._merge_suggestions(results),
        }
```

## 五、专家会诊流程（争议解决）

### 5.1 争议触发条件

| 条件 | 阈值 | 处理方式 |
|------|------|----------|
| 专家评分差异 | >3分 | 触发会诊 |
| 存在P0争议 | 任意 | 立即会诊 |
| 总分处于边界 | 68-72分 | 会诊确认 |

### 5.2 会诊流程

```
争议检测
    ↓
┌─────────────────────────────────┐
│  专家会诊室 (Consultation Room)  │
├─────────────────────────────────┤
│  1. 陈述观点 (2分钟/专家)        │
│  2. 交叉辩论 (5分钟)             │
│  3. 寻求证据 (3分钟)             │
│  4. 投票表决 (1分钟)             │
│  5. 生成共识报告                 │
└─────────────────────────────────┘
    ↓
共识决策 或 CEO仲裁
```

### 5.3 会诊决策规则

```yaml
会诊决策规则:
  一致通过:
    - 所有无反对意见
    - 直接通过

  多数通过:
    - ≥2/3专家同意
    - 记录少数意见
    - 通过

  争议未决:
    - 无法达成共识
    - 升级CEO仲裁
    - CEO决策为最终
```

## 六、质量门禁机制

### 6.1 三级质量门控

| 门控级别 | 检查内容 | 阈值 | 工具 |
|---------|---------|------|------|
| **L1 基础** | 格式/元数据/Schema | 100% | 自动检查 |
| **L2 内容** | 专业度/可读性/逻辑 | ≥80分 | 专家评审 |
| **L3 风险** | 安全/合规/敏感内容 | 0缺陷 | 安全专家 |

### 6.2 PASS/FAIL判定

```python
class QualityGate:
    """质量门控"""

    L1_THRESHOLD = 1.0    # 100% 通过
    L2_THRESHOLD = 0.8    # 80分
    L3_THRESHOLD = 0      # 0缺陷

    def evaluate(self, artifact):
        """评估是否通过"""

        # L1: 基础检查
        l1_result = self.run_l1_checks(artifact)
        if not l1_result.passed:
            return GateResult(
                pass=False,
                level=1,
                reason=l1_result.errors
            )

        # L2: 内容检查
        l2_result = self.run_l2_checks(artifact)
        if l2_result.score < self.L2_THRESHOLD:
            return GateResult(
                pass=False,
                level=2,
                reason=f"内容评分{ l2_result.score } < 80"
            )

        # L3: 风险检查
        l3_result = self.run_l3_checks(artifact)
        if l3_result.blockers > 0:
            return GateResult(
                pass=False,
                level=3,
                reason=f"发现{ l3_result.blockers }个风险"
            )

        return GateResult(
            pass=True,
            scores={
                "l1": 1.0,
                "l2": l2_result.score,
                "l3": 0
            }
        )
```

### 6.3 门禁放行标准

```yaml
放行标准:
  必须满足:
    - L1基础检查: 100%通过
    - L2内容评分: ≥80分
    - L3风险检查: 0个P0/P1风险
    - 无单项否决

  有条件放行:
    - L2内容评分: 70-79分
    - P2风险: ≤3个
    - 需PM确认
    - 需记录改进计划

  不予放行:
    - L1检查失败
    - L2内容评分: <70分
    - 存在P0/P1风险
```

## 八、QA循环控制机制

### 8.1 循环限制

| 保护层 | 限制 | 动作 |
|--------|------|------|
| 单次修复 | 3次重试 | 第3次失败 → 升级 |
| L1检查 | 5次修复循环 | 强制升级 |
| L2检查 | 3次修复循环 | 强制升级 |
| L3检查 | 2次修复循环 | 强制升级 |
| 全链路 | 10次总循环 | 强制升级人工 |

### 8.2 超时保护

```yaml
超时配置:
  l1_check_timeout: 30秒
  l2_check_timeout: 300秒(5分钟)
  l3_check_timeout: 600秒(10分钟)
  fix_timeout: 120秒/次
  total_loop_timeout: 1800秒(30分钟)
```

### 8.3 升级路径

```
QA审查失败
    ↓
修复 + 重试 (最多3次)
    ↓
仍然失败？
    ↓ 是
┌─────────────────────────┐
│   升级决策树            │
├─────────────────────────┤
│ P0/P1问题 → 升级技术负责人│
│ P2问题 → 升级PM确认      │
│ 争议问题 → 专家会诊      │
│ 超时问题 → 升级CEO       │
└─────────────────────────┘
```
