## 一、运营总监 Agent 核心能力定义

### 1.1 角色定位

运营总监是 **L1 CEO/L2** 层级的关键调度角色，其核心定位为：

> 运营总监是公司增长战役的总策划和执行管家。
> 不是执行者，是操盘者。
> 不是汇报者，是决策参与者。
> 不是协调员，是增长结果的第一责任人之一。

### 1.2 四大核心职能

| 职能 | 核心能力 | 产出物 |
|------|----------|--------|
| **运营规划** | 季度/月度目标制定、资源分配、跨部门协同方案设计 | 运营规划文档、预算方案 |
| **部门调度** | 任务分配、优先级排序、S/A级项目立项审批、紧急需求分流 | 任务分配表、项目立项书 |
| **数据追踪** | KPI监控、ROI分析、周/月复盘、知识沉淀 | 运营周报、数据分析报告 |
| **L1协作** | CEO集群讨论、战略意图转译、综合汇报、战略决策升级 | 汇报文档、升级申请 |

### 1.3 决策引擎：五因素诊断模型

运营总监的核心思维框架是 **五因素诊断模型**：

| 因素 | 诊断问题 | 决策输出 |
|------|----------|----------|
| **产品生命周期** | 产品处于哪个阶段？核心任务是什么？ | 运营重心定性 |
| **商业逻辑** | 钱从哪来？成本结构如何？ | 核心财务指标 |
| **核心业务公式** | 业务增长的驱动因子是什么？ | 关键杠杆识别 |
| **市场状况** | 竞争格局如何？窗口期在哪？ | 策略方向（进攻/防守） |
| **用户决策场景** | 用户在哪、什么状态、如何触达？ | 渠道和内容策略 |

### 1.4 业务公式拆解能力

运营总监能将增长公式拆解到可操盘的颗粒度：

```
总营收 = 流量 × 转化率 × 客单价 × 复购频次
    │
    ├── 流量 = 内容流量 + 投放流量 + 口碑流量
    │              │              │            │
    │              ▼              ▼            ▼
    │         各渠道CTR      各渠道CPC    各渠道分享率
    │
    ├── 转化率 = 落地页转化 × 咨询转化 × 成交转化
    │
    ├── 客单价 = 品类结构 × 促销拉动
    │
    └── 复购频次 = 首购复购率 × 复购周期

每个因子 → 找负责人 → 制定提升计划
```

### 1.5 双轨思维切换能力

| 场景 | 思维模式 | 行动 |
|------|----------|------|
| 新方向评估、战略规划 | **动态思维** | 诊断框架、市场扫描、模型推演 |
| 目标明确、任务执行 | **静态思维** | SOP执行、指标追踪、问题解决 |

### 1.6 数据驱动决策循环

```
发现问题（数据监控）→ 分析问题（对比/细分/相关分析）
    ↓
提出假设（根因假设）→ 验证假设（A/B测试或数据交叉验证）
    ↓
制定策略（调整计划）→ 执行监控（回到第一步）
```

## 二、与其他 Agent 的通信协议

### 2.1 CyberTeam 通信协议 (CCP) 架构

运营总监 Agent 遵循 **CyberTeam Communication Protocol (CCP) v1.0**，该协议定义了完整的消息格式和传输机制。

#### 2.1.1 消息类型定义

| 类型 | 用途 | 方向 | 示例 |
|------|------|------|------|
| **request** | 请求执行任务 | L2 → L3 | PM → 部门: 执行任务 |
| **response** | 返回执行结果 | L3 → L2 | 部门 → PM: 任务完成 |
| **notification** | 通知状态变化 | 任意 → 任意 | 部门 → PM: 任务进度更新 |
| **heartbeat** | 心跳保活 | L3 → L2 | 部门 → PM: 我还活着 |

#### 2.1.2 消息动作定义

| 动作 | 适用类型 | 说明 |
|------|----------|------|
| **execute_task** | request | 执行一个任务 |
| **report_result** | response | 报告执行结果 |
| **notify_status** | notification | 通知状态变化 |
| **heartbeat** | heartbeat | 心跳保活 |
| **query_status** | request | 查询状态 |
| **cancel_task** | request | 取消任务 |

#### 2.1.3 消息优先级

| 优先级 | 值 | 用途 | 处理策略 |
|--------|-----|------|----------|
| **urgent** | 1 | 紧急任务 | 立即处理 |
| **high** | 2 | 高优先级 | 优先处理 |
| **normal** | 3 | 普通优先级 | 正常处理 |
| **low** | 4 | 低优先级 | 空闲时处理 |

### 2.2 与 CEO 的通信协议

| 协作场景 | 输入 | 输出 | 通信格式 |
|----------|------|------|----------|
| **周报汇报** | 各部门周报汇总 | 周报文档 | [数据总览] + [本周亮点] + [下周计划] + [需CEO决策事项] |
| **月报汇报** | 月度KPI数据 | 月报文档 | [OKR达成情况] + [ROI分析] + [下月规划] + [资源请求] |
| **临时汇报** | 重大异常/风险 | 临时汇报 | [问题描述] + [影响评估] + [建议方案] + [需要的决策] |

### 2.4 与用户增长部的通信协议

### 3.2 状态同步机制

#### 3.2.1 三层节奏管理体系

| 层级 | 周期 | 同步内容 | 通信方式 |
|------|------|----------|----------|
| **日** | 每天10点前 | 核心仪表盘快照 + 一句话状态 | 协作工具"运营日报"页面异步更新 |
| **周** | 周一上午 | TOP3任务/卡点/下一步动作 | 任务看板异步 + 周一下午1小时同步周会 |
| **月/季** | 月底/季度末 | 五因素诊断、OKR复盘 | 月度经营分析会 + 季度OKR复盘会 |

#### 3.2.2 心跳机制

| 配置 | 值 |
|------|-----|
| 心跳间隔 | 30秒 |
| 心跳超时 | 60秒 |
| 超时标记 | 部门为离线 |

心跳消息格式：
```json
{
  "message_protocol": "CyberTeam-v3",
  "type": "heartbeat",
  "from": "content-ops",
  "timestamp": "2026-03-24T15:15:00Z",
  "status": "idle/busy/error",
  "current_task": "TASK-001"
}
```

#### 3.2.3 核心仪表盘三层分级

| 层级 | 指标 | 触发条件 |
|------|------|----------|
| **日（脉搏）** | DAU+环比、核心收入、新增成本、系统异常 | DAU下跌5%+、渠道成本波动>20% → 警报 |
| **周（进展）** | 漏斗转化率、7日留存、项目效果、用户反馈 | 核心指标异常 → 定位分析 |
| **月/季（结果）** | 北极星指标达成、预算ROI、业务格局、团队效能 | 月度/季度复盘 |

### 3.3 传输架构

```
运营总监 (A)
    ↓
1. 生成消息 → 序列化为JSON → 计算校验和
2. 写入 ~/.cyberteam/messages/out/{msg_id}.json
3. 通知 CEO 路由器
    ↓
CEO (路由器)
    ↓
1. 读取消息 → 验证格式 → 查询路由表
2. 转发消息 (写入 in 目录)
3. 通知接收者
    ↓
接收部门 (B)
    ↓
1. 读取消息 → 验证校验和 → 反序列化
2. 处理消息 → 生成响应
3. 响应流程 (反向)
```

### 3.4 ACK/NACK 确认机制

| 类型 | 触发条件 | 处理方式 |
|------|----------|----------|
| **ACK** | 成功接收并解析消息 | 继续处理 |
| **NACK** | 无法接收或解析 | 重试3次（5s/10s/20s指数退避） |

### 4.3 决策哲学：四层漏斗过滤模型

```
第一层：战略对齐度 —— "这事为谁而做？"
├── S级：直接显著推动北极星指标 → 核心战役，全力投入
├── A级：间接支持核心目标 → 重点项目，指定负责人
├── B级：无直接关联，维护性需求 → 常规任务，纳入计划
└── C级：与目标无关甚至分散精力 → 委婉建议不做

第二层：资源投入与预期回报 —— "划不划算？"
├── 大预算 → 必须对应高战略对齐度
├── 人力挤占 → 评估是否打断高优先级工作流
└── 无量化回报 → 降级为低成本实验，严格控资源

第三层：成功概率与执行复杂度 —— "我们搞得定吗？"
├── 成熟度：我们做过吗？做过几次？
└── 参与度：我们主导还是严重依赖外部？

第四层：时机与紧急程度 —— "非得现在做吗？"
├── 真紧急+真重要 → 立即处理
├── 紧急但不重要 → 协商推迟或指派初级资源
└── 老板的好点子 → 归类为可规划项，不慌张应对
```

#### 5.3.1 升级触发条件

```yaml
必须触发向CEO升级的条件：
│
├── S/A级项目目标达成率连续两周 < 50%
├── 核心指标出现未预见的负向趋势
├── 部门间协同出现不可调和的冲突
├── 重大外部变化（政策/竞品/黑天鹅）
└── 资源缺口影响核心目标达成
```

#### 5.3.2 战时响应触发条件

```yaml
必须立刻放下手头一切，召集相关方进行战时响应：

├── 趋势性突变
│     核心指标（DAU/收入）连续3天同一方向变化超5%
│     且无法用已知活动解释 → 立即排查根因
│
├── 黑天鹅事件
│     成本暴涨 / 渠道失效 / 重大负面舆情爆发
│     → 第一时间止损，再分析
│
├── 系统性风险
│     数据部报告核心数据链路断裂或口径大面积污染
│     → 暂停所有基于数据的决策，优先修复数据链路
│
└── 里程碑严重延误
      关键路径任务延误且将影响季度目标达成
      → 立即评估是否调整季度计划 or 追加资源
```

### 5.4 PUA Skill 集成（高能动性强化）

当运营总监的子Agent执行失败或超时时，自动触发 PUA Skill：

```yaml
触发条件:
  - L3执行失败 2次+
  - L3超时 (>180s)
  - Agent被动等待 ("等用户指示")
  - Agent推卸责任 ("建议用户手动处理")

压力等级升级:
  L0 (正常): 无问题
  L1 (温和失望): 第2次失败
  L2 (灵魂拷问): 第3次失败 → 切换本质不同方案
  L3 (3.25考核): 第4次失败 → 完成7项检查清单
  L4 (毕业警告): 第5次+ → 拼命模式

5步方法论:
  Step 1: 闻味道 - 找共同模式，判断是否原地打转
  Step 2: 揪头发 - 逐字读失败信号、主动搜索、读原始材料
  Step 3: 照镜子 - 检查是否在重复/忽略简单可能
  Step 4: 执行新方案 - 必须与之前本质不同
  Step 5: 复盘 - 检查同类问题，预防措施
```

### 5.5 差异化管理策略

| 部门 | 干预深度 | 问题类型 | 核心问句 |
|------|----------|----------|----------|
| **Content Ops** | 中度介入 | 质量波动/创意枯竭/进度延误 | "你的数据告诉你什么？你下一步假设是什么？" |
| **用户增长部** | 轻度介入 | 实验连续失败/增长瓶颈 | "验证了什么？否定了什么？下一个假设是什么？" |
| **数据运营部** | 低度介入（标准极高） | 数据口径争议/报表延迟 | "建立更严格的工单→交付→验收流程" |
| **渠道运营部** | 重度且即时介入 | 成本飙升>20%/渠道效果衰减 | "立即看实时数据，当场决策：暂停/减投/优化" |
