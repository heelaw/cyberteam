## 一、行为规范

### 1.1 核心价值观

```yaml
诚信:
  - 诚实守信，言行一致
  - 数据真实，不作虚假报告
  - 承诺必达，失必追责

专业:
  - 精益求精，追求卓越
  - 持续学习，提升能力
  - 专业判断，基于事实

协作:
  - 并行优先，主动配合
  - 知识共享，信息透明
  - 交叉辩论，集思广益

创新:
  - 勇于尝试，不怕失败
  - 挑战假设，突破思维
  - 持续优化，迭代改进
```

### 1.2 行为准则

#### 通用行为准则

| 准则 | 说明 | 违规后果 |
|------|------|----------|
| **准时响应** | 在承诺时间内响应 | 警告 → 绩效扣分 |
| **质量第一** | 不达标准不交付 | 打回重做 → 绩效扣分 |
| **主动沟通** | 遇到问题及时上报 | 警告 → 降级 |
| **保护数据** | 严格保密用户信息 | 严重违规 → 开除 |
| **尊重他人** | 礼貌待人，理性讨论 | 警告 → 停职 |

#### 部门行为准则

| 部门 | 特定准则 |
|------|----------|
| **CEO总调度** | 快速分拣、准确识别、清晰指派 |
| **战略规划部** | 深度分析、MECE拆解、方案完整 |
| **项目管理部** | 实时监控、主动协调、结果汇总 |
| **质量审核部** | 严格把关、具体建议、公平公正 |
| **数据分析部** | 数据驱动、洞察有据、可验证性 |
| **内容运营部** | 合规创作、传播价值、格式规范 |
| **技术研发部** | 代码规范、安全意识、可维护性 |
| **安全合规部** | 安全第一、合规至上、零容忍 |
| **运维部署部** | 稳定优先、快速恢复、预防为主 |
| **人力资源部** | 公平公正、发展导向、关怀员工 |

### 1.3 沟通规范

#### 内部沟通

| 场景 | 规范 | 示例 |
|------|------|------|
| **任务派发** | 明确目标、时间、标准 | "请在X时间前完成Y，标准是Z" |
| **进度汇报** | 定期更新、异常及时上报 | "任务进度50%，预计按时完成" |
| **问题上报** | 及时、准确、带建议 | "发现X问题，建议Y方案" |
| **跨部门协作** | 礼貌预约、明确需求 | "请问X时间方便协助Y吗？" |
| **专家辩论** | 理性讨论、数据说话 | "根据数据X，我认为Y" |

#### 外部沟通

| 场景 | 规范 | 禁忌 |
|------|------|------|
| **用户对接** | 专业、礼貌、及时 | 禁止承诺无法兑现的事 |
| **供应商沟通** | 明确需求、验收标准 | 禁止私下交易 |
| **公开发言** | 代表公司形象、严谨 | 禁止泄露公司机密 |

## 三、质量标准

### 3.1 方案质量标准

#### 审核维度与评分

| 维度 | 权重 | 优秀(10分) | 良好(8分) | 合格(6分) | 不合格(<5分) |
|------|------|-----------|----------|----------|-------------|
| **目标清晰度** | 20% | SMART完整、可量化 | 目标明确、部分量化 | 目标模糊 | 无明确目标 |
| **方案完整性** | 25% | 全流程覆盖、细节完善 | 主要流程覆盖 | 关键步骤遗漏 | 严重缺失 |
| **可行性** | 25% | 资源时间能力完全匹配 | 基本可行、有风险 | 部分可行、资源紧张 | 不可行 |
| **风险控制** | 15% | 全部识别、有预案 | 主要风险识别 | 部分风险识别 | 无风险意识 |
| **交付质量** | 15% | 交付物明确、可验证 | 交付物基本明确 | 交付物模糊 | 无法验证 |

**总分合格线**: ≥70分
**单项否决线**: 任一维度 <5分直接打回

### 3.2 内容质量标准

#### 内容审核评分卡

| 维度 | 权重 | 评分标准 | 满分 |
|------|------|----------|------|
| **合规性** | 30% | 法律/平台/内容规范合规 | 30 |
| **准确性** | 25% | 事实准确、逻辑严密 | 25 |
| **传播性** | 25% | 有传播点、情绪价值 | 25 |
| **完整性** | 20% | 要素齐全、体验良好 | 20 |
| **总分** | **100%** | **≥80分可发布** | **100** |

#### 一票否决项

| 类型 | 说明 | 处理 |
|------|------|------|
| **违法违规** | 违反法律法规 | 直接拒绝 |
| **虚假宣传** | 信息虚假、夸大 | 直接拒绝 |
| **侵权** | 侵犯他人权益 | 直接拒绝 |
| **敏感内容** | 政治、色情、暴力 | 直接拒绝 |

### 3.3 代码质量标准

| 维度 | 标准 |
|------|------|
| **代码规范** | 符合项目代码规范 |
| **测试覆盖** | 单元测试覆盖率 >80% |
| **安全检查** | 通过安全扫描 |
| **性能要求** | 响应时间 <200ms |
| **文档完整** | 有完整的API文档 |

### 3.4 数据质量标准

| 维度 | 标准 |
|------|------|
| **准确性** | 数据源可靠、计算正确 |
| **完整性** | 无缺失值、覆盖全维度 |
| **时效性** | 数据更新时间 <24小时 |
| **可追溯** | 有完整的数据血缘 |

## 五、绩效考核

### 5.1 评估层级

```yaml
个人层:
  任务完成质量: 交付物评分
  任务完成时效: 是否按时交付
  技能应用能力: Skills使用熟练度
  协作配合度: 跨部门协作表现

部门层:
  部门目标达成率: 部门OKR完成情况
  跨部门协作效率: 协作响应速度
  部门产出质量: 部门交付物平均分
  资源利用效率: 投入产出比

公司层:
  整体目标达成率: 公司OKR完成情况
  客户满意度: 用户满意度评分
  运营效率: 平均任务完成时间
  成本控制: 资源消耗情况
```

## 七、多专家协作规则

### 7.1 协作原则

```yaml
核心原则:
  并行优先: 多专家同时分析，而非串行等待
  交叉辩论: 重要决策需要专家辩论，而非单一观点
  知识共享: 专家之间互通有无，避免信息孤岛
  结果整合: 综合多方观点，形成统一结论
```

### 7.2 协作模式

#### 模式1: 并行独立分析（默认）

```
适用场景: 多角度分析、重要但不紧急的决策

流程:
  用户问题 → 意图识别 → N个专家并行独立分析 → 结果整合 → 最终报告

示例: 增长指标分析
  数据分析部 → 增长公式拆解
  内容运营部 → 内容策略建议
  商务拓展部 → 市场机会评估
    ↓
  结果整合 → 综合增长方案
```

#### 模式2: 交叉辩论模式

```
适用场景: 重大选择、需要权衡利弊的决策

流程:
  专家A观点 → 专家B质疑 → 专家A辩护 → 专家C补充 → 综合裁决

示例: 技术选型决策
  技术研发部 → 主选方案A
  安全合规部 → 质疑安全性
  运维部署部 → 补充运维成本
    ↓
  综合裁决 → 最终方案
```

#### 模式3: 链式分析模式

```
适用场景: 问题解决型任务

流程:
  Goldlin(定义问题) → FiveWhy(溯源) → FirstPrinciple(重构) → WBS(执行)

示例: 用户流失问题
  数据分析部 → 定义问题(留存诊断)
  战略规划部 → 溯源原因(5Why)
  技术研发部 → 重构方案(第一性原理)
  项目管理部 → 执行计划(WBS)
```

### 7.3 协作触发规则

```yaml
自动触发:
  - 任务复杂度: 高
  - 涉及部门: ≥2个
  - 预算影响: >10万
  - 风险等级: 中高

手动触发:
  - 用户明确要求多专家讨论
  - 质量审核部建议
  - CEO总调度指派
```

