---

## 一、规范背景

### 1.1 为什么需要这个规范

| 风险类型 | 影响 |
|----------|------|
| **工具链兼容** | 部分 CI/自动化工具对中文路径支持差 |
| **跨平台协作** | 跨操作系统（Mac/Windows/Linux）可能出现编码问题 |
| **代码集成** | 配置文件、脚本中引用中文路径会导致错误 |
| **未来扩展** | 预留自动化和 AI Agent 接入的技术基础 |

### 1.2 规范原则

```
优先级1: 可读性（人能看懂）
优先级2: 可排序（目录自动按时间/类型排序）
优先级3: 唯一性（同一目录下不重名）
优先级4: 可扩展（未来新增不会破坏结构）
```

## 三、文件命名规范

### 3.1 核心模板文件

```
workflow/00-init/
├── task-brief-template.md           ✓ 英文
└── [项目名]-init-YYYYMMDD.md        ✓ 中英混合（项目名用拼音或英文缩写）

workflow/01-user-research/
├── strategy-brief-template.md        ✓ 英文
├── user-profile-template.md          ✓ 英文
└── [项目名]-strategy-brief-YYYYMMDD.md

workflow/02-selling-points/
├── content-plan-template.md          ✓ 英文
├── selling-points-analysis-template.md
└── [项目名]-selling-points-YYYYMMDD.md
```

### 3.2 项目产出文件

```
[project]-[stage]-[yyyymmdd]-[version]
  │
  ├── project: 项目缩写（英文或拼音，≤10字符）
  ├── stage: 阶段缩写（见下表）
  ├── yyyymmdd: 日期
  └── version: 版本号（v01, v02...）

阶段缩写对照表：
├── brief       → 任务简报
├── research    → 用户研究
├── strategy    → 策略简报
├── points      → 卖点分析
├── pain        → 痛点定位
├── plan        → 内容策划
├── draft       → 文案初稿
├── draft-v2    → 文案二稿
├── review      → 审核意见
├── final       → 定稿
├── publish     → 发布报告
└── retro       → 复盘报告

示例：
├── spring-festival-brief-20260301.md
├── spring-festival-strategy-20260302.md
├── spring-festival-draft-20260305-v01.md
├── spring-festival-draft-20260306-v02.md
├── spring-festival-review-20260307.md
├── spring-festival-final-20260308.md
└── spring-festival-retro-20260408.md
```

### 3.3 命名单字禁止

```
禁止字符：
├── 空格         → 用 - 或 _ 代替
├── 中文字符     → 用拼音或英文代替
├── 特殊符号     → 仅允许 - _ .
├── 超过30字符   → 缩写到30字符以内
└── 重复命名     → 日期+版本区分
```

## 四、新旧迁移策略

### 4.1 迁移原则

```
新项目：100% 遵循本规范
存量项目：按需迁移，不搞批量重命名
```

### 4.2 迁移触发条件

存量项目在以下情况必须迁移：

| 触发条件 | 说明 |
|----------|------|
| 项目被重新激活 | 再次使用时必须规范命名 |
| 接入新自动化系统 | CI/CD 或 AI Agent 接入时 |
| 纳入代码仓库 | 提交到 Git 时 |
| 跨团队共享 | 给其他部门使用时 |

### 4.3 迁移检查清单

```
□ 目录名是否包含中文？
□ 文件名是否包含中文？
□ 路径中是否有空格？
□ 引用此路径的配置文件是否同步更新？
□ 迁移后自动化工具测试是否通过？
□ 文档内链接是否更新？
```

### 4.4 遗留中文路径处理

```
已识别的中文路径（暂不迁移）：
├── references/ContentOps部门章程.md
├── references/新人三段式成长路径.md
├── references/内容生产命名与路径规范.md
└── skills/content-review/

迁移触发：接入自动化工具时
预计处理时间：2小时内（通过脚本批量重命名）
```

## 五、配置与脚本规范

### 5.1 代码配置引用

```bash
# ❌ 错误：中文路径
path: "02_Skill研发/cyberteam搭建/..."  # 不可用

# ✅ 正确：英文路径
path: "skills/writing/workflow/..."

# ✅ 正确：拼音缩写
path: "02_skill_yf/cyberteam_dj/cyberteam-v3/"
```

### 5.2 Git 提交规范

```
提交信息格式：
[type] [scope] [description]

type: add | fix | update | refactor | docs
scope: workflow | references | agents | skills
description: 简短描述

示例：
[docs] workflow: add demand-classification-sop
[fix] references: rename Chinese filenames in temp/
[add] agents: update content-ops SOUL.md
```

## 六、检查与验收

### 6.1 PR 提交检查

在提交 PR 前，必须确认：

```
□ 新建文件/目录不包含中文命名
□ 修改文件名后，文档内引用同步更新
□ 如涉及中文路径，迁移到英文路径
□ Git 提交信息符合规范
```

### 6.2 自动化检查（未来）

当 CI 工具接入后，自动检查：

```bash
# 检查所有新增文件是否包含中文
find . -name "*.md" -newer .git/HEAD | xargs grep -l "[一-龥]"
# 如有输出，PR 自动拒绝
```
