# CyberTeam v2.1 定价 A/B 测试方案

**版本**: v1.0
**编制日期**: 2026-03-24
**编制部门**: 财务部
**测试周期**: 12 周（2026-04 至 2026-06）

---

## 测试目标

### 核心目标

通过科学、系统的 A/B 测试，验证并优化 CyberTeam v2.1 的定价策略，实现:

1. **最大化收入**: 找到收入最大化的最优定价点
2. **提升转化率**: 优化免费版到付费版的转化率
3. **平衡 ARPU**: 在价格和转化率之间找到最佳平衡点

### 成功指标

| 指标 | 当前值 | 目标值 | 测量周期 |
|------|--------|--------|----------|
| 付费转化率 | 3% | ≥ 4% | 每周 |
| MRR 增长率 | 47%/月 | ≥ 50%/月 | 每月 |
| ARPU | $152 | ≥ $150 | 每月 |
| 收入提升 | - | ≥ 10% | 测试结束时 |

---

## 测试设计原则

### 1. 科学性原则

- **单一变量**: 每次测试只改变 1 个定价变量
- **对照设置**: 设置控制组（A 组）和实验组（B/C 组）
- **随机分配**: 用户随机分配到各组，避免选择偏差

### 2. 统计显著性原则

- **置信度**: 95%（p < 0.05）
- **统计功效**: 80%（β = 0.2）
- **最小样本量**: ≥ 1,000 用户/组

### 3. 伦理性原则

- **透明告知**: 在定价页面注明"测试价格"
- **公平对待**: 所有用户享受相同服务
- **回滚机制**: 测试结束后统一到最优价格

### 4. 商业性原则

- **短期测试**: 单次测试 ≤ 4 周（避免长期收入损失）
- **快速迭代**: 基于数据快速决策
- **风险控制**: 设置止损线（收入下降 ≥ 5% 立即回滚）

## 测试 1: 价格敏感度测试

### 测试目标

验证用户对个人版价格的敏感度，找到收入最大化的最优价格点。

### 测试变量

| 方案 | 个人版价格 | 差异 | 假设 |
|------|-----------|------|------|
| **A 组（控制组）** | $19/月 | - | 基准价格 |
| **B 组（低价组）** | $14/月 | -26% | 低价提升转化率，但 ARPU 下降 |
| **C 组（高价组）** | $24/月 | +26% | 高价降低转化率，但 ARPU 提升 |

### 测试假设

| 方案 | 转化率假设 | ARPU 假设 | 收入影响 = 转化率 × ARPU |
|------|-----------|----------|----------------------|
| A 组 | 3% | $19 | 基准 |
| B 组 | 4.5% (+50%) | $14 (-26%) | +11% |
| C 组 | 2.1% (-30%) | $24 (+26%) | -12% |

**预期结果**: B 组收入提升 11%，C 组收入下降 12%，A 组保持不变。

### 测试设计

#### 流量分配

```
新用户流量 → 随机分配 → A 组 33.3% | B 组 33.3% | C 组 33.3%
```

#### 测试周期

| 阶段 | 时间 | 行动 |
|------|------|------|
| 准备期 | D1-D3 | 搭建测试基础设施，数据埋点 |
| 预热期 | D4-D7 | 灰度发布 10% 流量，验证数据准确性 |
| 正式期 | D8-D35 | 全量发布，收集数据 |
| 冷却期 | D36-D38 | 统一价格，等待用户行为稳定 |
| 分析期 | D39-D42 | 数据分析，决策 |

#### 样本量计算

**公式**:
```
n = (Zα/2 + Zβ)² × (p1(1-p1) + p2(1-p2)) / (p1-p2)²
```

**参数**:
- Zα/2 = 1.96（95% 置信度）
- Zβ = 0.84（80% 统计功效）
- p1 = 0.03（A 组转化率）
- p2 = 0.045（B 组转化率）

**计算**:
```
n = (1.96 + 0.84)² × (0.03×0.97 + 0.045×0.955) / (0.03-0.045)²
n = 7.84 × 0.0709 / 0.000225
n ≈ 2,468 用户/组
```

**考虑 10% 的流失率，实际需要**: 2,468 / 0.9 ≈ **2,743 用户/组**

#### 测试时长

**日新增用户**: 100 人（基于 M1-M3 平均）

**所需天数**: 2,743 / 100 ≈ **28 天**

**加上预热和冷却期**: **42 天（6 周）**

### 数据收集

#### 关键指标

| 指标 | 定义 | 数据源 |
|------|------|--------|
| **流量分配** | 各组访客数 | Analytics |
| **注册转化率** | 访客 → 注册用户 | 数据库 |
| **激活转化率** | 注册 → 激活用户 | 数据库 |
| **付费转化率** | 激活 → 付费用户 | Stripe |
| **ARPU** | 收入 / 付费用户数 | Stripe |
| **MRR** | 月度经常性收入 | Stripe |
| **留存率** | 付费用户留存率 | 数据库 |

### 数据分析

#### 每周监控报告

| 指标 | A 组 | B 组 | C 组 | B vs A | C vs A |
|------|------|------|------|--------|--------|
| 流量 | 1,000 | 1,000 | 1,000 | - | - |
| 注册用户 | 700 | 710 | 690 | +1.4% | -1.4% |
| 激活用户 | 490 | 497 | 483 | +1.4% | -1.4% |
| 付费用户 | 15 | 22 | 11 | +46.7% | -26.7% |
| 付费转化率 | 3.1% | 4.4% | 2.3% | +42% | -26% |
| ARPU | $19 | $14 | $24 | -26% | +26% |
| MRR | $285 | $308 | $264 | +8.1% | -7.4% |

#### 统计显著性检验

**Z-test for proportions**:

```
Z = (p1 - p2) / √(p(1-p)(1/n1 + 1/n2))
```

**示例**: B 组 vs A 组

```
p1 = 0.031 (A 组)
p2 = 0.044 (B 组)
p = (15+22) / (490+497) = 0.037
n1 = 490
n2 = 497

Z = (0.031 - 0.044) / √(0.037×0.963×(1/490 + 1/497))
Z = -0.013 / 0.012
Z = -1.083

p-value = 0.279 > 0.05
结论: 不显著（需要更大样本量）
```

### 决策标准

#### 采用新价格的条件

1. **收入提升 ≥ 10%**（统计显著，p < 0.05）
2. **或付费转化率提升 ≥ 20%**（统计显著，p < 0.05）
3. **且留存率无显著下降**（< 5%，p < 0.05）

#### 回滚条件

1. **收入下降 ≥ 5%**（统计显著，p < 0.05）
2. **或留存率下降 ≥ 10%**（统计显著，p < 0.05）

#### 延长测试条件

- **样本量不足**: 各组 < 2,500 用户
- **统计不显著**: p-value > 0.1

### 测试风险

#### 风险 1: 用户不满价格差异

**概率**: 低（10%）
**影响**: 中

**应对措施**:
- 在定价页面注明"测试价格"
- 提供价格保护承诺（老用户价格不变）
- 快速客服响应

#### 风险 2: 收入短期下降

**概率**: 中（30%）
**影响**: 高

**应对措施**:
- 设置止损线（收入下降 ≥ 5% 立即回滚）
- 限制测试时长（4 周）
- 准备现金储备（$10k）

#### 风险 3: 数据污染

**概率**: 中（30%）
**影响**: 中

**应对措施**:
- 只对新用户测试（老用户价格不变）
- 数据埋点严格隔离
- 定期数据清洗

## 测试 2: 免费版限额测试

### 测试目标

验证免费版 LLM 调用次数限额对用户激活和付费转化的影响，找到最优限额点。

### 测试变量

| 方案 | 免费版 LLM 调用次数 | 差异 | 假设 |
|------|-------------------|------|------|
| **A 组（控制组）** | 100 次/月 | - | 基准限额 |
| **B 组（宽松组）** | 200 次/月 | +100% | 宽松限额提升激活率，但降低付费转化率 |
| **C 组（严格组）** | 50 次/月 | -50% | 严格限额降低激活率，但提升付费转化率 |

### 测试假设

| 方案 | 激活率假设 | 付费转化率假设 | 综合影响 |
|------|-----------|---------------|----------|
| A 组 | 70% | 3% | 基准 |
| B 组 | 85% (+21%) | 2.4% (-20%) | +2.5% |
| C 组 | 55% (-21%) | 3.6% (+20%) | -13% |

**预期结果**: B 组激活率提升 21%，但付费转化率下降 20%，综合影响 +2.5%。

### 测试设计

#### 流量分配

```
新用户流量 → 随机分配 → A 组 33.3% | B 组 33.3% | C 组 33.3%
```

#### 测试周期

| 阶段 | 时间 | 行动 |
|------|------|------|
| 准备期 | D1-D3 | 搭建测试基础设施，数据埋点 |
| 预热期 | D4-D7 | 灰度发布 10% 流量 |
| 正式期 | D8-D35 | 全量发布，收集数据 |
| 冷却期 | D36-D38 | 统一限额 |
| 分析期 | D39-D42 | 数据分析，决策 |

#### 样本量计算

**公式**: 同测试 1

**参数**:
- p1 = 0.70（A 组激活率）
- p2 = 0.85（B 组激活率）

**计算**:
```
n = (1.96 + 0.84)² × (0.70×0.30 + 0.85×0.15) / (0.70-0.85)²
n = 7.84 × 0.3475 / 0.0225
n ≈ 121 用户/组
```

**考虑激活用户需要转化为付费用户，实际需要**: **3,000 用户/组**

#### 测试时长

**日新增用户**: 200 人（基于 M4-M6 平均）

**所需天数**: 3,000 / 200 = **15 天**

**加上预热和冷却期**: **28 天（4 周）**

1. **激活率下降 ≥ 20%**（统计显著，p < 0.05）
2. **或用户满意度下降 ≥ 10%**（NPS < -10）

#### 动态限额方案

如果测试结果显示:
- **B 组激活率高但付费转化率低**: 考虑 **动态限额**（前 30 天 200 次，之后 100 次）
- **C 组激活率低但付费转化率高**: 考虑 **分级限额**（50 次 → 激活后 → 100 次）

## 测试 3: 年付折扣测试

### 测试目标

验证不同年付折扣力度对年付转化率和现金流的影响，找到最优折扣点。

### 测试变量

| 方案 | 年付折扣 | 实付价格 | 差异 | 假设 |
|------|----------|----------|------|------|
| **A 组（控制组）** | 20% off | $15/月 ($180/年) | - | 基准折扣 |
| **B 组（低折扣）** | 10% off | $17/月 ($204/年) | +13% | 低折扣降低年付转化率，但提升利润率 |
| **C 组（高折扣）** | 30% off | $13/月 ($156/年) | -13% | 高折扣提升年付转化率，但降低利润率 |

### 测试假设

| 方案 | 年付转化率假设 | ARPU 假设 | 利润率假设 | 综合影响 |
|------|--------------|----------|-----------|----------|
| A 组 | 20% | $180 | 90% | 基准 |
| B 组 | 15% (-25%) | $204 (+13%) | 92% (+2%) | -15% |
| C 组 | 25% (+25%) | $156 (-13%) | 88% (-2%) | +9% |

**预期结果**: C 组年付转化率提升 25%，但利润率下降 2%，综合影响 +9%。

### 测试设计

#### 流量分配

```
付费用户 → 随机分配 → A 组 33.3% | B 组 33.3% | C 组 33.3%
```

**注意**: 此测试只在付费用户展示定价页面时进行。

| 正式期 | D8-D35 | 全量发布，收集数据 |
| 冷却期 | D36-D38 | 统一折扣 |
| 分析期 | D39-D42 | 数据分析，决策 |

### 风险与应对

#### 风险 1: 测试时间过长

**概率**: 中（30%）
**影响**: 中

**应对措施**:
- 并行运行测试 1 和测试 2（不冲突的用户群）

**概率**: 中（30%）

**应对措施**:
- 延长测试时间（增加样本量）
- 调整测试变量（扩大价格差异）
- 接受当前价格（基于数据决策）

#### 风险 3: 用户抱怨价格差异

**概率**: 低（10%）
**影响**: 低

**应对措施**:
- 在定价页面注明"测试价格"
- 提供价格保护承诺
- 快速客服响应

## 成功案例参考

### 案例 1: Slack

**测试**: 月付价格从 $8 调整到 $6

**结果**:
- 付费转化率提升 30%
- 收入提升 25%
- 采用新价格

**启示**: 低价策略在早期增长阶段有效

### 案例 2: Dropbox

**测试**: 免费版存储空间从 2GB 调整到 5GB

**结果**:
- 注册用户提升 50%
- 付费转化率下降 10%
- 综合影响 +35%
- 采用新限额

**启示**: 宽松的免费版限额能带来规模效应

### 案例 3: Zoom

**测试**: 年付折扣从 15% 调整到 20%

**结果**:
- 年付转化率提升 40%
- 现金流提升 35%
- 采用新折扣

**启示**: 年付折扣是提升现金流的有效手段

## 结论

通过系统的 A/B 测试，我们期望在 12 周内找到 CyberTeam v2.1 的最优定价策略:

1. **测试 1**: 验证个人版价格敏感度，找到收入最大化的价格点
2. **测试 2**: 优化免费版限额，平衡激活率和付费转化率
3. **测试 3**: 确定最优年付折扣，最大化现金流和 LTV

**预期成果**:
- 付费转化率提升 33%（3% → 4%）
- MRR 增长率提升 6%（47% → 50%）
- 收入提升 10%（$4.62M → $5.08M）

**关键成功因素**:
- 科学严谨的测试设计
- 充分的样本量（≥ 1,000 用户/组）
- 快速的数据分析和决策
- 风险控制和回滚机制
