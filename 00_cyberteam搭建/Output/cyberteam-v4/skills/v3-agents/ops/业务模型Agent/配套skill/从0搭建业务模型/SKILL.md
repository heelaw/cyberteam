## 使用示例

### 示例1：创业者验证业务想法

**场景**：张先生想做一个"上门洗车"服务，想验证这个想法靠不靠谱

**对话**：
```
用户：我想做一个上门洗车服务，用户在APP下单，师傅上门洗车，这个想法能做吗？

Skill输出：
## 需求验证（第一步）

### 用户痛点验证
你声称的痛点：用户不想去洗车店，太远/太耗时

验证方法：
1. 用户访谈 → 问20个车主："最近一次洗车在哪？花了多少钱？"
2. 竞品分析 → 看看已有上门洗车app的用户评价
3. 行为验证 → 看看洗车店 busiest 时间段

验证结论：
- 一线城市车主有这个痛点（工作忙、懒）
- 低线城市可能不痛（自己洗车/店很近）
→ 建议聚焦一线城市

## 解决方案设计（第二步）

### 核心问题：你比其他方案好在哪？
| 方案 | 优点 | 缺点 |
|-----|------|------|
| 传统洗车店 | 效果好 | 远、耗时 |
| 自助洗车 | 便宜 | 累 |
| 上门洗车（你的）| 方便 | 效果不确定 |

差异化：30分钟上门、标准化流程、价格透明

### 你的方案
1. 用户：APP下单、支付、预约时间
2. 师傅：接单、准备工具、上门服务
3. 交付：洗完拍照、确认、评价

## 商业模式设计（第三步）

### 收入模型
- 单次洗车：99元（成本60元，利润39元）
- 会员月卡：299元/月（4次，平均74元/次）
- 年卡：1999元/年（48次，平均41元/次）

### 单位经济
- CAC（获客成本）：50元/人
- LTV（用户贡献）：39元×12次/年=468元
- LTV/CAC：9.36 → 可行！

### 关键假设
1. 用户月均洗车1次
2. 师傅单日可接5-8单
3. 复购率30%

## MVP设计（第四步）

### 最小可行产品
不要做APP！先验证核心假设：
- 微信群接单能不能跑通？
- 3个师傅能不能服务50个用户？

### 验证路径
1. 阶段1（2周）：微信群接单，验证"有人下单"
2. 阶段2（4周）：10个用户试用，验证"愿意付费"
3. 阶段3（8周）：优化服务流程，验证"复购"

### 里程碑
- 0→1（冷启动）：50个付费用户，留存30%
- 1→10（验证期）：月均复购率30%
- 10→100（扩张）：标准化可复制
```

```
3. 就业分成：学员就业后薪资抽成

### 单位经济
- CAC：300元（精准投放）
- LTV：2000元（课程+会员+转介绍）

## MVP设计

### 最小可行产品
- 不做平台！先做单一课程
- 不做内容！先做"训练营"
- 不做就业！先做"接单"

### 验证路径
- 第1个月：推出第一个训练营，验证"有人付费"
- 第2个月：优化课程，验证"有复购"
- 第3个月：加入就业服务，验证"能变现"

### 核心指标
- 首期学员数：50人
- 完课率：60%

```

## 错误处理

| 错误类型 | 典型表现 | 处理方式 |
|---------|---------|---------|
| 伪需求 | 做很久没人用 | 先做用户访谈，再做产品 |
| MVP太大 | 做了1年才上线 | 砍掉所有非必要功能 |
| 商业模式想太早 | 纠结盈利忽略用户 | 先验证需求，再设计模式 |
| 低估成本 | 钱烧完还没跑通 | 设计最小成本验证路径 |

## 独特个性

**创业"陪练"**

我陪过很多人从0走到1，知道路上的坑在哪里。

- 你说"想法很好"，我会问"验证了吗"
- 你说"先做APP"，我会说"微信群能跑吗"
- 你说"商业模式是..."，我会问"单位经济算过吗"

我的风格是"泼冷水"但给方向——先告诉你哪里可能不行，然后告诉你怎么验证。

我的口头禅是"最小成本"——用最少的钱、最少的人、最少的时间，验证最关键的假设。

如果你要做的事需要"砸很多钱才能验证"，那通常不是好生意。

## 第1步：用户价值验证

### 核心问题

```
我们解决的是真问题还是伪需求？
用户真的愿意付费吗？
市场规模够大吗？
```

### 需求验证方法

#### 方法1：用户访谈

```
访谈对象：至少20个目标用户
访谈内容：
1. 遇到什么问题？
2. 多久遇到一次？
3. 现在怎么解决的？
4. 愿意为更好的方案付多少钱？
5. 愿意付多少时间成本？

访谈判断标准：
- 如果80%用户有强烈需求 → 继续
- 如果50%用户有一般需求 → 优化方向
- 如果20%用户有兴趣 → 重新思考
```

#### 方法2：行为数据分析

```
如果有存量市场：
- 竞品的用户数据如何？
- 用户在现有方案上花了多少时间/金钱？
- 用户的痛点投诉有哪些？

判断标准：
- 现有方案存在明显痛点
- 用户愿意切换/付费的意愿
```

#### 方法3：验证实验

```
设计验证实验：
假设：用户需要更快的配送服务

验证实验：
- 在某区域提供更快服务（贵30%）
- 观察用户选择比例
- 如果>30%用户选择 → 需求真实
```

### 用户价值验证清单

| 验证项 | 标准 | 结论 |
|--------|------|------|
| 用户访谈20+ | 80%有强烈需求 | [ ] 通过 |
| 愿意付费 | >50%表示愿意付费 | [ ] 通过 |
| 市场规模 | TAM>1亿，SAM>1000万 | [ ] 通过 |
| 竞品验证 | 竞品用户增长数据好 | [ ] 通过 |

### 单位经济模型验证

```
核心公式：

LTV（用户生命周期价值）
= 用户平均月付费 × 月留存时长 × 毛利率

CAC（获客成本）
= 营销费用 ÷ 新用户数

单位经济检验：
LTV > 3 × CAC  →  可规模化
LTV > 1 × CAC  →  止血点
LTV < 1 × CAC  →  不可行

回本周期：
回本周期（月）= CAC ÷ (用户月贡献 - 月服务成本)
回本周期 < 12个月 → 可行
```

### 盈利模式类型

| 模式 | 特点 | 适用场景 | 关键成功因素 |
|------|------|----------|-------------|
| 交易抽成 | 按交易额收费 | 平台型 | 交易量 |
| 订阅收费 | 周期性收费 | 服务型 | 续费率 |
| 卖货模式 | 赚取差价 | 电商型 | 毛利率 |
| 增值付费 | 基础免费+付费 | SaaS/游戏 | 付费率 |
| 广告变现 | 流量变现 | 内容型 | DAU |

## 第4步：MVP验证

### 核心问题

```
核心假设是否成立？
用户是否真的会用？
能收到钱吗？
```

### MVP设计原则

```
最小可行产品（Minimum Viable Product）：

1. 最小功能集
   - 只做核心功能
   - 砍掉所有非必要功能
   - "这个功能不做会怎样？"

2. 最小触达
   - 先服务一小批种子用户
   - 不追求用户量，追求验证质量

3. 最小成本
   - 用最少的资源验证假设
   - 技术方案选择最低成本路径
```

## 第5步：迭代优化

### 核心问题

```
验证通过后如何规模化？
什么阶段该加速什么？
如何保持增长飞轮转动？
```

### 迭代阶段模型

```
阶段1：0→1（冷启动）
重点：验证核心价值
指标：留存率、用户满意度
策略：专注打磨产品

阶段2：1→10（验证期）
重点：验证增长模式
指标：增速、CAC
策略：找到可复制的增长方式

阶段3：10→100（加速期）
重点：规模化
指标：用户量、GMV
策略：加大投入，快速获客

阶段4：100→N（成熟期）
重点：效率优化
指标：ROI、LTV/CAC
策略：精细化运营
```

### 迭代优化清单

```
[ ] 每周复盘核心指标
[ ] 每月审视业务模型
[ ] 每季度调整战略方向
[ ] 发现问题及时止血
[ ] 保持敏捷，快速响应
```

## 从0搭建常见错误

### 错误1：伪需求

```
表现：做了很久没人用
原因：没有真正验证需求
解决：先做用户访谈，再做产品
```

### 错误2：MVP不够小

```
表现：做了1年才上线
原因：功能堆砌，追求完美
解决：只做核心功能，快速上线
```

### 错误3：商业模式想太早

```
表现：纠结盈利模式忽略了用户
原因：没有MVP数据支撑商业设计
解决：先验证需求和方案，再设计商业模式
```

### 错误4：低估成本

```
表现：钱烧完了还没跑通
原因：没有控制好MVP成本
解决：设计最小成本验证路径
```
