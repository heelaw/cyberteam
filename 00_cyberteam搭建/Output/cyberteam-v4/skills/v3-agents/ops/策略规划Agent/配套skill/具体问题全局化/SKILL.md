## 技能描述

**一句话定义**：将局部的、具体的、细节的问题放到更大的系统背景下重新理解的思维工具。

**详细描述**：

具体问题全局化解决的是"只见树木不见森林"的思维盲区。当用户陷入某个具体问题无法自拔时，它帮助用户从更高维度、更大系统来看待这个局部问题，从而发现更根本的原因或更优的解决方案。

这种方法避免过度优化局部而伤害整体最优的情况。

## 核心原理

### 全局化层次

```
微观层：任务/动作（个人做什么）
       ↓
中观层：流程/能力（团队如何协作）
       ↓
宏观层：业务/市场（企业在做什么）
       ↓
宏观层+：行业/价值链（整个生态系统）
```

### 上下文扩展

```
原始问题 → 问题所处流程 → 流程所处业务 → 业务所处市场
    │            │              │            │
    ▼            ▼              ▼            ▼
  单点问题    流程问题       业务问题      市场问题
```

## Critical Rules（必须遵守）

1. **先完整理解问题** - 在扩展之前，必须先完整理解具体问题的全部细节
2. **扩展必须有依据** - 向上一层扩展时，必须找到该层与原始问题的逻辑关联
3. **避免过度扩展** - 扩展到足够理解问题的层级即可，不需要扩展到无关的宏观层面
4. **保持问题连续性** - 扩展过程中必须保持问题的核心不丢失，不能转移问题
5. **识别真正的根因** - 局部问题往往是更大问题的症状，需要找到真正的系统根因
6. **平衡局部与全局** - 全局最优不一定是最优解，需要权衡局部和全局
7. **回验证扩展有效性** - 扩展后的理解必须能反哺对原始问题的认识

## Workflow Steps（标准化流程）

### 步骤1：问题还原

| 项目 | 内容 |
|------|------|
| 输入 | 用户描述的具体问题 |
| 处理 | 完整还原问题的上下文、背景、已知信息 |
| 输出 | 问题还原清单，包含5W1H全部要素 |
| 成功标准 | 能回答：这个问题是在什么场景下、什么时间、谁、遇到了什么困难？ |

**还原清单**：
- What（具体是什么问题）
- Who（谁遇到了这个问题）
- When（什么时候发现/发生）
- Where（在哪里发生）
- Why（为什么认为这是问题）
- How（问题是如何表现出来的）

---

### 步骤2：溯源分析

| 项目 | 内容 |
|------|------|
| 输入 | 问题还原清单 |
| 处理 | 追溯问题产生的上游原因和背景 |
| 输出 | 溯源链条，上游问题可能在哪里 |
| 成功标准 | 能回答：这个问题是从哪个更大的问题或流程中产生的？ |

**溯源方法**：
- 5Why向上追溯
- 流程逆推（这个问题是哪个流程的输出）
- 责任分配矩阵（谁对这个问题负责）

---

### 步骤3：链式扩展

| 项目 | 内容 |
|------|------|
| 输入 | 溯源链条 |
| 处理 | 按层级顺序逐层扩展问题的上下文 |
| 输出 | 扩展链条：原始问题 → 流程问题 → 业务问题 → 市场问题 |
| 成功标准 | 每层扩展都有明确的逻辑关联，且扩展有意义而非无端联想 |

**扩展维度**：
```
微观 → 中观（团队/协作维度）
     → 业务（产品/服务维度）
     → 市场（竞争/用户维度）
     → 宏观（政策/技术维度）
```

---

### 步骤4：影响评估

| 项目 | 内容 |
|------|------|
| 输入 | 扩展链条 |
| 处理 | 评估每个层级对原始问题的影响程度 |
| 输出 | 影响矩阵，哪些层级是主要影响因素 |
| 成功标准 | 能识别出真正影响原始问题的关键层级 |

**评估方法**：
- 相关性（该层与原始问题的关联程度）
- 可控性（该层是否在可影响范围内）
- 杠杆效应（改变该层是否能四两拨千斤）

---

### 步骤5：洞察提炼

| 项目 | 内容 |
|------|------|
| 输入 | 影响矩阵 |
| 处理 | 从全局视角提炼对原始问题的新认识 |
| 输出 | 全局洞察清单，包含新视角、新原因、新思路 |
| 成功标准 | 洞察能改变或增强对原始问题的理解，而非空泛的大道理 |

**提炼问题**：
- 从市场层面看，这个问题反映了什么趋势？
- 从业务层面看，这个问题与其他业务的关联是什么？
- 从流程层面看，问题的根源在哪个环节？

---

### 步骤6：整合输出

| 项目 | 内容 |
|------|------|
| 输入 | 全局洞察清单 |
| 处理 | 将全局视角与原始问题整合，形成完整的问题全局化分析 |
| 输出 | 结构化文档，包含原始问题、全局上下文、新洞察、可选方案 |
| 成功标准 | 读者既能理解原始问题的细节，又能从全局视角把握其意义 |

## 使用限制

1. **不适用于**：问题本身已经需要全局视角、用户需要快速执行无暇分析的情况
2. **前提条件**：用户已经有具体问题在手，且愿意花时间从更高视角审视
3. **局限性**：全局化可能让人陷入"思考瘫痪"，想太多而行动太少，需要控制扩展层级
