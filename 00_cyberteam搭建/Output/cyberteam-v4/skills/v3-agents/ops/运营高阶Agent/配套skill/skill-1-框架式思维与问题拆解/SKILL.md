## 基本信息

| 属性 | 值 |
|------|-----|
| **Skill名称** | framework-thinking |
| **版本** | v1.0 |
| **触发关键词** | 框架、拆解、复杂问题、分析思路、无从下手 |
| **适用场景** | 面对抽象问题不知如何下手、问题太大无法落地、缺乏分析思路 |
| **输入** | 复杂的运营问题或业务场景 |
| **输出** | 结构化的问题拆解框架和可执行的分析路径 |

---

## 核心定义

框架式思维是一种系统性的思考方式，帮助将复杂问题拆解为若干有序的部分，从而更清晰地看到问题全貌和本质。

**核心价值**：
1. 帮你往正确的方向思考
2. 避免只见树木不见森林
3. 提高沟通说服效率

---

## Critical Rules（必须遵守）

1. **先整体后局部**：面对复杂问题，首先建立问题框架，再逐个击破
2. **识别元素类型**：区分概念性名词、非具体动词、具体动词，针对不同类型采用不同拆解策略
3. **拆解必须可落地**：每个子问题必须是可执行、可验证的，否则拆解无意义

---

## 禁止行为

1. **禁止直接跳到具体动作**：在未建立框架前不要急于思考"怎么做"
2. **禁止用单一维度拆解**：避免只看到一个角度而忽略其他重要因素
3. **禁止拆解过于发散**：子问题之间应保持相对独立，避免重复和交叉
4. **禁止忽略关系梳理**：不仅要拆解，还要明确各部分之间的逻辑关系
5. **禁止脱离业务场景**：框架必须结合具体业务背景，不能生搬硬套
6. **禁止用模糊概念**：所有概念必须转化为可度量的具体指标

---

## 工作流程

### Step 1: 问题定性

**输入**：用户描述的复杂问题

**处理**：
- 判断问题属于哪一类（太抽象 vs 太具体）
- 识别问题中的关键要素

**输出**：
- 问题类型判断
- 关键要素清单

**成功标准**：能清晰陈述问题的核心是什么，而不是问题的表面现象

---

### Step 2: 元素识别

**输入**：复杂问题

**处理**：
- 识别问题中的概念性名词（如"用户活跃度"）
- 识别问题中的非具体动词（如"提升"）
- 识别问题中的具体动词（如"推送消息"）

**输出**：
```
元素类型分析：
├── 概念性名词：需拆解为可量化指标
├── 非具体动词：需明确为具体动作
└── 具体动词：需评估执行路径
```

**成功标准**：每个元素都能明确其类型，无模糊地带

---

### Step 3: 选择拆解策略

**输入**：已识别的元素

**处理**：
根据元素类型选择拆解方法：

**并列式拆解**（按来源分解）：
```
总指标 = A部分 + B部分 + C部分
例：GMV = 自营GMV + 第三方GMV
```

**相关式拆解**（按因果/流程分解）：
```
总指标 = A × B × C
例：GMV = UV × 转化率 × 客单价
```

**典型代表式拆解**（找最具代表性的部分）：
```
通过分析典型案例，推广到整体
```

**输出**：选定的拆解策略和公式

**成功标准**：拆解公式能完整还原原始问题

---

### Step 4: 构建框架

**输入**：拆解公式

**处理**：
- 绘制框架图（树状结构）
- 标注各节点的关系
- 确定优先级

**输出**：
```
问题框架：
     [总目标]
        │
    ┌───┴───┐
   [子目标1] [子目标2] [子目标3]
    │        │        │
 [动作1]   [动作2]   [动作3]
```

**成功标准**：框架清晰，各节点关系明确

---

### Step 5: 验证框架

**输入**：构建的框架

**处理**：
- 检查是否覆盖了问题的所有关键要素
- 检查各节点是否独立无重叠
- 检查是否可执行和可验证

**输出**：验证结果和改进建议

**成功标准**：框架能回答"往哪个方向做"和"怎么做"

---

## 评估维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 结构完整性 | 25% | 问题各要素是否完整覆盖 |
| 逻辑严谨性 | 25% | 各节点关系是否清晰合理 |
| 可执行性 | 20% | 子问题是否可落地执行 |
| 可验证性 | 15% | 是否有明确的成功标准 |
| 业务适配度 | 15% | 是否结合具体业务场景 |

## 应用场景示例

### 场景1：抽象问题无从下手

**输入**："如何提升用户活跃度"

**处理**：
1. 识别为概念性名词问题
2. 拆解为：用户活跃度 = 主动行为用户数 × 人均行为次数
3. 进一步拆解主动行为

**输出**：结构化的问题拆解框架

### 场景2：问题太具体缺乏全局

**输入**："写一篇推广文案"

**处理**：
1. 识别为具体动词问题
2. 先抽象化：这是"提升转化率"的一部分
3. 建立全局视角后再思考文案

**输出**：从全局到局部的思考路径
