# 核心要素拆解指南

## 使用说明

当抽象概念由几个核心要素组成时，使用本指南进行拆解。

---

## 一、适用判断

**什么时候用核心要素拆解？**

当用户描述的概念：
- 是一个"系统"或"体系"
- 可以分解为若干独立模块
- 各模块有明确的边界

**不适用情况**：
- 概念本身就是单一要素
- 各要素之间边界模糊
- 难以区分独立的执行模块

---

## 二、拆解方法

### 2.1 要素识别

**核心问题**：构成这个概念的最小单元是什么？

**拆解角度**：
1. **按流程拆解**：输入→处理→输出
2. **按模块拆解**：功能A、功能B、功能C
3. **按角色拆解**：发起者、执行者、接收者
4. **按资源拆解**：人、财、物、信息

## 三、拆解步骤

### Step 1: 明确概念边界
（这个概念包含哪些范围，不包含哪些）

### Step 2: 识别核心要素
（通过拆解角度找出核心要素）

### Step 3: 定义每个要素
（用一句话明确每个要素是什么）

### Step 4: 确定要素关系
（各要素之间的关联和依赖）

---

## 四、案例示范

### 案例：内容运营体系

**要素识别**：
| 要素 | 定义 | 核心职责 |
|------|------|----------|
| 选题 | 确定要生产什么内容 | 需求分析、热点追踪 |
| 创作 | 生产内容 | 写作、设计、制作 |
| 编辑 | 把控内容质量 | 审核、修改、优化 |
| 分发 | 让内容触达用户 | 渠道选择、推送策略 |

**要素关系**：
```
选题 → 创作 → 编辑 → 分发
         ↑________________↓（反馈循环）
```

---

## 五、注意事项

1. **要素数量适中**：一般3-7个核心要素
2. **要素独立完整**：每个要素可独立执行
3. **边界清晰**：要素之间无重叠
4. **关系明确**：标注要素之间的依赖
