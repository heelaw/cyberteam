# 需求分类决策树

## 快速判断流程

```
开始
  │
  ▼
这个功能用户能明显感知到吗？
  │
  ├─否─→ I (无差异属性)
  │
  └─是─→ 这个功能是产品"必须有"的吗？
          │
          ├─是─→ 用户会因为有这个功能而更满意吗？
          │       │
          │       ├─是─→ A (兴奋属性)
          │       │       (超出预期的惊喜)
          │       │
          │       └─否─→ M (必备属性)
          │               (没有会极度不满)
          │
          └─否─→ 没有这个功能用户会不满吗？
                  │
                  ├─是─→ O (期望属性)
                  │       (越有越满意，越没有越不满)
                  │
                  └─否─→ 有这个功能用户会不满吗？
                          │
                          ├─是─→ R (反向属性)
                          │       (建议移除或重新设计)
                          │
                          └─否─→ I (无差异属性)
```

---

## 简化判断题

### 判断1: 基础判断

| 问题 | 选项 | 结论 |
|------|------|------|
| 这个功能是同类型产品的标配吗？ | 是 | M |
| | 否 | 继续判断 |

### 判断2: 必要性判断

| 问题 | 选项 | 结论 |
|------|------|------|
| 如果没有这个功能，用户会流失吗？ | 是 | M |
| | 不确定 | 继续判断 |

### 判断3: 满意度判断

| 问题 | 选项 | 结论 |
|------|------|------|
| 有这个功能，用户满意度会明显提升吗？ | 是 | 继续判断 |
| | 否 | I |

### 判断4: 惊喜判断

| 问题 | 选项 | 结论 |
|------|------|------|
| 这个功能是用户"没想到但很惊喜"的吗？ | 是 | A |
| | 否 | O |

## 各类需求特征对比

### M必备属性特征

| 特征 | 描述 |
|------|------|
| 门槛性 | 是产品/服务的基本门槛 |
| 底线性 | 没有用户体验会崩溃 |
| 习惯性 | 用户认为"理所当然应该有" |
| 隐蔽性 | 用户不会主动提出 |

**典型案例**:
- 手机: 能打电话、发短信
- 电商: 能下单支付
- 银行App: 账户安全

### O期望属性特征

| 特征 | 描述 |
|------|------|
| 线性关系 | 满足程度与满意度呈线性关系 |
| 竞争性 | 竞品也有，差异在于做得好不好 |
| 可衡量性 | 有明确的质量好坏标准 |
| 可感知性 | 用户能明显感知到好坏 |

**典型案例**:
- 手机的续航、拍照质量
- 电商的物流速度、客服响应

### A兴奋属性特征

| 特征 | 描述 |
|------|------|
| 创新性 | 通常是创新功能 |
| 惊喜性 | 用户没想到但很喜欢 |
| 非预期性 | 用户不会主动要求 |
| 传播性 | 用户会主动分享 |

**典型案例**:
- iPhone初代的多点触控
- 微信的红包功能
- 抖音的沉浸式体验

### I无差异属性特征

| 特征 | 描述 |
|------|------|
| 边缘性 | 功能本身对用户价值不大 |
| 可替代性 | 有无不影响核心体验 |
| 过度性 | 可能是过度开发的功能 |

**典型案例**:
- App的启动画面动画
- 过多的设置选项
- 冗余的辅助功能

### R反向属性特征

| 特征 | 描述 |
|------|------|
| 干扰性 | 功能干扰了正常体验 |
| 强制性 | 功能违背用户意愿 |
| 复杂性 | 增加了用户的认知负担 |

**典型案例**:
- 无法跳过的广告
- 强制注册才能使用
- 过多且无意义的推送

## 快速分类工作表

```markdown
## 功能快速分类

功能名称: _______________

### Step 1: 基础判断
- 竞品普遍有这个功能吗？ 是/否
- 这个功能用户能直接感知吗？ 是/否

### Step 2: 如果没有...
- 没有这个功能，用户会流失吗？ 是/否
- 没有这个功能，用户会极度不满吗？ 是/否

### Step 3: 如果有...
- 有这个功能，用户满意度会明显提升吗？ 是/否
- 这个功能是"超出预期"的惊喜吗？ 是/否

### Step 4: 综合判断
根据以上问答，初步判断类别: M / O / A / I / R

### Step 5: 交叉验证
- 用户访谈中用户提过吗？如果多次被提→可能是M或O
- 用户主动要求过吗？如果是→更可能是M或O
- 用户知道这个功能存在吗？如果不知道→可能是I或A
```

## 常见误区

### 误区1: 把O当M

```
❌ 误: 用户说想要快速物流，这是必备的
✓ 正: 快速物流是期望属性，没有会不满但不会崩溃
```

### 误区2: 把A当O

```
❌ 误: 个性化推荐做好了是应该的
✓ 正: 个性化推荐是兴奋属性，没有也基本满意
```

### 误区3: 忽视M

```
❌ 误: 必备功能太基础，先做O和A
✓ 正: M没做好，用户会直接流失
```

### 误区4: 过度创新A

```
❌ 误: 只做兴奋属性，忽视基础体验
✓ 正: M是底线，先保证底线再追求惊喜
```
