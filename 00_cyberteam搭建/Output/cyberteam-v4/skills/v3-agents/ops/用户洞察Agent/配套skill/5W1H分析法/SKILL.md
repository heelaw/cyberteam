# 5W1H分析法

## 核心定义

5W1H分析法是一种六要素分析法，通过 What（是什么）、Why（为什么）、Who（是谁）、When（何时）、Where（何地）、How（如何）六个维度全面理解问题，确保分析全面、系统、不遗漏。

## 六要素定义

| 要素 | 定义 | 分析问题 |
|------|------|---------|
| What | 发生了什么问题 | 识别核心问题是什么 |
| Why | 为什么会发生 | 探究问题根因 |
| Who | 谁受到了影响 | 明确目标用户群体 |
| When | 什么时候发生 | 确定时间节点和周期 |
| Where | 在哪里发生 | 定位发生场景和渠道 |
| How | 怎么发生的 | 还原过程和机制 |

## 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| 要素信息不足 | 基于现有信息分析，标注待补充项 |
| Why分析困难 | 提出多个假设，标注需验证 |
| 用户群体泛化 | 引导用户细分用户群体 |
| 时间模糊 | 明确具体时间节点

## Critical Rules

### 必须遵守（5条）

1. **要素完整性** - 每个分析必须覆盖全部6个要素，不能遗漏
2. **因果逻辑性** - Why的分析必须有数据或用户反馈支撑
3. **群体区分性** - Who必须区分核心用户和边缘用户
4. **时间敏感性** - When必须明确是突发还是渐进式问题
5. **场景具体性** - Where必须具体到具体渠道或页面

### 禁止行为（5条）

1. **禁止要素缺失** - 不能跳过任何一个要素
2. **禁止主观臆断** - Why的分析不能仅凭假设必须有依据
3. **禁止泛化用户** - Who的分析不能笼统说"所有用户"
4. **禁止模糊时间** - When不能说"最近"必须具体
5. **禁止脱离场景** - Where不能说"线上"必须具体

## Workflow Steps

### Step 1: 明确分析问题

**输入**: 用户描述的业务问题

**处理**:
- 确认核心问题是什么（What）
- 与用户确认分析边界
- 确定分析目标

**输出**:
```json
{
  "核心问题": "用户下单转化率下降",
  "问题定义": "从15%下降到8%",
  "分析目标": "找出转化下降的根本原因",
  "分析边界": "聚焦App端渠道"
}
```

**成功标准**: 用户确认核心问题清晰、分析边界明确

---

### Step 2: 分析What（是什么）

**输入**: 核心问题

**处理**:
- 量化问题的具体表现
- 明确受影响的指标
- 确定问题的范围和程度

**输出**:
```json
{
  "What": {
    "问题": "下单转化率下降",
    "具体表现": "从15%下降到8%，降幅47%",
    "受影响指标": ["浏览-下单转化率", "加购-下单转化率"],
    "影响范围": "全品类商品"
  }
}
```

**成功标准**: 问题描述具体、可量化、有数据支撑

---

### Step 3: 分析Why（为什么）

**输入**: What的分析结果

**处理**:
- 收集用户反馈和行为数据
- 分析可能的根因假设
- 验证假设是否成立

**输出**:
```json
{
  "Why": {
    "根因假设": [
      {
        "假设": "价格敏感用户流失",
        "证据": "客单价下降20元、促销活动参与率下降",
        "置信度": "高"
      },
      {
        "假设": "竞品分流",
        "证据": "竞品同期做大幅促销",
        "置信度": "中"
      },
      {
        "假设": "用户体验下降",
        "证据": "页面加载时间增加2秒",
        "置信度": "中"
      }
    ],
    "核心结论": "价格和体验是主要因素"
  }
}
```

**成功标准**: 每个根因都有数据支撑，置信度评估合理

---

### Step 4: 分析Who/When/Where（谁/何时/何地）

**输入**: 问题背景

**处理**:
- 识别受影响的核心用户群体
- 确定问题发生的时间特征
- 定位问题发生的具体场景

**输出**:
```json
{
  "Who": {
    "核心受损群体": "中等消费能力用户（月均消费500-1500元）",
    "群体特征": "价格敏感度高、活跃度中等",
    "占比": "流失用户中占60%"
  },
  "When": {
    "问题起始": "双十一后开始",
    "时间特征": "工作日下午3-5点下单减少",
    "周期": "已持续3周"
  },
  "Where": {
    "主要场景": "App端商品详情页",
    "次要场景": "购物车页",
    "渠道分布": "App占75%、小程序占20%、PC占5%"
  }
}
```

**成功标准**: 三要素分析具体、相互印证

---

### Step 5: 分析How（怎么发生的）

**输入**: 前五个要素的分析结果

**处理**:
- 还原问题的发生过程
- 识别关键转化节点
- 确定流失路径

**输出**:
```json
{
  "How": {
    "发生过程": "浏览商品→加入购物车→浏览对比→放弃下单",
    "关键流失节点": [
      {
        "环节": "加购到提交订单",
        "流失率": "35%",
        "原因": "运费不透明"
      },
      {
        "环节": "提交订单到支付",
        "流失率": "20%",
        "原因": "支付方式不熟悉"
      }
    ],
    "流失路径": "主要在支付环节流失"
  }
}
```

**成功标准**: 过程还原清晰、节点定位准确

---

### Step 6: 输出综合结论

**输入**: 六个要素的完整分析

**处理**:
- 归纳核心洞察
- 给出优先级排序的问题原因
- 提出针对性的解决方案建议

**输出**:
```markdown
# 5W1H分析报告：下单转化率下降

## 综合结论

| 要素 | 核心发现 | 优先级 |
|------|---------|-------|
| What | 转化率从15%降至8% | - |
| Why | 价格敏感+体验下降 | P0 |
| Who | 中等消费力用户流失最多 | P0 |
| When | 双十一后开始，持续3周 | P1 |
| Where | App端商品详情页流失最大 | P0 |
| How | 支付环节流失最严重 | P0 |

## 根因排序

1. **P0 - 运费不透明**（Where+How）
   - 证据：加购到下单流失35%集中在运费确认环节
   - 建议：优化运费展示策略

2. **P0 - 中等用户价格敏感**（Who+Why）
   - 证据：客单价下降20元、此类用户流失60%
   - 建议：针对性优惠券策略

3. **P1 - 页面加载慢**（Why）
   - 证据：加载时间增加2秒
   - 建议：性能优化

## 立即行动

1. 运费透明化改造（本周启动）
2. 针对中等用户发专属优惠券（下周）
3. 页面性能优化（下月）
```

**成功标准**: 结论明确、优先级清晰、建议可落地
