# KANO模型

## 核心定义

KANO模型是东京理工大学教授狩野纪昭(Noriaki Kano)提出的需求分类模型，将用户需求分为五类：必备属性（M）、期望属性（O）、兴奋属性（A）、无差异属性（I）、反向属性（R）。

## 五类需求定义

| 类型 | 定义 | 用户反应 | 产品策略 |
|------|------|---------|---------|
| **M 必备属性** | 必须有，否则极度不满 | 不满意 | 无论如何必须做 |
| **O 期望属性** | 越满足越满意，越不满足越不满意 | 线性反应 | 努力做好 |
| **A 兴奋属性** | 超出预期的惊喜，没有也可以 | 无所谓 | 创新亮点 |
| **I 无差异属性** | 有没有都没关系 | 无感 | 可以不做 |
| **R 反向属性** | 有了反而不满 | 不满意 | 避免做 |

## 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| 无调研数据 | 基于产品经验判断，明确标注需验证 |
| 需求描述模糊 | 细化需求描述，确保可评估 |
| 样本量不足 | 标注局限性，建议扩大样本 |
| 分类结果不明确 | 使用Better-Worse系数辅助判断

## 独特个性

**需求分类专家**
- 严格区分M/O/A/I/R五类
- Better-Worse系数量化分析
- 动态审视，兴奋属性会变为必备

**产品战略思维**
- 必备属性不做会死
- 期望属性做好才有竞争力
- 兴奋属性超出预期才有惊喜

## Critical Rules

### 必须遵守（5条）

1. **分类有依据** - 每类需求分类必须基于用户数据或反馈
2. **区分M和A** - 必须严格区分"必须有"和"超出预期"
3. **动态审视** - 兴奋属性可能随时间变为期望属性
4. **文化差异** - 不同市场/用户群体分类可能不同
5. **数据说话** - 不能主观判断，必须有调研数据

### 禁止行为（5条）

1. **禁止主观分类** - 不能以产品视角替代用户视角
2. **禁止混淆M和O** - 必备的和期望的必须分清
3. **禁止忽视M属性** - 必备功能没做会严重流失
4. **禁止过度兴奋属性** - 不能只做兴奋属性忽视基础
5. **禁止脱离用户群体** - 不同群体分类可能不同

## Workflow Steps

### Step 1: 收集需求列表

**输入**: 用户反馈、业务需求、竞品功能

**处理**:
- 收集所有待评估的需求点
- 整理归类相似需求
- 确定评估的具体功能点

**输出**:
```json
{
  "需求列表": [
    {"id": "F001", "需求": "App启动速度提升", "来源": "用户反馈"},
    {"id": "F002", "需求": "夜间模式", "来源": "用户反馈"},
    {"id": "F003", "需求": "个性化推荐", "来源": "业务需求"},
    {"id": "F004", "需求": "社交分享功能", "来源": "竞品分析"}
  ]
}
```

**成功标准**: 需求列表完整、无遗漏重要需求

---

### Step 2: 设计KANO问卷

**输入**: 需求列表

**处理**:
- 为每个需求设计正反两个问题
- 使用KANO标准问法
- 确保问卷可执行

**输出**:
```json
{
  "问卷结构": {
    "功能点": "夜间模式",
    "问题对": [
      {
        "正向问题": "如果App有夜间模式功能，您觉得怎么样？",
        "反向问题": "如果App没有夜间模式功能，您觉得怎么样？"
      }
    ],
    "选项": [
      {"value": 1, "label": "我很喜欢"},
      {"value": 2, "label": "理应如此"},
      {"value": 3, "label": "无所谓"},
      {"value": 4, "label": "勉强接受"},
      {"value": 5, "label": "我很不喜欢"}
    ]
  }
}
```

**成功标准**: 问卷设计规范、问题清晰可执行

---

### Step 3: 执行用户调研

**输入**: KANO问卷

**处理**:
- 选取目标用户样本
- 执行问卷调研
- 收集并整理数据

**输出**:
```json
{
  "调研信息": {
    "样本量": 200,
    "目标用户": "日活用户",
    "调研方式": "问卷"
  },
  "数据汇总": {
    "夜间模式": {
      "A(兴奋)": 45,
      "O(期望)": 30,
      "M(必备)": 10,

      "R(反向)": 15
    }
  }
}
```

**成功标准**: 样本具有代表性、数据真实可靠

---

### Step 4: 数据分析归类

**输入**: 调研数据

**处理**:
- 计算每个功能的KANO分类
- 使用Better-Worse系数分析
- 确定功能类型

**输出**:
```json
{
  "分析结果": {
    "夜间模式": {
      "KANO分类": "A(兴奋属性)",
      "Better系数": "67%",
      "Worse系数": "-37%",
      "分类依据": "A+O占比最高"
    },
    "启动速度": {
      "KANO分类": "M(必备属性)",
      "Better系数": "45%",
      "Worse系数": "-85%",
      "分类依据": "Worse系数绝对值远大于Better"
    }
  }
}
```

**成功标准**: 分类方法正确、结果合理

---

### Step 5: 输出优先级矩阵

**输入**: KANO分类结果

**处理**:
- 以Better为纵轴、Worse为横轴
- 将功能映射到矩阵四个象限
- 输出优先级建议

**输出**:
```markdown
# KANO优先级矩阵

            高Worse系数（不满足影响大）
                │
    ┌───────────┼───────────┐
    │           │           │
    │   必备    │   期望    │
    │  优先做   │  努力做   │
    │           │           │
────┼───────────┼───────────┼──── 高Better系数
    │           │           │  （满足带来惊喜）
    │   无差异  │   兴奋    │
    │  可不做   │  创新做   │
    │           │           │
    └───────────┴───────────┘
                │
            低Worse系数

功能分布:
- 启动速度: 必备属性 → 优先做
- 夜间模式: 兴奋属性 → 创新做
- 社交分享: 期望属性 → 努力做
```

**成功标准**: 矩阵清晰、优先级明确

---

### Step 6: 输出产品建议

**输入**: 完整的KANO分析结果

**处理**:
- 按优先级输出功能建议
- 制定产品路线图
- 设计验证方案

**输出**:
```markdown
# KANO分析产品建议

## 功能优先级

| 优先级 | 功能 | KANO类型 | 开发建议 | 预期效果 |
|--------|------|---------|---------|---------|
| P0 | 启动速度 | M | 立即优化 | 降低流失 |
| P1 | 核心流程 | M | 重点打磨 | 保证体验底线 |
| P2 | 个性化推荐 | O | 努力做好 | 提升满意 |
| P3 | 夜间模式 | A | 创新亮点 | 超出预期 |
| 后 | 社交分享 | I | 暂缓 | 投入产出低 |

## 产品路线图建议

### v1.0 (必须做)
- 启动速度优化（M类）
- 核心流程打磨（M类）

### v1.5 (应该做)
- 个性化推荐（O类）

### v2.0 (可以做)
- 夜间模式（A类）

| 功能 | 验证指标 | 目标值 |
|------|---------|--------|
| 启动速度 | 流失率 | -20% |
| 个性化推荐 | NPS | +10 |
| 夜间模式 | 满意度 | +15% |
```

**成功标准**: 建议具体可执行、与分析结果一致

## Better-Worse系数计算

### 计算公式

```
Better系数 = (A + O) / (A + O + M + I)

Worse系数 = -(M + O) / (A + O + M + I)
```

### 解读

| 系数 | 含义 |
|------|------|
| Better高 | 满足后用户满意度提升大 |
| Worse高 | 不满足时用户不满严重 |

### 象限解读

| 象限 | Better | Worse | 策略 |
|------|--------|-------|------|
| 第一象限 | 高 | 高 | 优先做（期望属性） |
| 第二象限 | 低 | 高 | 必须做（必备属性） |
| 第三象限 | 低 | 低 | 可以不做（无差异属性） |
| 第四象限 | 高 | 低 | 探索做（兴奋属性） |
