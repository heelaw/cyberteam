# 思维天团输出层级体系

> **版本**: v1.0
> **目的**: 定义不同场景下应该输出什么层级的内容
> **核心原则**: 不是每个问题都需要完整报告，合适的才是最好的

---

## 一、层级体系概述

### 1.1 为什么要分层？

当前问题：
- 所有输出都是"完整报告"格式
- 简单问题被过度分析（杀鸡用牛刀）
- 缺乏"快速响应"模式
- 无法满足不同场景需求（发微博 vs 写报告 vs 内部讨论）

### 1.2 四级体系

```
┌─────────────────────────────────────────────────────────────┐
│  L4 完整报告                                                │
│  场景：深度分析、正式交付、客户报告                          │
│  长度：800-1200字                                           │
│  结构：完整框架展开 + 数据支撑 + 行动建议                    │
├─────────────────────────────────────────────────────────────┤
│  L3 标准输出                                                │
│  场景：内部汇报、中等复杂度问题、COE审核                     │
│  长度：400-600字                                             │
│  结构：问题→分析→结论→建议                                   │
├─────────────────────────────────────────────────────────────┤
│  L2 短分析                                                  │
│  场景：快速决策、初步讨论、社群分享                          │
│  长度：150-300字                                             │
│  结构：判断→核心逻辑→行动要点                                │
├─────────────────────────────────────────────────────────────┤
│  L1 微洞察                                                  │
│  场景：微博/朋友圈、即兴发言、快速判断                        │
│  长度：50-100字                                              │
│  结构：1个核心观点 + 1个支撑                                 │
└─────────────────────────────────────────────────────────────┘
```

## 二、层级详细定义

### 2.1 L1 微洞察

**定位**：一句话级别的判断或洞察

| 属性 | 内容 |
|------|------|
| **长度** | 50-100字 |
| **核心特征** | 1个核心观点 + 1个支撑逻辑 |
| **适用场景** | 微博/朋友圈、即时讨论、快速判断 |
| **禁止场景** | 需要深度分析的问题、客户交付 |
| **格式** | 自然段落，不用bullet |

**输出格式**：
```
[emoji] [核心判断，20字以内]
[支撑逻辑/数据，1-2句]
```

**示例**：

```
🎯 卡尼曼视角：
"面对不确定性，人会高估小概率事件。
A选项的『可能血本无归』比『80%概率赚钱』更刺激你的系统1。
——这就是为什么多数人宁愿赌一把也不选确定的收益。"

📊 SWOT视角：
"拼多多能成，核心不是『便宜』，而是『占便宜的感觉』。
同样的商品，标价从29.9划到19.9，比直接标19.9多赚50%。"
```

### 2.2 L2 短分析

**定位**：公众号文章级别，一次性可读完

| 属性 | 内容 |
|------|------|
| **长度** | 150-300字 |
| **核心特征** | 问题→分析→判断，完整但不冗长 |
| **适用场景** | 公众号/社群、初步讨论、快速决策参考 |
| **禁止场景** | 需要多轮辩论的问题、正式客户交付 |
| **格式** | 1-2段文字 + 可选1-2个emoji |

**输出格式**：
```
[emoji] [核心判断/结论]
[分析逻辑，2-4句]
[为什么这个判断是对的，1-2句]
→ [一句话行动建议]
```

**示例**：

```
🎯 选B，但不是因为"B更好"。

你的纠结本质上是"确定的平庸 vs 不确定的成长"的选择。
系统1讨厌B的不确定性——它把"可能失败"放大成"必然失败"，
同时把"稳定"等同于"安全"。

但数据不说谎：
国企5年期望收益 = 有限升职 + 有限加薪 = 天花板明确
创业公司5年期望收益 = 0.2×(爆发) + 0.8×(快速成长) = 上限未知

→ 如果你还能承受2年低成本生活，选B。
```

### 2.3 L3 标准输出

**定位**：正式内部报告，可用于COE汇报

| 属性 | 内容 |
|------|------|
| **长度** | 400-600字 |
| **核心特征** | 完整分析框架 + 洞察 + 建议 |
| **适用场景** | 内部汇报、中等复杂度问题、COE审核参考 |
| **禁止场景** | 极简单问题（杀鸡用牛刀） |
| **格式** | 多段 + emoji锚点 + 少量bullet |

**输出格式**：
```
[emoji] [问题定性/核心判断]

[背景：为什么这个问题重要，2-3句]
[分析框架应用，3-5句]
  • [维度1发现]
  • [维度2发现]
  • [维度3发现]
[反直觉洞察，1-2句]
[结论，1句]
→ [可执行建议，2-3条]
```

**示例**：

```
🎯 这个选择本质上是"风险偏好 vs 成长空间"的博弈

【背景】
职业选择是人生最重要的复利决策之一。
选国企不是选择稳定，而是选择"放弃非线性增长的可能性"。
选创业不是选择冒险，而是选择"用确定性时间换潜在指数回报"。

【分析】
从卡尼曼视角扫描：
⚠️ 损失厌恶：你对"失去稳定收入"的恐惧被系统1放大了3倍
⚠️ 锚定效应："国企=稳定"这个等式是被社会建构的，不是客观事实
⚠️ 现状偏见："一直这样"是你给自己的最贵借口

从机会成本看，国企的真正成本是"你在创业公司能获得的成长"——
这个成本被你的系统1悄悄隐藏了。

【洞察】
最大的风险不是选B，而是选A后"后悔没试过"的长期成本。
统计上，35岁前没创过业的人，35岁后创业成功率开始下降——
不是因为能力，而是因为锚定效应已经定型。

→ 建议：给自己设一个2年窗口期，用"学到了什么"而非"赚了多少"评估。
```

### 2.4 L4 完整报告

**定位**：深度分析报告，可用于正式客户交付

| 属性 | 内容 |
|------|------|
| **长度** | 800-1200字 |
| **核心特征** | 完整框架 + 多维度数据 + 风险预案 |
| **适用场景** | 深度分析、正式交付、战略规划 |
| **禁止场景** | 简单问题、时间紧迫场景 |
| **格式** | 完整报告结构，包含executive summary |

**输出格式**：
```
═══════════════════════════════
[专家名称] · [问题类型] 分析报告
═══════════════════════════════

【Executive Summary】(50字以内)
[核心结论，1-2句]

【问题定义】
[重构后的问题，2-3句]
[边界条件，1-2句]

【分析框架】
[框架选择理由，1句]
[完整分析，5-8句]
  • [维度1：数据/事实]
  • [维度2：风险/障碍]
  • [维度3：机会/价值]
  • [维度4：替代方案]

【洞察】(必须包含"反直觉"内容)
[洞察1：多数人没想到的]
[洞察2：如果假设不成立的场景]

【建议】
→ [短期行动，1-2条]
→ [中期行动，1-2条]
→ [止损条件，1-2条]

【风险预案】
如果[假设X不成立]，则[建议Y]
```

## 三、层级选择指南

### 3.1 选择决策树

```
问题来了 →
  ├─ 需要即时判断？ → L1
  ├─ 需要快速结论、可以延后深入？ → L2
  ├─ 需要正式汇报、涉及多方决策？ → L3
  └─ 需要深度分析、影响战略？ → L4
```

### 3.2 按场景选择

| 场景 | 推荐层级 | 理由 |
|------|---------|------|
| 朋友圈/微博 | L1 | 注意力短，需要快速抓住眼球 |
| 社群讨论 | L1-L2 | 互动场景，需要快速响应 |
| 公众号推文 | L2 | 可深度阅读但要控制长度 |
| 内部讨论 | L2-L3 | 需要一定深度但不需正式报告 |
| 初步方案 | L3 | 需要完整分析但还在迭代中 |
| 正式汇报 | L3-L4 | 需要完整框架支撑决策 |
| 客户交付 | L4 | 需要完整报告和风险预案 |
| CEO决策 | L3 | 快速抓到重点就行，不要废话 |

### 3.3 按问题复杂度选择

| 问题类型 | 复杂度 | 推荐层级 |
|---------|--------|---------|
| "选A还是B？" | 低 | L1-L2 |
| "这个方向对吗？" | 中 | L2-L3 |
| "要不要进入新市场？" | 高 | L3-L4 |
| "未来5年战略怎么定？" | 极高 | L4 |

## 四、层级转换规则

### 4.1 何时需要升级？

```
触发升级的条件：
□ 问题复杂度超出当前层级能处理的范围
□ 接收方要求更深入的分析
□ 发现需要多轮迭代才能得出结论
□ 涉及重大决策，需要完整支撑
□ 需要与其他专家的输出合并
```

### 4.2 升级流程

```
L1 → L2：
触发条件：需要进一步解释或补充逻辑
操作：保留核心判断，展开分析逻辑
时间增量：+5分钟

L2 → L3：
触发条件：需要提交正式汇报或审核
操作：按L3模板重构，增加维度覆盖
时间增量：+15分钟

L3 → L4：
触发条件：需要深度战略分析或客户交付
操作：按L4模板重构，增加风险预案
时间增量：+30分钟
```

### 4.3 降级规则

```
降级是允许的，但不推荐：
□ 如果问题被证明比预期简单 → 可以降级
□ 如果时间不够 → 明确告知对方"L3降级到L2"，不能偷偷降级
□ 降级时必须说明："核心判断不变，但分析深度缩减，原因：[时间/复杂度]"
```

## 五、层级与Gate的关系

```
层级定义了"输出多长"
Gate定义了"输出多好"

L1 → Gate 1(70分) 即可通过
L2 → Gate 1 + Gate 2(80分) 才可通过
L3 → Gate 1 + Gate 2 + Gate 3(70分) 才可通过
L4 → 所有Gate必须通过，且Gate3>=85
```

## 六、层级与专家类型的匹配

### 6.1 各专家的典型输出层级

| 专家 | 典型层级 | 特殊说明 |
|------|---------|---------|
| 卡尼曼决策 | L2-L3 | 决策问题通常L2足够，重大决策L3 |
| 六顶思考帽 | L2-L3 | 六帽扫描适合L2-L3，太短失去意义 |
| 第一性原理 | L3-L4 | 需要层层递进，L1-L2无法承载 |
| Pre-Mortem | L3 | 风险预案需要完整展开 |
| 逆向思维 | L2 | 一个反直觉观点L2足够 |
| 博弈论 | L3-L4 | 多方博弈需要完整推演 |

### 6.2 专家层级偏好设置

每个专家的AGENT.md应该声明：

```
## 典型输出层级
- 主要层级：L2-L3
- 最长层级：L4
- 最短层级：L1（仅用于快速判断）
```

## 七、多专家协作时的层级管理

### 7.1 协作时的层级规则

```
单一专家 → 输出对应层级
多专家协作 → 取最高层级作为基准
如果专家A输出L2，专家B输出L3 → 合并后应为L3
```

### 7.2 协作输出模板

```
## 多专家协作输出模板

【问题】[问题描述]

【专家A - L3】[输出内容]

【专家B - L2】[输出内容]

【综合结论 - L2/L3】
[综合各专家观点，给出统一结论]
[如果结论冲突，说明分歧点]

→ [统一建议]
```
