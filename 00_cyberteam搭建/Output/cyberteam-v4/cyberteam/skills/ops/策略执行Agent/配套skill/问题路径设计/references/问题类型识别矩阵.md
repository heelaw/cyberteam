# 问题类型识别矩阵

## 使用说明

在设计执行路径前，首先需要准确识别问题类型。本矩阵帮助快速判断问题属于A类还是B类。

---

## 一、识别维度

### 1.1 六大识别维度

| 维度 | A类问题特征 | B类问题特征 |
|------|-------------|-------------|
| **目标明确性** | 目标模糊、开放，难以量化 | 目标清晰、可量化 |
| **周期长短** | 周期长，通常>3个月 | 周期短，通常<3个月 |
| **成功标准** | 定性为主，成功难以精确衡量 | 定量为主，成功可以精确衡量 |
| **实现路径** | 路径不清晰，需要探索 | 路径相对清晰，可设计 |
| **风险类型** | 方向风险（是否做正确的事） | 执行风险（是否把事做正确） |
| **调整频率** | 需要频繁调整和迭代 | 按计划执行，减少调整 |

### 1.2 典型场景对照

| 场景 | 类型 | 说明 |
|------|------|------|
| 品牌建设 | A类 | 心智建立难以量化，路径需要探索 |
| 用户增长 | A/B混合 | 获取可量化（A），留存运营偏B |
| 产品打磨 | A类 | 用户需求不明确，需要探索 |
| 活动策划 | B类 | 目标明确，周期短 |
| 转化优化 | B类 | 目标明确，可分解 |
| 团队建设 | A类 | 能力提升难以量化 |
| 市场开拓 | A类 | 市场反应需要探索 |
| 项目交付 | B类 | 明确交付物和时间 |

## 二、快速判断流程

### 2.1 第一步：问目标

```
目标是否能用数字明确描述？
├── 能 → 跳转到第2步
└── 不能 → A类问题
```

### 2.2 第二步：问周期

```
距离截止日期是否>3个月？
├── 是 → 跳转到第3步
└── 否 → B类问题
```

### 2.3 第三步：问路径

```
实现路径是否清晰可设计？
├── 是 → B类问题
└── 否/不确定 → A类问题
```

---

## 三、复合问题识别

### 3.1 问题分解原则

很多问题包含A类和B类两部分，需要分解处理：

| 复合问题 | A类部分 | B类部分 |
|----------|---------|---------|
| "如何提升用户留存" | 留存问题根因分析 | 针对根因的具体措施 |
| "如何做好品牌传播" | 品牌定位和传播策略探索 | 具体传播活动执行 |
| "如何提升产品竞争力" | 用户需求和竞争分析 | 具体功能开发 |

### 3.2 处理策略

```
复合问题 → 分解为A类部分 + B类部分
              ↓
        先处理A类部分（分析+探索）
              ↓
        再处理B类部分（执行+交付）
```

---

## 四、类型转换

### 4.1 A类转B类

当A类问题经过探索后，可以转为B类执行：

**转换条件**：
- 核心假设已验证
- 成功路径已明确
- 目标可以量化

**转换时机**：
- 探索期完成并确认方向后
- 验证期成功并准备扩展前

### 4.2 B类转A类

当B类问题遇到方向性问题时，可能需要转A类处理：

**转换信号**：
- 执行遇到根本性障碍
- 目标无法达成
- 外部环境重大变化

**处理方式**：
- 暂停当前执行
- 重新分析问题本质
- 调整方向后再执行

## 六、常见误区

### 6.1 A类问题误判为B类

| 误区 | 后果 | 正确做法 |
|------|------|----------|
| 把"品牌知名"当成可量化目标 | 过度追求短期数字 | 先探索真正的品牌建设目标 |
| 把"用户信任"当成简单任务 | 执行流于表面 | 需要长期积累和验证 |
| 把"产品优化"当成纯执行任务 | 方向可能错误 | 先做用户需求分析 |

### 6.2 B类问题误判为A类

| 误区 | 后果 | 正确做法 |
|------|------|----------|
| 活动策划过度灵活 | 错失时机 | 明确目标后严格执行 |
| 转化优化过度分析 | 行动迟缓 | 小步快跑快速验证 |
| 项目交付频繁调整 | 难以交付 | 确定需求后按计划执行 |

### 6.3 混合问题不分解

| 误区 | 后果 | 正确做法 |
|------|------|----------|
| 不分解A/B部分 | 策略和执行混淆 | 先分析再执行 |
| 用A类方式处理B类 | 效率低下 | B类问题要强调执行 |
| 用B类方式处理A类 | 方向错误 | A类问题要强调探索 |

## 七、自检清单

### 7.1 问题类型判断自检

| 检查项 | 是 | 否 | 备注 |
|--------|---|---|------|
| 能用数字描述最终目标 | □ | □ | |
| 周期在3个月以内 | □ | □ | |
| 实现路径相对清晰 | □ | □ | |
| 成功标准可以量化 | □ | □ | |
| 执行过程不需要频繁调整 | □ | □ | |

**判断结果**：
- 4-5项为"是" → B类问题
- 0-1项为"是" → A类问题
- 2-3项为"是" → 需要进一步分析，可能需要分解

### 7.2 路径设计匹配自检

| 检查项 | A类路径 | B类路径 |
|--------|---------|---------|
| 强调目标分解 | ○ | ● |
| 强调持续迭代 | ● | ○ |
| 强调阶段评审 | ○ | ● |
| 强调风险预案 | ● | ● |
| 强调复盘机制 | ● | ● |

●=必须 ○=可选
