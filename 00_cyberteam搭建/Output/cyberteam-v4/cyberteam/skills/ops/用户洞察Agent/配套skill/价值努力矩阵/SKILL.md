# 价值努力矩阵

## 核心定义

价值努力矩阵（Value-Effort Matrix）是一种优先级排序工具，通过评估每个选项的价值（业务价值/用户价值）和努力（实现成本/开发工作量），帮助团队决定先做什么。

## 矩阵结构

| 维度 | 定义 | 评估标准 |
|------|------|---------|
| **价值** | 功能对用户或业务的帮助程度 | 高/中/低 |
| **努力** | 实现该功能需要的资源和成本 | 高/中/低 |

## 错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| 价值评估分歧 | 团队讨论达成共识，标注不同观点 |
| 努力估算困难 | 给出区间估算，明确不确定性 |
| 选项过多 | 分批评估，优先聚焦核心选项 |
| 四象限不均衡 | 检查评估标准，避免极端情况

---

## 独特个性

**优先级决策专家**
- 价值与努力双维度评估
- 四象限清晰，决策有据
- ROI导向，资源最优配置

**务实高效**
- 高价值低努力立即做
- 避免低价值高努力的陷阱
- 考虑依赖关系和风险

## 四象限定义

| 象限 | 价值 | 努力 | 策略 | 行动 |
|------|------|------|------|------|
| **做** | 高 | 低 | 立即做 | 最高优先级 |
| **计划** | 高 | 高 | 规划做 | 合理安排资源 |
| **委托** | 低 | 高 | 考虑外包 | 评估是否值得 |
| **消除** | 低 | 低 | 快速做或不做 | 最小投入或放弃 |

## Critical Rules

### 必须遵守（5条）

1. **价值主观但需共识** - 价值评估可能有主观成分，但必须团队达成共识
2. **努力需实际估算** - 不能拍脑袋，必须有实际工作量估算
3. **四象限必须填满** - 每个象限至少有一个选项（除非确实没有）
4. **定期重新评估** - 随着时间推移，价值和努力可能变化
5. **考虑依赖关系** - 有些选项可能依赖于其他选项先完成

### 禁止行为（5条）

1. **禁止只看价值** - 不能只做高价值的，忽视努力
2. **禁止只比价格** - 不能只选低努力的，忽视价值
3. **禁止静态评估** - 不能一次评估后不再更新
4. **禁止忽略风险** - 高努力选项可能有风险
5. **禁止平均主义** - 不能所有选项都放中间位置

## Workflow Steps

### Step 1: 收集待评估选项

**输入**: 需求列表、功能点、问题或解决方案

**处理**:
- 列出所有需要排序的选项
- 确认每个选项的清晰定义
- 去除明显不属于本范围的选项

**输出**:
```json
{
  "待评估选项": [
    {"id": "A", "名称": "用户登录优化", "来源": "用户反馈"},
    {"id": "B", "名称": "新用户引导", "来源": "产品设计"},
    {"id": "C", "名称": "数据导出功能", "来源": "客户需求"},
    {"id": "D", "名称": "界面风格升级", "来源": "运营需求"}
  ]
}
```

**成功标准**: 选项列表完整、无遗漏重要项

---

**处理**:

- 估算每个选项的相对工作量

### Step 4: 绘制价值努力矩阵

**输入**: 价值评估 + 努力评估

**处理**:
- 创建四象限矩阵
- 将每个选项放到对应位置
- 标注优先级

**输出**:
```markdown
# 价值努力矩阵

            高努力
              │
    ┌─────────┼─────────┐
    │         │         │
    │ 委托    │  计划   │
    │ C(数据导出)│ A+B(登录+引导)
    │         │         │
────┼─────────┼─────────┼──── 高价值
    │         │         │
    │ 消除    │  做     │
    │ D(界面升级)│ B(新用户引导)
    │         │         │
    └─────────┴─────────┘
              │
           低努力

优先级排序:
1. B (高价值+低努力) → 立即做
2. A (高价值+中努力) → 尽快做
3. C (中价值+高努力) → 考虑外包或延后
4. D (低价值+高努力) → 尽量不做
```

**成功标准**: 矩阵清晰直观，位置合理

---

### Step 5: 输出执行建议

**输入**: 价值努力矩阵

**处理**:
- 按象限给出行动建议
- 制定具体的执行计划
- 确定时间节点和责任人

**输出**:
```markdown
# 执行建议

## 立即做 (高价值+低努力)

| 功能 | 行动 | 工期 | 负责人 |
|------|------|------|--------|
| B-新用户引导 | 设计引导流程+开发 | 1周 | @产品A |

## 尽快做 (高价值+高努力)

| 功能 | 行动 | 工期 | 负责人 |
|------|------|------|--------|
| A-登录优化 | 重构登录流程 | 3周 | @后端B |

## 延后做 (中价值+高努力)

| 功能 | 行动 | 工期 | 建议 |
|------|------|------|------|
| C-数据导出 | 评估外包可能性 | 4周 | 评估后再定 |

## 尽量不做 (低价值+高努力)

| 功能 | 行动 | 建议 |
|------|------|------|
| D-界面升级 | 暂缓，纳入下版本规划 |

## Sprint建议

### Sprint 1 (2周)
- 新用户引导 (B)
- 登录优化前期准备

### Sprint 2 (2周)
- 登录优化上线 (A)
```

**成功标准**: 建议具体可执行，有明确时间表

## ROI估算模板

### 价值量化

```markdown
## 价值量化估算

| 功能 | 预期收益 | 收益类型 | 量化方式 |
|------|---------|---------|---------|
| 新用户引导 | 激活率+15% | 转化提升 | 估算LTV增加 |
| 登录优化 | 流失率-10% | 留存提升 | 估算DAU增加 |
```

### ROI计算

```markdown
## ROI估算

功能 = B (新用户引导)
- 预期价值: 激活率提升15%，预计月新增活跃用户2000
- 估算月价值: ¥50,000
- 开发成本: ¥20,000 (1周人力)
- ROI = (50,000 - 20,000) / 20,000 = 150%
- 建议: 立即做
```

## 典型案例

### 案例: App迭代优先级

**输入**: "有5个功能要做，怎么排优先级"

**分析**:
```
功能清单:
1. 登录优化 - 高价值/中努力
2. 新手引导 - 高价值/低努力
3. 数据导出 - 中价值/高努力
4. 界面改版 - 低价值/高努力
5. 分享功能 - 中价值/低努力

矩阵结果:
- 做: 新手引导、分享功能
- 计划: 登录优化
- 委托: 数据导出
- 消除: 界面改版

结论:
Sprint1: 新手引导 + 分享功能
Sprint2: 登录优化
```
