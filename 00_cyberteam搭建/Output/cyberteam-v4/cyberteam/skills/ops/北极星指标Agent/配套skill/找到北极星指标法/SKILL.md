## 错误处理

| 错误类型 | 典型表现 | 处理方式 |
|---------|---------|---------|
| 虚荣指标 | 选"看起来好看"的指标 | 必须回答"用户真的重视吗" |
| 照搬竞品 | 直接复制别人家的北极星 | 每个业务不一样 |
| 多中心 | 同时定多个"核心指标" | 一个时期只能有一个 |
| 滞后太长 | 用一年后的结果做指标 | 指标要有反馈周期 |
| 脱离用户 | 用业务指标替代价值指标 | 必须是用户价值导向 |

## 核心定义

北极星指标（North Star Metric）是业务中唯一最重要的指标，它能够清晰地反映用户从产品中获得的核心价值，是团队统一努力方向的核心。

## 北极星指标的特征

| 特征 | 说明 | 验证问题 |
|------|------|---------|
| **价值导向** | 直接反映用户获得的价值 | "用户重视这个吗？" |
| **可衡量** | 可以被精确测量 | "能拿到数据吗？" |
| **可驱动** | 团队行为能影响它 | "我们能做什么影响它？" |
| **线性相关** | 与业务增长正相关 | "指标涨，业务真的好吗？" |
| **单一聚焦** | 一个时期只有一个核心 | "团队能记住吗？" |

## Critical Rules

### 必须遵守（5条）

1. **价值第一** - 北极星指标必须直接反映用户价值
2. **业务适配** - 不同业务阶段需要不同的北极星指标
3. **数据可得** - 指标必须能够被测量和追踪
4. **团队可控** - 团队行为必须能影响该指标
5. **阶段适配** - 成长期和成熟期指标选择不同

### 禁止行为（5条）

1. **禁止虚荣指标** - 不能选择看起来好看但无实际价值的指标
2. **禁止照搬竞品** - 不能直接复制其他公司的北极星指标
3. **禁止多中心** - 一个时期只能有一个北极星指标
4. **禁止滞后指标** - 不能选择结果滞后太久的指标
5. **禁止脱离用户** - 不能用业务指标替代用户价值指标

## Workflow Steps

### Step 1: 确定业务阶段

**输入**: 业务类型、商业模式、当前发展阶段

**处理**:
- 判断业务处于哪个阶段（探索期/成长期/成熟期/衰退期）
- 识别当前阶段的核心任务
- 确定指标制定的重点

**输出**:
```json
{
  "业务阶段": "成长期",
  "阶段特征": "用户量快速增长、需明确增长模式",
  "阶段任务": "找到可复制的增长模式",
  "指标重点": "用户价值验证、增长效率"
}
```

**成功标准**: 阶段判断准确、任务明确

---

### Step 2: 识别用户价值

**输入**: 业务阶段 + 用户研究数据

**处理**:
- 明确产品为用户解决什么问题
- 识别用户使用产品的核心场景
- 找到用户获得价值的关键时刻
- 区分核心价值和附属功能

**输出**:
```json
{
  "核心价值": "帮助用户高效学习",
  "用户场景": ["主动学习时", "完成作业时", "获得反馈时"],
  "关键时刻": ["学完一个知识点", "独立完成作业", "获得老师批改"],
  "价值要素": ["学习效果", "时间成本", "学习动力"]
}
```

**成功标准**: 用户价值清晰、场景具体

---

### Step 3: 找到指标候选

**输入**: 用户价值分析结果

**处理**:
- 列出所有可能反映用户价值的指标
- 从多个维度评估每个指标
- 筛选出3-5个候选指标

**评估维度**:
| 维度 | 权重 | 评估问题 |
|------|------|---------|
| 相关性 | 30% | 与用户价值直接相关？ |
| 可衡量性 | 20% | 能拿到准确数据？ |
| 可驱动性 | 25% | 团队能影响它？ |
| 时效性 | 15% | 能快速反馈？ |
| 清晰度 | 10% | 团队能理解？ |

**输出**:
```json
{
  "候选指标": [
    {"指标": "作业提交率", "得分": 85, "优势": "强相关、可驱动", "劣势": "需区分科目"},
    {"指标": "课程完播率", "得分": 78, "优势": "数据准确、易衡量", "劣势": "不直接反映效果"},
    {"指标": "7日留存率", "得分": 72, "优势": "行业通用、可比", "劣势": "与价值关联弱"}
  ]
}
```

**成功标准**: 候选指标3-5个、评估维度清晰

---

### Step 4: 验证指标特性

**输入**: 候选指标列表

**处理**:
- 验证每个指标是否符合北极星特征
- 检查指标的历史数据趋势
- 评估指标的潜在负面激励
- 确认指标的可持续性

**验证清单**:
```
[ ] 指标是否反映用户价值？
[ ] 指标是否可被精确测量？
[ ] 团队行为是否能影响该指标？
[ ] 指标与业务增长是否正相关？
[ ] 是否有负面激励风险？
[ ] 数据是否能持续获取？
```

**输出**:
```json
{
  "指标验证": {
    "作业提交率": {
      "价值导向": true,
      "可衡量": true,
      "可驱动": true,
      "线性相关": true,
      "潜在风险": "可能刷题应付"
    }
  },
  "最终候选": ["作业提交率", "课程完播率"]
}
```

**成功标准**: 每个指标都通过特征验证

---

### Step 5: 确定北极星指标

**输入**: 指标验证结果

**处理**:
- 选择最符合特征的指标
- 明确指标的精确定义
- 确定指标的衡量方式
- 设定指标的目标值

**输出**:
```json
{
  "北极星指标": "核心课程作业提交率",

  "衡量方式": "按周统计、按科目汇总",
  "数据来源": "学习平台后台",
  "目标值": {
    "当前值": "65%",
    "下季度目标": "75%",
    "年度目标": "85%"
  },
  "选择理由": "直接反映学习投入、与用户价值强相关、团队可影响"
}
```

**成功标准**: 指标明确、定义清晰、目标可达成

### Step 6: 分解子指标

**输入**: 北极星指标

**处理**:
- 将北极星指标拆解为可直接追踪的子指标
- 建立指标之间的关联关系
- 确定各子指标的负责团队
- 设计数据追踪看板

**分解逻辑**:
```
北极星指标: 核心课程作业提交率
    │
    ├── 分子: 实际提交作业数
    │   ├── 各科目提交数
    │   ├── 各班级提交数
    │   └── 各老师催交效果
    │
    └── 分母: 应提交作业数
        ├── 实际上课人数
        ├── 课程作业数量
        └── 作业布置及时率
```

**输出**:
```markdown
# 北极星指标体系

## 北极星指标
- 核心课程作业提交率

## 一级子指标
| 指标 | 定义 | 负责团队 |
|------|------|---------|
| 实际提交数 | 每周提交作业总数 | 教学团队 |
| 应提交数 | 每周应提交作业总数 | 产品团队 |
| 完播率 | 课程视频完播比例 | 内容团队 |

## 二级子指标
| 指标 | 定义 | 负责团队 |
|------|------|---------|
| 各科目提交率 | 分科目统计 | 学科运营 |
| 各班级提交率 | 分班级统计 | 班主任 |

```

   │              │
   ↓              ↓

```
- 核心矛盾: 获客成本高 vs 续费率不稳定

Step 2: 识别价值
- 用户价值: 孩子学到编程能力、有成就感
- 关键时刻: 完成作品、比赛获奖、作品分享

Step 3: 找到候选
- 候选1: 完课率 (65分)
- 候选2: 作品提交率 (80分)
- 候选3: 续费率 (70分)
- 候选4: 体验课转化率 (55分)

Step 4: 验证特性
- 完课率: 可衡量但与价值关联中等
- 作品提交率: 强相关、可驱动、无负面风险
- 续费率: 滞后性强、周期长

Step 5: 确定北极星
推荐: 作品提交率
定义: 每月学员独立完成并提交的作品数 / 报名学员数

Step 6: 分解子指标
- 作品提交率 → 完课率 × 作业完成率 × 主动创作率
```

**结论**:
```
北极星指标: 学员月均作品提交数

理由:
1. 直接反映孩子学习投入和产出
2. 与"学到编程能力"强相关
3. 作品可展示、有成就感、驱动续费
4. 团队可影响(通过课程设计、老师催交)

子指标:
- 周完课率: 反映学习进度
- 作品获奖率: 反映学习效果
- 续费率: 反映商业转化
```
