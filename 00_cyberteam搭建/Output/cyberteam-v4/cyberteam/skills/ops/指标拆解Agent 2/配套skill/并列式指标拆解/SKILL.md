## 使用示例

### 示例1：收入结构拆解

**场景**：想了解公司收入有哪些构成

**对话**：
```
用户：我们公司有多个收入来源，但一直理不清到底有哪些，你能帮我拆解吗？

Skill输出：
## 拆解维度
建议按"收入类型"维度拆解

## 拆解公式
总收入 = 产品收入 + 服务收入 + 平台收入

## 子指标详情

| 收入类型 | 定义 | 占比 | 趋势 | 优先级 |
|---------|------|-----|------|-------|
| 产品收入 | 硬件/软件销售 | 60% | 稳定 | 核心 |
| 服务收入 | 实施/咨询/培训 | 25% | +10%/月 | 增长 |
| 平台收入 | 抽成/流量变现 | 15% | 新兴 | 探索 |

## 验证
- 占比总和：60%+25%+15% = 100% ✓
- 无重叠：各类型互斥 ✓

## 分析洞察

### 收入健康度评估
1. 产品收入占比60%，收入结构偏"重"
   - 风险：硬件毛利低、库存重
   - 建议：提升服务收入占比

2. 服务收入增长快（+10%/月），是增长引擎
   - 优势：毛利高、复购好
   - 建议：加大服务团队投入

3. 平台收入占比低但有潜力
   - 机会：未来可能成为第二曲线
   - 建议：持续观察，不急于投入

## 行动建议
1. [P0] 提升服务收入占比到35%
2. [P1] 优化产品毛利结构
3. [P2] 探索平台收入模式
```

## 错误处理

| 错误类型 | 典型表现 | 处理方式 |
|---------|---------|---------|
| 维度混乱 | 把渠道和地区混在一起拆 | 确保同一层级的维度一致 |
| 重复计算 | 子指标之间有重叠 | 检验MECE原则 |
| 模糊分类 | 用"其他"类概括 | 每个分类必须有明确定义 |
| 基准缺失 | 不提供各子指标的数据 | 必须有数据支撑，否则无意义 |
| 浅尝辄止 | 只拆到第一层就停止 | 需要时可深入拆解第二层 |

## 核心定义

并列式指标拆解是将一个总体指标按照**并列（加法）关系**拆解为多个子指标的方法。每个子指标独立存在，共同构成总体指标。

## 核心公式

```
总指标 = 子指标1 + 子指标2 + 子指标3 + ... + 子指标N
```

## 适用场景

| 场景类型 | 示例 |
|---------|------|
| 业务构成分析 | 用户运营 = 获取 + 激活 + 留存 + 付费 + 传播 |
| 收入结构拆解 | 总收入 = 收入1 + 收入2 + 收入3 |
| 成本分类 | 总成本 = 固定成本 + 变动成本 |
| 渠道分类 | 总流量 = 渠道A + 渠道B + 渠道C |

## Critical Rules

### 必须遵守（5条）

1. **业务相关性** - 每个子指标必须与业务目标直接相关
2. **独立可测** - 每个子指标必须能够独立测量和量化
3. **无遗漏** - 子指标加起来必须等于总指标，不能有遗漏
4. **MECE原则** - 子指标之间不能有重叠，保持互斥性
5. **层级一致** - 同一层级的子指标必须在同一维度

### 禁止行为（5条）

1. **禁止维度混乱** - 不能将不同维度的指标并列（如地区+产品类别）
2. **禁止重复计算** - 子指标之间不能有从属或重叠关系
3. **禁止模糊命名** - 不能使用"其他"、"misc"等模糊分类
4. **禁止缺失基准** - 不能不提供各子指标的基准数据
5. **禁止只看表面** - 不能只拆解到第一层就停止，需深入分析

## Workflow Steps

### Step 1: 明确总指标

**输入**: 用户描述的业务目标或总体指标

**处理**: 
- 确认总指标的定义和衡量单位
- 确认业务范围和时间周期
- 与用户确认指标边界

**输出**: 
```json
{
  "总指标": "用户运营健康度",
  "衡量单位": "百分比",
  "业务范围": "App全链路用户生命周期",
  "时间周期": "月度"
}
```

**成功标准**: 用户确认总指标定义清晰、无歧义

---

### Step 2: 识别并列维度

**输入**: 总指标定义

**处理**:
- 分析总指标的构成要素
- 确定拆解的维度（来源/类型/阶段/渠道等）
- 选择最适合业务理解的维度

**输出**:
```json
{
  "拆解维度": "用户生命周期阶段",
  "子指标数量": 5,
  "各维度定义": {
    "维度1": "用户获取",
    "维度2": "用户激活",
    "维度3": "用户留存",
    "维度4": "用户付费",
    "维度5": "用户传播"
  }
}
```

**成功标准**: 维度选择符合业务逻辑，便于理解和执行

---

### Step 3: 量化子指标

**输入**: 拆解后的子指标列表

**处理**:
- 为每个子指标定义衡量公式
- 确认各子指标的数据来源
- 收集或估算各子指标的当前值

**输出**:
```json
{
  "子指标详情": [
    {
      "名称": "用户获取",
      "衡量公式": "新用户数",
      "数据来源": "埋点系统/渠道后台",
      "当前值": "5000/月",
      "占比": "15%"
    },
    {
      "名称": "用户激活",
      "衡量公式": "完成首任务的新用户数",
      "数据来源": "行为埋点",
      "当前值": "2500",
      "占比": "50%"
    }
  ]
}
```

**成功标准**: 每个子指标都有明确的计算公式和数据来源

---

### Step 4: 验证完整性

**输入**: 各子指标的量化数据

**处理**:
- 计算各子指标之和，验证是否等于总指标
- 检查是否有遗漏的重要子指标
- 检查子指标之间是否有重叠

**输出**:
```json
{
  "验证结果": {
    "子指标之和": "100%",
    "与总指标一致": true,
    "无重叠": true,
    "无遗漏": true
  }
}
```

**成功标准**: 验证通过，各子指标完整且无重叠

---

### Step 5: 输出拆解结果与建议

**输入**: 完整的拆解验证结果

**处理**:
- 生成标准格式的拆解图表
- 根据各子指标的当前值和占比
- 输出优先级排序的执行建议

**输出**:
```markdown

## 公式

## 各子指标详情
| 指标 | 当前值 | 占比 | 优先级 |
|------|--------|------|--------|
| 用户获取 | 5000 | 15% | 中 |
| 用户激活 | 2500 | 25% | 高 |
| 用户留存 | 1500 | 30% | 高 |
| 用户付费 | 1000 | 20% | 中 |
| 用户传播 | 500 | 10% | 低 |

## 执行建议
1. 优先提升激活率（转化漏斗最大瓶颈）
2. 重点关注留存（决定LTV核心）
3. 适度扩大获客量（配合激活和留存提升）
```

**成功标准**: 输出结构清晰、建议具体可执行
