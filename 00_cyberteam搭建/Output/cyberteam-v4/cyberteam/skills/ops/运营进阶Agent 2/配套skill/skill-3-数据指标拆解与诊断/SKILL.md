## 核心定义

### 什么是数据指标拆解？

将一个核心指标（金字塔式或漏斗式）拆分为多个子指标，找出影响核心指标的关键因素。

### 两种拆解方法

**1. 金字塔拆解法**
```
                    销售额
                   /      \
              客单价       付费用户数
             /    \         /      \
         ...     ...     新用户    老用户
```

**2. 漏斗拆解法**
```
曝光 → 点击 → 注册 → 付费
10000   5000   1000    100
(50%)   (20%)  (10%)
```

### 什么是异常数据诊断？

识别数据中的异常波动（突然上升/下降/持续波动），并分析原因、制定对策。

---

## Critical Rules（必须遵守）

1. **先明确分析目的** - 不是所有数据都有价值，要选择关键指标
2. **警惕虚荣指标** - 看起来好看但无实际价值的数据要排除
3. **相关性≠因果性** - 不能看到两个数据相关就说有因果关系
4. **排除统计误差** - 确认数据确实异常后再分析
5. **对比才有意义** - 要和历史、竞品、行业对比才有判断依据

---

## 禁止行为

1. **禁止眉毛胡子一把抓** - 不是所有数据都要分析
2. **禁止只看百分比** - 要结合绝对值一起看
3. **禁止小样本下结论** - 样本量不够不能得出结论
4. **禁止用过时数据** - 数据有时效性，要用最新数据
5. **禁止为分析而分析** - 先明确问题再找数据
6. **禁止过度解读** - 正常波动不要过度解读
7. **禁止忽视基数效应** - 从100到200是100%，但从100万到100.1万不是

## Workflow Steps

### 步骤1：明确分析目的（输入→处理→输出）

**输入**：
- 业务背景描述
- 决策问题

**处理**（明确分析三问）：
| 问题 | 示例 |
|------|------|
| 分析什么？ | 销售额下降的原因 |
| 为什么分析？ | 找出问题，制定对策 |
| 用什么数据？ | 销售数据、用户数据、渠道数据 |

**输出**：
- 分析目的声明
- 关键指标清单（不超过5个）
- 数据来源和时间范围

**成功标准**：有明确的目的和聚焦的指标清单

---

### 步骤2：数据收集与预处理（输入→处理→输出）

**输入**：
- 分析目的
- 数据来源清单

**处理**：
```
1. 收集数据
   - 确保数据完整
   - 确保时间范围正确
   - 记录数据口径

2. 预处理数据
   - 清洗异常值
   - 统一数据格式
   - 计算衍生指标

3. 验证数据质量
   - 检查是否有缺失
   - 检查是否有明显错误
   - 确认统计口径一致
```

**输出**：
- 清洗后的数据集
- 数据质量报告
- 关键指标汇总

**成功标准**：数据完整、口径一致、质量可靠

---

### 步骤3：指标拆解分析（输入→处理→输出）

**输入**：
- 关键指标
- 历史数据

**处理**（选择拆解方法）：

**方法A：金字塔拆解**
```
步骤1：确定要拆解的指标（如销售额）
步骤2：问"什么因素影响这个指标"
步骤3：拆解为两个子因素（如客单价×用户数）
步骤4：对每个子因素继续拆解
步骤5：直到找到可操作的最小单元
```

**方法B：漏斗拆解**
```
步骤1：画出用户转化路径
步骤2：标注每步的转化率
步骤3：找出转化率异常的环节
步骤4：分析该环节的影响因素
```

**输出**：
- 指标拆解树状图
- 各子因素贡献度
- 关键发现清单

**成功标准**：拆解到可操作的最小单元，并量化各因素影响

---

### 步骤4：异常数据诊断（输入→处理→输出）

**输入**：
- 历史数据
- 当前数据
- 业务背景

**处理**（4步诊断法）：

**Step 1：确认异常**
| 检查项 | 方法 |
|--------|------|
| 数据真的异常吗？ | 与历史均值对比 |
| 异常幅度多大？ | 计算偏差百分比 |
| 是短期还是持续？ | 观察时间趋势 |

**Step 2：定位问题**
| 问题维度 | 具体问题 |
|----------|---------|
| 哪个指标异常？ | 转化率？客单价？ |
| 哪个渠道异常？ | 付费？自然？ |
| 哪个用户群异常？ | 新用户？老用户？ |
| 什么时候开始？ | 确定变化节点 |

**Step 3：分析原因**
```
内部因素：
- 产品变化（新功能、bug）
- 运营活动（促销、内容更新）
- 技术问题（页面加载慢）

外部因素：
- 竞品变化（活动、政策）
- 季节性因素（节假日）
- 市场变化（趋势、政策）
```

**Step 4：制定对策**
| 原因 | 对策 | 负责人 | 时间 |
|------|------|--------|------|
| 原因1 | 对策1 | 张三 | 3天 |
| 原因2 | 对策2 | 李四 | 1周 |

**输出**：
- 异常诊断报告
- 原因分析清单
- 行动计划表

**成功标准**：找到根本原因，制定可执行的对策

---

### 步骤5：撰写分析报告（输入→处理→输出）

**输入**：
- 所有分析结果
- 关键发现

**处理**（标准报告结构）：

```markdown
# 数据分析报告

## 一、报告概述
- 分析背景
- 分析目的
- 数据来源

## 二、核心发现
- 主要结论1
- 主要结论2
- 主要结论3

## 三、详细分析
- 维度1分析
- 维度2分析

## 四、问题诊断
- 问题1：描述+原因
- 问题2：描述+原因

## 五、建议与行动计划
- 建议1：具体行动+负责人+时间
- 建议2：具体行动+负责人+时间
```

**输出**：
- 完整数据分析报告
- 执行建议清单

**成功标准**：报告结构清晰，结论有数据支撑，建议可执行

## 评估维度

| 维度 | 定义 | 权重 |
|------|------|------|
| 目的明确性 | 分析目的是否清晰 | 15% |
| 数据质量 | 数据是否完整、准确 | 20% |
| 拆解逻辑性 | 拆解是否有逻辑 | 25% |
| 诊断准确性 | 原因分析是否到位 | 25% |
| 建议可行性 | 对策是否可执行 | 15% |

## 常用分析框架

### 5W2H分析法
| 维度 | 问题 |
|------|------|
| What | 发生了什么问题？ |
| Why | 为什么会发生？ |
| When | 什么时候发生的？ |
| Where | 在哪里发生的？ |
| Who | 涉及到谁？ |
| How | 怎么发生的？ |
| How much | 影响了多少？ |

### 对比分析法
| 类型 | 说明 |
|------|------|
| 环比 | 与上一周期对比 |
| 同比 | 与去年同期对比 |
| 横比 | 与竞品/行业对比 |

## 常见误区速查表

| 误区 | 表现 | 正确做法 |
|------|------|---------|
| 相关性当成因果性 | 看到相关就说因果 | 需要进一步验证 |
| 样本偏差 | 用小样本或偏差样本 | 确保样本代表性 |
| 忽视基数效应 | 只看百分比 | 结合绝对值 |
| 过度解读 | 数据1%波动就下结论 | 区分正常波动 |
