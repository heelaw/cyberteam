## 一、第一性原理追问

### 问题1：CyberTeam v4 的状态机为什么存在但没有被真正使用？

**表面现象**：
- `engine/state_machine.py` 存在，定义了完整的状态机
- `engine/coo/coordinator.py` 有 COO 工作流
- 但两者是**两套独立的系统**

**追问**：为什么设计了两套系统？

**根本原因**：
1. 状态机是**设计时拍脑袋想出来的**，没有考虑如何与 COO workflow 集成
2. COO workflow 是**另一个设计**，没有复用状态机的状态定义
3. 没有一个**统一的任务上下文**贯穿整个执行流程

**结论**：架构设计缺乏统一的任务生命周期管理。

---

### 问题2：edict 的核心价值是什么？

**edict 的本质**：
- 一种**工作流状态机**，用于任务生命周期管理
- 核心价值：**严格的状态流转校验 + 完整的流转日志 + 部门职责映射**

**CyberTeam v4 缺少什么**：
1. **状态流转校验**：新状态是否能转换，有严格规则
2. **流转日志**：每一步转换都有记录，可以追溯
3. **部门职责**：任务与执行部门绑定

---

### 问题3：什么是最小集成方案？

**最小可行集成**：
1. **清理历史术语**：去掉 "三省六部"、"太子" 等，改用现代公司治理术语
2. **统一任务上下文**：让 TaskState 成为贯穿整个流程的统一数据结构
3. **部门真实化**：把六部改成真实的 Department Agent

## 三、核心问题：双重系统

### 3.1 系统A：StateMachine

```
TaskState (状态机上下文)
    ├── state: EdictState
    ├── current_node: CyberTeamNode
    ├── transitions: List[StateTransition]
    └── node_results: Dict[CyberTeamNode, dict]
```

### 3.2 系统B：COOWorkflow

```
COOWorkflow
    ├── layer1: Layer1Alignment
    ├── layer2: Layer2Strategy
    ├── layer3: Layer3Debate
    └── layer4: Layer4Reporting
```

### 3.3 问题

这两个系统**没有共享同一个任务上下文**。

- StateMachine 管理任务的**执行状态**
- COOWorkflow 管理任务的**讨论流程**

**但一个任务既有执行状态，又有讨论流程，这两者应该统一。**

## 四、第一性原理重构

### 4.1 任务 = 讨论 + 执行

从第一性原理出发，一个任务包含：
1. **讨论层**：专家讨论、策略制定、风险预案
2. **执行层**：部门执行、状态流转、质量验收

### 4.2 统一的任务模型

```python
@dataclass
class UnifiedTask:
    """统一任务上下文"""
    task_id: str

    # 身份信息
    title: str
    description: str
    user_input: str
    priority: str

    # 状态机核心
    state: TaskState  # 执行状态
    workflow: COOWorkflow  # 讨论流程

    # 流转日志
    flow_log: List[FlowEntry]  # 状态转换记录
    progress_log: List[ProgressEntry]  # 进展记录

    # 部门执行
    assigned_department: str  # 当前执行部门
    department_results: Dict[str, DeptResult]  # 各部门产出

    # 时间戳
    created_at: datetime
    updated_at: datetime
    completed_at: Optional[datetime]
```
