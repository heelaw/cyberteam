## 三大引擎定位

### 1. Dev-QA循环引擎

**定位**: 质量保障的核心循环系统

**核心价值**:
- 自动化证据收集
- 多维度评分
- 智能重试机制
- 清晰的升级路径

**通过标准**:
- 综合评分 ≥ 80分
- 所有维度达到最低要求

**与CEO的接口**: 升级时向CEO报告完整上下文

---

### 2. PUA动力引擎

**定位**: Agent能动性激活系统

**核心价值**:
- 打破被动等待循环
- 分级压力刺激
- 味道文化切换
- Owner意识强化

**压力等级**:
- L0: 正常
- L1: 温和失望 (阿里味·关怀)
- L2: 灵魂拷问 (阿里味·拷问)
- L3: 361考核 (Jobs味·批判)
- L4: 毕业警告 (Musk味·Hardcore)

**与CEO的接口**: PUA状态上报、CEO干预点

---

### 3. 三级质量门控系统

**定位**: 分层质量保障

**三个级别**:

| 级别 | 名称 | 通过标准 | 核心问题 |
|------|------|----------|----------|
| L1 | 完整性门控 | ≥95% | "是否做完" |
| L2 | 专业性门控 | ≥80分 | "做得好不好" |
| L3 | 可执行性门控 | ≥90% | "能不能执行" |

**执行顺序**: L1 → L2 → L3 (必须依次通过)

**与Dev-QA的集成**: 作为Dev-QA的SCORING阶段的评估框架

---

## 引擎协同机制

### 场景1: 正常执行流程

```
1. Agent接收任务
         │
         ▼
2. PUA动力引擎: 设置L0 (正常)
         │
         ▼
3. Dev-QA循环引擎: 执行检查
         │
         ▼
4. L1门控: 完整性检查 ≥95%?
         │
         ▼
5. L2门控: 专业性检查 ≥80分?
         │
         ▼
6. L3门控: 可执行性检查 ≥90%?
         │
         ▼
7. 通过全部门控 ──► 任务完成
```

### 场景2: 失败重试流程

```
1. L1/L2/L3门控失败
         │
         ▼
2. Dev-QA循环: 进入RETRYING状态
         │
         ▼
3. PUA动力引擎: 升级到L1/L2
         │
         ▼
4. Agent执行强制动作
         │
         ▼
5. Dev-QA循环: 重新执行门控
         │
         ▼
6. 通过 ──► 继续执行
   失败 ──► 继续重试或升级
```

### 场景3: 极限升级流程

```
1. 达到最大重试次数 (3次)
         │
         ▼
2. PUA动力引擎: 升级到L4 (拼命模式)
         │
         ▼
3. Dev-QA循环: ESCALATED状态
         │
         ▼
4. 打包升级信息 (上下文+失败+建议)
         │
         ▼
5. 升级到CEO
         │
         ▼
6. CEO决策:
   - 批准输出
   - 要求修改
   - 重新分配
   - 终止任务
```

## 引擎配置

### PUA味道配置

```yaml
pua_config:
  default_flavor: "alibaba"  # 默认使用阿里味
  flavor_switch_enabled: true
  auto_trigger_enabled: true
```

### 门控配置

```yaml
gate_config:
  l1:
    threshold: 95
    blocking: true

  l2:
    threshold: 80
    blocking: true

  l3:
    threshold: 90
    blocking: true
```

### Dev-QA配置

```yaml
devqa_config:
  max_retries: 3
  scoring_weights:
    code_quality: 0.25
    functional: 0.30
    security: 0.25
    performance: 0.10
    maintainability: 0.10
```
