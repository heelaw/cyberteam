# 考核标准

- 是否根据用户状态、证据深度和平台做结构选择。
- 是否解释选择原因，而不是只报名称。
- 是否避免结构与场景错配。
- 是否给出可执行的写作顺序、备用方案和排除理由。
- 是否能明确主结构和备用结构的边界。

## 量化门禁

- 必须给出主结构和至少 1 个备用结构。
- 必须说明为什么选它，而不是别的结构。
- 必须把结构和用户状态、证据深度、阅读时长挂钩。
- 必须给出可执行的写作顺序。
- 必须说明为什么不选其他结构。
- 不能只报结构名。
- 不能让结构选择和平台预设冲突。

## 通过判定

- 6 项以上满足，且主结构与场景一致 -> pass。
- 少于 6 项或出现硬冲突 -> fail。

## 一票否决补充

- 先喜欢结构再硬套。
- 结构顺序写反。
- 需要解释却用纯直述。
- 需要比较却不给维度。
- 用户还没认知问题就上 PAS。
- 平台限制很短却选了长结构。
