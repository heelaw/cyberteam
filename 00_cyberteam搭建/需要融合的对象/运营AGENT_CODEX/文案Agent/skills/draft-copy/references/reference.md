# 初稿生成

## 适用场景

- 上游已经给出用户、卖点、场景和结构。
- 只需要把材料组装成可读、可改、可继续加工的初稿。
- 需要先交付一个可工作版本，而不是假装一次成稿。

## 先决条件

写之前先确认 7 件事：

1. 谁是主写对象。
2. 这篇要说的主卖点是什么。
3. 当前触点是短、中、长，还是决策触点。
4. 结构已经选定。
5. 至少有一条证据或例子。
6. 主动作是什么。
7. 哪些限制条件不能丢。

少一项，就先回上游，不要硬写。

## 生成顺序

1. 先写一句主张，只放最核心的结论。
2. 再写理由，解释为什么这件事值得看。
3. 再写证据，把能证明的事实放进去。
4. 再写限制，告诉用户适用条件和边界。
5. 再写行动，只保留一个主动作。
6. 最后把缺口单独列出，不把缺口硬补成内容。

## 写作骨架

### 短触点

- 主张：一句话给判断。
- 利益点：只保留一个用户结果。
- 证据：最多一个最强证据。
- 动作：只保留一个动作。

### 中触点

- 场景：先告诉用户现在为什么要看。
- 主张：再给核心判断。
- 理由：说明为什么成立。
- 证据：给 1-2 个可感知证据。
- 动作：收口到 1 个动作。

### 长触点

- 场景。
- 解释。
- 主张。
- 证据链。
- 反对意见。
- 限制条件。
- 动作。

## 句群规则

- 一段只解决一个问题。
- 一段只保留一个主动作。
- 需要解释时，把解释放在结论后面，不要先绕圈。
- 需要证据时，优先放事实、案例、数据、过程，而不是形容词。
- 如果内容太长，先拆成层次，不要压成一坨。
- 如果同一句里同时出现结论、证据和 CTA，优先拆开。

## 平台化写法

### 海报 / 广告位

- 只保留一个主张。
- CTA 只能是一个。
- 不能塞第二个卖点。

### 抖音

- 前面先放钩子。
- 中段放冲突或结果。
- 最后给动作。
- 前 3 秒必须出现可感知入口。

### 小红书

- 先场景，再体验，再细节，再对比。
- 少用内部术语，多用用户语言。
- CTA 先弱后强，常见是收藏、评论、私信。

### 公众号

- 可以先解释，再证明，再推进。
- 允许更完整的证据链。
- 适合把限制条件和 FAQ 放到后段。

### 详情页

- 首屏先判断。
- 中段再展开证据和反对意见。
- 最后再收口动作。

## 缺口处理

- 缺用户：回 `segment-users`。
- 缺卖点：回卖点 Skill。
- 缺场景：回 `judge-scenario`。
- 缺结构：回 `select-structure`。
- 缺证据：先保留缺口，不要硬写结论。
- 缺限制条件：把它写成待确认项，不要擅自补完。

## 质量检查

- 是否真的只写了第一版。
- 是否保留了可继续删改的空间。
- 是否把缺口显式列出。
- 是否只有一个主动作。
- 是否没有把空话包装成完整交付。
- 是否保留了可回退的边界条件。

## 典型好稿

- 先给判断，再给理由。
- 先给结果，再给证据。
- 先给主动作，再补必要说明。
- 先给限制，再给承诺。

## 典型坏稿

- 先讲一大段背景，最后才说重点。
- 同时催用户做三件事。
- 用抽象词替代理由和证据。
- 把不确定内容写成确定结论。
- 用“看起来完整”假装“已经可交付”。

## 绝对不能

- 不能在卖点、场景、结构不清时直接写成稿。
- 不能把口号当正文。
- 不能把缺口当作已完成内容。
- 不能在短触点里塞长解释。
- 不能写多个互相打架的 CTA。
- 不能把边界条件删掉来换顺口。

## 输出骨架

- 文案初稿
- 可删改段落
- 待确认信息
- 结构说明
- 风险 / 限制
- 主动作
