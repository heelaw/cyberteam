# 考核标准

- 是否逐一分析七个方向。
- 是否每个方向至少有候选，而不是只写结论。
- 是否区分功能、收益、证据。
- 是否说明为什么选这个方向，而不是别的方向。
- 是否保留证据缺口，避免硬编。

## 量化门禁

- 七个方向必须全部过一遍，不能只挑 2-3 个写。
- 每个方向至少要有 1 个候选，不允许空白。
- 主卖点必须能回到至少 1 条证据来源。
- 必须区分功能、结果、证据，不能混写。
- 必须写出为什么选这个方向，而不是只写”这是主卖点”。

## 一票否决补充

- 功能描述直接等于卖点。
- 优惠折扣直接当长期卖点。
- 品牌口号直接当证据。
- 只给一个结论，没有候选池。

## 七方向分项考核

### 方向一：功能独特性

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 差异识别 | 明确写出与竞品/默认方案的差异点 | 只写功能名，无对比 |
| 可感知性 | 翻译成用户能感受到的结果 | 直接写功能参数，未翻译 |
| 证据质量 | 有竞品对比数据或技术说明 | 无来源，或仅凭”我们觉得” |
| 候选分级 | 区分”有差异”和”用户真的在意” | 把所有差异都当主候选 |

### 方向二：效果承诺

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 可量化性 | 有具体数字、时间、比例 | 只写”更好””更高” |
| 结果层级 | 找到最终结果，或清楚标注退到中间/过程结果 | 把过程当结果，或不标注结果层级 |
| 证据溯源 | 每条承诺有对应数据来源 | “根据数据”但说不出哪个数据 |
| 边界标注 | 一次性案例已标注”单个案例” | 把单个案例当普遍结论 |

### 方向三：解决问题

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 用户语言 | 有原始用户评论或原话引用 | 全程用自己的话翻译，无来源 |
| 场景还原 | 写清楚问题发生在哪一步 | 只写抽象的”效率低” |
| 代价拆解 | 写出问题不解决的损失 | 只写问题，不写代价 |
| 产品匹配 | 问题层次（业务/操作/心理）与产品能力对应 | 产品无法接住所写问题 |

### 方向四：用户身份

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 身份场景 | 有具体场景的体面/效率/掌控感表达 | 只写职业标签 |
| 用户语言 | 有用户自己如何称呼/描述自己的记录 | 全程用企业语言描述用户 |
| 产品匹配 | 身份表达与产品核心价值一致 | 身份和功能逻辑矛盾 |
| 角色拆分 | 多角色产品已拆分各写 | 混在一起写一个身份 |

### 方向五：价格优势

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 比较对象 | 明确写出与谁比（竞品/外包/人工/时间成本） | 只说”便宜”不说和谁比 |
| 计算前提 | 算出综合成本或性价比，并标注假设 | 无数据支撑的估算当结论 |
| 效果前提 | 便宜的同时效果有保障 | 便宜但效果差 |
| 公平比较 | 用正价vs正价，或相同条件下的对比 | 用促销价和竞品正价比 |

### 方向六：服务保障

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 边界清晰 | 写出范围、时效、条件、赔付机制 | 只写”服务好” |
| 可兑现性 | 承诺能被写进合同，有实际执行记录 | 口头承诺大于书面承诺 |
| 可感知性 | 承诺在用户使用中能被感知到 | 只有合同里的承诺，用户日常感受不到 |
| 不过度承诺 | 承诺标准与企业实际服务能力匹配 | 写得远高于实际能兑现的 |

### 方向七：情感价值

| 考核项 | 通过标准 | 否决标准 |
|---|---|---|
| 用户来源 | 有用户主动表达的原文，非品牌自创 | 全部是品牌口号，无用户语言 |
| 场景落地 | 情绪对应到具体产品使用场景 | 只写”安心””幸福”等空泛词 |
| 逻辑关联 | 情感结果有明确的产品行为路径 | 情感和功能/效果之间无合理关联 |
| 差异化 | 该情感价值非所有竞品通用 | 每家都能写的情感不选为主卖点 |

## 综合评分

| 等级 | 要求 |
|---|---|
| 优秀 | 七个方向全部展开，每个方向有候选、有证据标注、有失败预防，主卖点三维度全部达标 |
| 合格 | 七个方向全部展开，每个方向有候选，主要卖点有证据，证据缺口已标注 |
| 待改进 | 七向不全，或方向全但有”一票否决”问题，或证据缺口未标注 |
| 不合格 | 缺少产品资料就发散，或功能描述直接当卖点，或优惠当长期卖点 |
