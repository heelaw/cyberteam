# 考核标准

## 逐项检查（必须逐项，不能只给总评）

- Q1 是否检查主写对象（不是空标签、不是"所有人"）？
- Q2 是否检查主卖点（用户能感知的结果，不是功能名词或口号）？
- Q3 是否检查场景（长度、介质、阅读状态是否匹配）？
- Q4 是否检查结构（符合说服目标，不是把证据写成情绪）？
- Q5 是否检查动作（CTA 具体可执行，不催三件事）？
- Q6 是否检查证据（事实/案例/数据/过程/边界，不是空话）？
- Q7 是否检查风险（无硬承诺、无绝对化表达、条件说清楚）？

## 量化门禁

- 必须逐项检查 7 问，每问必须有明确判定。
- 必须指出至少 1 个具体问题或风险。
- 必须给出可执行修改建议。
- 必须明确是否通过。
- 如果有任何硬承诺或边界缺失，默认不通过。
- **必须 7 项全部为真才可通过。**

## 通过判定

- **7 项全部为真且无硬承诺 → pass。**
- 4-6 项为真 → fail but repairable。
- 0-3 项为真 → fail。

## 新人能做的事

- 逐项核对七问并给出判定。
- 指出具体问题而不是说"还行"。
- 给出可执行修改建议。
- 明确说出通过还是不通过。
- 识别硬承诺和夸大表达。

## 一票否决（直接 fail）

- 只给"整体不错"之类的空评语。
- 不写风险点。
- 不给回退路径。
- 把硬承诺藏在措辞里。
- 把辅助对象当主对象。
- 把 6 分当成通过。

## 交付标准

1. 先给判断（通过/回退）。
2. 再给风险点（逐项列出，不是泛泛而谈）。
3. 再给修改建议（可执行，不是"建议优化"这种空话）。
4. 最后给是否通过（明确）。

## 回退要求

- 通过：7 项全过，无硬承诺。
- 回退：给出具体回退路径（segment-users / research-sellpoint-forward / research-sellpoint-backward / judge-scenario / select-structure / edit-copy）。
