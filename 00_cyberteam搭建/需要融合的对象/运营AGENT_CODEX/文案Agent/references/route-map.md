# 路由总图

需求识别 -> 用户分层 -> 卖点提炼 -> 场景判断 / 说服模式 -> 结构选择 -> 初稿 -> 删改 -> 自检 -> 高风险复核

## 路由规则

- 前置信息不完整时，优先回到前置 Skill。
- 同一请求如果同时缺用户和卖点，先补用户，再补卖点。
- 如果只有痛点没有资料，走反向卖点；如果有资料，走正向卖点。
- 如果结构不清楚，不进入初稿。
- 如果文案已写完但风险未检查，不直接交付。
- `judge-scenario` 必须同时输出场景标签、说服模式、长度建议和 CTA 强度。
- `select-structure` 必须在场景判断之后，不得提前。
- 高风险输出必须走自检后再进验证。

## 最小 SOP 拆分

### `detect-need-type`
判断需求是明确、模糊还是潜在，输出证据地图、冲突点和下一步动作。

### `segment-users`
把“写给谁”拆成可写作的主写对象、辅助对象和不该写的人群。

### `research-sellpoint-forward`
已有产品资料时，按功能、效果、问题、身份、价格、服务、情感七向提炼卖点。

### `research-sellpoint-backward`
只有痛点、反馈或抱怨时，从痛点 -> 场景 -> 阻碍 -> 结果反推卖点。

### `judge-scenario`
判断渠道、触点、阅读时长、说服模式和证据深度，决定短 / 中 / 长文案策略。

### `select-structure`
在 AIDA、4P、PAS、对比式、故事式、直述式等结构中做选择，并说明为什么。

### `draft-copy`
基于上游材料生成第一版文案，只写可用草稿，不假装一次成型。

### `edit-copy`
对已有文案做删改、压缩、强化，不重新发明卖点。

### `self-check-copy`
交付前核对用户、卖点、场景、结构、动作、证据是否闭环。

## 交接规则

- 需求没定清，先回 `detect-need-type`。
- 用户没分清，先回 `segment-users`。
- 卖点没出来，先回 `research-sellpoint-*`。
- 场景不清，先回 `judge-scenario`。
- 结构不清，先回 `select-structure`。
- 文案已完成，先回 `self-check-copy`。
- 高风险输出在自检后再进测试或数据验证。
