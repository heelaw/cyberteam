# 运营AGENT_CODEX 记忆

## 当前结论

文案 Agent 不能只做目录收敛，必须做到“新人拿着 SOP 就能独立执行、判断、回退和验收”。

当前这套 `文案Agent` 已经具备可用的知识层，但还没有达到“自动化编排层可直接依赖”的程度。核心原因不是结构不清，而是执行层与验收层还不够硬。

## 这次暴露的问题

1. 早期版本只有骨架，没有足够的判断密度。
2. 每个 Skill 只写了“做什么”，没有写透“怎么做、何时停、何时回退、何为错误”。
3. 缺少完整样例，导致新人不知道好稿长什么样、坏稿长什么样。
4. 缺少失败案例和边界条件，导致遇到模糊输入时容易误判。
5. 缺少绝对不能做的红线，导致输出标准不够硬。
6. scripts 目前仍是占位层，不能承担真实自动化逻辑。
7. 部分 Skill 还缺平台预设、量化验收和超时兜底。

## 根因

第一性原理上，文案工作不是“把文字写出来”，而是“在信息不完整时完成判断，再把判断转成可交付文本”。

因此，一个可上岗的文案 Agent 必须同时具备：
- 输入分辨能力
- 证据采集能力
- 卖点收敛能力
- 场景判断能力
- 结构选择能力
- 初稿生成能力
- 删改优化能力
- 交付自检能力

如果这些能力只写成标题，没有写成具体 SOP，批量化时就会复制出一堆“看起来有结构、实际上不能执行”的 Skill。

更深一层的根因是：我前期把“知识完整”当成了“可上岗”，但这两者不是一回事。知识完整只说明知道什么是对的，可上岗还要求知道怎样在不完整输入、冲突输入、模糊输入下做判断。

## 以后必须遵守的标准

1. 每个 Skill 必须能独立完成一个最小 SOP 单元。
2. 每个 Skill 必须有：作用、SOP、输出、参考、考核。
3. 每个关键 Skill 必须补齐：典型例子、失败案例、边界条件、交接判断、绝对不能。
4. 每个 Agent 必须明确：岗位职责、适用范围、路由逻辑、回退规则。
5. 任何看起来像“总结”的内容，都要检查是否还能指导新人真实执行。
6. 如果 Skill 需要自动化，就必须有真实脚本；如果只是提示词型，就明确声明，不写空脚本。
7. 需要批量化时，必须先做单个 Skill 的完整演练，再复制到其他 Skill。

## 评估暴露的优先修正点

1. 脚本层要么补真实逻辑，要么明确降级为纯提示词型。
2. 根 Agent 需要补超时和兜底思路，避免模糊输入来回循环。
3. 场景层需要加入平台级预设，不能只停留在短/中/长抽象层。
4. assessment 要逐步从定性检查变成可打分、可门禁的验收标准。
5. 如果后续要接入 CyberTeam 调度，需要补 Agent 角色定义文件，而不是只保留 Skill。

## 递进式排错问题

1. 这个任务的输入到底是什么，是否已经足够清楚？
2. 现在缺的是事实、判断，还是表达？
3. 当前最该回退到哪个前置 Skill？
4. 这个 Skill 的输出，是否能直接被下游使用？
5. 如果给一个新人，他能不能靠这份 SOP 自己做对？
6. 这份内容里，什么是必须保留的判断，什么是可以删掉的修辞？
7. 是否已经写清“什么是好”“什么是绝对不能” ？

## 批量化方法

1. 先确定一个 Agent 的职责边界。
2. 再把这个 Agent 拆成最小 SOP 单元。
3. 每个单元补齐：输入样例、判断规则、反例、边界、回退、输出模板。
4. 再做一轮新人视角自检：是否能独立执行，是否能判错，是否能接到下一步。
5. 最后才批量复制到其他 Agent。

## 批量化前的提问顺序

1. 这个单元到底解决哪一个最小问题？
2. 输入是否足够，还是还需要上游补证据？
3. 输出能不能直接给下游用？
4. 遇到不完整输入时，回退到哪一层？
5. 有哪些情况绝对不能做？
6. 有没有完整样例证明它真的能跑？
7. 新人照着做，是否能分辨好坏？

## 当前工作方式

- 不再只做摘要式收敛。
- 改为按“输入 -> 判断 -> 输出 -> 反例 -> 边界 -> 回退”逐层加厚。
- 优先补最薄、最容易误判、最影响下游的 Skill。

## 经验教训

- 不能把“目录齐了”误判成“能力齐了”。
- 不能把“写了规则”误判成“能执行”。
- 不能把“补了案例”误判成“已可批量化”。
- 必须持续做新人视角自检：我拿到这份东西，能不能独立做对、做完、做稳。
