# 偏差分析与应对模板

## 一、偏差分析基础

### 什么是进度偏差
```
进度偏差 = 实际进度 - 计划进度

偏差为正：进度提前
偏差为零：按计划
偏差为负：进度延误
```

### 偏差计算示例
```
计划进度：本周应完成10个行动
实际进度：本周只完成7个行动

进度偏差 = 7 - 10 = -3（延误3个行动）
进度偏差率 = -3 / 10 = -30%（延误30%）
```

### 偏差分类
| 类型 | 定义 | 处理方式 |
|------|------|----------|
| 可控偏差 | 由内部管理因素导致的偏差 | 立即改进 |
| 不可控偏差 | 由外部因素导致的偏差 | 调整计划 |

## 二、5Why分析法

### 方法说明
通过连续追问"为什么"，找到偏差的根本原因

### 分析步骤
1. 描述偏差现象
2. 追问第一个"为什么"
3. 继续追问，直到找到根本原因
4. 通常5次追问能到达根本原因

### 分析模板
```
偏差描述：____________

Why 1：为什么会发生？
回答：____________

Why 2：为什么（上面回答）会发生？
回答：____________

Why 3：为什么（上面回答）会发生？
回答：____________

Why 4：为什么（上面回答）会发生？
回答：____________

Why 5：为什么（上面回答）会发生？
回答：____________

根本原因：____________
```

## 三、偏差分析案例

### 案例1：供应商延迟

```
偏差描述：因供应商原材料延迟，导致生产计划延误2周

Why 1：为什么会延误？
回答：供应商没有按时交货

Why 2：为什么供应商没有按时交货？
回答：供应商接了太多订单，生产排期满了

Why 3：为什么供应商接了太多订单？
回答：供应商销售为了冲业绩，接了超出产能的订单

Why 4：为什么供应商销售要冲业绩？
回答：供应商公司的考核只看订单量，不看交付

Why 5：为什么供应商考核只看订单量？
回答：供应商公司管理粗放，考核体系不完善

根本原因：供应商管理不完善，考核体系有缺陷

解决方案：
□ 更换供应商
□ 提前下单，增加交期buffer
□ 与供应商深度绑定，签订框架协议
□ 自主储备关键原材料
```

### 案例2：团队能力不足

```
偏差描述：新功能开发进度延误1周

Why 1：为什么会延误？
回答：开发人员能力不足，代码质量差，返工多

Why 2：为什么开发人员能力不足？
回答：新人占比高，缺少培训

Why 3：为什么新人占比高？
回答：业务快速发展，大量招聘，但入职培训不系统

Why 4：为什么入职培训不系统？
回答：培训部门资源有限，只能做基础培训

Why 5：为什么培训资源有限？
回答：培训预算被削减，优先保障业务发展

根本原因：业务扩张与人才培养脱节

解决方案：
□ 减少业务扩张速度
□ 增加培训预算
□ 建立导师制度
□ 引入外部培训资源
□ 调整招聘策略，提高人才门槛
```

## 四、偏差应对措施库

### 可控偏差应对

#### 进度延误应对
| 措施 | 适用场景 | 成本 | 风险 |
|------|----------|------|------|
| 赶工（加班）| 短期延误 | 人力成本增加 | 质量下降、人员疲劳 |
| 增加资源 | 资源不足 | 预算增加 | 资源冲突 |
| 换人 | 人员能力问题 | 重新培训成本 | 交接延迟 |
| 优化流程 | 流程低效 | 改造成本 | 适应期 |
| 缩小范围 | 无法挽回 | 影响目标达成 | 目标不完整 |

#### 范围蔓延应对
| 措施 | 适用场景 | 成本 | 风险 |
|------|----------|------|------|
| 冻结范围 | 蔓延可控 | 无 | 需求积压 |
| 优先级重排 | 蔓延不严重 | 无 | 影响原计划 |
| 增加迭代 | 有足够时间 | 周期延长 | 影响最终交付 |
| 拒绝变更 | 蔓延严重 | 干系人不满 | 需求冲突 |

### 不可控偏差应对

#### 外部依赖延误
| 措施 | 适用场景 | 成本 | 风险 |
|------|----------|------|------|
| 等待 | 短期可等待 | 无 | 继续延误 |
| 替换方案 | 有备选方案 | 额外成本 | 质量差异 |
| 升级沟通 | 沟通不畅 | 无 | 可能无效 |
| 接受风险 | 无法解决 | 影响目标 | 需上级确认 |

#### 市场/政策变化
| 措施 | 适用场景 | 成本 | 风险 |
|------|----------|------|------|
| 调整目标 | 变化根本性 | 目标变更 | 信任损失 |
| 调整策略 | 变化可应对 | 战略调整 | 转型成本 |
| 加快执行 | 变化是机会 | 加速成本 | 质量风险 |
| 暂停/终止 | 变化是威胁 | 沉没成本 | 机会损失 |

## 五、偏差应对决策树

```
发现偏差
    │
    ├── 偏差类型
    │   ├── 可控偏差
    │   │   ├── 延误程度 < 10%
    │   │   │   ├── 是否可以通过赶工解决？
    │   │   │   │   ├── 是 → 加班/增加工时
    │   │   │   │   └── 否 → 继续分析
    │   │   │   └── 是否需要增加资源？
    │   │   │       ├── 是 → 申请额外资源
    │   │   │       └── 否 → 优化流程
    │   │   │
    │   │   └── 延误程度 10-30%
    │   │       ├── 能否调整范围？（保核心）
    │   │       │   ├── 是 → 缩小范围
    │   │       │   └── 否 → 申请延期
    │   │       └── 是否需要换人？
    │   │           ├── 是 → 人员调整
    │   │           └── 否 → 其他措施
    │   │
    │   └── 不可控偏差
    │       ├── 能否找到替代方案？
    │       │   ├── 是 → 启动替代方案
    │       │   └── 否 → 继续
    │       ├── 偏差是否影响最终目标？
    │       │   ├── 否 → 接受偏差，继续执行
    │       │   └── 是 → 升级决策
    │       └── 是否需要调整目标？
    │           ├── 是 → 启动目标调整流程
    │           └── 否 → 重新评估策略
```

## 七、常见偏差处理参考

### 延误处理话术
```
向上级汇报延误时：
"目前进度延误了___天（延误___%），原因是___。

我们评估了以下方案：
1. ___（优势：___，风险：___）
2. ___（优势：___，风险：___）

建议采用方案1，理由是___。

需要的支持是___。"

向团队说明时：
"___遇到了问题，会影响___的进度。

原因是___，不是___（团队/个人）的错。

我们决定采用___方案应对，需要大家___（具体行动）。

目标是___（具体目标），大家一起加油。"
```
