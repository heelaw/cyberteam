# 复盘机制设计指南

## 总览

复盘是执行路径的重要组成部分。本指南提供A类和B类问题的复盘机制设计模板。

---

## 一、复盘基础

### 1.1 复盘目的

| 目的 | 说明 |
|------|------|
| 经验沉淀 | 把个人经验转化为组织知识 |
| 问题发现 | 及时发现执行中的问题 |
| 持续改进 | 根据反馈不断优化 |
| 责任追溯 | 明确责任，总结教训 |

### 1.2 复盘原则

| 原则 | 说明 |
|------|------|
| 及时性 | 问题发生后尽快复盘 |
| 客观性 | 基于事实，不带情绪 |
| 深入性 | 找到根本原因，不停留表面 |
| 建设性 | 重点在于改进，而非追责 |

## 二、A类问题复盘机制

### 2.1 复盘频率

| 复盘类型 | 频率 | 时机 | 参与者 |
|----------|------|------|--------|
| 周迭代回顾 | 每周 | 每周五 | 执行团队 |
| 阶段复盘 | 每2-4周 | 每个里程碑 | 项目组+管理层 |
| 方向复盘 | 每季度 | 重大节点 | 高层+执行层 |

### 2.2 A类问题复盘框架

```
KISS复盘法：
├── Keep（保持）：哪些做法有效，需要保持？
├── Improve（改进）：哪些做法可以改进？
├── Stop（停止）：哪些做法应该停止？
└── Start（开始）：哪些新做法应该开始？
```

### 2.3 A类问题复盘模板

#### 周迭代回顾

**时间**：每周五 XX:00

**参与者**：执行团队

**议程**：
1. 本周进展（15分钟）
   - 完成了什么？
   - 验证了哪些假设？
2. 发现与洞察（15分钟）
   - 发现了什么新信息？
   - 对问题的理解有什么变化？
3. 下周计划（15分钟）
   - 下周目标
   - 关键行动
4. 需要支持（10分钟）
   - 资源需求
   - 决策需求

**产出**：周报（模板如下）

```
# 周报：[日期]

## 本周进展
- 验证假设：...
- 关键产出：...

## 关键发现
- 新信息：...
- 认知变化：...

## 下周计划
- 目标：...
- 关键行动：...

## 支持需求
- 资源需求：...
- 决策需求：...
```

#### 阶段复盘

**时间**：每个里程碑结束时

**参与者**：项目组 + 管理层

**议程**：
1. 阶段目标回顾（30分钟）
   - 目标达成情况
   - 差距分析
2. 假设验证回顾（30分钟）
   - 核心假设验证结果
   - 假设调整
3. 策略调整讨论（45分钟）
   - 是否需要调整方向
   - 是否需要调整资源
4. 下阶段规划（30分钟）
   - 下阶段目标和策略
   - 资源配置

### 2.4 A类问题预警机制

| 预警指标 | 安全线 | 预警线 | 行动 |
|----------|--------|--------|------|
| 假设验证进度 | 按计划验证 | 延迟1周以上 | 重新评估假设 |
| 用户反馈 | 正向反馈>60% | 正向反馈<40% | 调整策略 |
| 资源消耗 | 按预算消耗 | 消耗>120% | 评估是否继续 |
| 团队士气 | 稳定 | 明显下降 | 团队沟通 |

## 三、B类问题复盘机制

### 3.1 复盘频率

| 复盘类型 | 频率 | 时机 | 参与者 |
|----------|------|------|--------|
| 日站会 | 每天 | 固定时间 | 执行团队 |
| 阶段检查 | 每个子阶段 | 关键节点 | 执行团队+PM |
| 项目复盘 | 项目结束 | 项目结束后1周内 | 全体 |

### 3.2 B类问题复盘框架

```
5Why分析法：
问题 → Why1 → Why2 → Why3 → Why4 → Why5 → 根本原因
                          ↓
                       解决方案
```

### 3.3 B类问题复盘模板

#### 日站会（15分钟）

**时间**：每天 XX:00

**参与者**：执行团队

**议程**（每人2分钟）：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么阻碍？

**产出**：每日任务清单更新、阻碍问题升级

#### 阶段检查

**时间**：每个子阶段结束时

**参与者**：执行团队 + PM

**议程**：
1. 阶段交付物检查（20分钟）
   - 是否符合标准
   - 问题清单
2. 进度评估（10分钟）
   - 进度偏差分析
   - 调整措施
3. 风险检查（10分钟）
   - 已知风险状态
   - 新风险识别
4. 下阶段准备（10分钟）
   - 交接事项
   - 资源确认

#### 项目复盘（Post-mortem）

**时间**：项目结束后1周内

**参与者**：项目全体

**议程**：
1. 项目回顾（30分钟）
   - 项目目标达成情况
   - 关键里程碑回顾
2. 问题分析（45分钟）
   - 问题清单及根因分析
   - 重大决策回顾
3. 经验沉淀（30分钟）
   - 成功要素
   - 教训总结
4. 改进建议（20分钟）
   - 流程改进
   - 工具改进
   - 技能提升

**产出**：项目复盘报告

```
# 项目复盘报告：[项目名称]

## 项目概况
- 项目目标：
- 实际结果：
- 偏差分析：

## 做得好的
1.
2.
3.

## 需要改进的
1.
2.
3.

## 根因分析
- 问题1：根本原因是...
- 问题2：根本原因是...

## 改进建议
| 改进项 | 优先级 | 负责人 | 完成时间 |
|--------|--------|--------|----------|
| | | | |

## 附件
- 问题日志
- 决策记录
- 重要文档
```

### 4.2 复盘检查清单

#### 复盘前准备
- [ ] 收集相关数据和文档
- [ ] 通知参与者复盘时间和议程
- [ ] 指定记录人

#### 复盘过程
- [ ] 目标回顾：最初目标是什么？
- [ ] 结果对比：实际结果和目标差距？
- [ ] 原因分析：为什么有这个差距？
- [ ] 经验提炼：学到了什么？
- [ ] 改进建议：下次如何改进？

#### 复盘后跟进
- [ ] 复盘报告发出
- [ ] 改进措施分配到人
- [ ] 改进措施跟踪机制建立

## 五、复盘文化建设

### 5.1 心理安全

| 要求 | 说明 |
|------|------|
| 不追责 | 复盘目的不是追责，而是学习和改进 |
| 开放心态 | 鼓励坦诚表达，不隐藏问题 |
| 建设性 | 聚焦于改进，而非批评 |
| 持续性 | 复盘是习惯，不是临时活动 |

### 5.2 常见问题处理

| 问题 | 处理方式 |
|------|----------|
| 复盘变成批斗会 | 明确复盘目的，强调学习导向 |
| 问题表面化 | 使用5Why等工具深入分析 |
| 改进措施不落地 | 明确责任人、时间节点、验收标准 |
| 复盘流于形式 | 领导以身作则，持续强调复盘价值 |
