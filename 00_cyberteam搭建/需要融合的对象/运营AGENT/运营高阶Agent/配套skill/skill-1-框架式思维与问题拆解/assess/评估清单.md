# 框架式思维与问题拆解 - 评估清单

## 评估维度

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 结构完整性 | 25% | 问题各要素是否完整覆盖 |
| 逻辑严谨性 | 25% | 各节点关系是否清晰合理 |
| 可执行性 | 20% | 子问题是否可落地执行 |
| 可验证性 | 15% | 是否有明确的成功标准 |
| 业务适配度 | 15% | 是否结合具体业务场景 |

## 评分等级

| 等级 | 分数 | 说明 |
|------|------|------|
| 优秀 | 90-100 | 全面满足所有维度，质量极高 |
| 良好 | 75-89 | 满足大部分维度，质量较高 |
| 合格 | 60-74 | 基本满足维度要求，有改进空间 |
| 不合格 | <60 | 存在明显缺陷，需重大改进 |

## 逐项检查清单

### 结构完整性 (25分)

- [ ] 问题类型判断正确（抽象/具体）
- [ ] 所有关键要素都被识别
- [ ] 无遗漏的重要维度
- [ ] 子问题之间保持独立

### 逻辑严谨性 (25分)

- [ ] 拆解策略选择合理
- [ ] 公式逻辑成立
- [ ] 各节点因果关系清晰
- [ ] 不存在循环依赖

### 可执行性 (20分)

- [ ] 每个子问题都可转化为具体动作
- [ ] 执行顺序合理
- [ ] 无超出能力范围的执行项
- [ ] 考虑了资源约束

### 可验证性 (15分)

- [ ] 每个节点都有明确的验收标准
- [ ] 成功条件可量化
- [ ] 有明确的复盘节点
- [ ] 可以追踪执行效果

### 业务适配度 (15分)

- [ ] 充分考虑了业务场景
- [ ] 符合行业特点和用户特征
- [ ] 考虑了竞争环境因素
- [ ] 执行方案有业务背景支撑

## 常见问题

### 问题1：拆解过于发散
**表现**：子问题数量过多，缺乏优先级
**修复**：筛选最关键的3-5个子问题

### 问题2：概念模糊
**表现**：使用"用户活跃"等模糊概念
**修复**：转化为"DAU>1000"等可量化指标

### 问题3：脱离业务
**表现**：套用通用框架不考虑行业特点
**修复**：结合具体产品形态和用户特征
