## 技能描述

**一句话定义**：将模糊的抽象概念分解为具体、可分析、可操作的具体维度的思维工具。

**详细描述**：

抽象问题拆解器解决的是"我知道问题存在，但说不清问题是什么"的困境。它将高层级的概念（如"用户体验不好"、"竞争力不足"）分解为可以具体分析、可以采取行动的维度。

这种方法帮助用户在投入解决方案之前，先正确地定义问题。

## 核心原理

### 抽象层次模型

```
Level 5: 哲学层面（存在、意义）
Level 4: 概念层面（竞争力、用户体验）
Level 3: 维度层面（速度、成本、质量）
Level 2: 指标层面（响应时间、错误率）
Level 1: 行为层面（点击、输入、等待）
```

### 拆解漏斗

```
    [抽象概念]
         │
         ▼
    [核心问题]
         │
         ▼
    [关键维度]
         │
         ▼
    [具体指标]
         │
         ▼
    [可操作项]
```

## Critical Rules（必须遵守）

1. **先定义再拆解** - 必须先对抽象概念给出明确、可理解的操作性定义，才能开始拆解
2. **维度必须独立** - 拆分出的维度之间必须相互独立，避免重复交叉
3. **维度必须全面** - 维度之和必须能覆盖原始概念的主要含义，不能有重大遗漏
4. **每维必须可测** - 每个维度必须能够被度量或评估，否则无法判断改善情况
5. **拆解深度适中** - 一般拆解到2-3层即可，过深会失去指导意义
6. **禁止循环定义** - 不能用抽象概念定义抽象概念，必须落到具体可感的层面
7. **验证完整性** - 拆解后必须验证：用维度组合能否还原原始概念的主要含义

## Workflow Steps（标准化流程）

### 步骤1：概念锁定

| 项目 | 内容 |
|------|------|
| 输入 | 用户描述的抽象问题或概念 |
| 处理 | 通过追问明确概念的具体含义 |
| 输出 | 概念的操作性定义（1-2句话） |
| 成功标准 | 定义能被一个从未听过这个概念的人理解 |

**追问清单**：
- "你说的XX，具体指什么？"
- "XX的反义词是什么？"
- "XX好/不好，怎么看出来？"
- "如果向小学生解释，你会怎么说？"

---

### 步骤2：维度发散

| 项目 | 内容 |
|------|------|
| 输入 | 概念的操作性定义 |
| 处理 | 列出所有可能解释这个概念的维度 |
| 输出 | 维度候选清单（通常10-15个） |
| 成功标准 | 覆盖概念的所有主要含义，无重大遗漏 |

**发散方法**：
- 同行维度（类似的词有哪些）
- 反向维度（它的反面是什么）
- 经典框架（PEST、SWOT等是否能套用）
- 用户旅程（从哪些角度可以描述它）

---

### 步骤3：维度收敛

| 项目 | 内容 |
|------|------|
| 输入 | 维度候选清单 |
| 处理 | 按MECE原则筛选和归类 |
| 输出 | 3-7个核心维度（符合"7±2"原则） |
| 成功标准 | 维度之间相互独立，且整体全面 |

**收敛标准**：
- 删除：与其他维度高度重叠的
- 删除：过于细节、不是本质的
- 合并：可以归并为同一主题的
- 保留：最能反映概念核心含义的

---

### 步骤4：维度定义

| 项目 | 内容 |
|------|------|
| 输入 | 核心维度清单 |
| 处理 | 为每个维度给出明确的可测量定义 |
| 输出 | 维度定义表，包含名称、定义、度量方法 |
| 成功标准 | 每个维度都能回答：如何知道它好还是不好？ |

**定义模板**：
```
维度名称：[维度名]
操作性定义：[一句话定义]
度量方法：[如何测量]
数据来源：[从哪里获取]
判断标准：[好/中/差的分界线]
```

---

### 步骤5：维度验证

| 项目 | 内容 |
|------|------|
| 输入 | 维度定义表 |
| 处理 | 用维度定义表反推，看能否还原原始概念 |
| 输出 | 验证报告，标注是否有遗漏或冗余 |
| 成功标准 | 主要情况下，用这些维度能正确描述原始概念 |

---

### 步骤6：拆解整合

| 项目 | 内容 |
|------|------|
| 输入 | 经过验证的维度定义表 |
| 处理 | 整合为结构化的抽象问题拆解框架 |
| 输出 | 完整拆解文档 |
| 成功标准 | 文档能帮助用户从"说不清问题"转变为"能清晰描述问题" |

## 使用限制

1. **不适用于**：问题已经非常具体、用户需要的是执行方案而非问题定义的情况
2. **前提条件**：用户愿意投入时间澄清概念，不能急于求成跳过定义阶段
3. **局限性**：拆解结果依赖于对概念的初始理解，理解偏差会导致拆解偏差
