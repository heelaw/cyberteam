# 五步诊断详解

## 五步诊断法概述

五步诊断法是一套系统性的问题分析方法，通过明确现象→拆分维度→定位问题→提出假设→验证解决的结构化流程，帮助快速准确定位产品问题的根因。

---

## 第一步：明确现象

### 1.1 核心要素

| 要素 | 说明 | 示例 |
|-----|-----|-----|
| 数据表现 | 具体指标的变化 | DAU从100万跌至80万 |
| 时间范围 | 问题持续多久 | 连续5天下跌 |
| 影响范围 | 影响多少用户/订单 | 影响约20万用户 |
| 严重程度 | P0/P1/P2/P3 | P1 |

### 1.2 严重程度判断标准

| 级别 | 定义 | 响应时间 | 示例 |
|-----|-----|---------|-----|
| P0 | 重大损失/功能不可用 | 立即处理 | 核心功能崩溃、营收骤降50%+ |
| P1 | 明显影响用户体验 | 24小时内 | DAU跌20%、转化率跌30% |
| P2 | 中度影响 | 3天内 | 某渠道下跌、某功能异常 |
| P3 | 轻微影响 | 1周内 | 指标小幅波动 |

### 1.3 对比基准选择

| 基准类型 | 适用场景 | 价值 |
|---------|---------|-----|
| 环比 | 日常监控、短期异常 | 及时发现 |
| 同比 | 考虑周期性因素 | 排除周期干扰 |
| 目标对比 | 有明确KPI时 | 评估目标达成 |
| 竞品对比 | 市场环境分析 | 判断市场位置 |

---

## 第二步：拆分维度

### 2.1 时间维度拆分

**拆分思路**：将问题拆解到不同时间粒度，识别规律或异常

| 时间粒度 | 分析要点 | 常见问题发现 |
|---------|---------|-------------|
| 按小时 | 日内波动、峰值时段 | 某个时段集中下跌 |
| 按日 | 日趋势、涨跌规律 | 持续下跌或涨跌交替 |
| 按周 | 周规律、工作日vs周末 | 周末效应异常 |
| 按月 | 月规律、月初月末 | 月底冲量后回落 |

**同比分析**：
- 与上周同期对比（环比）
- 与上月同期对比（环比）
- 与去年同期对比（同比，考虑季节性）

### 2.2 渠道维度拆分

**拆分思路**：按流量来源和入口拆分，识别渠道问题

| 渠道类型 | 拆分维度 | 分析价值 |
|---------|---------|---------|
| 流量渠道 | App/小程序/Web/H5 | 平台差异 |
| 来源渠道 | 搜索/社交/付费/直连/分享 | 渠道效率 |
| 入口页面 | 首页/活动页/详情页 | 入口质量 |
| 地域渠道 | 一二线/三四线/不同城市 | 地域差异 |

**渠道异常判断**：
- 某渠道占比突然变化
- 某渠道转化率异常
- 某渠道CPA突然上升

### 2.3 用户维度拆分

**拆分思路**：按用户特征拆分，识别用户群体问题

| 拆分维度 | 具体分类 | 分析价值 |
|---------|---------|---------|
| 用户类型 | 新用户/老用户/活跃用户 | 流失分析 |
| 用户画像 | 年龄/性别/地域/职业 | 画像差异 |
| 用户行为 | 高频用户/低频用户 | 行为差异 |
| 用户价值 | 高价值/中价值/低价值 | 价值分析 |
| 用户生命周期 | 引入期/成长期/成熟期/休眠/流失 | 周期分析 |

**关键指标**：
- 各用户群体的数量占比
- 各用户群体的跌幅差异
- 各用户群体的贡献度

### 2.4 功能维度拆分

**拆分思路**：按产品功能模块拆分，识别功能问题

| 拆分维度 | 具体指标 | 分析价值 |
|---------|---------|---------|
| 功能使用 | 各功能UV、点击量 | 功能渗透 |
| 功能转化 | 使用到下阶段的转化率 | 漏斗分析 |
| 功能停留 | 各功能使用时长 | 体验分析 |
| 功能路径 | 用户功能使用路径 | 路径分析 |

**功能问题识别**：
- 某功能使用率突然下降
- 某功能转化率异常
- 某功能入口流量下跌

---

## 第三步：定位问题

### 3.1 问题类型判断

| 问题类型 | 特征 | 识别方法 | 应对策略 |
|---------|-----|---------|---------|
| 趋势性下跌 | 连续多天持续下跌 | 看日环比趋势 | 系统性排查 |
| 突发事件 | 单日大幅下跌 | 看单日跌幅 | 快速定位触发事件 |
| 结构性变化 | 逐步变化但持续 | 看移动平均 | 深度分析用户 |
| 周期性波动 | 规律性涨跌 | 看同期对比 | 正常波动，无需干预 |

### 3.2 问题焦点确定

**交叉分析**：将多个维度的异常点进行交叉，找到共性

```
渠道异常 + 时间异常 → 某渠道在某时段的问题
用户异常 + 功能异常 → 某用户群体对某功能的问题
渠道异常 + 用户异常 → 某渠道某用户群体的专属问题
```

### 3.3 排除干扰因素

| 干扰因素 | 排除方法 |
|---------|---------|
| 外部事件 | 排查是否有节假日、竞品动作 |
| 技术问题 | 排查是否有系统升级、故障 |
| 数据问题 | 排查数据采集、统计口径 |
| 正常波动 | 对比历史同期，排除周期性 |

---

## 第四步：提出假设

### 4.1 假设生成框架

每个假设必须包含：

| 要素 | 说明 |
|-----|-----|
| 假设内容 | 用一句话描述可能原因 |
| 数据依据 | 支撑这个假设的数据是什么 |
| 概率评估 | 高/中/低 |
| 验证方法 | 如何验证这个假设 |

### 4.2 常见假设类型

| 类型 | 示例假设 | 典型数据特征 |
|-----|---------|-------------|
| 产品问题 | 新功能体验差/旧功能退化 | 某功能使用率下降 |
| 技术问题 | 页面加载慢/支付故障 | 技术指标异常 |
| 运营问题 | 活动效果差/推送过度 | 运营指标变化 |
| 市场问题 | 竞品分流/需求下降 | 整体市场下跌 |
| 用户问题 | 目标用户流失 | 某类用户下跌明显 |

### 4.3 假设优先级排序

| 优先级 | 排序依据 |
|-------|---------|
| P0 | 高概率 + 高影响 + 验证成本低 |
| P1 | 高概率 + 可影响 或 中概率 + 高影响 |
| P2 | 中概率 + 可影响 或 低概率 + 高影响 |
| P3 | 低概率 + 低影响 |

---

## 第五步：验证解决

### 5.1 验证方法

| 验证类型 | 方法 | 适用场景 |
|---------|-----|---------|
| 数据验证 | 直接分析数据确认 | 数据充足时 |
| 对比验证 | A/B测试或分群对比 | 可做实验时 |
| 用户验证 | 用户调研或访谈 | 需要定性确认时 |
| 技术验证 | 技术排查或日志分析 | 技术问题时 |
| 外部验证 | 竞品分析或市场调研 | 外部因素时 |

### 5.2 验证结果判断

| 结论 | 说明 |
|-----|-----|
| 假设成立 | 数据明确支撑，可作为根因 |
| 假设不成立 | 数据不支持，需重新假设 |
| 部分成立 | 有一定支撑，但非主因 |

### 5.3 解决方案制定

| 要素 | 要求 |
|-----|-----|
| 具体措施 | 要有可执行的行动项 |
| 优先级 | 按重要程度和紧急程度排序 |
| 资源需求 | 明确需要的资源和支持 |
| 预期效果 | 要有可量化的预期 |
| 时间周期 | 明确完成时间 |
| 验收标准 | 如何判断问题解决 |
