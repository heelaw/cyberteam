# 计算方法论

## 1. LTV计算流程总览

### 1.1 五步计算法

```
Step 1: 业务分析
    ↓
Step 2: 数据收集与验证
    ↓
Step 3: 模型选择
    ↓
Step 4: LTV计算
    ↓
Step 5: 决策建议
```

### 1.2 各阶段时间分配

| 阶段 | 建议时间 | 重要性 |
|------|----------|--------|
| 业务分析 | 15分钟 | ★★★ |
| 数据收集 | 25分钟 | ★★★ |
| 模型选择 | 10分钟 | ★★★ |
| LTV计算 | 25分钟 | ★★☆ |
| 决策建议 | 15分钟 | ★★☆ |

## 2. Step 1: 业务分析

### 2.1 产品类型识别

#### 消磨时间产品
- **特征**: 内容消费、沉浸体验、社交互动
- **典型**: 短视频、社交媒体、内容平台
- **数据重点**: 活跃度、留存、付费率

#### 效率提升产品
- **特征**: 工具属性、任务导向、付费意愿高
- **典型**: SaaS、工具软件、协作平台
- **数据重点**: 功能使用、续费率、ARPU

#### 交易类产品
- **特征**: 交易导向、供需双边、复购重要
- **典型**: 电商、本地生活、出行平台
- **数据重点**: 复购率、客单价、交易频次

#### 虚拟产品/游戏
- **特征**: 娱乐导向、付费分层明显
- **典型**: 游戏、虚拟物品、增值服务
- **数据重点**: 付费深度、分层分布、ARPPU

### 2.2 商业模式确定

| 商业模式 | 收入来源 | LTV计算重点 |
|----------|----------|-------------|
| 广告变现 | 广告点击/展示 | 用户注意力价值 |
| 订阅制 | 会员费/月费 | 续费率、订阅时长 |
| 交易抽成 | 交易佣金 | 交易额、抽成比例 |
| 增值服务 | 虚拟物品/功能 | 付费率、付费深度 |
| 混合模式 | 多种组合 | 综合计算 |

### 2.3 计算目标明确

| 目标 | 需要的LTV类型 | 决策问题 |
|------|---------------|----------|
| 获客预算 | LTV/CAC比值 | 能否规模化获客？ |
| 会员定价 | 利润LTV | 定价是否合理？ |
| 产品优先级 | 各类用户LTV | 哪些功能值得投入？ |
| 投资估值 | 预测LTV | 公司值多少钱？ |
| 渠道优化 | 渠道LTV | 哪个渠道最有效？ |

## 3. Step 2: 数据收集与验证

### 3.1 核心数据清单

#### 必填数据
| 数据项 | 符号 | 说明 |
|--------|------|------|
| ARPU | A | 平均每用户收入 |
| 流失率 | C | 月流失比例 |
| 用户生命周期 | T | 平均存活月数 |

#### 选填数据（增强准确性）
| 数据项 | 符号 | 说明 |
|--------|------|------|
| CAC | - | 获客成本 |
| 客单价 | - | 平均订单金额 |
| 复购率 | - | 用户复购比例 |
| 毛利率 | - | 收入转化为利润比例 |

### 3.2 数据获取渠道

| 数据类型 | 推荐来源 | 注意事项 |
|----------|----------|----------|
| ARPU | 财务系统/BI | 明确时间周期 |
| 流失率 | 运营后台 | 区分用户/收入流失 |
| CAC | 投放平台 | 区分不同渠道 |
| 复购率 | 交易数据库 | 定义复购周期 |

### 3.3 数据验证方法

#### 一致性检验
```
检查项:
- 时间范围是否一致
- 用户口径是否一致
- 收入定义是否一致
```

#### 交叉验证
```
示例:
- ARPU验证: 收入/用户数 = 收入/付费用户数 × 付费率
- 流失率验证: 期末用户 = 期初用户 - 流失 + 新增
```

#### 异常值处理
```
步骤:
1. 识别异常值（3σ原则或IQR方法）
2. 分析异常原因
3. 决定保留/修正/剔除
4. 记录处理过程
```

### 3.4 置信度评估

| 置信度 | 条件 | 影响 |
|--------|------|------|
| 高 | 多数据源一致、周期长 | 可直接使用 |
| 中 | 单数据源、周期中等 | 建议敏感性分析 |
| 低 | 数据缺失、周期短 | 标注为估算值 |

## 4. Step 3: 模型选择

### 4.1 模型对比

| 模型 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **Simple LTV** | 流失稳定的产品 | 简单、快速 | 不够精确 |
| **Cohort LTV** | 有明确Cohort的产品 | 准确、动态 | 需要足够数据 |
| **Marginal LTV** | 付费分层明显的产品 | 考虑差异 | 计算复杂 |
| **Predictive LTV** | 需要预测未来 | 考虑多种因素 | 需要模型能力 |

### 4.2 模型选择决策树

```
产品有足够历史数据（≥6个月）？
│
├─ 是 → 数据按Cohort可分？
│       │
│       ├─ 是 → 使用 Cohort LTV
│       │
│       └─ 否 → 流失率是否稳定？
│               │
│               ├─ 是 → 使用 Simple LTV
│               │
│               └─ 否 → 使用 Predictive LTV
│
└─ 否 → 产品有明显的付费分层？
        │
        ├─ 是 → 使用 Marginal LTV
        │
        └─ 否 → 使用 Simple LTV（估算）
```

### 4.3 各模型详细说明

#### Simple LTV（简单LTV）
```
公式: LTV = ARPU × (1 / 月流失率)

适用条件:
- 流失率相对稳定
- ARPU较为稳定
- 产品进入成熟期

示例（月流失率5%）:
LTV = ¥100 × 20 = ¥2000
```

#### Cohort LTV（Cohort分析）
```
公式: LTV = Σ(Cohort月收入 × 存活率) / 初始用户数

适用条件:
- 有至少6个月的Cohort数据
- 不同时间段的Cohort表现可能有差异
- 需要分析用户生命周期变化

示例:
Month 1: 收入¥100, 存活100%
Month 2: 收入¥80, 存活70%
Month 3: 收入¥60, 存活50%
LTV = (100+80+60) / 1 = ¥240
```

#### Marginal LTV（边际LTV）
```
公式: LTV = Σ(每层付费额 × 该层用户占比)

适用条件:
- 付费用户差异大（如游戏）
- 需要考虑不同用户群的不同价值

示例（游戏付费分层）:
免费用户(60%): ¥0
小R(25%): ¥30
中R(10%): ¥200
大R(5%): ¥2000
LTV = 0×0.6 + 30×0.25 + 200×0.1 + 2000×0.05 = ¥7.5 + ¥20 + ¥100 = ¥127.5
```

#### Predictive LTV（预测LTV）
```
公式: LTV = Σ [P(存活_t) × ARPU_t × 折现率]

适用条件:
- 需要预测未来
- 数据量足够训练模型
- 产品变化较快

常用方法:
- 回归模型
- 机器学习模型
- 贝叶斯模型
```

## 5. Step 4: LTV计算

### 5.1 Simple LTV计算示例

```
步骤1: 确定ARPU
- 月收入: ¥500,000
- 月活用户: 10,000
- ARPU = 500,000 / 10,000 = ¥50

步骤2: 确定流失率
- 月初用户: 10,000
- 月末用户: 9,500
- 流失率 = (10,000-9,500)/10,000 = 5%

步骤3: 计算LTV
- LTV = ¥50 × (1/0.05) = ¥50 × 20 = ¥1000
```

### 5.2 Cohort LTV计算示例

```
数据: 某电商产品1月份Cohort（1000用户）

Month 1:
- 收入: ¥100,000
- 用户: 1000（100%存活）
- ARPU: ¥100

Month 2:
- 收入: ¥70,000
- 用户: 700（70%存活）
- ARPU: ¥100

Month 3:
- 收入: ¥55,000
- 用户: 500（50%存活）
- ARPU: ¥110

LTV计算:
Cohort LTV = (100×1.0 + 100×0.7 + 110×0.5) / 1.0
           = 100 + 70 + 55
           = ¥225
```

### 5.3 敏感性分析

#### 为什么要做敏感性分析？
- LTV计算基于假设，假设可能变化
- 关键变量变化对结果影响程度不同
- 帮助理解结果的稳健性

#### 敏感性分析方法

```
步骤:
1. 识别关键变量（流失率、ARPU、周期）
2. 设定变化范围（±10%, ±20%）
3. 逐一计算影响
4. 绘制 Tornado 图

示例:
基准LTV = ¥1000

流失率±10%变化:
- 流失率 4.5%: LTV = ¥1111 (+11%)
- 流失率 5.5%: LTV = ¥909 (-9%)

ARPU±10%变化:
- ARPU ¥55: LTV = ¥1100 (+10%)
- ARPU ¥45: LTV = ¥900 (-10%)
```

## 6. Step 5: 决策建议

### 6.1 LTV/CAC健康度评估

| LTV/CAC | 健康等级 | 颜色 | 建议 |
|---------|----------|------|------|
| > 5:1 | 优秀 | 绿色 | 大幅增加获客投入 |
| 3-5:1 | 健康 | 绿色 | 稳步获客，优化效率 |
| 1-3:1 | 警告 | 黄色 | 优化获客渠道，提升LTV |
| < 1:1 | 危险 | 红色 | 暂停获客，重新审视模式 |

### 6.2 具体建议框架

#### 基于LTV/CAC的建议
```
LTV/CAC > 5:1:
- 加速获客
- 测试新渠道
- 适当提高CAC

LTV/CAC 3-5:1:
- 保持当前获客规模
- 优化转化漏斗
- 提升用户留存

LTV/CAC 1-3:1:
- 诊断哪个环节出问题
- 优先提升LTV还是降低CAC
- 分析用户分群

LTV/CAC < 1:1:
- 暂停大规模获客
- 深入分析商业模式
- 产品/market fit问题
```

#### 基于产品阶段的建议
```
早期产品（数据少）:
- 使用Simple LTV
- 定期更新
- 关注趋势而非绝对值

成长期产品（数据增长）:
- 建立Cohort分析
- 按渠道分群计算
- 关注LTV变化趋势

成熟期产品（数据丰富）:
- 使用Predictive LTV
- 精细化运营
- A/B测试验证
```

### 6.3 风险提示

| 风险类型 | 描述 | 应对方法 |
|----------|------|----------|
| 数据不准确 | 基础数据有误 | 交叉验证 |
| 假设变化 | 市场/产品变化 | 敏感性分析 |
| 季节性 | 周期性波动 | 使用同期对比 |
| 新用户差异 | 历史不预测未来 | 分群分析 |

### 6.4 后续验证计划

```
建议:
1. 每月更新LTV计算
2. 追踪LTV趋势变化
3. 按渠道/用户群分群计算
4. 对比预测LTV与实际LTV
5. 持续优化数据质量
```
