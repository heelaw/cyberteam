# 编码风格

## 不变性（关键）

始终创建新对象，永远不要改变现有对象：```
// Pseudocode
WRONG:  modify(original, field, value) → changes original in-place
CORRECT: update(original, field, value) → returns new copy with change
```理由：不可变数据可以防止隐藏的副作用，使调试更容易，并实现安全并发​​。

## 文件组织

很多小文件 > 很少的大文件：
- 高内聚、低耦合
- 典型 200-400 行，最多 800 行
- 从大型模块中提取实用程序
- 按功能/领域而不是类型进行组织

## 错误处理

始终全面处理错误：
- 在每个级别明确处理错误
- 在面向 UI 的代码中提供用户友好的错误消息
- 在服务器端记录详细的错误上下文
- 永远不要默默地吞下错误

## 输入验证

始终在系统边界进行验证：
- 在处理之前验证所有用户输入
- 在可用的情况下使用基于模式的验证
- 快速失败并提供清晰的错误消息
- 永远不要相信外部数据（API 响应、用户输入、文件内容）

## 代码质量检查表

在标记工作完成之前：
- [ ] 代码可读且命名良好
- [ ] 函数很小（<50 行）
- [ ] 文件被聚焦（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 正确的错误处理
- [ ] 无硬编码值（使用常量或配置）
- [ ] 无突变（使用不可变模式）
