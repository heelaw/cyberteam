# Ralphinho RFC 管道

受到 [ humanplane ](https://github.com/ humanplane) 风格的 RFC 分解模式和多单元编排工作流程的启发。

当某个功能对于单个代理传递来说太大并且必须拆分为可独立验证的工作单元时，请使用此技能。

## 管道阶段

1. RFC摄入
2. DAG分解
3. 单元分配
四、单位实施
5. 单元验证
6. 合并队列和集成
7. 最终系统验证

## 单位规格模板

每个工作单元应包括：
- `id`
- `取决于`
- `范围`
- `验收测试`
- `风险级别`
- `回滚计划`

## 复杂度等级

- 第 1 层：独立文件编辑、确定性测试
- 第 2 层：多文件行为变化，中等集成风险
- 第 3 层：架构/身份验证/性能/安全更改

## 每单位的质量管道

1. 研究
2、实施方案
3. 实施
4. 测试
5. 审查
6. 合并就绪报告

## 合并队列规则

- 切勿合并具有未解决的依赖性故障的单元。
- 始终在最新的集成分支上重新调整单元分支的基础。
- 每次排队合并后重新运行集成测试。

## 恢复

如果设备停转：
- 从活动队列中逐出
- 快照发现
- 重新生成缩小的单位范围
- 使用更新的约束重试

## 输出

- RFC执行日志
- 单位记分卡
- 依赖图快照
- 整合风险总结
