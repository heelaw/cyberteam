# 构建并修复

通过最少的安全更改逐步修复构建和类型错误。

## 步骤 1：检测构建系统

识别项目的构建工具并运行构建：

|指标|构建命令|
|------------|--------------|
|带有“build”脚本的“package.json”| `npm run build` 或 `pnpm build` |
| `tsconfig.json`（仅限 TypeScript）| `npx tsc --noEmit` |
| `Cargo.toml` | `货物构建 2>&1` |
| `pom.xml` | `mvn 编译` |
| `build.gradle` | `./gradlew编译Java` |
| `go.mod` | `去构建./...` |
| `pyproject.toml` | `python -m py_compile` 或 `mypy 。` |

## 步骤 2：解析和分组错误

1. 运行构建命令并捕获stderr
2. 按文件路径对错误进行分组
3.按依赖顺序排序（修复逻辑错误之前的导入/类型）
4. 统计错误总数以跟踪进度

## 步骤 3：修复循环（一次一个错误）

对于每个错误：

1. **读取文件** — 使用读取工具查看错误上下文（错误周围 10 行）
2. **诊断** — 确定根本原因（缺少导入、类型错误、语法错误）
3. **最小修复** — 使用编辑工具进行可解决错误的最小更改
4. **重新运行构建** — 验证错误已消失并且没有引入新的错误
5. **移至下一个** — 继续处理剩余的错误

## 步骤 4：护栏

停下来询问用户是否：
- 修复引入的错误**多于它解决的错误**
- **3次尝试后仍然存在相同的错误**（可能是更深层次的问题）
- 该修复需要**架构更改**（不仅仅是构建修复）
- 构建错误源于**缺少依赖项**（需要“npm install”、“cargo add”等）

## 步骤 5：总结

显示结果：
- 修复了错误（带有文件路径）
- 剩余错误（如果有）
- 引入了新的错误（应该为零）
- 针对未解决的问题建议的后续步骤

## 恢复策略

|情况|行动|
|------------|--------|
|缺少模块/导入 |检查包是否安装；建议安装命令 |
|类型不匹配 |阅读两个类型定义；修复较窄的类型 |
|循环依赖|用导入图识别周期；建议提取|
|版本冲突 |检查 `package.json` / `Cargo.toml` 的版本限制 |
|构建工具配置错误 |读取配置文件；与工作默认值比较|

为了安全起见，一次修复一个错误。与重构相比，更喜欢最小的差异。
