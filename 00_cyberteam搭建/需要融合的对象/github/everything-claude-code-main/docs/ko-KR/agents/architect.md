# ADR-001: Use Redis for Semantic Search Vector Storage

## Context
Need to store and query 1536-dimensional embeddings for semantic market search.

## Decision
Use Redis Stack with vector search capability.

## Consequences

### Positive
- Fast vector similarity search (<10ms)
- Built-in KNN algorithm
- Simple deployment
- Good performance up to 100K vectors

### Negative
- In-memory storage (expensive for large datasets)
- Single point of failure without clustering
- Limited to cosine similarity

### Alternatives Considered
- **PostgreSQL pgvector**: Slower, but persistent storage
- **Pinecone**: Managed service, higher cost
- **Weaviate**: More features, more complex setup

## Status
Accepted

## 프로젝트별 아키텍처 (예시)

AI 和 SaaS 的应用：

### 현재 아키텍처
- **前端**：Next.js 15（Vercel/Cloud Run）
- **后端**：FastAPI 또는 Express（Cloud Run/Railway）
- **数据库**：PostgreSQL (Supabase)
- **缓存**：Redis（Upstash/Railway）
- **AI**：具有结构化输出的 Claude API
- **实时**：Supabase 订阅

### 주요 설계 결정
1. **하브리드 배포**：최적 성능을 위한 Vercel（前端）+ Cloud Run（后端）
2. **AI 통합**: 타입 안전성을 위한 Pydantic/Zod 기반 结构化输出
3. **실시간 업데֊**: 라ּ브 데ք를 위한 Supabase 订阅
4. **불변 패턴**: 예측 і능한 상태를 위한 扩展运算符
5. **작은 파일 다수**: 높은 응집도, 낮은 결합도

### 확장성 계획
- **1만 사용자**: 현재 아키텍처로 충분
- **10 部分**：Redis 服务、CDN 服务
- **100만사용자**: 마마크로서비스 아키텍처, 읽기/쓰기 데dling터베비스 분리
- **1000만 사용자**：事件驱动架构、분산 캐싱、멀티 리전

**기억하세요**: 좋은 아키텍처는 빠른 개발, 쉬운 유지보수, 그리고 자신 있는 확장을 и능하게 합니다。 최고의 아키텍처는 단순하고、명확하며、검증된 패턴을 따릅니다。
