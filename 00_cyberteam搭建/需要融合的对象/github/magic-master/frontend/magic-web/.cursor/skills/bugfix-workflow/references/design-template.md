# design.md 模板（弹性版）

这个模板强调“要求导向”，而不是固定章节导向。
目标是保证质量下限，同时给不同复杂度问题足够的输出自由度。

## 一、最小必需要求（任何复杂度都必须满足）

- 必须说清 3 件事：
  - 根因是什么
  - 修复方案是什么
  - 如何验证修复且不回归
- 必须引用至少 1 条可追溯证据（代码路径、日志、截图、复现片段、测试之一）
- 必须明确本次修复边界（改什么 / 不改什么）
- 若存在 `Unchanged Behavior`，必须给对应验证方式

> 最小必需要求是硬约束；文档结构不是硬约束。

## 二、按复杂度选择输出深度

### L1 轻量问题（单点修复、影响面小）

推荐包含：
- 问题现象（1 段）
- 根因（1-2 条）
- 修复要点（1-3 条）
- 验证结果（自动化或手工）

图示建议：
- 可选 0-1 张简图（仅当图比文字更清晰时）

### L2 标准问题（涉及多模块或存在相邻回归风险）

推荐包含：
- 问题复盘（触发条件 + 实际/期望）
- 根因证据链（代码 + 复现证据）
- 方案对比（至少 2 个候选）
- 选型理由与改动范围
- 测试策略（复现 / 修复 / 回归）

图示建议：
- 建议 1-2 张图（例如失败路径图、修复后路径图）

### L3 复杂问题（跨链路、时序敏感、易回归）

推荐包含：
- 系统化复盘（上下游链路、状态转换、边界条件）
- 完整证据链与未证实假设
- 多方案权衡表（改动面 / 风险 / 验证成本）
- 发布与回滚预案
- 后续技术债治理建议

图示建议：
- 建议 2-4 张图（时序图、状态图、架构图、决策图择优）

## 三、可选内容组件库（按需组合）

以下是“可选组件”，不是必须章节。AI 可按问题选择：

- 概览：范围、非目标、约束
- 问题复盘：触发条件、实际/期望
- 失败路径图：时序或流程
- 最小复现结构：DOM/数据结构/状态图
- 根因分解：表象 -> 直接原因 -> 深层原因
- 候选方案对比表
- 详细改动点：代码路径 + 前后逻辑
- 修复后路径图
- 测试矩阵
- 风险与后续

## 四、图示使用准则（强调“有用”而非“数量”）

- 图是为了降低理解成本，不是为了凑格式
- 优先画“文字难解释清楚”的部分：
  - 时序关系
  - 多节点命中关系
  - 状态转换
  - 决策分支
- 若问题足够简单，可以不画图；中高复杂度问题建议至少 1 张关键图

## 五、推荐骨架（可增删改）

~~~md
# 设计：[缺陷标题]

## 问题复盘（可选）
- 触发条件：
- 实际结果：
- 期望结果：

## 根因分析（必需）
- 根因结论：
- 证据：

## 修复方案（必需）
- 选定方案：
- 边界（改 / 不改）：

## 验证计划（必需）
- 修复验证：
- 回归验证：

## 其它（按需）
- 方案对比 / 图示 / 风险 / 后续
~~~

## 六、质量检查清单

- [ ] 是否明确根因，而非只描述现象
- [ ] 是否说明“为何选这个方案”
- [ ] 是否覆盖修复验证 + 回归验证
- [ ] 是否标注证据与假设边界
- [ ] 文档复杂度是否与问题复杂度匹配（不过度，也不欠缺）
