# CyberTeam 四个核心 Agent 设计书

## 1. 设计原则

- 第一版只做 4 个核心 Agent
- 每个 Agent 都要有明确职责边界
- 每个 Agent 都要能被映射到实际产品模块
- 每个 Agent 都要能服务于 CyberTeam 的主链路

## 2. 四个核心 Agent

### 2.1 CEO / 产品 Agent

**参考来源**：CEO视角分析器、策略执行、框架思维

#### 定位
负责产品方向、范围控制、优先级排序、商业判断。

#### 核心职责
- 评估项目值不值得做
- 判断产品方向是否正确
- 控制 MVP 范围
- 决定先做什么、后做什么
- 用 CEO 视角审查方案

#### 输入
- 产品想法
- 功能需求
- 商业目标
- 用户反馈
- 技术方案

#### 输出
- 产品判断
- 方向建议
- 范围裁剪
- 优先级排序
- CEO 审核意见

#### 适合任务
- 产品定义
- 方案评审
- MVP 裁剪
- 战略判断
- 风险识别

#### 提示词方向
要从 CEO 的经营管理链条看问题，优先判断商业可行性、产品方向、范围大小和执行重点。

---

### 2.2 架构 Agent

**参考来源**：框架思维Agent、业务模型Agent、全局业务模型构建器

#### 定位
负责整体系统框架、模块拆解、数据结构、目录设计、技术边界。

#### 核心职责
- 建立系统框架
- 划分模块
- 设计目录结构
- 定义数据模型
- 拆解复杂问题
- 输出技术方案

#### 输入
- 产品目标
- 功能清单
- 业务边界
- 技术约束
- 当前系统状态

#### 输出
- 架构设计
- 模块图
- 目录结构
- 数据模型
- 任务拆解

#### 适合任务
- 技术方案设计
- 架构评审
- 数据库规划
- 系统拆分
- 代码组织

#### 提示词方向
用系统性框架思维把复杂问题拆成层次清晰、可执行的结构，并给出技术实现路径。

---

### 2.3 前端 Agent

**参考来源**：文案Agent、活动策划Agent、用户激励Agent、活动运营进阶Agent

#### 定位
负责界面表现、交互体验、聊天界面、组织界面、Playground 视觉化。

#### 核心职责
- 设计聊天页
- 设计组织架构页
- 设计 Playground 页
- 打磨类微信视觉
- 处理交互细节
- 提升用户激活和留存

#### 输入
- 页面需求
- 交互目标
- 用户场景
- 视觉要求

#### 输出
- 页面方案
- 组件结构
- 交互流程
- UI 文案
- 视觉建议

#### 适合任务
- 页面原型
- UI 细化
- 聊天体验设计
- 组件设计
- 前端文案

#### 提示词方向
围绕真实聊天软件的体验做设计，让 Agent 的身份感、沟通感和组织感都变强。

---

### 2.4 基础设施 Agent

**参考来源**：策略执行Agent、工作管理Agent、进度追踪器、行动框架Agent

#### 定位
负责 Electron、Claude Code 接入、SQLite、本地配置、运行链路、最小可运行系统。

#### 核心职责
- 搭建桌面壳
- 连接 Claude Code
- 管理本地数据
- 实现会话流式输出
- 处理 IPC / SSE / 存储
- 保证系统可运行

#### 输入
- 技术方案
- 目录结构
- 环境约束
- 运行要求

#### 输出
- 启动代码
- 接入代码
- 存储代码
- 通信代码
- 骨架代码

#### 适合任务
- Electron 初始化
- SQLite schema
- Claude Code bridge
- IPC 通信
- 运行时接入

#### 提示词方向
以“先跑起来”为目标，把系统底座、会话、存储和通信打通。

---

## 3. 四个 Agent 的协作方式

```text
用户需求
   ↓
CEO / 产品 Agent
   ↓
架构 Agent
   ↓
前端 Agent + 基础设施 Agent
   ↓
输出到 CyberTeam 主项目
```

### 协作规则
- CEO / 产品 Agent 决定方向和范围
- 架构 Agent 拆系统和目录
- 前端 Agent 负责体验
- 基础设施 Agent 负责能跑起来

## 4. 第一版不要做的 Agent

- QA Agent：后面再补
- 文档 Agent：可先由人类和 gstack 补
- 市场 Agent：后面再做
- PUA Agent：只保留扩展接口，不纳入核心执行链路
