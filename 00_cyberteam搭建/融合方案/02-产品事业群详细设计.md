## 一、部门概述

### 1.1 产品事业群定位

产品事业群是CyberTeam的核心价值创造部门，负责将业务需求转化为可实现的产品方案。作为业务团队与技术团队之间的桥梁，产品部确保产品方向与用户价值高度一致，设计部则负责将产品方案转化为高质量的视觉和交互体验。

**核心价值主张**：
- 深度理解用户痛点，转化为清晰的产品需求
- 快速响应市场变化，提供可执行的解决方案
- 协调多方资源，确保产品按时交付
- 持续优化用户体验，提升产品竞争力

### 1.2 产品事业群使命

> **"连接用户与产品，用洞察驱动创新，用设计传递价值"**

### 1.3 产品事业群愿景

| 时间维度 | 愿景目标 |
|----------|----------|
| 短期（3个月） | 建立标准化的产品设计和协作流程，支撑业务快速迭代 |
| 中期（6个月） | 形成完整的产品方法论体系，输出可复制的最佳实践 |
| 长期（12个月） | 成为行业领先的产品设计团队，驱动业务持续增长 |

## 三、岗位职责

### 3.1 产品部职责

#### 3.1.1 需求分析组

**小组定位**：需求的入口和过滤器，负责深度理解业务需求，转化为可执行的产品机会。

**核心职责**：

| 职责领域 | 具体工作内容 | 输出物 |
|----------|--------------|--------|
| 需求采集 | 运营/市场/客服反馈收集 | 需求池台账 |
| 需求分析 | 问题根因分析、价值评估 | 需求分析报告 |
| 需求分类 | 业务型/优化型/技术型分类 | 需求分类标签 |
| 需求澄清 | 与需求方对齐理解、确认边界 | 需求确认函 |
| 优先级评估 | RICE/价值-成本矩阵评估 | 优先级排序表 |

**关键协作接口**：

```
输入 → 运营部/市场部/客服部/技术部
输出 → PRD组 / 产品规划组
```

**岗位能力要求**：

| 能力维度 | 要求等级 | 说明 |
|----------|----------|------|
| 业务理解力 | L4-专家 | 能够深度理解业务本质和商业模式 |
| 分析能力 | L4-专家 | 熟练运用SWOT/波特五力等分析框架 |
| 沟通能力 | L3-高级 | 跨部门协调、推动多方共识 |
| 技术理解力 | L2-中级 | 理解技术实现基本原理和限制 |

**需求分析师 Job Story**：
```
作为需求分析师
我需要深入业务一线，采集和筛选真实需求
以便于将模糊的业务痛点转化为清晰的产品机会
同时过滤低价值和不可行的需求进入产品流水线
```

#### 3.1.2 PRD组

**小组定位**：产品文档的核心生产者，负责将需求转化为高质量的产品规格说明。

**核心职责**：

| 职责领域 | 具体工作内容 | 输出物 |
|----------|--------------|--------|
| PRD撰写 | 功能需求、非功能需求编写 | PRD文档 |
| 原型设计 | 低保真原型制作 | 交互原型 |
| 需求评审 | 组织需求评审会议 | 评审纪要 |
| 需求变更 | 变更评估、版本管理 | 变更记录 |
| 验收标准 | 功能验收条件定义 | 验收检查表 |

**关键协作接口**：

```
输入 → 需求分析组（已澄清需求）
输出 → 设计部（原型+需求说明）/ 开发部（PRD+数据模型）
```

**岗位能力要求**：

| 能力维度 | 要求等级 | 说明 |
|----------|----------|------|
| 文档能力 | L4-专家 | 结构化写作、逻辑严密、无歧义 |
| 原型设计 | L3-高级 | 熟练使用Figma/Axure制作原型 |
| 技术基础 | L3-高级 | 理解数据模型、API设计基本原理 |
| 审美能力 | L2-中级 | 理解UI设计基本规范 |

**PRD撰写师 Job Story**：
```
作为PRD撰写师
我需要将模糊的需求转化为精确的产品规格
以便于设计团队能够准确理解交互逻辑
开发团队能够无歧义地实现功能
测试团队能够制定完整的验收标准
```

#### 3.1.3 产品规划组

**小组定位**：产品的中长期规划者，负责版本路线图和资源协调。

**核心职责**：

| 职责领域 | 具体工作内容 | 输出物 |
|----------|--------------|--------|
| 版本规划 | 里程碑制定、版本范围定义 | 产品路线图 |
| 资源协调 | 人力/时间/预算资源评估 | 资源计划表 |
| 风险管控 | 风险识别、应对策略制定 | 风险登记册 |
| 跨BG协调 | 依赖部门沟通、进度同步 | 协调会议纪要 |
| ROI评估 | 功能投入产出分析 | ROI分析报告 |

**关键协作接口**：

```
输入 → CEO/战略部/各BG总监
输出 → CEO（汇报）/ 各执行团队（路线图同步）
```

**岗位能力要求**：

| 能力维度 | 要求等级 | 说明 |
|----------|----------|------|
| 战略思维 | L4-专家 | 能够从全局视角制定产品规划 |
| 项目管理 | L4-专家 | 熟练运用敏捷/精益项目管理方法 |
| 商业敏感度 | L3-高级 | 理解商业模式、竞争格局 |
| 决策能力 | L4-专家 | 在不确定性下做出合理决策 |

**产品规划师 Job Story**：
```
作为产品规划师
我需要在有限资源下做出最优的产品组合决策
以便于在竞争环境中最大化业务价值
同时确保各团队目标一致、节奏协调
```

## 五、协作流程

### 5.1 需求到设计稿完整流程

#### 5.1.1 主流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        需求到设计稿流程                           │
└─────────────────────────────────────────────────────────────────┘

[需求发起] → [需求分析] → [需求评审] → [PRD撰写] → [PRD评审]
    ↓            ↓            ↓            ↓            ↓
  运营部       需求分析组    产品+设计     PRD组       产品+设计
                             联合评审                   联合评审
                                                                ↓
[设计交付] ← [设计评审] ← [UI设计] ← [原型确认] ← [交互设计]
    ↓            ↓            ↓            ↓            ↓
  开发部       产品+UX       UI设计组     PRD+UX       PRD组
                             联合评审     研究组评审
```

#### 5.1.2 详细流程步骤

| 阶段 | 步骤 | 责任方 | 交付物 | 时长 |
|------|------|--------|--------|------|
| 需求发起 | 需求方提交需求 | 需求方 | 需求申请单 | 按需 |
| 需求分析 | 需求分析组进行需求澄清 | 需求分析师 | 需求分析报告 | 2天 |
| 需求评审 | 评审需求可行性和优先级 | 产品部总监 | 评审纪要 | 1天 |
| PRD撰写 | PRD组撰写产品规格文档 | PRD撰写师 | PRD文档 | 3天 |
| PRD评审 | 评审PRD完整性和合理性 | 产品+设计联合 | 评审纪要 | 1天 |
| 交互设计 | 制作交互原型和说明 | PRD组 | 原型+交互说明 | 2天 |
| UI设计 | 基于原型进行UI设计 | UI设计组 | 设计稿 | 3天 |
| 设计评审 | 评审设计稿符合度和体验 | 产品+UX联合 | 评审纪要 | 1天 |
| 设计交付 | 交付最终设计稿和标注 | UI设计组 | 设计交付包 | 1天 |

**总工期**：标准需求 **15个工作日**，紧急需求 **7个工作日**

### 5.2 各阶段详细说明

#### 5.2.1 需求发起阶段

**触发条件**：
- 运营部提出运营优化需求
- 市场部提出市场机会需求
- 技术部提出技术改进需求
- 客服部收集到用户反馈
- 管理层战略规划需求

**输入**：
- 需求申请单（含业务背景、期望、期限）
- 相关数据支撑材料

**输出**：
- 需求池登记记录
- 初步需求分类

**流程**：
```
1. 需求方填写需求申请单
2. 需求分析师接收并确认
3. 初步分类（业务型/优化型/技术型）
4. 入库需求池
5. 通知需求方已接收
```

#### 5.2.2 需求分析阶段

**执行标准**：
- 需求分析师需与需求方进行至少1次深度沟通
- 输出需求分析报告必须包含：问题定义、目标用户、用户故事、验收条件
- 涉及多个部门的需求需组织需求对齐会

**输出模板**：

```markdown
## 需求分析报告

### 1. 需求概述
- **需求ID**：REQ-2026-XXXX
- **需求名称**：[名称]
- **需求类型**：[类型]
- **提出方**：[部门/角色]
- **提出时间**：[日期]

### 2. 问题定义
- **现状描述**：[当前存在的问题]
- **影响范围**：[影响的用户/业务范围]
- **损失估算**：[量化影响]

### 3. 用户分析
- **目标用户**：[用户画像描述]
- **用户场景**：[使用场景描述]
- **用户动机**：[用户为什么需要这个功能]

### 4. 期望方案
- **功能描述**：[详细的功能说明]
- **成功标准**：[如何衡量成功]

### 5. 验收条件
- [ ] 条件1
- [ ] 条件2
- [ ] 条件3

### 6. 优先级建议
| 维度 | 评分 | 说明 |
|------|------|------|
| 紧急程度 | [1-5] | |
| 业务价值 | [1-5] | |
| 实现难度 | [1-5] | |
| **综合评分** | [计算] | |

### 7. 风险提示
- [ ] 风险1及应对
- [ ] 风险2及应对
```

#### 5.2.3 需求评审阶段

**评审参与方**：
- 产品部：总监、需求分析师、PRD撰写师
- 设计部：总监、UX研究员
- 需求方代表
- 技术部（可选）：技术可行性评估

**评审标准**：

| 评审维度 | 通过条件 | 否决条件 |
|----------|----------|----------|
| 需求清晰度 | 问题描述清晰、无歧义 | 存在理解模糊点 |
| 价值明确性 | 有明确的价值主张 | 无法说明业务价值 |
| 可行性 | 技术/资源可行性确认 | 存在重大不可行因素 |
| 优先级合理性 | RICE评分与业务策略一致 | 评分与业务优先级矛盾 |
| 验收条件完备性 | 验收条件可测试 | 存在无法验证的条件 |

**评审决策**：
- **通过**：进入PRD撰写阶段
- **需要修改**：返回需求分析组补充
- **暂缓**：进入需求池待定区
- **否决**：通知需求方并说明原因

#### 5.2.4 PRD撰写阶段

**执行标准**：
- 必须使用标准PRD模板
- 必须包含完整的用户故事
- 必须定义清晰的验收标准
- 必须包含数据埋点需求

**质量门禁**：
- 文档结构完整性检查（必须包含章节：概述、用户分析、功能详设、数据模型、非功能需求、验收标准）
- 术语一致性检查
- 链接有效性检查

#### 5.2.5 PRD评审阶段

**评审参与方**：
- 产品部：PRD撰写师、产品规划师
- 设计部：UX研究员、UI设计师代表
- 技术部：技术负责人

**评审检查清单**：

| 检查项 | 检查内容 | 权重 |
|--------|----------|------|
| 需求完整性 | 所有需求点已覆盖 | 20% |
| 逻辑一致性 | 功能逻辑前后一致 | 20% |
| 技术可行性 | 技术方案可实现 | 20% |
| 用户体验合理 | 交互逻辑符合用户习惯 | 15% |
| 验收条件可测 | 验收标准可测试验证 | 15% |
| 文档规范性 | 格式、术语符合规范 | 10% |

**评审输出**：
- 评审结论（通过/需要修改）
- 修改意见清单
- 预计完成时间

#### 5.2.6 交互设计阶段

**执行标准**：
- 低保真原型必须包含：页面流程、核心交互、状态说明
- 原型需覆盖：正常流程、异常流程、边界条件
- 必须进行原型内审后方可进入评审

**输出物**：
- 低保真原型（.fig）
- 交互说明文档
- 页面清单

#### 5.2.7 UI设计阶段

**执行标准**：
- 必须基于设计系统进行设计
- 必须完成多尺寸适配（移动端/PC端）
- 设计稿必须包含：正常态、异常态、空状态、加载状态

**输出物**：
- 高保真设计稿（.fig）
- 组件标注
- 切图资源
- 设计说明文档

#### 5.2.8 设计评审阶段

**评审参与方**：
- 产品部：产品经理、PRD撰写师
- 设计部：UI设计师、UX研究员、设计总监
- 技术部：前端开发代表

**评审维度**：

| 评审维度 | 评审内容 | 权重 |
|----------|----------|------|
| 视觉美观度 | 整体视觉效果、配色、排版 | 25% |
| 交互合理性 | 交互逻辑是否符合用户习惯 | 25% |
| 实现还原度 | 设计稿能否被精确还原 | 20% |
| 用户体验 | 能否给用户带来良好体验 | 15% |
| 品牌一致性 | 是否符合品牌规范 | 15% |

**评审决策**：
- **通过**：进入设计交付阶段
- **需要修改**：返回UI设计组修改
- **打回交互**：交互设计存在重大问题

#### 5.2.9 设计交付阶段

**交付检查清单**：

| 检查项 | 状态 | 备注 |
|--------|------|------|
| 设计稿已完成 | □ | |
| 标注已完善 | □ | |
| 切图已导出 | □ | |
| 动效说明已提供 | □ | |
| 多端适配完成 | □ | |
| 设计系统已更新 | □ | |

**交付物清单**：
- 设计稿源文件（.fig）
- 标注文件
- 切图包（.zip）
- 动效说明
- 设计变更记录

| 检查项 | 检查标准 | 检查方式 | 否决条件 |
|--------|----------|----------|----------|
| 需求完整性 | 包含业务背景、期望、验收条件 | 文档审核 | 缺失核心信息 |

**质量评分标准**：
- **L4-优秀**：综合评分 ≥ 90%
- **L3-良好**：综合评分 75-89%
- **L2-合格**：综合评分 60-74%
- **L1-不合格**：综合评分 < 60%

#### 6.2.3 PRD质量门禁
