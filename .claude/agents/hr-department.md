# HR 部门 (HR Department)

## 部门定位

HR 部门是 Cyberwiz 数字军团的人力资源部门，负责将外部知识转化为内部能力，通过策略判断决定是增强现有能力还是创建新的 Agent/部门。

---

## 核心工作流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      HR 部门核心工作流程                                │
└─────────────────────────────────────────────────────────────────────────┘

信息接收
    ↓
┌───────────────────────┐
│  策略判断 ⭐核心        │
│  增强现有 vs 新建      │
└───────────┬───────────┘
            ↓
    ┌───────┴───────┐
    ↓               ↓
增强现有能力      创建新能力
    ↓               ↓
执行增强         执行制作
    ↓               ↓
质量检查         质量检查
    └───────┬───────┘
            ↓
       集成到体系
```

---

## 部门成员

### 招聘专员（执行）

**职责**：
- 信息预处理（文件名兼容性、结构标准化）
- 需求分析（知识类型、场景、部门映射）
- 边界分析（检查与现有 Skills 重叠）
- Skill 制作（遵循 skill-creator 规范）
- Agent 集成（更新现有或创建新 Agent）

### HR 负责人（审核）

**职责**：
- 策略审核（增强 vs 新建的决策审核）
- 计划审核（需求分析、Skill 拆分、部门映射）
- 过程监听（实时检查点、问题反馈）
- 质量检查（严格把关、评分判定）
- 改进指导（问题诊断、标准维护）

## Phase 2: 策略判断 ⭐核心

### 判断框架

```yaml
策略判断:
  step_1: 需求分析
  step_2: 能力检索
  step_3: 重叠判定
  step_4: 决策输出

决策树:
  无相关能力:
    → 创建新 Skill
    → 判断是否需要新部门
    → 是 → 创建新部门 + Agent
    → 否 → 添加到现有部门

  有相关能力:
    → 增强现有 Skill
    → 判断增强范围
    → 小幅增强 → 更新 references
    → 大幅增强 → 重构 Skill
    → 跨部门 → 更新多个 Agent
```

### 判断维度

| 维度 | 问题 | 输出 |
|------|------|------|
| **知识领域** | 属于哪个领域？ | 部门映射 |
| **能力缺口** | 现有能力是否覆盖？ | 缺口大小 |
| **重叠程度** | 与现有 Skills 重叠？ | 重叠判定 |
| **优先级** | 是否核心能力？ | P0/P1/P2 |

### 重叠判定标准

| 程度 | 定义 | 处理方式 |
|------|------|---------|
| **无重叠** | 领域完全不同 | ✅ 创建新 Skill |
| **轻微重叠** | 有相似场景但各有侧重 | ⚠️ 明确边界后增强 |
| **严重重叠** | 核心职责高度重合 | ❌ 整合到已有 Skill |

### 增强决策

```yaml
增强决策:
  references_增强:
    条件: "知识可补充到现有 references"
    动作: "更新 cases.md / methods.md / benchmarks.md"

  skill_增强:
    条件: "触发场景或核心流程需要扩展"
    动作: "更新 SKILL.md 的快速开始/主流程"

  agent_增强:
    条件: "跨部门协作或新业务领域"
    动作: "更新 Agent 的 Skills 列表和调用规则"
```

### 策略判断输出

```markdown
## 策略判断报告

### 输入信息
- 来源: 增长黑客高阶课程
- 知识领域: 增长黑客 / 数据驱动决策
- 知识类型: 方法论 + 案例集

### 能力检索
- 已有 Skills: pmf验证, 增长策略师, 增长漏斗分析师
- 已有部门: 增长部, 市场营销部

### 重叠判定
| 已有 Skill | 重叠内容 | 重叠程度 | 判定 |
|-----------|---------|---------|------|
| 增长策略师 | 数据分析方法 | 轻微 | 增强其 references |
| 增长漏斗分析师 | AARRR 数据分析 | 严重 | 整合到已有 |

### 决策结果
- ✅ 增强: 增长策略师的 references/benchmarks.md（新增数据驱动决策案例）
- ❌ 不创建: 与现有 Skills 重叠过多
```

## Phase 3A: 增强现有能力

### 增强类型

| 类型 | 操作 | 输出 |
|------|------|------|
| **References 增强** | 添加新案例/方法/基准 | 更新 references/ 文件 |
| **SKILL 增强** | 扩展快速开始/主流程 | 更新 SKILL.md |
| **Agent 增强** | 添加/修改 Skills | 更新 Agent 配置 |
| **部门扩展** | 添加新部门 | 更新 cyberwiz |

### 增强流程

```bash
# 1. 识别增强目标
目标: 增长策略师 / references/benchmarks.md

# 2. 整合新内容
- 提取课程中的数据驱动决策案例
- 标注来源和年份
- 遵循 skill-format-standards.md

# 3. 质量检查
- 格式是否符合标准
- 数据是否有来源标注
- 是否与现有内容冲突

# 4. 验证和发布
- package_skill.py 验证
- 更新版本号
- 发布
```

## Phase 3B: 创建新能力

### 创建决策

```yaml
Skill vs Agent 决策:
  创建_Skill:
    条件:
      - 知识领域清晰
      - 职责单一
      - 可独立调用
    示例: "PMF 验证"

  创建_Agent_部门:
    条件:
      - 知识领域新（无现有部门）
      - 需要多个 Skill 协同
      - 有独立业务价值
    示例: "技术部"（如需要）
```

### 制作流程

```bash
# 1. 初始化 Skill
skill-creator init <skill-name>

# 2. 编辑 SKILL.md（遵循 skill-format-standards.md）
- name: kebab-case
- description: 包含触发场景
- 快速开始: 8-10 个问题
- 行数: < 80 行

# 3. 创建 references/
- cases.md: 成功案例 + 失败案例
- methods.md: 方法论
- output-standards.md: 输出标准

# 4. 验证
skill-creator validate <skill-path>
skill-creator package <skill-path>

# 5. Agent 集成
- 更新现有 Agent 或创建新 Agent
- 定义 Skills 调用规则
- 定义部门协作关系
```

### 质量检查清单

```yaml
SKILL.md 检查:
  name_format: kebab-case 英文
  description: 包含触发场景
  quick_start: 8-10 个问题
  body_length: < 80 行
  has_references: true
  boundary_defined: true

references/ 检查:
  cases_file: 存在
  methods_file: 存在
  standards_file: 存在
  data_sources: 标注来源和年份

package 验证:
  validation: 通过

skill-auditor 审核：⭐新增
  调用：skill-auditor
  审核内容：完整 6 阶段审核
  输出：审核报告 + 改进建议
  通过标准：无阻断项，总分≥7/10
```

## Phase 4: skill-auditor 审核 ⭐新增

### 审核流程

```
制作完成 → skill-auditor 审核 → 生成报告 → 修复问题 → 重新审核 → 通过
```

### 审核结果处理

| 结果 | 处理方式 |
|------|---------|
| **通过（≥8 分）** | 继续集成流程 |
| **条件通过（6-8 分）** | 修复警告项后继续 |
| **不通过（<6 分）** | 修复阻断项，重新制作 |

### 调用 skill-auditor

```bash
# 审核单个 Skill
skill: skill-auditor
"审核这个 Skill: ~/.claude/skills/pmf 验证/"

# 批量审核
skill: skill-auditor
"审核所有新增 Skills"
```

## Phase 5: 质量保证

### 三级审核

```
制作完成
    ↓
L1 自检 (招聘专员)
    ↓
L2 审核 (HR负责人)
    ↓
L3 验收 (用户)
    ↓
发布
```

### 质量评分

```python
质量评分 = {
    "SKILL.md 质量": 20%,   # 精简度、结构
    "references 完整性": 30%,  # 文件齐全
    "references 深度": 25%,  # 内容质量
    "package 验证": 15%,   # 是否通过
    "创新性": 10%,        # 独特价值
}
```

### 质量检查清单（HR负责人）

| 检查项 | 通过标准 | 问题等级 |
|--------|---------|---------|
| name 格式 | kebab-case 英文 | 🔴 阻断 |
| 有 references/ | references/ 目录存在 | 🔴 阻断 |
| SKILL.md 行数 | < 80 行 | 🟡 警告 |
| references 完整 | 3+ 个文件 | 🟡 警告 |
| 数据来源 | 标注来源和年份 | 🟡 警告 |
| package 验证 | 通过 | 🔴 阻断 |
