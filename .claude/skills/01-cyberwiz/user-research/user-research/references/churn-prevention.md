# 流失预警与召回策略

## 一、流失用户定义

### 流失定义三要素

**1. 关键行为识别**
- 用户不再做的核心行为
- 示例：电商用户不再购买、内容用户不再访问、社交用户不再互动

**2. 时间长度设定**
- 多长时间不做该行为算流失
- 示例：30天不登录、60天不购买、90天不访问

**3. 回访率验证**
- 确保定义的合理性
- 回访率<5%：定义合理
- 回访率>20%：定义过严，需调整

### 不同产品的流失定义

| 产品类型 | 关键行为 | 流失时间定义 |
|----------|----------|--------------|
| 电商 | 购买 | 90天无购买 |
| 内容 | 访问 | 30天无访问 |
| 社交 | 登录/互动 | 60天无登录 |
| 工具 | 使用 | 60天无使用 |
| 游戏 | 登录 | 30天无登录 |
| SaaS | 登录/使用 | 30天无登录 |

## 二、流失预警机制

### 流失征兆分析

**数据指标变化**：
- 使用频次下降
- 使用时长减少
- 核心行为减少
- 社交互动减少
- 负面反馈增加

**预警指标体系**：

| 预警等级 | 特征 | 示例 |
|----------|------|------|
| 高风险 | 核心行为停止超过阈值 | 7天未登录 |
| 中风险 | 使用频次显著下降 | 活跃度下降50% |
| 低风险 | 使用模式轻微变化 | 使用时长减少20% |

### 预警模型建立

**1. 规则预警**
```
IF 用户连续X天未登录
    OR 使用频次下降Y%
    OR 核心行为下降Z%
THEN 触发预警
```

**2. 机器学习预警**
- 使用历史流失用户数据训练模型
- 识别流失前的行为模式
- 预测流失概率

**特征选择**：
- 使用频次
- 使用时长
- 核心行为
- 社交互动
- 投诉反馈
- 付费行为

**算法选择**：
- 逻辑回归
- 决策树
- 随机森林
- XGBoost

### 预警等级划分

**三级预警**：

| 等级 | 流失概率 | 干预策略 |
|------|----------|----------|
| 红色预警 | >70% | 紧急干预，强力激励 |
| 橙色预警 | 40-70% | 主动关怀，提供优惠 |
| 黄色预警 | 20-40% | 关注监测，适度提醒 |

## 四、召回策略

### 召回时机

**最佳时机**：
1. 流失预警触发时
2. 产品重大更新时
3. 节假日营销时
4. 用户生日时
5. 特定活动时

### 召回渠道

**1. APP推送**
- 优点：直接、及时
- 缺点：可能被关闭
- 适用：轻度流失用户

**2. 短信**
- 优点：打开率高
- 缺点：成本高、打扰性强
- 适用：中高价值流失用户

**3. 邮件**
- 优点：内容丰富、成本低
- 缺点：打开率低
- 适用：详细召回内容

**4. 社交媒体**
- 优点：社交化、自然
- 缺点：覆盖面有限
- 适用：活跃于社交媒体的用户

**5. 电话回访**
- 优点：效果好、深度沟通
- 缺点：成本高、效率低
- 适用：高价值流失用户

### 召回内容

**1. 情感唤醒**
- 回忆美好时光
- 分享其他用户故事
- 情感化内容

**2. 价值吸引**
- 新功能介绍
- 优质内容推荐
- 独家优惠

**3. 激励刺激**
- 回归礼包
- 专属优惠
- 限时活动

**4. 产品改进**
- 告知问题已解决
- 介绍产品改进
- 征求意见

### 召回A/B测试

**测试维度**：

| 维度 | 测试内容 |
|------|----------|
| 召回时机 | 第1天 vs 第7天 vs 第30天 |
| 召回渠道 | 推送 vs 短信 vs 邮件 |
| 召回内容 | 情感型 vs 利益型 vs 功能型 |
| 激励强度 | 低激励 vs 中激励 vs 高激励 |
| 触达频次 | 1次 vs 3次 vs 5次 |

**评估指标**：
- 召回率
- 召回成本
- 召回用户留存
- 召回用户LTV

## 六、召回效果评估

### 关键指标

**召回效果**：
- 召回率 = 召回用户数 / 触达用户数
- 回流率 = 活跃召回用户 / 召回用户数
- 召回ROI = 召回用户价值 / 召回成本

**召回质量**：
- 召回用户留存率
- 召回用户活跃度
- 召回用户LTV

### 数据分析

**召回渠道效果对比**：

| 渠道 | 召回率 | 成本 | ROI | 适用用户 |
|------|--------|------|-----|----------|
| APP推送 | 5% | 低 | 高 | 轻度流失 |
| 短信 | 15% | 中 | 中 | 中度流失 |
| 邮件 | 3% | 低 | 中 | 有邮箱用户 |
| 电话 | 30% | 高 | 低 | 高价值 |

**召回内容效果对比**：

| 内容类型 | 召回率 | 适用场景 |
|----------|--------|----------|
| 情感唤醒 | 10% | 长期流失用户 |
| 价值吸引 | 15% | 功能敏感型用户 |
| 激励刺激 | 20% | 价格敏感型用户 |
| 产品改进 | 12% | 因问题流失用户 |

## 七、常见问题

### Q1: 召回效果越来越差怎么办？

**原因**：
- 召回过于频繁，用户疲劳
- 召回内容单一，缺乏吸引力
- 召回渠道不当，触达率低

**解决方案**：
1. 控制召回频次
2. 丰富召回内容
3. 优化召回渠道组合
4. 提升召回吸引力

### Q2: 召回成本过高怎么办？

**解决方案**：
1. 精准召回，只召回高价值用户
2. 优先使用低成本渠道
3. 优化召回内容，提升召回率
4. 设置召回成本上限

### Q3: 召回后再次流失怎么办？

**原因**：
- 召回只是短暂回归
- 根本问题未解决

**解决方案**：
1. 深入了解流失原因
2. 解决根本问题
3. 召回后持续关注
4. 提供持续价值

### SQL查询示例

```sql
-- 流失用户识别
SELECT
    user_id,
    MAX(last_active_date) AS last_active,
    DATEDIFF(CURRENT_DATE, MAX(last_active_date)) AS days_inactive,
    CASE
        WHEN DATEDIFF(CURRENT_DATE, MAX(last_active_date)) >= 30 THEN 'Churned'
        WHEN DATEDIFF(CURRENT_DATE, MAX(last_active_date)) >= 7 THEN 'At Risk'
        ELSE 'Active'
    END AS status
FROM user_activities
GROUP BY user_id;

-- 流失预警
SELECT
    user_id,
    last_active_date,
    activity_count_7d,
    activity_count_30d,
    CASE
        WHEN activity_count_7d = 0 THEN 'High Risk'
        WHEN activity_count_7d < activity_count_30d / 4 THEN 'Medium Risk'
        ELSE 'Low Risk'
    END AS risk_level
FROM user_activity_summary;
```
