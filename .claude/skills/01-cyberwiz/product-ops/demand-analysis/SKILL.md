# 需求分析方法论

## 概述

需求分析是产品工作的核心，指通过系统化的方法收集、分析、整理用户和业务需求，转化为可执行的产品方案的过程。

## 核心定义

### 什么是需求？

**需求 = 用户需要 + 业务目标 + 技术可行**

**Kano 模型 - 需求分类**:

```
         │
满意    │     │  兴奋需求
程度    │    ╱
        │   ╱  期望需求
        │  ╱
        │ ╱_______基本需求
        │─────────────────→
                 需求满足程度
```

| 需求类型 | 定义 | 特点 | 示例 |
|---------|------|------|------|
| **基本需求** | 必须有的，否则无法接受 | 不满足就失望 | 手机能打电话 |
| **期望需求** | 越多越好，线性关系 | 满足越多越满意 | 手机续航越长越好 |
| **兴奋需求** | 没想到的，惊喜 | 没有不会不满，有了惊喜 | 手机面部识别 |

### 需求层次

```
┌─────────────────────────────────────────────────────────────┐
│                     需求层次金字塔                           │
├─────────────────────────────────────────────────────────────┤
│  业务需求 (Business): 为什么做？商业目标是什么？              │
│       ↓                                                       ↓
│  用户需求 (User): 用户要解决什么问题？                        │
│       ↓                                                       ↓
│  功能需求 (Feature): 产品要提供什么功能？                     │
│       ↓                                                       ↓
│  技术需求 (Technical): 如何实现？技术要求是什么？             │
└─────────────────────────────────────────────────────────────┘
```

## 需求收集方法

### 定性方法

#### 1. 用户访谈
**适用**: 探索性研究，深入了解需求

**访谈技巧**:
- 开放式提问: "请描述一下您..."
- 追问细节: "为什么这么说？"
- 场景还原: "在什么情况下..."
- 观察行为: 注意言行不一致

**访谈框架**:
```
开场 → 建立信任 → 了解现状 → 深挖问题 → 探索需求 → 总结确认
```

#### 2. 焦点小组
**适用**: 生成创意，了解群体动态

**组织要点**:
- 6-8人为宜
- 同质性与异质性平衡
- 专业主持人引导
- 避免群体压力

#### 3. 观察法
**适用**: 发现用户未表达的需求

**观察维度**:
- 行为观察: 用户实际做了什么
- 环境观察: 使用场景特点
- 互动观察: 与他人/工具的互动
- 情绪观察: 情绪变化和反应

### 定量方法

#### 1. 问卷调查
**适用**: 验证假设，量化需求

**设计原则**:
- 明确调研目标
- 问题简洁明了
- 避免诱导性提问
- 合理设置选项

**问卷结构**:
```
开场说明 → 基本信息 → 核心问题 → 验证问题 → 开放问题 → 结束感谢
```

#### 2. 数据分析
**适用**: 从行为数据中发现需求

**分析维度**:
- 使用频率: 使用多少次
- 功能使用: 用了哪些功能
- 路径分析: 怎么使用的
- 流失分析: 在哪里流失

## 需求分析方法

### 需求优先级矩阵

**价值 vs 成本矩阵**:

```
高价值 │ P0    │ P1
      │ (必须做)│ (应该做)
──────┼───────┼──────
      │ P2    │ P3
低价值 │ (可以做)│ (不做)
      │低成本 │高成本
              └──────→
```

**优先级规则**:
- **P0**: 高价值低成本 → 立即做
- **P1**: 高价值高成本 → 计划做
- **P2**: 低价值低成本 → 有空做
- **P3**: 低价值高成本 → 不做

### RICE 优先级模型

```yaml
RICE:
  R (Reach): 触达用户数
  I (Impact): 每个用户的影响程度
  C (Confidence): 信心程度
  E (Effort): 工作量（人月）

计算:
  RICE = (R × I × C) / E

使用:
  - 对所有需求打分
  - 按RICE分数排序
  - 优先处理高分需求
```

### 需求依赖分析

**依赖类型**:
```yaml
dependencies:
  mandatory: "必须先完成的需求"
  optional: "建议先完成的需求"
  parallel: "可以并行开发的需求"
  conflicting: "相互冲突的需求（需要取舍）"

可视化:
  使用依赖图展示需求之间的关系
```

## 需求验证

### 假设验证

**假设驱动开发**:
```
假设 → 设计实验 → 收集数据 → 分析结果 → 验证/推翻
```

**假设模板**:
```
我们相信 [用户群体]
想要 [需求/功能]
这将带来 [预期价值]
我们将通过 [指标] 来验证
```

### MVP 验证

**MVP (最小可行产品)**:
- **目标**: 用最小成本验证核心假设
- **原则**: 只做核心功能，不做加法
- **衡量**: 关注验证指标，非虚荣指标

**MVP 类型**:
| 类型 | 描述 | 适用场景 |
|------|------|----------|
| **Smoke Test** | 烟雾测试页面 | 验证兴趣 |
| **Concierge** | 手动服务模式 | 验证需求 |
| **Wizard of Oz** | 假装自动化 | 验证体验 |
| **Prototype** | 可交互原型 | 验证方案 |

## 需求文档

### PRD 结构

```markdown
# 产品需求文档 (PRD)

## 1. 背景与目标
- 为什么做这个功能？
- 要达到什么目标？

## 2. 用户与场景
- 目标用户是谁？
- 典型使用场景是什么？

## 3. 功能需求
### 3.1 功能点1
- 功能描述
- 用户故事
- 验收标准

### 3.2 功能点2
...

## 4. 非功能需求
- 性能要求
- 安全要求
- 兼容性要求

## 5. 数据指标
- 成功指标
- 衡量方式

## 6. 依赖与风险
- 技术依赖
- 资源依赖
- 潜在风险
```

## 需求变更管理

### 变更控制

**变更评估**:
```yaml
change_request:
  rationale: "变更理由"
  impact: "影响评估"
  cost: "成本估算"
  priority: "优先级调整"

decision:
  approve: "批准"
  reject: "拒绝"
  defer: "延后"
  modify: "修改后批准"
```

### 变更流程

```
变更申请 → 影响评估 → 利益相关者评审 → 决策 → 沟通 → 实施
```

## 常见误区

### 误区 1: 用户说的就是需求
**纠正**: 用户说的是解决方案，不是真正的需求。需要深挖"为什么"。

### 误区 2: 需求一成不变
**纠正**: 需求是动态的，市场在变，用户在变，需求也在变。

### 误区 3: 听所有人意见
**纠正**: 需要识别目标用户，聚焦核心用户的需求。

### 误区 4: 功能越多越好
**纠正**: 少即是多，聚焦核心价值，避免功能膨胀。
